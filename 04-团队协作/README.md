# 团队协作（Working Agreement）

> 目标：让多人协作时**交付更快、质量更稳、沟通更少**。可直接复制到团队项目里作为默认协作规范。

---

## 🎯 协作原则

- **单一事实来源**：需求/决策/进度以文档为准，聊天记录只做提醒。
- **小步提交**：小改动频繁提交，降低冲突与返工成本。
- **先对齐再开工**：对齐目标、范围、验收标准，再写代码/写内容。

---

## 🧱 角色与责任（最小集合）

- **Owner（负责人）**：定义目标、范围、验收标准；做关键取舍；推进落地。
- **Contributor（贡献者）**：按任务拆分交付；确保自测；写清楚变更说明。
- **Reviewer（评审者）**：关注风险与质量；给出可执行建议；避免“我觉得”。

---

## 🧾 需求与任务管理（轻量版）

### 1) 需求模板（建议放在 Issue / Notion / 飞书）

- **背景/问题**：为什么要做？
- **目标**：要达成什么？
- **非目标**：明确不做什么（防范围膨胀）
- **验收标准**：可测试、可判定
- **风险/依赖**：外部依赖、未知点

### 2) 任务拆分标准

- **每个任务可独立验收**
- **1-2天内可完成**（更长就再拆）
- **明确输入/输出**

---

## 🔀 Git 协作（推荐流程）

### 分支命名

- `feat/<topic>`
- `fix/<topic>`
- `docs/<topic>`
- `refactor/<topic>`

### 提交信息（与知识库保持一致）

- `feat:` 新增
- `fix:` 修复
- `update:` 更新
- `docs:` 文档
- `refactor:` 重构

### PR / MR 规范（最小要求）

- **标题**：一句话说明改了什么
- **描述**：为什么改、怎么改、影响范围
- **截图/证据**（如有 UI/数据变化）
- **风险点**：可能出问题的地方
- **回滚方式**：如何撤回

---

## ✅ 评审清单（Reviewer 参考）

- **正确性**：是否满足验收标准？
- **可维护性**：命名/结构/重复是否可控？
- **一致性**：是否符合仓库约定（路径、格式、术语）？
- **风险**：有没有隐性破坏（同步脚本、路径、断链）？

---

## 🧠 决策记录（建议）

当出现分歧或重要取舍，记录一条：

- **决策**：我们选择了什么？
- **理由**：为什么？
- **替代方案**：考虑过什么？
- **影响**：对未来的影响与约束
