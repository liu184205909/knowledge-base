# Agent 开发指南：Framework + Tools + Sandbox

> **从自动化到智能化** | 2026-01-05

---

## 📋 文档定位

**什么是 Agent？**

> **Agent = 能自主规划、执行、迭代的智能体**

与自动化流水线不同：
- **自动化流水线**：批量处理重复任务（50-151个），规则明确
- **Agent**：处理复杂、多步骤的自主任务，需要决策和迭代

**典型案例**：
- **Manus**：通用智能体，被 Meta 以 20 亿美金收购
- **Devin**：首个 AI 软件工程师
- **Claude Code**：6 个月 10 亿美金 ARR 的 Coding Agent

---

## 🎯 Agent vs 自动化流水线

### 核心区别

| 维度 | 自动化流水线 | Agent（Manus级别） |
|------|------------|-------------------|
| **任务类型** | 50-151个重复性任务 | 复杂、多步骤的自主任务 |
| **隔离机制** | CSV外挂记忆 | Sandbox（沙盒） |
| **可视化** | 无（命令行） | 有（可视化每个步骤） |
| **交互方式** | 无人值守 | Human-in-the-loop |
| **技术门槛** | 高（MCP+测试） | 极高（Framework+Tools+Sandbox） |
| **开发成本** | 2-4小时准备 | 10人团队×3个月（Manus） |
| **适用场景** | 批量处理重复任务 | 通用的智能体 |

### 关系

```
自动化流水线 = Agent 的一种特殊形式（无 Sandbox）

Agent = 自动化流水线 + Sandbox + 可视化 + 人工交互
```

**举例**：
- **自动化流水线**：批量生成 100 个静态页面（规则明确）
- **Agent**：从零开发一个完整的 Web 应用（需要决策、迭代、用户反馈）

---

## 🏗️ Agent 三大件

基于 Manus 等成功案例的分析，Agent 产品需要三大核心组件：

### 1. Framework（框架）

**作用**：提供 Agent 骨架

**核心能力**：
- **Plan（规划）**：拆解任务，制定执行计划
- **Act（执行）**：执行具体步骤，调用工具
- **Memory（记忆）**：记住上下文和历史
- **Context Manager（上下文管理）**：管理 token 使用

**现状**：
- 现有框架：LangGraph、AutoGen、CrewAI、LlamaIndex、Mastra
- 问题：过于底层，虽然封装了很多功能，但很难开箱即用

**目标**：
```
Next.js（Web 开发框架）→ ShipAny（一小时上线 AI SaaS）
LangGraph（Agent 框架）→ ？（一天内上线 Manus 级别 Agent）
```

**类比**：
- Next.js 封装了路由、服务端渲染、中间件等基础功能
- 但对于不熟悉 MVC 的开发者，上手成本还是高
- 需要像 ShipAny 一样的"开箱即用"框架

**关键问题**：
- 独立开发者或小团队，如何在一周内或一月内，做出一个 Manus 级别的 Agent 产品？

---

### 2. Tools（工具）

**作用**：Agent 的工具箱，原子能力

**典型案例**：

**Manus**（29个工具）：
- 浏览器操作
- Shell 操作
- 文件操作
- 信息查询
- 消息交互

**Deep Research Agent**：
- 联网搜索
- 阅读网页
- 文件解析

**播客生成 Agent**：
- 联网搜索
- TTS（文本转语音）
- 音频合成

**核心问题**：
> 如何给 Agent 提供必要的工具？

**解决方案**：
- **MCPRouter**：基于 MCP 的原子能力平台
- **原理**：给 Agent 提供各种类型的 Tools，让 Agent 专注于业务逻辑
- **优势**：
  - Agent 开发者不需要开发工具
  - 按需挂载适合自己业务场景的工具
  - 大大降低 Agent 开发门槛

**工具来源**：
1. **MCP 服务器**：标准化的工具接入方式
2. **Skills 脚本**：Python/Bash 脚本提供的自定义能力
3. **Plugins**：封装的专业能力

---

### 3. Sandbox（沙盒）⭐ 最关键

**作用**：
1. **提供完整的隔离机制**（安全性、可控性）
2. **可视化输出每个步骤**（用户体验）
3. **方便 human-in-the-loop**（及时纠正或终止）

**为什么 Sandbox 最关键？**

**没有 Sandbox 的问题**：
```
用户提交任务 → Agent 在后台运行 20 分钟
→ 用户不知道 Agent 在干什么
→ 产出了什么也不清楚
→ 用户体验非常不好
```

**有 Sandbox 的好处**：
```
用户提交任务 → Agent 在云端虚拟机中运行
→ 每个步骤清晰可见（代码、文件、输出）
→ 用户看到 AI 在干活（智能产品的样子）
→ 可以及时纠正或终止（Human-in-the-loop）
```

**典型案例**：

**Manus**：
- 基于云端虚拟机（Sandbox）
- 可视化每个步骤的产出
- 用户可以实时查看进度

**技术实现方式**：
1. **自建**：K8S 集群 + 容器化
2. **接入第三方**：
   - **e2b**：代码执行沙盒
   - **Webcontainers**：前端开发沙盒

**技术难点**：
- 集群部署
- 动态扩容
- 毫秒级启动
- 并发控制
- 资源隔离
- 数据持久化

**最大卡点**：
- 独立开发者很难搞定 Sandbox
- 需要一定的容器化经验
- 不熟悉 K8S 的开发者，一般很难搞定这些技术细节
- 成本管控问题

**期待**：
- 2026 年，会有更轻量级的 Sandbox 解决方案
- 成本可控，接入便利
- 让 Agent 开发者可以专注业务逻辑

## 🚀 如何开发 Agent 产品？

### 场景1：独立开发者

**建议**：从自动化流水线开始

**原因**：
- 门槛相对较低（只需 MCP + 测试）
- 可以快速验证想法
- 不需要 Sandbox（成本可控）

**路径**：
```
1. 学习 MCP 和 Skills
2. 用自动化流水线处理重复任务
3. 逐步增加复杂性
4. 后期再考虑加入 Sandbox
```

**案例**：
- CopyWeb：网页复刻工具（ARR 2.4w 美金）
- Same：4 个月 300 万美金 ARR（后来转向全场景）

---

### 场景2：小团队（3-5人）

**建议**：聚焦垂直场景

**方向1：垂类场景**
- 验证了需求（CopyWeb、Same）
- Same 后来转向全场景，但深耕垂类可能更好

**方向2：基于模板做增量填充**
- Lovable 的问题：从零生成，质量不可控
- 更好的解法：基于成熟模板做增量
- 本质：让 AI 的产出更加收敛，效果更加可控

**路径**：
```
1. 选择垂直场景（如：网页复刻）
2. 开发或集成 Framework
3. 接入第三方 Sandbox（e2b、Webcontainers）
4. 按需挂载 Tools（基于 MCP）
5. 快速迭代，收集用户反馈
```

---

### 场景3：大团队（10+人）

**建议**：全栈 Agent

**参考**：Manus（10人团队 × 3个月）

**核心组件**：
- 自研或深度定制 Framework
- 自建 Sandbox（K8S 集群）
- 开发专有 Tools

**目标**：
- Manus 级别的通用 Agent
- 或者垂直领域的超级 Agent

---

## 🎓 Agent vs Skills vs MCP

### 三者关系

```
Skills = 工作流的标准化（告诉 AI "怎么做"）
  ↓ 可以挂载
MCP = 工具的连接器（给 AI 提供外部工具）
  ↓ 组合成
Agent = 自主决策的智能体（自主规划、执行、迭代）
```

### 实际应用

**场景：开发 Python API**

**使用 Skills**：
```
AI 知道：Python 最佳实践
```

**使用 MCP**：
```
AI 可以：搜索文档、读取网页、运行测试
```

**使用 Agent**：
```
AI 能够：
1. 规划：拆解任务，制定计划
2. 执行：编写代码、运行测试、修复 Bug
3. 迭代：根据测试结果优化代码
4. 交互：在 Sandbox 中可视化每个步骤
```
