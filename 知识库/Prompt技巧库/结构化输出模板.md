# Prompt结构化输出模板

> **统一输出格式** | 减少后处理 | 提升效率

**核心理念**：通过明确的输出格式规范，让AI按照预期的结构输出结果，减少后续手动调整工作。

## 为什么需要结构化输出

**问题**：没有结构化要求时，AI输出格式不确定，需要手动调整

**解决**：明确输出格式规范，让AI按照预期结构输出

**价值**：
- ✅ 减少后处理：直接获得可用格式
- ✅ 提升一致性：多次运行格式统一
- ✅ 便于集成：可程序化解析
- ✅ 提升质量：明确结构促使AI思考更全面

---

## 通用结构化框架

### 框架结构

```markdown
【输出格式要求】
1. 格式类型：[Markdown表格/JSON/列表/代码]
2. 包含字段/列：[列出所有必需字段]
3. 字段说明：[每个字段的含义和类型]
4. 排序要求：[按什么字段排序]
5. 约束条件：[必填/可选/格式要求]

【示例】（可选）
提供一个完整的输出示例
```

---

### 框架应用步骤

1. **确定输出类型**：表格/JSON/列表/代码
2. **定义字段结构**：需要哪些列或键
3. **说明字段含义**：每个字段的类型和内容
4. **添加约束条件**：必填、格式、范围
5. **提供示例**（可选）：展示期望的输出

## 常见输出格式模板

### 模板1：Markdown表格

#### 适用场景
- 数据对比
- 问题清单
- 参数说明
- 任务列表

#### Prompt模板

```markdown
【输出格式】
以Markdown表格形式输出，包含以下列：

| [列1] | [列2] | [列3] | [列4] |
|-------|-------|-------|-------|
| [说明] | [说明] | [说明] | [说明] |

【字段说明】
- [列1]：[字段类型，例如：字符串/数字/枚举]
- [列2]：[字段类型]
- [列3]：[字段类型]
- [列4]：[字段类型]

【排序要求】
按[列名]升序/降序排列

【示例】
| 优先级 | 问题类型 | 严重程度 | 优化建议 |
|--------|---------|---------|---------|
| P1     | 性能问题 | 高       | 使用缓存 |
```

---

#### 实战案例：代码问题分析

**Prompt**：
```markdown
分析以下代码的性能问题，以Markdown表格输出：
| 问题ID | 问题类型 | 严重程度 | 问题描述 | 优化建议 | 预期提升 |

要求：
- 问题ID按严重程度排序（P1 > P2 > P3）
- 问题描述简洁（1句话）
- 优化建议具体可行

**AI输出**：
```markdown
| 问题ID | 问题类型   | 严重程度 | 问题描述                     | 优化建议                 | 预期提升 |
|--------|------------|---------|------------------------------|--------------------------|---------|
| P1     | 算法复杂度 | 高      | 双重循环，时间复杂度O(n²)    | 使用集合降低到O(n)        | 100x    |
| P2     | 代码风格  | 中      | not in duplicates是O(n)操作  | 使用set去重               | 2x      |
| P3     | 可维护性  | 低      | 变量命名不够描述性           | 改为更清晰的名称          | -       |
```

### 模板2：JSON格式

#### 适用场景
- API响应
- 配置文件
- 数据交换
- 程序化处理

#### Prompt模板

```markdown
【输出格式】
以JSON格式输出，符合以下结构：

```json
{
  "[字段1]": "[类型说明]",
  "[字段2]": "[类型说明]",
  "[字段3]": {
    "[子字段1]": "[类型说明]",
    "[子字段2]": "[类型说明]"
  },
  "[字段4]": [
    {
      "[数组元素字段1]": "[类型说明]",
      "[数组元素字段2]": "[类型说明]"
    }
  ]
}
```

【字段说明】
- [字段1]：[说明，必填/可选，格式要求]
- [字段2]：[说明，必填/可选，格式要求]
- [字段3]：[说明，嵌套结构说明]
- [字段4]：[说明，数组元素说明]

【约束条件】
- 所有字符串使用双引号
- 日期格式：ISO 8601 (YYYY-MM-DDTHH:MM:SSZ)
- 布尔值：true/false（小写）
- null值：使用null（而非None或nil）

【示例】
```json
{
  "task_id": "TASK-001",
  "status": "completed",
  "result": {
    "code": 200,
    "message": "Success"
  },
  "items": [
    {"id": 1, "name": "Item 1"},
    {"id": 2, "name": "Item 2"}
  ]
}
```
```

### 模板3：列表格式

#### 适用场景
- 步骤清单
- 检查列表
- 要点总结
- 行动建议

#### Prompt模板

```markdown
【输出格式】
以编号列表形式输出，每个列表项包含：

【格式】
[序号]. **[标题]**：[描述]
   - [子要点1]
   - [子要点2]
   - [子要点3]

【内容要求】
- 标题：简洁有力（5-10个字）
- 描述：详细说明（1-2句话）
- 子要点：3-5个
- 每个列表项之间空一行

【示例】
1. **环境准备**：安装必要的开发工具
   - 安装Python 3.9+
   - 安装Node.js 16+
   - 配置PostgreSQL数据库

2. **项目初始化**：创建项目基础结构
   - 初始化Git仓库
   - 创建虚拟环境
   - 安装项目依赖
```

#### 实战案例：部署步骤清单

**Prompt**：
```markdown
提供一个FastAPI应用的部署步骤清单

【应用信息】
- 框架：FastAPI
- 数据库：PostgreSQL
- 部署环境：Ubuntu 20.04服务器
- 使用Docker容器化

【输出格式】
以编号列表形式输出：

[序号]. **[阶段名称]**：[阶段描述]
   - [具体步骤1]
   - [具体步骤2]
   - [具体步骤3]

【内容要求】
- 分为5个阶段：准备、配置、构建、部署、验证
- 每个阶段3-5个步骤
- 每个步骤包含具体命令（如适用）
- 包含验证命令确保每步成功
```

**AI输出**：
```markdown
1. **环境准备**：安装Docker和Docker Compose
   - 更新系统：`sudo apt update && sudo apt upgrade -y`
   - 安装Docker：`curl -sSL https://get.docker.com/ | sh`
   - 验证安装：`docker --version`

2. **配置文件**：创建Dockerfile和docker-compose.yml
   - Dockerfile：定义应用镜像构建步骤
   - docker-compose.yml：定义服务编排配置
   - .env：配置环境变量

3. **构建镜像**：`docker-compose build`

4. **部署应用**：启动服务 `docker-compose up -d`

5. **验证部署**：健康检查 `curl http://localhost:8000/health`
```

### 模板4：代码结构输出

#### 适用场景
- 多文件代码生成
- 项目结构生成
- 配置文件生成

#### Prompt模板

```markdown
【输出格式】
以代码块形式输出，每个文件一个代码块：

【格式】
```[语言]
// 文件路径：[文件路径]
[文件内容]
```

```[语言]
// 文件路径：[文件路径]
[文件内容]
```

【内容要求】
- 明确每个文件的路径
- 使用正确的语法高亮标记
- 包含必要的注释
- 遵循代码规范

【文件列表】
1. [文件1路径]
2. [文件2路径]
3. [文件3路径]
```

### 技巧4：验证规则

在Prompt中包含验证规则：

**Prompt**：
```markdown
【输出格式】
JSON格式，遵循以下验证规则：

- 所有email字段必须符合email格式
- 所有date字段必须符合ISO 8601格式
- 所有price字段必须是正数，保留2位小数
- 所有status字段必须是枚举值（pending/active/inactive）

【验证示例】
✅ 有效的price：19.99, 100.00, 0.99
❌ 无效的price：19, 19.9, -10
```

### 格式选择决策树

```
你的数据需要程序化处理吗？
│
├── 是 → 使用JSON格式
│
└── 否 → 需要对比多个项目吗？
    │
    ├── 是 → 使用Markdown表格
    │
    └── 否 → 需要展示步骤吗？
        │
        ├── 是 → 使用编号列表
        │
        └── 否 → 需要生成代码吗？
            └── 是 → 使用代码块格式
```
