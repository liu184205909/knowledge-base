## 四大核心原则

### 原则1：明确性（Clarity）

**定义**：清楚、无歧义地表达你的需求

#### ✅ 好的Prompt
```markdown
写一篇1000字的技术文章，介绍如何使用Docker部署Node.js应用。
目标受众是初级开发者，假设他们知道基本的JavaScript概念，
但不熟悉Docker。文章需要包含：Docker基础概念、Dockerfile编写、
docker-compose使用、常见问题解决。
```

**特点**：
- ✅ 明确字数（1000字）
- ✅ 明确主题（Docker部署Node.js）
- ✅ 明确受众（初级开发者）
- ✅ 明确内容结构（4个部分）

#### ❌ 差的Prompt
```markdown
写一篇关于Docker的文章
```

**问题**：
- ❌ 字数不明确
- ❌ 没有说明具体主题
- ❌ 没有说明目标受众
- ❌ 没有说明内容结构

---

### 原则2：上下文（Context）

**定义**：提供充分的背景信息，帮助AI理解场景

#### ✅ 好的Prompt
```markdown
我正在开发一个在线教育平台，使用Python + Django + PostgreSQL。

当前问题：用户注册时，系统发送的欢迎邮件经常延迟5-10分钟，
影响用户体验。技术栈：Celery + Redis作为任务队列，邮件服务使用SendGrid。

请帮我分析可能的性能瓶颈，并给出优化建议。
```

**特点**：
- ✅ 说明项目背景（在线教育平台）
- ✅ 说明技术栈（Python + Django + PostgreSQL）
- ✅ 说明具体问题（邮件延迟）
- ✅ 说明相关技术（Celery + Redis + SendGrid）

#### ❌ 差的Prompt
```markdown
邮件发送很慢，帮我优化
```

**问题**：
- ❌ 没有项目背景
- ❌ 没有技术栈信息
- ❌ 没有说明"慢"的具体情况
- ❌ 缺少关键信息（邮件服务、任务队列）

---

### 原则3：具体化（Specificity）

**定义**：给出具体的要求、约束和输出格式

#### ✅ 好的Prompt
```markdown
分析以下Python代码的性能问题：

[代码内容]

请以Markdown表格形式输出分析结果，包含以下列：
| 问题类型 | 问题描述 | 严重程度 | 优化建议 | 预期提升 |

严重程度分为：P1（严重）、P2（中等）、P3（轻微）
```

**特点**：
- ✅ 明确输入（代码内容）
- ✅ 明确输出格式（Markdown表格）
- ✅ 明确列名和结构
- ✅ 定义评分标准（P1/P2/P3）

#### ❌ 差的Prompt
```markdown
优化这段代码
```

**问题**：
- ❌ 没有说明分析维度
- ❌ 没有输出格式要求
- ❌ 没有评分标准

---

### 原则4：迭代优化（Iteration）

**定义**：通过多轮调整和改进，逐步完善Prompt

#### 迭代流程示例

**第1版Prompt**：
```
帮我写一个用户登录功能
```

**AI输出结果**：代码过于简单，缺少错误处理

**第2版Prompt**：
```
帮我写一个用户登录功能，使用JWT认证，包含错误处理
```

**AI输出结果**：代码完整，但缺少常见安全措施

**第3版Prompt（最终版）**：
```
帮我写一个用户登录功能：

技术栈：FastAPI + PostgreSQL + JWT

功能要求：
1. 用户登录（邮箱+密码）
2. JWT Token生成和验证
3. 密码加密存储（bcrypt）
4. 登录失败次数限制（5次/15分钟）
5. 错误处理（用户不存在、密码错误、账号锁定）
6. 日志记录（登录成功/失败）

安全要求：
- 防止SQL注入
- 防止暴力破解
- Token过期时间：7天

输出：完整的代码 + 单元测试 + 使用说明
```

**AI输出结果**：代码完整、安全、符合生产标准

## Prompt结构框架

### 通用框架

```
1. 角色定位（可选）
2. 任务描述
3. 背景信息/上下文
4. 具体要求/约束
5. 输出格式
6. 示例（可选）
```

### 框架应用示例

#### 示例1：代码生成

```markdown
【角色定位】
你是一位资深的后端工程师，精通Python和API设计

【任务描述】
开发一个用户管理API，包含增删改查功能

【背景信息】
- 项目：电商平台后端
- 技术栈：FastAPI + SQLAlchemy + PostgreSQL
- 数据模型：User(id, username, email, created_at, updated_at)

【具体要求】
1. RESTful API设计
2. 数据验证（Pydantic）
3. 错误处理
4. 分页查询（默认20条/页）
5. 单元测试覆盖率 > 80%

【输出格式】
- 代码文件（main.py, models.py, schemas.py）
- API文档（README.md）
- 测试文件（test_main.py）
```

#### 示例2：内容生成

```markdown
【任务描述】
写一篇关于"远程工作最佳实践"的博客文章

【背景信息】
- 目标受众：团队管理者
- 发布平台：公司内部博客
- 目的：帮助团队提升远程协作效率

【具体要求】
1. 字数：2000-2500字
2. 结构：引言 + 5个核心建议 + 总结
3. 语气：专业但不失亲和
4. 包含：真实案例、数据支撑、可落地的建议
5. SEO优化：关键词"远程工作"、"团队管理"、"效率提升"

【输出格式】
- Markdown格式
- 包含H1/H2/H3标题
- 包含表格对比
- 包含行动清单
```

#### 示例3：问题分析

```markdown
【任务描述】
分析Web应用性能下降的原因

【背景信息】
- 应用：在线教育平台
- 技术栈：React + Node.js + MongoDB
- 症状：响应时间从200ms增加到2s
- 时间：最近一周逐渐恶化
- 用户量：从1000增加到2000（翻倍）

【具体要求】
1. 从以下维度分析：
   - 前端性能
   - 后端性能
   - 数据库性能
   - 网络性能
2. 给出优先级排序的问题列表
3. 每个问题提供：
   - 可能原因
   - 验证方法
   - 解决方案
   - 预期效果

【输出格式】
| 优先级 | 问题类型 | 可能原因 | 验证方法 | 解决方案 | 预期效果 |
|--------|---------|---------|---------|---------|---------|
```

## 常见误区与正确做法

### 误区1：Prompt越长越好

#### ❌ 错误做法
```markdown
写一篇非常非常非常非常非常详细的文章，包含很多很多很多内容，
要非常详细非常详细非常详细，每一个部分都要写得很详细很详细...
```

**问题**：
- 重复啰嗦，浪费Token
- 信息密度低
- AI理解困难

#### ✅ 正确做法
```markdown
写一篇2500字的技术教程，包含5个章节，每个章节500字，
涵盖原理、实现、示例、最佳实践、常见问题。
```

**优势**：
- 简洁明确
- 信息密度高
- 易于理解

---

### 误区2：一次性写出完美Prompt

#### ❌ 错误做法
```
期望第一次就写出完美的Prompt，不允许调整
```

**问题**：
- 不现实
- 浪费时间过度思考
- 难以发现潜在问题

#### ✅ 正确做法
```
快速迭代法：
1. 第1版：粗糙的Prompt（1分钟）
2. 根据AI输出调整（2分钟）
3. 第2版：细化具体要求（1分钟）
4. 继续调整...
5. 最终版：高质量Prompt（总耗时<10分钟）
```

**优势**：
- 快速验证
- 持续改进
- 节省时间

---

### 误区3：忽视AI反馈

#### ❌ 错误做法
```
User: "帮我写代码"
AI: "请问您需要什么类型的代码？具体功能是什么？"
User: "就写代码啊！你不会吗？"
```

**问题**：
- 错过澄清需求的机会
- 浪费Token反复尝试
- 输出结果不满意

#### ✅ 正确做法
```
User: "帮我写用户登录代码"
AI: "请问使用什么技术栈？需要包含哪些功能？"
User: "FastAPI + JWT，包含注册、登录、密码重置"
AI: "好的，我现在为您生成代码..."
```

**优势**：
- 需求更明确
- 输出更准确
- 减少反复修改

---

### 误区4：照搬他人Prompt

#### ❌ 错误做法
```
直接使用网上找到的Prompt，不修改任何内容
```

**问题**：
- 可能不符合你的需求
- 缺少上下文
- 输出结果不理想

#### ✅ 正确做法
```
1. 找到优秀的Prompt模板
2. 理解Prompt的结构和原理
3. 根据自己的需求调整：
   - 修改角色定位
   - 补充背景信息
   - 调整具体要求
   - 修改输出格式
4. 测试并迭代优化
```

**优势**：
- 更贴合实际需求
- 理解Prompt设计原理
- 逐步提升Prompt能力

## 迭代优化方法

### 优化流程图

```
初始Prompt
    ↓
测试执行
    ↓
分析输出结果
    ↓
┌─────────────────┐
│ 输出满意？      │
└─────────────────┘
    ↓ 是          ↓ 否
结束            识别问题
    ↓              ↓
              ┌─────────────────┐
              │ 问题类型？      │
              └─────────────────┘
                  ↓ ↓ ↓ ↓
            ┌─────┬─────┬─────┬─────┐
            │不清楚│不准确│不完整│格式错│
            └─────┴─────┴─────┴─────┘
                ↓     ↓     ↓     ↓
            增加明 补充细 补充要 修改格
            确信息 节内容 求细节 式要求
                ↓     ↓     ↓     ↓
                  修改Prompt
                      ↓
                  重新测试
                      ↓
                  （循环）
```

### 优化技巧

#### 技巧1：反向提问法

当Prompt不够明确时，先让AI提问：

```markdown
我想要优化一个Python函数的性能。
在给出代码之前，请你先问我5个问题，
帮助我们更好地理解需求和背景。
```

**优势**：
- 补充遗漏的关键信息
- 发现没考虑到的问题
- 获得更准确的解决方案

---

#### 技巧2：分步骤法

将复杂任务拆分成多个简单步骤：

**❌ 一口气完成**：
```markdown
帮我从零开始搭建一个电商网站，包括前端、后端、数据库、支付、物流...
```

**✅ 分步骤完成**：
```markdown
【步骤1：需求分析】
帮我分析搭建一个电商网站需要哪些核心功能

【步骤2：技术选型】
基于上述功能，推荐技术栈并说明理由

【步骤3：数据库设计】
设计数据库表结构和关系

【步骤4：API设计】
设计RESTful API接口

【步骤5：实现代码】
基于上述设计，实现核心功能
```

**优势**：
- 每一步都更可控
- 便于验证和调整
- 降低AI出错概率

---

#### 技巧3：示例引导法

提供期望输出的示例：

```markdown
请帮我分析用户评价的情感倾向。

输入示例：
"这个产品太棒了，我非常喜欢！"
输出：正面

"质量一般，价格有点贵"
输出：中性

"很差，再也不买了"
输出：负面

现在请分析以下评价：
[你的评价内容]
```

**优势**：
- 明确输出格式
- 建立评分标准
- 提升一致性

---

#### 技巧4：约束条件法

明确约束条件，避免超出范围：

```markdown
帮我设计一个数据库表结构：

【约束条件】
- 只使用MySQL 8.0支持的特性
- 表数量不超过5个
- 避免使用外键约束（用应用层控制）
- 索引数量 < 10个
- 预期数据量：100万条/年

【功能需求】
用户管理、订单管理、商品管理
```

**优势**：
- 控制输出范围
- 避免过度设计
- 符合实际情况

## 实战案例分析

### 案例1：代码生成优化

#### 初始Prompt
```markdown
写一个Python函数，计算斐波那契数列
```

**问题分析**：
- ❌ 没有说明性能要求
- ❌ 没有说明输入范围
- ❌ 没有说明返回格式

#### 优化后Prompt
```markdown
写一个Python函数，计算斐波那契数列：

【功能要求】
- 函数名：fibonacci
- 输入：整数n（0 <= n <= 1000）
- 输出：返回第n个斐波那契数

【性能要求】
- 时间复杂度：O(n)
- 空间复杂度：O(1)
- 支持大数（n=1000时也能快速计算）

【代码要求】
- 包含类型注解
- 包含文档字符串
- 包含单元测试（至少5个测试用例）

【输出格式】
```python
def fibonacci(n: int) -> int:
    """
    [文档字符串]
    """
    # 实现代码

# 单元测试
if __name__ == "__main__":
    # 测试代码
```
```

**优化效果**：
- ✅ 代码质量更高
- ✅ 性能更优
- ✅ 可维护性更好
- ✅ 包含测试

---

### 案例2：文档生成优化

#### 初始Prompt
```markdown
帮我写一个API文档
```

**问题分析**：
- ❌ 没有说明API功能
- ❌ 没有说明文档格式
- ❌ 没有说明目标读者

#### 优化后Prompt
```markdown
为以下API接口编写文档：

【API信息】
接口：GET /api/users/:id
功能：获取指定用户信息
认证：需要Bearer Token

【请求参数】
- id (path): 用户ID，整数，必填

【响应示例】
成功（200）：
{
    "id": 123,
    "username": "john_doe",
    "email": "john@example.com",
    "created_at": "2025-01-01T00:00:00Z"
}

失败（404）：
{
    "error": "User not found",
    "code": "USER_NOT_FOUND"
}

【文档要求】
1. 格式：Markdown
2. 包含：接口描述、请求方法、URL、参数、响应示例、错误码
3. 目标读者：前端开发者
4. 语言：简洁、清晰、易懂
5. 包含cURL调用示例
```

**优化效果**：
- ✅ 文档结构完整
- ✅ 信息准确
- ✅ 便于前端集成
- ✅ 包含实际可用示例

---

### 案例3：问题诊断优化

#### 初始Prompt
```markdown
我的网站很慢，帮我分析原因
```

**问题分析**：
- ❌ 缺少关键信息
- ❌ 无法定位问题
- ❌ 建议泛泛而谈

#### 优化后Prompt
```markdown
帮我诊断网站性能问题：

【网站信息】
- 类型：博客网站
- 技术栈：WordPress + PHP 7.4 + MySQL 5.7
- 服务器：阿里云ECS，2核4G
- 访问量：日PV 5000

【性能问题】
- 首页加载时间：5-8秒（目标<2秒）
- 数据库查询慢：某些查询>3秒
- 偶尔出现502错误

【已尝试】
- 启用了WordPress缓存插件
- 优化了图片大小
- 效果不明显

【需要】
1. 分析可能的原因（按优先级排序）
2. 给出验证方法（如何确认是这个问题）
3. 给出解决方案（具体步骤）
4. 预期效果（改善幅度）
```

**优化效果**：
- ✅ 分析更有针对性
- ✅ 方案可落地
- ✅ 包含验证方法
- ✅ 预期效果明确
