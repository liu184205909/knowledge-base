# SPC 苏格拉底式需求梳理提示词

**开发前与 AI 梳理需求的标准化流程** | v1.0

---

## 核心理念

> **SPEC 是 AI 时代的新"源代码"。**

AI 可以帮你写代码，但没办法帮你想清楚到底要写什么。SPEC（需求说明文档）的价值不在于文档本身，而在于**写 SPEC 的过程就是在帮你把想法想清楚**。

从"我想做个XX"到一份完整 SPEC，往往需要经历多轮与 AI 的对话讨论。很多时候你以为自己想清楚了，一写 SPEC 就发现漏洞百出 — 这是正常的，也是必要的。

---

## 提示词模板

直接复制以下模板，替换最后一行的描述即可开始：

```
我要做一个：[一句话描述你的想法]

请你先不要写代码，你需要：
1. 像苏格拉底一样，帮助我梳理需求
2. 你需要一个问题一个问题地问，每次都给出带编号的选项
3. 直到我们能够形成一份完整的SPEC，符合以下标准：
   - Problem Statement（问题陈述）
   - Proposed Solution（方案描述）
   - Technical Constraints（技术约束）
   - Non-goals（明确不做的事）
   - Success Criteria（成功标准）
```

---

## SPEC 的五个标准字段

| 字段 | 回答的问题 | 示例 |
|------|-----------|------|
| **Problem Statement** | 要解决什么问题？给谁解决？为什么值得做？ | "职场新人看不懂互联网黑话，在会议和文档中经常卡壳" |
| **Proposed Solution** | 你打算怎么做？核心功能是什么？用户交互流程是怎样的？ | "一个输入黑话、输出通俗解释的翻译工具，带示例场景" |
| **Technical Constraints** | 技术边界、平台限制、性能要求、依赖条件 | "必须支持中英双语，响应时间<2s，部署为 Web 应用" |
| **Non-goals** | 明确排除的范围，防止需求蔓延 | "不做专业术语数据库，不做社区讨论功能" |
| **Success Criteria** | 什么算"做完了"？可衡量的验收标准 | "用户输入一条黑话后，3秒内获得准确的通俗解释" |

---

## 补充：完整 SPEC 的四个维度

在五个标准字段之外，一份面向开发的完整 SPEC 还应覆盖：

1. **产品定位** — 这是什么？给谁用？
2. **核心功能** — 要做什么？（优先级排序）
3. **交互逻辑** — 怎么做？（用户操作流程）
4. **边界条件** — 不做什么？遇到异常怎么处理？

## 使用要点

### 为什么要让 AI "一个一个问题地问"？

- 防止你跳过关键思考环节
- 每个问题附带编号选项，降低回答成本
- 确保需求梳理的**完整性**和**无歧义性**

### 与 feature-dev 流程的关系

SPC 是 feature-dev 开发 SOP（[03-单AI开发-feature-dev开发流程.md](03-单AI开发-feature-dev开发流程.md)）的**上游环节**：

```
SPC 需求梳理 → Plan 模式讨论 → 编码实现 → 验证反馈循环
```

feature-dev 强调"plan before you act"，而 SPC 就是在帮你形成那个 plan 的输入。

### 适用场景

- 产品想法还比较模糊，需要梳理
- 多人协作时需要对齐需求
- 复杂功能开发前需要确认边界
- 接手一个需求描述不清晰的任务
