# Ralph Loop - 强制迭代工具

> **给 Claude Code 装个"不许下班"的监工** | 强制AI持续优化直到完美 | 2026-01-09

---

## 🎯 核心价值

**Ralph Loop = 治好AI的"交差心态"**

- ✅ **强制迭代**：按着AI的头让它一遍遍优化，直到真正完美
- ✅ **自动循环**：无需人工介入，AI自主判断并继续优化
- ✅ **死磕精神**：像监工一样，不达目标不罢休
- ✅ **成本控制**：设置最大迭代次数，防止烧光预算

---

## 💡 为什么需要 Ralph Loop？

### AI的通病：懒

**一次性输出（One-shot）的问题**：
```
你：写个贪吃蛇游戏
AI：给你一个能运行但极其简陋的版本
AI：搞定！✨

你：这也太简陋了吧...
AI：但你没说要做得好啊
```

**AI的"交差心态"**：
- 只满足于"能用"
- 不会主动优化
- 一次完成就交差

### Ralph Loop 的解决方案

**核心机制**：强行插入一个"停止钩子"（Stop Hook）

**之前的工作流**：
```
接收任务 → 执行 → 结束
```

**Ralph Loop 的工作流**：
```
接收任务 → 执行 → 检查 → 不够好？重来！ → 再执行...
          ↑___________________________________|
                    循环直到达成目标
```

**比喻**：就像一个不知疲倦的监工，拿着鞭子让AI一遍遍优化

## 🎯 核心机制

### Stop Hook（停止钩子）

**原理**：拦截AI的退出动作

当 Claude Code 认为任务完成并试图退出时：
1. Ralph 插件拦截这个动作
2. 把原本的提示词再次"喂"回给AI
3. 强制开启下一轮对话

### 技术实现

```bash
# 简化的伪代码
while :; do
  cat PROMPT.md | claude
  # 只要没达成条件，就无限循环
done
```

### 当前会话（Current Session）

**官方插件已将逻辑内置到当前会话**：

- ✅ **上下文连续**：AI能看到自己上一轮的修改和 git diff
- ✅ **完成承诺**（Completion Promise）：设置"通关密语"（如 DONE）
- ✅ **最大迭代**（Max Iterations）：硬性停止轮数，防止烧光预算

## 📖 使用方法

### 基本命令

```bash
/ralph-loop "任务描述" --completion-promise "DONE" --max-iterations 20
```

**参数说明**：
- `任务描述`：你希望AI完成的目标
- `--completion-promise`：通关密语（AI输出这个词才停止）
- `--max-iterations`：最大迭代次数（安全网）

### 实际案例

#### 案例1：测试迁移

```bash
/ralph-loop "将所有 Jest 测试迁移到 Vitest" \
  --completion-promise "DONE" \
  --max-iterations 20
```

**效果**：
- AI会一遍遍检查和优化
- 直到所有测试都成功迁移到Vitest
- 或达到20次迭代上限

#### 案例2：功能优化

```bash
/ralph-loop "优化这个React组件的性能" \
  --completion-promise "PERFECT" \
  --max-iterations 10
```

**提示词建议**：
```
请优化这个组件：
1. 减少不必要的re-render
2. 使用memo和useCallback
3. 优化state管理
4. 代码风格要清晰

如果完成了以上所有优化，请回复 "PERFECT"
```

### 紧急停止

如果发现AI开始胡言乱语或陷入错误循环：

```bash
# 随时可以强行终止
Ctrl+C
```

## 🎨 提示工程（Prompt Engineering）

### 成功的关键

**Ralph Loop 的威力取决于你的提示词质量**

### ❌ 错误示例

```
做得更好
```

**问题**：太模糊，AI不知道"好"的标准

### ✅ 正确示例

```
请优化这个模块，必须满足：
1. ✅ 无 Linter 报错
2. ✅ 测试覆盖率 > 80%
3. ✅ 代码复杂度 < 10
4. ✅ 所有函数有JSDoc注释
5. ✅ 性能测试通过

只有满足以上所有条件，才回复 "DONE"
```

### 分阶段提交（大任务）

**对于复杂任务，让AI分阶段输出**：

```
第1阶段：基础实现
- 实现核心功能
- 通过单元测试

第2阶段：优化
- 优化性能
- 减少代码复杂度

第3阶段：完善
- 添加文档
- 边界情况处理

每个阶段完成后告诉我，然后进入下一阶段。
```

### TDD 模式（最稳健）

**先写测试，再让循环跑通测试**：

```bash
# Step 1: 先写一个会失败的测试
cat > test.spec.js << 'EOF'
test('should return correct result', () => {
  expect(func()).toBe('expected');
});
EOF

# Step 2: 用 Ralph Loop 跑通测试
/ralph-loop "让这个测试通过" \
  --completion-promise "TEST_PASSED" \
  --max-iterations 15
```

## 📊 适用场景

### ✅ 何时使用（适用场景）

#### 1. 大规模重构

**示例**：
- 框架迁移（React → Vue）
- API升级（v1 → v2）
- 成百上千个文件的机械修改

**为什么适用**：
- 有明确的"成功标准"（编译通过、测试通过）
- 机械性任务，AI可以持续优化

#### 2. 测试覆盖

**示例**：
```
为 src 目录下所有函数补充单元测试
直到覆盖率达到 80%
```

**为什么适用**：
- 有明确的量化指标（80%覆盖率）
- 可以自动化验证

#### 3. Greenfield 搭建

**示例**：
```
从零开始搭建一个博客系统
要求包含：用户系统、文章管理、评论系统
```

**为什么适用**：
- 有明确的功能清单
- 可以分阶段实现和验证

### ❌ 何时不用（不适用场景）

#### 1. 架构决策

**示例**：
```
设计一个微服务架构
```

**为什么不用**：
- 需要人类的直觉与经验
- 没有绝对的"正确答案"
- AI可能陷入局部最优

#### 2. 模糊需求

**示例**：
```
做一个好用的界面
```

**为什么不用**：
- 如果连你都定义不出什么是"好"
- 循环永远不会收敛
- 浪费时间和金钱

#### 3. 安全敏感代码

**示例**：
```
实现支付系统
```

**为什么不用**：
- 严禁全自动生成
- 必须人工每步审核
- 错误代价太大

**虽然单次API调用便宜，但循环起来就是碎钞机**

**预估成本**：
- 小代码库迭代 20 轮：$10-30
- 大代码库迭代 50 轮：$50-100+
- 长时间运行（数小时）：$200+

**建议**：
- 💰 务必设置 `--max-iterations`
- 💰 建议分批次运行
- 💰 设置 API 额度硬顶
- 💰 监控 Token 使用量

## 🔒 风险与最佳实践

### 风险控制

#### 1. 成本控制（Cost Control）

```bash
# 永远、永远要设置 max-iterations
--max-iterations 20

# 对于大型任务，分批次运行
/ralph-loop "第1批：优化前10个文件" --max-iterations 10
/ralph-loop "第2批：优化后10个文件" --max-iterations 10

# 设置 API 额度硬顶
# 在 Claude Code 设置中配置月度限额
```

#### 2. 验证优先（Verification First）

**不要依赖 AI 说"我做好了"**

**使用可编程验证**：
```javascript
// 自动化测试
npm test

// 构建脚本
npm run build

// 静态检查
npm run lint

// 编译器不会撒谎
```

**AI的主观判断往往过于自信**

#### 3. 提示工程调优

**从"无护栏搭乐园"到"加提示牌"**

如果Ralph总是犯同样的错：

```bash
# 在 Prompt 里显式加入"禁止项"
❌ 不要使用嵌套三元运算符
❌ 不要使用 eval
❌ 不要省略错误处理

✅ 使用 if-else 或 switch
✅ 使用显式错误处理
✅ 添加边界检查
```

**记住**：LLM 是操作员技能的镜子

#### 4. 安全边界

**引入人工 Review 节拍**：

```bash
# 每 10 轮暂停一次
/ralph-loop "..." --max-iterations 10 --checkpoint-interval 10

# 在涉及数据迁移或破坏性操作时
# 绝不让 Ralph 独自在生产环境裸奔
```

## 🎯 总结

### Ralph Loop 告诉我们什么？

**AI 的潜力不在于**：
- ❌ 单次推理有多快
- ❌ 一次性输出多完美

**AI 的潜力在于**：
- ✅ 能多不知疲倦地修改自己的错误
- ✅ 能持续迭代直到达成目标
- ✅ 能从错误中学习

### 核心哲学

> 在这个人人都想"一次过"的快节奏时代，
> 给你的AI一点"傻傻的执着"，
> 或许才是弯道超车的关键。

### 落地行动清单

1. **小步快跑**：先在小模块上试用
2. **明确停止条件**：量化满意度
3. **监控与审查**：每N轮加入人工Review
4. **资源预算**：设置硬锁（时间或成本）
