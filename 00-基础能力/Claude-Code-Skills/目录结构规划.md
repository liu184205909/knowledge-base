## 🎯 设计原则

### 1. 按领域分类

```
marketing/     → 营销相关工作流
development/    → 开发相关工作流
geo/           → GEO相关工作流
common/        → 跨领域通用工具
```

**为什么这样设计？**
- ✅ 清晰的领域边界
- ✅ 便于团队协作（不同团队负责不同领域）
- ✅ 便于选择性安装（可以只安装某个领域的Skills）

---

### 2. 主控Skill模式

每个领域都有一个主控Skill：

```
marketing/
└── marketing-master/              # 调用领域内所有子Skill

development/
└── dev-master/                    # 调用领域内所有子Skill

geo/
└── geo-master/                    # 调用领域内所有子Skill
```

**优势**：
- ✅ 一句话触发完整工作流
- ✅ 自动调用子Skill（递归）
- ✅ 标准化执行流程

---

### 3. References机制

每个Skill都包含参考文档：

```
market-analysis/
├── SKILL.md                       # 可执行的工作流
└── references/                    # 参考文档（学习用）
    ├── 01-市场分析.md             # 流程文档
    └── 选品评估框架.md            # 评估标准
```

**双轨设计**：
- **SKILL.md**: 可执行版本（AI自动执行）
- **references/**: 学习版本（人类可读）

**为什么需要references？**
- ✅ 理解Skill的工作原理
- ✅ 学习最佳实践
- ✅ 自定义Skill时参考

## 🚀 安装策略

### 策略1: 全部安装（推荐）

## 📝 创建新Skill的流程

### Step 1: 规划

在本文档中添加：
```markdown
| 领域 | Skill名称 | 优先级 | 说明 |
| development | code-review | P2 | 代码审查工作流 |
```

---

### Step 2: 开发

创建Skill文件：
```bash
mkdir -p development/code-review
touch development/code-review/SKILL.md
```

编写SKILL.md：
```yaml
---
name: code-review
description: |
  代码审查技能。使用此技能当用户需要:
  1. 审查XX代码的质量
  2. 检查XX代码的安全性
  3. 优化XX代码的性能

  触发关键词: "审查代码"、"代码review"、"检查代码质量"

  执行流程:
  1. 代码风格检查
  2. 安全性审查
  3. 性能优化建议
  4. 最佳实践建议

language: chinese
---

# 代码审查 Skill
...
```

---

### Step 3: 测试

```
"测试code-review Skill"
"审查这段代码: [代码]"
```

### Step 4: 部署

```bash
# 安装到Claude Code
xcopy "development" "%USERPROFILE%\.claude\skills\development\" /E /I /Y

# 验证
"检查已安装的Skills列表"
```

## 🎯 最佳实践

### 1. 命名规范

**Skill名称**：
- 使用小写字母
- 使用连字符分隔
- 描述性命名（market-analysis, code-review）

**触发关键词**：
- 简洁明确（"分析XX"、"审查代码"）
- 覆盖常用表达（"分析"、"评估"、"研究"）
- 避免歧义（不用"处理"、"操作"等通用词）

---

### 2. 文档组织

**SKILL.md**：
- 简洁明了（AI执行用）
- 专注于执行流程
- 包含触发关键词

**references/**：
- 详细完整（人类学习用）
- 包含理论背景
- 包含最佳实践

---

### 3. 版本管理

**Git策略**：
- ✅ 将整个`Claude-Code-Skills/`纳入知识库Git

- ✅ references/文档版本管理
