## 核心概念

**RLM = 分而治之的工作流设计方法**

```
传统方式: 用户 → AI → 一次性输出（容易遗漏、泛泛而谈）

RLM方式（流程文档）:
用户 → 主控流程文档 → 调用子流程1 → 调用子流程2 → 汇总报告

RLM方式（Skills）:
用户 → 触发主控Skill → 自动调用子Skill1 → 自动调用子Skill2 → 汇总报告
```

---

## 四种实现方式

| 方式 | 适用场景 | 优势 | 劣势 |
|-----|---------|-----|------|
| **Claude Code Skills** | 固定工作流（自动化触发） | ✅ 自动识别触发<br>✅ 可包含脚本和参考文档<br>✅ 支持递归调用子Skill | ❌ 需要安装配置 |
| **流程文档递归** | 固定工作流（产品分析、内容生产） | ✅ 结构化、可复用<br>✅ 无需安装，直接使用 | ❌ 需要手动指定文档路径 |
| **MCP工具** | 需要联网/文件操作 | ✅ 实时获取信息<br>✅ 工具生态丰富 | ❌ 每次要描述工作流 |
| **Prompt** | 临时任务、快速验证 | ✅ 最简单、立即可用 | ❌ 难以复用 |

**选择建议**：
- ✅ **有Claude Code Skills** → 优先使用（自动触发，体验最好）
- ✅ **有现成流程文档** → 直接用（无需配置）
- ✅ **需要搜索/读网页** → MCP工具
- ✅ **一次性任务** → Prompt

**Skills vs 流程文档**：
- **Skills**: 适合频繁使用、需要自动触发的场景（如"分析产品"、"生成文章"）
- **流程文档**: 适合项目特定、临时性的工作流（如"XX项目的部署流程"）

## 降级方案（Skill/流程文档不可用时）

当没有现成的Skill或流程文档时，可以使用Prompt降级方案：

```markdown
请像项目经理一样，分步骤完成"XX产品"分析：

第1步：市场分析
第2步：竞品分析
第3步：用户洞察
第4步：差异化策略

最后汇总成完整报告。
```

**注意**: 这种方式每次都需要重新描述工作流，无法复用。

## 如何创建新的RLM工作流

### 方式1: 创建Claude Code Skill（推荐⭐）

**适合**: 需要自动触发、频繁使用的工作流

**步骤**：
```
1. "帮我创建一个[XX功能]的Skill"
2. AI会自动创建Skill目录和SKILL.md
3. 配置description中的触发关键词
```

**优势**：
- ✅ Claude自动识别关键词并触发（无需手动指定）
- ✅ 支持递归调用子Skill（主控Skill → 子Skill1 → 子Skill2）
- ✅ 可以包含脚本和参考文档
- ✅ 热重载（修改后立即生效）

**RLM递归示例**：
```yaml
# 主控Skill: product-analysis
description: |
  产品分析技能。使用此技能当用户需要:
  1. 分析产品市场潜力
  2. 评估竞争情况
  
  触发关键词: "分析XX产品"、"评估XX产品"
  
  执行流程:
  1. 调用market-research子Skill（市场研究）
  2. 调用competitor-analysis子Skill（竞品分析）
  3. 调用user-insights子Skill（用户洞察）
  4. 汇总完整报告
```

**详细指南**: [04-Skills完整指南.md](./04-Skills完整指南.md)

### 方式2: 创建流程文档

**适合**: 临时项目、简单工作流、项目特定需求

直接告诉AI：
```
"帮我创建一个[XX功能]的流程文档，包含以下步骤：..."
```

AI会自动参考现有流程文档的格式创建。

**流程文档递归示例**：
```
主控文档: 01-市场分析.md
  → 调用子流程: 产品评估
  → 调用子流程: 竞品分析
  → 调用子流程: 用户洞察
  → 汇总报告
```
