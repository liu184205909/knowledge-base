# 宝玉：Skills的正确用法

> **来源**: 微信文章 | 作者：宝玉

**核心观点**: Skills不是越多越好。真正的用法是因需而建、可组合、可迭代——先干活，干到卡壳再封装，让解决方案能串联、能进化。

---

## Skills四问判断法

### 1. 你真的需要Skills吗?

**定义**: Skill是用来补充Agent本身不具备、而你又反复需要的信息

**❌ 不需要Skill的情况**:
- Agent已经知道怎么做（如下载YouTube视频，流行的开源项目它都训练过）
- Agent能通过搜索自己搞定
- 过度封装只会增加复杂度，不带来额外价值

**✅ 真正需要Skill的场景**:
- 反复踩坑后，发现某个地方**每次都要解释一遍**

**实战案例**:
- `release-skills`: 每次发布都要写changelog、更新README、写commit message、决定版本号 → 封装成Skill

---

### 2. Skills是最佳方案吗?

**Skills的甜蜜点**:

```
简单提示 ←──── Skills ────→ Web应用
  太简单           最佳方案          太复杂
```

**判断标准**:
- 提示词能解决 → 用提示词
- 需要写Web应用 → 成本太高
- **多步骤串联但不值得开发系统** → Skill最佳方案

**实战案例 - 文章配图**:
- ❌ 提示词做不到（只能分析文章、生成画图提示词，但要人工一张张生成、一张张插入）
- ❌ 写程序成本高（要搭Web应用，接LLM API和画图API）
- ✅ Skill完美：Agent分析需要多少配图 → 生成提示词 → 调图像API → 插入合适位置，全程自动化

---

### 3. 你的Skill能和其他Skills组合吗?

**核心思想**: 模块化，像乐高一样拼起来

**❌ 避免**:
- "巨无霸"Skill（包揽所有看似省事，实则堵死组合可能性）
- 功能重叠

**✅ 提倡**:
- 单一职责（每个Skill做好一件事）
- 可组合（一组可组合的Skills能解决一类问题）

**实战案例 - 写作流程**:
```
素材 → 分析Skill → 提纲Skill → 写作Skill
```

分析Skill和提纲Skill在写作时用，写PPT时又能重用，避免重复造轮子。

---

### 4. 这个Skill值得你持续迭代吗?

**核心观点**: 好的Skill不是写出来的，是**用出来的**

**迭代成本极低**: 用的过程中发现问题，马上让Agent帮忙优化，省去了重现描述的步骤

**实战案例 - 小红书Skill迭代**:
1. 最开始：简单的提示词
2. 发现样式单一 → 添加不同样式风格
3. 发现信息密度低 → 添加Layout选项
4. 内容长了质量不够 → 增加分析并提炼大纲
5. 大纲不是想要的 → 一次性出多个版本供选择
6. 加上水印支持

**判断标准**: 如果一个Skill做完之后再也不想碰了，那大概率这个Skill本身就不该做

---

## Skills三原则

1. **因需而建** - 先干活，干到某个地方反复卡壳，再用最小上下文解决这个卡壳
2. **可组合** - 让解决方案和其他Skill组合
3. **可迭代** - 持续优化，在使用中完善

---

## Skill的本质

**给Agent写Skill，就像给新员工写入职指南**

不会在第一天就把所有文档都塞给他，而是根据要做的事，给他最需要的那几份。
