# Claude Routines — AI 全天候自动化

> Claude Code 从「需要盯着屏幕的交互式工具」→「云端 24 小时自动运行的系统」

---

## 核心概念

**无头 AI（Headless AI）**：不需要聊天界面，不需要人盯着。在云端服务器监听触发器（邮件、订单、定时），触发后自动启动、思考、执行、汇报。

**每次运行都是全新快照**：Anthropic 启动云端容器 → 克隆你的 GitHub 仓库 → 执行任务 → 销毁容器。两次运行之间的「记忆」必须存在仓库里。

---

## Routines vs n8n — 互补而非替代

| 维度 | n8n / Zapier | Claude Routines |
|------|-------------|-----------------|
| 执行方式 | 刚性流程图，严格按脚本 | 根据上下文动态推理 |
| 异常处理 | 未定义的情况直接报错 | 尝试理解意图并灵活处理 |
| 配置难度 | 需懂 API 和数据结构 | 会写清晰 Prompt 即可 |
| 适用场景 | 确定性强、规则明确的任务 | 模糊性高、需要判断的任务 |

**最佳实践**：用 n8n 触发 Routines → Routines 思考完 → 交回 n8n 执行后续动作。

## 触发方式

### 事件触发（门铃）
特定事件发生时响应：新邮件、新订单、差评、购物车放弃等。

### 定时触发（闹钟）
按周期执行：每天/每周定时扫描、汇总、生成报告。

**执行配额**：Pro 5次/天、Max 15次/天、Team/Enterprise 25次/天。

## 实用场景

### 业务增长类
- **购物车放弃恢复**：分析客户浏览历史，写个性化挽留邮件
- **差评危机公关**：收到 1-2 星评价时自动分析、起草回复、通知负责人
- **流失风险应对**：客户点击取消时，根据使用历史写挽留方案
- **销售线索挖掘**：新线索录入后自动搜索公司信息，生成破冰指南

### 效率提升类
- **早晨简报**：扫描 Slack、邮件、日历，只推送紧急事项和待决策
- **竞品雷达**：每周扫描竞品官网、Release Notes、社交媒体动态
- **内容矩阵生成**：一篇长博客 → 拆解为 Twitter / LinkedIn / 短视频脚本
- **团队周报**：读取 GitHub 提交 + Jira 状态，自动汇总

### 跨境电商类
- **智能时差客服**：北美客户凌晨发邮件时，自动查询订单物流并回复
- **爆款预警**：监控 TikTok 标签播放量激增，通知采购备货
- **竞品价格监控**：每天抓取竞品定价页，发现调价立刻通知销售

## Prompt 六要素

一个好的 Routine Prompt 必须包含：

1. **角色和上下文** — 你是谁，在什么环境里工作
2. **具体任务** — 清晰、无歧义的目标
3. **数据来源** — 信息从哪来，变量名叫什么
4. **步骤流程** — 按顺序列出多步骤任务
5. **输出格式** — 明确定义输出长什么样
6. **失败处理** — 出错时怎么办（如发 Slack 告警而非直接执行）

## 仓库结构设计

```
/
├── CLAUDE.md          ← 全局指令（原则和约束）
├── skills/            ← 技能库（模块化 Prompt）
│   ├── daily-briefing.md
│   ├── weekly-report.md
│   └── content-draft.md
├── outputs/           ← 输出目录
│   ├── reports/
│   └── drafts/
└── memory/            ← 记忆系统（跨运行保持状态）
    └── last-run-data.json
```

**模块化思维**：把复杂指令拆成 Markdown 文件放在 `skills/` 下，让 Routines 按需读取，而不是塞进一个巨型 Prompt。

## 记忆系统

每次运行都是全新容器，如何保持连续性？

**方案**：让 Routine 自己写日志。每次运行结束时把状态写入 `last-run-data.json`，下次运行时先读取算差异，只报告变化部分。

```json
{
  "lastRun": "2026-04-21T08:00:00Z",
  "trackedItems": {
    "competitor-pricing": { "lastChecked": "...", "changes": [] }
  }
}
```

## 避坑指南

### 1. 最小权限原则
只给 Routine 完成任务所需的最小权限。先在沙盒测试，再接生产数据。

### 2. 设置断路器
防止死循环触发。如："同一客户 24 小时内最多触发一次"。在 n8n 中用防抖节点。

### 3. 禁止幻觉约束
在 Prompt 中加入强约束：
> 绝对禁止编造链接、数据或政策。如果知识库中找不到答案，回复"需要向主管确认"，并标记为人工介入。

### 4. 敏感信息管理
API Key 配置在 Cloud Environment 中，不要把 `.env` 提交到 GitHub。
