## URL 标准化规则

`normalized_url` 必须统一：

- 小写
- 去掉 `http://` 或 `https://`
- 去掉 `www.`
- 去掉查询参数 `?`
- 去掉锚点 `#`
- 去掉末尾 `/`

同一路径可能出现在多个竞品中，关联时必须使用 `competitor_domain + normalized_url`，不能只用 URL 路径。

## 1C 使用规则

- Seed-Master 是本项目唯一关键词主表；Seed-* 工作表只是导入暂存表。
- Top Pages / Top Keywords 不直接合并进关键词主表。
- `TopPages_All` / `TopKeywords_All` 是过滤后的证据索引，不是原始数据全量备份。
- `Keyword-Page-Proof` 只负责给关键词主表补竞品成功证据，不能把关键词流量为0的排名行当成成功案例。
- 竞品验证中发现的增量词，只能以 Candidate 状态补回 Seed-Master，并标记 `Source Type` / `Source Detail`。

## 数据架构协议

> 以下规则适用于本项目的关键词数据处理，详见 [01-关键词研究方法论.md](../../../../01-营销方法论基础/01-关键词研究方法论.md)。

**核心原则**：
- All 层负责发现全貌，Seed-Master 负责决策，Proof 负责验证
- 主题发现发生在 TopKeywords_All 层，不发生在 Proof 层
- 主题是关键词级属性，不是竞品级属性。竞品可以有主题倾向（`known_topic_hint`），但关键词才有最终主题归属
- Competitor-Sheet-Map 的 `known_topic_hint` 只是可选先验提示，不参与强制分组
- Seed-Master 是唯一关键词主表，暂存表只做导入区
- 未知主题先进入 Topic-Discovery 候选，不自动变成正式关键词库

**Seed-Master 合并规则**：
1. 暂存表优先（Source Type = Seed），TopKeywords_All 增量词为辅助（Source Type = Competitor_Top_Keyword）
2. 同 keyword 已存在时不新增重复行，只补来源和 proof
3. 暂存表之间 Topic Pillar 冲突时不自动覆盖，进入人工审核
4. 增量词只能进 Candidate 状态，不能直接变 Approved

**Proof scope 切换规则**：
- Seed-Master 为空时，Proof 脚本仍读取所有 Seed-* 暂存表
- 切换条件：Seed-Master 首轮合并完成（当前已满足）
- 旧 Proof 数据保留，但不作为长期依据，新流程重跑后覆盖

**去重键**：`Normalized Keyword`（全小写、去首尾空格、多空格合并为单空格，不做词形归并和同义词归并）

**Draft 标记**：
- Competitor-Sheet-Map 已跳过，直接用全部 36 竞品工作表
- Seed-Master v2 + 轨道D + Page Type 映射已完成（44,999 条 × 24列），1B 轨道 B+C+D 全部完成
