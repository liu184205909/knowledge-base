### 1.2 行业最佳实践总结

**A. 抽牌交互（参与感三档）**
1. **高参与感（主流）**：牌背全展（78 张网格/扇形）→ 用户**主动点选牌背**抽牌（Tarotap/Evatarot/Trusted Tarot）。仪式感最强，用户"亲手抽"的心理投入大。**这是行业绝对主流**，4/6 自助工具采用。
2. **中参与感**：分步向导选牌组+牌阵+问题 → 系统抽（Llewellyn）。清晰但抽牌动作弱。
3. **低参与感**：点一个按钮直接出结果（早期工具，已被淘汰）。

→ **v3 决策：采用"高参与感"档**——牌背全展 + 用户点选翻牌。这与 v2 设计方向一致，但 v2 因 JS bug 没跑通；v3 修复后是行业主流模式。

**B. 牌背设计**
- 行业共识：**统一牌背**（不随牌变化），深色基调 + 金色符号 + 星空/几何纹理（Tarotap 紫金∞、Labyrinthos 金箔、Evatarot 深色神秘）。
- 牌背要"神秘但克制"——不能抢牌面风头，但要好看到用户想点。
→ **v3 决策**：深 `#1A1A2E`→`#2D2D52` 渐变 + 金 `#CFAA3E` 边框 + 中心符号（用塔罗符号如 ☽/✦/☉），统一牌背。v2 已是这个方向，v3 保留并优化。

**C. 翻牌动画**
- 主流：CSS 3D `rotateY` 翻转（牌背→牌面），时长 0.4-0.6s。
→ **v3 决策**：CSS `rotateY` 翻牌 0.5s（v2 有但 bug 阻断，v3 修复）。

**D. 解读深度（行业分两派）**
- **浅派**：关键词+短牌义（多数免费工具）。
- **深派**：多视角（Tarot.com 多位置、Trusted Tarot 注册后深度、Biddy 牌义标杆）。
- **我们差异化**：tarot-knowledge.json 字段极丰富（psychological_lens / eastern_anchors / recommended_practice / chakra / related_cards / 5 颗水晶），**深度可对标 Tarot.com 付费层，但免费给 + 水晶电商独家维度**。这是 hint 规划里"卖确定性幻觉+实物"的核心。
→ **v3 决策**：深派。每张牌结果卡包含：正/逆位关键词 + 牌义 + 心理学视角 + 东方视角 + 反思问题（recommended_practice）+ 脉轮 + 关联牌 + 2 颗水晶（含 reason+shop）。**行业里没有一个免费工具给到这个深度**。

**E. 牌阵**
- 最高频：**单牌（Daily）+ 三牌阵（Past/Present/Future）**，这是所有竞品的标配入门。
- 进阶：Yes/No、Celtic Cross、Love。
→ **v3 决策**：单牌 + 三牌阵（Past/Present/Future）两种，覆盖 80% 入门需求。Yes/No 留作同引擎变体（规划文档已列）。

**F. Freemium**
- 行业：免费给基础抽牌+浅解读，深度解读/PDF/特定牌阵付费（Tarot.com VIP / Trusted Tarot 注册 / hint 订阅）。
- **我们（电商站）策略不同**：不卖订阅，**免费给深度解读 → Shop 水晶变现**。规划文档明确"订阅留到 5 万 UV"，现阶段免费深度+电商是差异化。
→ **v3 决策**：无付费墙。完整深度免费，水晶 Shop CTA 是变现路径（三级降级）。

**G. SEO**
- 主词：`free tarot reading` / `daily tarot` / `tarot card meaning` / `crystals for tarot cards`（我们独有，低竞争长尾）。
- 结构：FAQPage schema（how tarot works / predict future / bad card / crystals for tarot）+ 深度 seo-content accordion。
- **我们的 SEO 差异化词**：`crystal tarot reading` / `crystals for tarot cards` / `tarot card crystals`——竞品都没水晶维度，这是我们独家词族（22 篇文章 + 工具双向集群）。

## 二、v2 bug 分析（真实根因，已通过线上 curl + Playwright pageerror 定位）

### 2.1 现象
- 线上 `goearthward.com/tools/crystal-tarot-reading/` 点 Shuffle → Playwright/CDP 实测：`ECTaro is not defined`，牌背数 = 0，抽牌完全跑不通。
- 但 `ect-focus` / `ect-spread` / `ect-wrap` 等 HTML 元素都在（静态 HTML 渲染正常）。

### 2.2 真正根因（三重，逐层定位）

Playwright 捕获到 `PAGE ERRORS: ['Invalid or unexpected token']`，这是 IIFE 整个不执行的原因。`node --check` 提取线上 script 后定位到第 51 行：

```
}else if(focus==='love' &#038;&#038; c.crystals.best_love){
```

**根因 = WordPress 的 `wp_kses_post` / `the_content` 过滤器对 `<!-- wp:html -->` 块内 `<script>` 里的 HTML 特殊字符做了实体转义**：
- `&&` → `&#038;&#038;`（最致命，JS 逻辑运算符变 HTML 实体 → 语法错误）
- `&` → `&amp;`
- `<` → `&lt;`（JS 比较 `a<b` 变 `a&lt;b`）
- `>` → `&gt;`
- 此外非 ASCII 字符（em dash `—` U+2014、en dash `–` U+2013、`✦` U+2726、CJK 佛教术语）经 WP 数据库字符集存储后字节破坏，也是 `Invalid token` 的次要来源。

**这是 WordPress HTML 块内嵌 JavaScript 的已知硬限制**：即使 `<!-- wp:html -->` 块"理论上"保留原始 HTML，WP 仍会对内容做实体转义，破坏 JS 语法。v2 的内联 `onclick` 和内联 `var CARDS=${JSON}` 双重触发此问题。

### 2.3 v3 规避策略（最终方案：Base64 包装可执行 JS）

经过 3 轮迭代定位，最终采用的方案：

1. **可执行 JS 用 Base64 编码**：APP_JS 作为 generate.js 里的可读模板字符串常量，输出时 `Buffer.from(asciiEscape(APP_JS)).toString('base64')`，放进 `<script type="text/plain" id="ect-app">`。一个微型 loader `atob(textContent) → eval`。Base64 是纯 ASCII（A-Za-z0-9+/=），**WP 任何过滤器都无法破坏**，彻底绕过 `&&`/`<`/`>`/`&` 实体化和非 ASCII 字节破坏。
2. **JSON 数据放独立 `<script type="application/json" id="ect-data">`** + `asciiJSON`（非 ASCII 转 `\uXXXX`），JS 运行时 `JSON.parse(textContent)`。
3. **事件用运行时 `addEventListener`**（不用内联 onclick）。
4. **FAQ schema 也 `asciiEscape`**（em dash → `—`），避免 Google 解析 ld+json 时 `&mdash;` 实体化破坏 JSON.parse。
5. **loader script 必须在 ect-app 块之后**（DOM 顺序，否则 loader 执行时元素不存在）。

**safeJSON 约束遵守**：safeJSON（给 FAQ text 英文字段）仍只 `.replace(/<\//g,'<\\/')`。新增的 `asciiEscape`/`asciiJSON` 是针对 WP 部署通道的额外保护，不替代 safeJSON。

**教训沉淀（写入 memory）**：WP `<!-- wp:html -->` 块内嵌 JS 必须 Base64 包装，不能裸写（`&&`/`<`/`>`/`&` 会被实体化破坏语法）。这是 `generate-js-html-newline-trap` 之外的另一个 WP 部署陷阱。

## 三、T14 v3 交互设计

### 3.1 用户流程（5 步，基于竞品"高参与感"最佳实践）

```
Step 1: 选 focus（Love / Career / Healing / Guidance / General）—— 下拉
Step 2: 选 spread（Single Card / Past-Present-Future）—— 下拉
Step 3: 点 "Shuffle the Deck"
        → 显示洗牌动画（牌堆交错动画 0.6s）
        → 揭示牌背（1 张或 3 张，金色边框深色牌背 + 中心符号 + "Tap to reveal"）
Step 4: 用户点牌背
        → CSS rotateY 翻牌动画 0.5s
        → 揭示完整结果卡（深色头部 + 完整内容）
Step 5: 结果卡底部 CTA
        → "Read the full [Card] guide" （/tarot-{slug}-crystals/）

        → 三牌阵时 3 张依次翻开
```

### 3.3 牌阵
