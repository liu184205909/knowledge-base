## §9 候选策略(→ 1H)

### 候选模仿
1. **模仿 AI 辅助规模化内容生产(ChatGPT + 数据库 + 人工审核)** — 证据:[§6] homepage "using multiple databases of dream reports with the aid of ChatGPT"。适用:我们水晶/解梦内容用 AI 辅助批量生产基础条目(每种水晶/每个符号),人工审核矿物学 + 灵性准确性
2. **模仿"真实生活事件关联"方法论** — 证据:[§2]。适用:水晶/解梦内容嵌入"真实用户故事"(UGC 评论/访谈),比纯理论更有说服力 + 转化力
3. **模仿 3 国 IP + 商标品牌保护** — 证据:[§1]。适用:Earthward 品牌 + 核心系列在美/英/欧/加注册商标
4. **模仿 Request Symbol UGC 驱动内容生产** — 证据:[§3]。适用:用户提交"想查 X 梦境/ X 水晶",驱动长尾 + 参与度

### 候选超越
1. **超越 jQuery UI + viewport=600 古早栈** — 我们用现代响应式 + 移动优先,移动端体验代差
2. **超越分类浅(8 主题 vs dreammoods 26)** — 我们建多轴矩阵(A-Z + 主题 + 类型 + 理论家),深度 + 广度双胜
3. **超越无社区** — 我们建梦境日记/水晶日记社区,UGC 双向
4. **超越单一变现** — 叠加水晶电商 + AI 工具

### 候选差异化
1. **AI 辅助 × 水晶实物闭环** — Dream Bible AI 辅助只做字典;我们 AI 辅助生产 + 解梦工具 + 自动推荐水晶 + Shop
2. **真实用户数据库 × 梦境水晶日记** — Dream Bible 访谈梦境;我们收集"用户做了什么梦 + 用了什么水晶 + 效果",数据驱动推荐,独家
3. **现代科学方法论文档化** — Dream Bible 方法论只在 homepage 一段;我们做"How We Interpret Dreams"独立方法论页(荣格 + 现代睡眠科学),E-E-A-T + AIO 引用

### 11.2 SEMrush 数据补的"初版未提及"新发现

#### 1) "6666 符号" 不可 URL join 说明(用户严边界)

> 用户明确边界:**不要写"6666 symbols 里多少有流量"**,6666 是站内声称/结构规模,SEMrush 只捕捉有流量页,是两个集合。

dreambible 涉及 **4 个互不重叠的集合**,任何两两 join 都不成立:

| 集合 | 数量 | 来源 | 性质 |
|---|---|---|---|
| **A. 站内声称符号总数** | 6,666 | homepage 计数器(2026-06-06 Last Updated) | dreambible 自报的结构规模 |
| **B. sitemap 暴露的静态 URL** | 96 | sitemap.xml(实测) | 静态页(/Symbols.htm、/CommonDreams/*.htm、字母索引页等) |
| **C. SEMrush Top Pages 捕捉的有流量 URL** | 516 | SEMrush organic TP(2026-07-08) | 实际被搜索触达的页(503 个 search.php 动态 + 12 静态 + 1 首页) |
| **D. SEMrush Top Keywords 捕捉的有流量词** | 1,597 | SEMrush organic TK(2026-07-08) | 实际被搜索的关键词 |

**为何 A∩C 不可计算**:6666 符号没有独立静态 URL 清单。它们通过 `/search.php?q={Symbol_Name}` 动态加载(如 q=Tigers / q=Blood / q=Funeral),**没有 sitemap 暴露、没有静态 URL 列表**。我们能拿到的只是 SEMrush 通过反向搜索流量捕捉到的"被实际触达"的 search.php 子集(503 个)。**这个子集不等于 6666 符号的随机抽样**——它是有流量符号的偏态样本(高需求符号如 tiger/blood/birth 等),低需求/无需求符号 SEMrush 根本捕捉不到。

**唯一可写的实测分布陈述**:SEMrush 在 dreambible.com 上**捕捉到 516 个有流量 URL + 1,597 个有流量关键词**——这是 dreambible **可见 SEO 资产**的实测下限。⚠️ **不能推导 6666 符号整体覆盖率**:516 URL 中 503 个是 /search.php?q=X 动态页,一个 q 参数页可对应多词,去重前不等同于独立符号数量;6666 符号无全量 URL 清单做 join(详 §11.2.1)。能确认的是:有一批高需求 symbol 被搜索触达(tiger/blood/birth/celebrity 等),其余符号的流量状态未知。

#### 2) /search.php?q=X 动态 URL 是 dreambible 的流量主引擎(初版未发现)

SEMrush Top Pages 516 行 URL 模式分布:

| URL 模式 | 页数 | 占比 | 典型页面 |
|---|---|---|---|
| **/search.php?q={Symbol}** | **503** | **97.5%** | /search.php?q=Blood / q=Tigers / q=Shooting / q=Rape / q=Birth |
| /(root 首页) | 1 | 0.2% | 首页(承接品牌词 + biblical/christian 词) |
| 其他静态页 | 12 | 2.3% | /CommonDreams/、/themes/、字母索引等 |

**核心洞察**:dreambible 把"**站内搜索结果页**"做成了 SEO 流量页。这是一种**边缘 SEO 策略**——Google 一般不喜欢索引搜索结果页(Google Search Central 文档明确指出"避免索引搜索结果页"),但 dreambible 通过以下方式让其生效:

1. search.php?q=X 返回的**不是搜索索引,而是符号详情内容**(类似 Wordpress 的"标签页"机制),所以 Google 视为内容页
2. URL 用 `?q=X` 参数形式,Google 将其视为同一路径(/search.php)的不同参数变体,部分变体被索引
3. 每个符号内容是 5600+ 字字典的真实条目,内容深度达标

**对标启示**(对 1H):
- **不学**:dreambible 的 search.php 模式是**历史遗留** + 古早栈限制(jQuery UI 不能做静态路由),新站不应复制。Google 对搜索结果页的容忍度随时可能收紧,这是**高风险模式**
- **可学**:用 /search.php?q=X 承接 "{X} in dream" / "dreaming of {X}" / "{X} dream meaning" 等多词变体到单页的做法,**精神可借鉴**——我们做静态 post 根级 `/{symbol}-dream-meaning/` 单页（B 方案，2A 裁决，非 /dream-dictionary/ 前缀），承接所有 "{symbol}" 词变体,获得同等效果但不冒 SEO 风险
