# T17 水晶手链定制工具 - v3 需求基线

> **状态**：开发前基线，2026-07-12 起作为后续需求、原型、后台和代码改造的唯一执行来源。  
> **定位**：移动端优先的 2D 水晶手链设计器；官方设计产品可直接购买，也可一键导入后继续定制。  
> **不直接覆盖**：v2 主文档、旧 UI 规范和 UI 修正记录保留为历史证据，不再作为新增功能的验收依据。

---

## 1. 为什么需要 v3

此前页面在现有 HTML/JS 工具上连续微调，导致视觉、交互和数据模型互相牵制。根因不是单个 CSS 问题，而是旧方案仍基于以下已过期的前提：

- 3D 手链与复杂模型资产。
- 全局 `Bead Size`，一条手链只能使用一种珠径。
- 直接从编辑器 Checkout。
- 静态 JSON 素材库和页面内嵌逻辑。
- 官方预设、现货商品、模板之间界限不清。

v3 先统一产品、数据和交易模型，再进入界面和代码重构。现有线上工具在 v3 核心完成前仅做阻断性 bug 修复，不再继续叠加局部 UI 调整。

## 2. 最终产品模型

### 2.1 两种设计来源

| 类型 | 本质 | 用户动作 | 管理位置 |
|---|---|---|---|
| 官方设计产品 | 一条可直接销售的官方手链设计，也是一份可编辑配方 | 直接购买 / Customize this design | WooCommerce 产品 + T17 配方字段 |
| 用户设计作品 | 用户保存、后续可选择公开的设计快照 | 保存 / 分享 / 借鉴并编辑 | T17 设计快照库；设计广场为后续阶段 |

官方预设模板和官方现货成品不是两套产品。它们统一称为 **Official Design Product**：同一份官方设计同时支持直接购买和导入编辑器。

### 2.2 两条购买路径

1. **直接购买官方设计产品**：进入 WooCommerce 产品页或工具下方官方设计列表，以 Woo 产品当前售价购买。
2. **定制后制作**：用户从官方设计、意图快捷入口或空白设计开始，在编辑器中修改；点击 `Finish Design` 进入确认页，再进入 WooCommerce 购物车与结账。

编辑器内不设置固定 Checkout 按钮。移动端底部仅在需要时显示设计操作或确认动作，避免与站点 Mobile Bottom Navbar、翻译浮窗和内容滚动冲突。

## 3. 页面与漏斗

### 3.1 工具页面

1. Hero：`Design Your Own` 与 `Shop Official Designs` 两个入口。
2. 官方设计产品列表：展示完整手链图、名称、用途/故事、Woo 售价、`Buy this design`、`Customize this design`。
3. 2D DIY 编辑器。
4. 折叠 SEO 内容、FAQ、购买须知。

`Shop Official Designs` 跳转到工具页内的官方设计产品列表；不是跳至普通水晶饰品归档页。

### 3.2 官方设计产品卡

每张卡必须直接传达“已完成的一条手链”，而不只是列出三种石头：

- 完整手链成品图。
- 名称、场景/意图、Woo 价格。
- 关键材料和配饰摘要。
- `Buy this design`：直接进入 Woo 购买路径。
- `Customize this design`：立即导入完整顺序、尺寸、配饰和包装默认值到编辑器。

导入后必须在设计区显示完整组合和当前材料明细；不能只给商品卡打勾，令用户猜测设计是否已载入。

### 3.3 意图入口

意图/场景既是低门槛入口，也是官方设计产品列表的分类体系。初始不限制为 6 个模板，而是按几十个官方设计产品建设；每个场景下配置多条人工设计的手链。MVP 不做复杂的意图推荐、五行匹配或自动选石算法。

建议首批以 30+ 个官方设计产品为目标，按 6-8 个浏览场景组织，例如 Calm & Grounding、Love & Connection、Joy & Happiness、Luck & Abundance、Protection、Confidence & Focus、New Beginnings。产品可有多个辅助标签，但只设置一个主场景，避免同一设计在列表中重复出现。

## 4. 交互基线

### 4.1 设备策略

- **移动端优先**：设计预览在上，素材库在下；商品卡两列；触控优先。
- **桌面端**：固定两栏，设计区 + 素材区；不做三栏响应式增强，不把摘要栏常驻为第三栏。
- **可视化**：仅 2D。桌面端和移动端均不提供 3D、OrbitControls 或 GLB 模型；保留现有木托盘作为设计区的核心视觉资产，珠子沿托盘内的手链轨道呈现。

### 4.2 设计区

- 手围入口合并为一个 `Wrist 16 cm` 芯片，点击进入手围设置。
- 设计区显示目标手围、已用长度、重量和适配状态；不常驻四张冗余数据卡。
- 木托盘保持完整比例和居中位置，顶部 Wrist 控件与底部操作区围绕木托盘排布，不压住托盘本体。
- 用户目标手围固定；新增材料只改变已用长度、重量和适配状态。超出适配范围时提示用户修改手围或调整材料。
- Reset 为清空当前设计；Save 为保存设计；Finish Design 为进入确认页。
- 选中单颗后提供移动、删除、翻转等上下文操作；移动端收纳到 Toolbox，桌面端可展开显示。

### 4.3 手围设置

- 主输入：手围 cm 数值和松紧偏好（Snug / Regular / Loose）。
- 辅助估算：身高、体重、骨架类型仅在“不知道手围”时出现，不作为必填。
- MVP 支持 Single wrap；Double/Triple wrap 只有在材料、计算和生产流程完整支持后才上线。

### 4.4 素材选择区

一级 Tab：

1. `Beads`
2. `Decor`：隔片、隔珠、花托、跑环、吊坠、魔盒等结构性配件
3. `Finish`：线材、包装及非结构性附加项

搜索采用放大镜入口，展开后显示搜索输入。筛选支持颜色、石种、尺寸、材质、价格、适配范围和上架状态；移动端不堆叠永久可见的复杂筛选器。

### 4.5 商品卡

每个卡片代表一个可加入设计的 **Variant**，包含：

- 透明背景图片。
- 英文商品名称。
- 尺寸，例如 `8 mm`。
- 单价。
- 已选数量。
- 必要时显示兼容提示或售罄状态。

不再使用全局 Bead Size。6mm、8mm、10mm 等是独立 Variant，可在同一条手链混用；不同尺寸在 2D 预览中必须按真实相对比例渲染。

## 5. 素材、价格和后台

### 5.1 管理边界

| 内容 | 所属系统 | 原因 |
|---|---|---|
| 珠子、结构性配件、线材、包装 | T17 WordPress 插件素材库 | 支持规格、兼容性、搜索和设计器使用，不污染普通产品目录 |
| 官方设计产品 | WooCommerce | 使用现有商品、库存、优惠、SEO、订单和售后能力 |
| 用户 DIY 成品 | Woo 隐藏承载产品 + 订单快照 | 不为每次用户设计新建 Woo 产品 |

**边界锁定：WooCommerce 不管理珠子、配件、线材、包装或各尺寸规格。** 它只管理可直接销售的官方设计产品，以及 DIY 下单时的隐藏结算承载产品。所有素材库的 CRUD、筛选、兼容关系、图片、规格和内部价格字段都由 T17 插件后台管理。

### 5.2 素材字段

每个 Material 至少包含：稳定 ID、英文名称、内部中文名、组件类型、石种/材质、主颜色与多颜色标签、功效标签、主图、状态、来源、备注。

每个 Variant 至少包含：稳定 ID、关联 Material、`size_mm`（允许小数）、形状、图片覆盖、单价、重量、占用长度、库存状态、库存数量、适配最小/最大珠径、上架状态。

`Decor` 还必须有：可插入位置、兼容规则、占用长度、是否替代珠子、生产注意事项。

### 5.3 图片标准

- 生产素材以高质量透明背景单珠/单配件图为主，保留来源、原始 URL、文件哈希、抓取日期和授权/替换状态。
- 国内小程序优先提供品类、尺寸、适配关系和移动交互参考；其缩略图不直接视为生产原图。
- MyAstris、Crystalis 是海外图片、英文名和价格基准的重点来源；Lucid 素材主要用于规格与价格结构参考，不用其低分辨率编辑器纹理作为商品主图。

### 5.4 价格体系

价格表独立维护并导出版本号。每个 Variant 记录国内零售价、海外零售价、供应成本、损耗、人工、包装、履约、建议售价、目标毛利、置信度和更新时间。

官方设计产品价格流程：

1. 根据配方和当前价格表计算建议售价。
2. 运营审核后更新 Woo 产品常规价/促销价。
3. Woo 产品记录 `price_version`；若配方或价格表变化，后台标记“需复核价格”。
4. 用户直接购买时以 Woo 当前商品价成交。
5. 用户进入 Customize 后，按当前 Variant 和附加项实时重算；确认页明确展示价格变化。

DIY 价格必须以后端计算为准。前端仅显示预估价，确认页、购物车和订单使用同一套服务端重算结果。

## 6. WooCommerce 与订单

### 6.1 官方设计产品字段

每个 Official Design Product 使用标准 Woo 产品，并增加 T17 自定义字段：

- `t17_recipe_json`：材料 Variant、顺序、配饰、线材、包装和默认手围。
- `t17_design_version`：配方版本。
- `t17_price_version`：价格表版本。
- `t17_customize_enabled`：是否允许导入编辑。
- `t17_preview_image`：设计预览图。
- `t17_recipe_status`：有效、缺货、待复核。

官方产品不需要另一份“模板表”。编辑界面通过 `product_id` 读取其配方，写入用户当前设计副本。

### 6.2 DIY 订单

使用一个隐藏 Woo 虚拟产品 `Custom Crystal Bracelet` 承接 DIY 下单。订单 line item 保存不可变快照：

- 设计预览图。
- 目标手围、松紧偏好、佩戴方式。
- Variant 明细、数量、顺序和配件位置。
- 线材、包装、附加项。
- 前端预估总价、服务端重算总价、价格表版本。
- 供生产查看的人类可读摘要。

支付完成后不新建公开 Woo 产品；生产、物流、退款和售后仍使用 WooCommerce 订单能力。

## 7. 设计保存与设计广场

### 7.1 MVP

- 匿名用户可在本地保存草稿；登录用户可保存私有设计。
- 每份设计保存完整 Variant 配方和版本；不得只保存总价或渲染图片。
- 支持从官方设计产品创建一个可编辑副本。

### 7.2 后续阶段

用户可选择公开投稿至设计广场。公开作品展示完整预览、名称、作者昵称和当前可制作状态；其他用户可点 `Use as a Starting Point` 导入配方。

导入公开作品后必须重新校验库存、兼容性、手围和当前价格。缺货组件提示替代，不静默删除；不得把历史订单金额复制为当前售价。

## 8. 技术架构

### 8.1 交付方式

从“页面内嵌 HTML/JS + Code Snippets”迁移为专用 WordPress 插件：

- Shortcode 或 Block 输出前端编辑器。
- 插件独立加载 JS/CSS 和 2D 渲染逻辑。
- WordPress 后台提供 Beads、Decor、Finish、Official Design Recipe 的管理界面。
- REST API 输出已上架素材和官方设计配方。
- 插件内部封装 Woo 加购、价格重算和订单快照。

WoodMart 仅负责站点头尾、页面外壳和主题通用能力，不拥有 T17 的业务逻辑。

### 8.2 不采用

- Three.js、Babylon.js、GLB、3D 相机与 3D 模型生成。
- 全局 Bead Size、单一珠径限制。
- 每颗珠子/配件作为普通 Woo 商品。
- 每次 DIY 创建一个公开 Woo 产品。
- 编辑器内直接 Checkout。
- MVP 阶段的公开设计广场、点赞和社交裂变。

## 9. 竞品结论如何使用

| 来源 | v3 使用方式 |
|---|---|
| 国内四个小程序 | 移动端编辑、珠子/配件分类、尺寸规格、兼容关系、模板与设计广场机制的主要一手来源 |
| MyAstris | 海外图片标准、英文资料、海外零售价、桌面信息架构参考 |
| Crystalis | 官方设计产品“双路径”、官方组合价格、透明单珠图、英文情绪文案参考 |
| Lucid Beads | 意图入口、非空白起步、基础价/Premium 加价参考；不采用空白 Freestyle 作为主入口 |
| Annflora / MoonArq | 分类、素材库与海外 DIY/购买衔接的补充参考 |

## 10. 当前实现审计与首轮重构范围

当前代码位于 `07-互动工具/crystal-bracelet-builder/`，核心事实如下：

| 当前模块 | 可保留 / 需要替换 | v3 处理 |
|---|---|---|
| `build/prototype.html` | Three.js 3D 原型 | 不继续演化；仅保留早期交互研究价值 |
| `data/parts.json` / `prices.json` | 静态数据，尺寸嵌套在珠子内 | 迁移为 Material / Variant / Decor 后台数据，不沿用全局珠径结构 |
| `woocommerce/t17-crystal-bracelet-woocommerce.php` | 服务端重算、购物车配置和订单 meta 思路可复用 | 重写为插件内 Woo 服务；配置改为 `variant_id`、顺序、配饰位置和版本快照 |
| 页面内嵌 HTML/JS 与 Code Snippets | 无后台、无 REST、主题耦合 | 替换为独立插件、Shortcode/Block、后台管理页和 REST API |

首轮开发不直接重写线上页面外观，按以下边界推进：

1. 创建可安装的 T17 插件骨架和后台素材数据模型。
2. 迁移首批素材为 Variant，保留原始 ID 到新 ID 的映射表。
3. 为 Woo 官方设计产品增加配方字段、导入接口和价格复核状态。
4. 在独立预览环境实现新的移动端 2D 编辑器，再替换线上旧工具。

这样可以让线上现有工具继续可用，同时避免新旧两套尺寸、计价和订单结构互相污染。

## 11. 分阶段实施顺序

### Phase 0 - 数据和视觉基线

1. 新建 EarthWard 价格表并确定首批 Material/Variant。
2. 建立图片暂存清单与来源记录。
3. 定义首批 30+ 个官方设计产品及其 Woo 配方、主场景、辅助标签和直接购买价格。
4. 完成移动端低保真原型，再决定视觉细节。

### Phase 1 - 插件与素材后台

1. 搭建 T17 插件骨架、素材 CRUD、媒体选择和 REST API。
2. 实现 Material / Variant / Decor 兼容性字段。
3. 为 Woo 官方设计产品增加配方编辑字段和价格复核状态。

### Phase 2 - 2D 编辑器

1. 实现手围、长度/重量/适配计算、Variant 混搭和 2D 圆环预览。
2. 实现移动端 Tab、搜索、筛选、两列商品卡和 Toolbox。
3. 实现 Save、Reset、Finish Design 和确认页。

### Phase 3 - Woo 交易闭环

1. 实现官方产品直接购买与 Customize 导入。
2. 实现 DIY 虚拟产品、后端重算、购物车/订单快照和生产摘要。
3. 在 staging 验证 WoodMart、优惠券、税费、翻译插件和移动底栏不冲突。

### Phase 4 - 内容与社区

1. 完成 Hero、官方设计产品列表、SEO 折叠内容和购买须知。
2. 上线登录用户私有设计保存。
3. 在审核、素材替代和隐私规则完整后，开放设计广场和一键借鉴。

## 12. 验收门槛

v3 进入开发前，以下事项必须明确：

- 首批 Material/Variant、配饰类别、图片来源和价格表已确认。
- 首批 30+ 个官方设计产品已定义 Woo 产品、T17 配方、主场景、标签、图片和直接购买价格。
- 官方直接购买与导入定制的价格规则已写清。
- 移动端设计区、素材区和确认页原型已审核。
- 订单快照字段与生产履约清单已审核。
