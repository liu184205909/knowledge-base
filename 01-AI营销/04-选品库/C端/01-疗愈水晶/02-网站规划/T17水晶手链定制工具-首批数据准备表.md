## 1. 本表解决什么

本表只定义首批数据进入生产前的准备、批准和回读动作。它不负责 UI、页面文案、Woo 上线操作或素材采购决策。

首批上线口径不变：至少 **32 个** Official Design Product，按 v3 基线列出的 **7 个主场景**浏览；每个设计只有一个主场景，可有多个辅助标签。32 个槽位均为 `unapproved`，不是已经存在的 Woo 商品，也不是可售配方。

## 4. Material / Variant 准备与导入

### 4.1 当前插件可导入列

`approved-production-catalog.import.csv` 的列头与现有插件导入器一致：

```text
material_key,component_type,name_en,primary_color,color_tags,intention_tags,material_image_url,material_status,variant_key,size_mm,shape,price,weight_g,occupied_length_mm,variant_image_url,stock_status,stock_quantity,compatibility,variant_status,source_name
```

一行对应一个 Variant。相同 Material 的多尺寸 Variant 重复其 Material 列是正常的；不得恢复旧数据的“一个珠子内嵌多个尺寸/全局 Bead Size”模型。

### 4.2 导入门槛

| 项目 | 完成定义 |
|---|---|
| 标识 | `material_key` 与 `variant_key` 已批准、稳定且不与现有记录冲突 |
| 类型 | `component_type` 为 `bead`、`decor` 或 `finish`；Decor 已有兼容关系和生产注意事项 |
| 规格 | `size_mm`、`shape`、`weight_g`、`occupied_length_mm` 已由生产确认 |
| 图片 | 透明背景主图/Variant 图已审核，URL、来源、原始文件哈希、采集日期和授权状态已写入素材溯源表 |
| 库存 | `stock_status` 和必要时的 `stock_quantity` 来自当前库存事实；未确认不可设为 `live` |
| 价格 | 价格审核表已有批准的 `price_version` 和单价；导入行使用该批准单价 |
| 状态 | 只有 Material 与 Variant 均完成全部门槛才可标记 `live`；其余使用 `draft` 或 `disabled` |

当前导入器会把空白数值转换为 `0`。因此生产 CSV 当前保持零数据行；任何未完成记录不能以空数值导入来“占位”。

## 5. Official Design Product 准备

每条设计必须同时满足以下四层，才可从 `unapproved` 变为 `approved`：

| 层 | 必须完成的事实 |
|---|---|
| Woo 产品 | 真实 Woo 产品 ID、SKU、英文名、常规/促销价、成品图、产品状态和 SEO 内容已由运营提供 |
| 场景分类 | 在登记表填一个 `primary_scene`，辅助标签仅用于筛选；不得因辅助标签重复进入多个主场景列表 |
| T17 配方 | `target_wrist_cm`、`fit_preference` 和按顺序排列的 `variant_id` 均已确认；每个 Variant 必须是 `live` |
| 价格与履约 | `price_version` 已批准，Woo 售价已审核，配方通过后端报价、库存校验及生产可行性复核 |

写入 Woo 的 T17 meta 为：

```text
_ew_t17_official_design=yes
_ew_t17_customize_enabled=yes
_ew_t17_primary_scene=<registry primary_scene>
_ew_t17_design_version=<approved version>
_ew_t17_price_version=<approved price version>
_ew_t17_recipe_json=<valid recipe JSON>
```

现有插件的配方 JSON 只允许 `target_wrist_cm`、`fit_preference` 和 `sequence`；`sequence` 中每个元素必须为 `{"variant_id":"<live variant key>"}`。`fit_preference` 只能是 `snug`、`regular` 或 `loose`，目标手围必须在 10-30 cm，序列长度为 1-80。模板内的 `null` 和 `replace-with-*` 均不可保存。

## 6. 遗留数据迁移规则

旧 `data/parts-source.json` 与 [商品与素材表](T17水晶手链定制工具-商品与素材表.md) 属于 v1/v2 静态、全局珠径、待补全来源。`legacy-provisional-candidates.NOT-FOR-IMPORT.csv` 仅摘录其稳定候选 ID 与名称，方便逐项重审。

下列旧值明确不迁移为生产事实：静态示例价格、纹理色、月搜索量、默认线材价格、饰件价格、`slotWeight`、3D/模型字段，以及未审核的 `pending` 状态。迁移时必须新建或确认独立 Variant、图片、重量、占位长度、库存、兼容性和批准价格版本。

公开竞品的零售价与规格只能作为研究输入，见 [竞品价格与规格基准](T17水晶手链定制工具-竞品价格与规格基准-20260713.md)。它不替代供应成本、库存、图片来源或 `price_version` 审批。

## 7. 执行与回读清单

1. 采购/生产负责人填写图片溯源和价格审核模板，指定批准人和时间。
2. 数据管理员只将批准记录复制到生产导入 CSV，先保持 `draft` 并在 staging 导入。
3. 回读插件后台与 `/wp-json/ew-t17/v1/catalog`，核对键、尺寸、图、状态、库存和价格；批准后才转 `live`。
4. 运营为每个已批准设计创建/关联 Woo 产品，填写登记表和有效配方 JSON。
5. 回读 Woo meta，验证 `Customize this design` 导入完整顺序，并验证官方直接购买与后端报价/订单快照。
6. 配方或价格版本变更时，把关联设计标为待复核；不得沿用旧成交价或静默移除缺货 Variant。

## 8. 仍需业务方提供

- 实际可生产的 Material/Variant 规格、兼容性、重量、占位长度和库存事实。
- 每个生产图片的文件、来源 URL、授权/替换状态、SHA-256 和采集日期。
- 供应商、供应商 SKU、供应成本、损耗、人工、包装、履约、建议售价、目标毛利、价格版本与审批记录。
- 32 个设计的正式英文名、真实 Woo 产品 ID/SKU、成品图、Woo 常规/促销价、SEO 内容、主场景/辅助标签、手围和完整 Variant 配方。
- 隐藏 Woo 承载商品 `Custom Crystal Bracelet` 的商品 ID，以及税费、运费、优惠券、退款和生产履约摘要的最终规则。
