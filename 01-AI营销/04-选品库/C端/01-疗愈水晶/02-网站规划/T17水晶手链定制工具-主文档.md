### 开发执行锁定（避免继续返工）

1. **一级 Tab 只保留商品类型**：水晶 / 配饰 / 吊坠 / 包装 / 手绳；"颜色"不是一级 Tab，而是水晶 Tab 内的分类视角。
2. **水晶卡不按尺寸拆分**：右侧商品卡代表水晶款式/石种；例如 Clear Quartz 下只展示 Clear Quartz (Pure)、Milky Quartz、Bridal Sparkle 等款式，不展示 6mm/8mm/10mm/12mm 四张重复卡。
3. **尺寸是左侧设计区顶部的全局控制**：6/8/10/12mm 放在左侧 `Bead Size` 胶囊内，和 `Hand Dimension` 并列；切换后统一影响所有水晶卡价格、2D/3D 珠径、轨道半径、目标槽位和总价。
4. **MVP 一串一个珠径**：暂不支持同一串混用 6mm/8mm/10mm/12mm。混合尺寸属于高级模式，会显著增加逐槽尺寸、计价、重排、履约拣货复杂度。
5. **桌面宽度锁定约 1360px**：线上工具不再追求近全屏；在宽屏上居中显示约 1360px，视口不足时退让到 `calc(100vw - 32px)`。
6. **左侧木托和珠子轨道锁定**：木托位于顶部控制和底部操作之间的视觉中心；12mm 珠子不能跑到外侧，应落在木托凹槽中间附近；珠径越小，珠子越小且轨道按比例内收。
7. **底部信息区锁定三胶囊**：Design Summary、Price Breakdown、Checkout 并列；Design Summary 字段内部 label/value 上下显示；Price Breakdown 左侧 2 行 2 列，右侧独立 Total 单列并用竖线分隔；Checkout 不放 Total。
8. **右侧 Strand Sequence 锁定**：位于 Tab 上方；左侧摘要字段内部左右显示（Beads 18 / Charms 0 这类），不包含 Size；右侧序列桌面端 4 列并保留行距，移动端 2 列。
9. **右侧商品卡要轻量精致**：水晶图和文字不能过大；卡片优先展示清晰单珠图，名称/价格/尺寸用小字号层级，避免现在这种大图大字的粗糙感。
10. **UI 细节以独立规范为准**：布局、弹窗、Tab、左侧分类、商品卡尺寸与字体详见 [T17水晶手链定制工具-UI交互规范.md](T17水晶手链定制工具-UI交互规范.md)。
11. **商品与素材表单独维护**：Crystals / Accessories / Charms / Packaging / Cord / Ready-made Presets 的图片、价格、SKU 状态和订单字段详见 [T17水晶手链定制工具-商品与素材表.md](T17水晶手链定制工具-商品与素材表.md)。

> **纠正历史错误**（原散落文档里的，本主文档已清掉）：

## 1.2 每家"我们抄什么"（具体到交互/数据结构/代码模式）

### Lucid Beads — 抄"默认珠子起步 + 3D 范式"

- **3D 范式**：Babylon.js + 每颗独立 .glb 模型 + ArcRotateCamera（球面坐标，天然 swipe 旋转 + pinch 缩放）。我们改用 Three.js（理由见 §1.3）。
- **默认珠子起步（Compose 模式）**：进 builder 默认预填 1 颗主石（如 Rose Quartz）+ 默认排列 Alternating + "Add 2nd Gemstone" 引导。**不是空白起步**（Freestyle 模式才是空白 + "Add First Bead"）。
  - 我们抄 Compose 模式：降低首次决策成本（用户面对 20 颗不知选啥时，已有 1 颗打底 + 排列引导）。
- **计价结构**：$29.99 起（含基础珠 + 弹力线）+ premium 珠按颗加价（Malachite 8mm +$0.4）+ charm 固定价 $10。基础珠不显式按颗，premium 按颗。
  - 我们调整：**所有珠子显式按颗累加**（更透明，符合"部件选购"定位），不抄"基础价含 N 颗"的扁平化。
- **3D 交互提示原文**："Swipe or drag to rotate" / "Pinch or scroll to zoom"。

### Crystalis — 抄"预设 combo 启动"

- **预设 combo**：Calm / Prosperity / Protection（一键预填珠子组合），降低空白页决策压力。
  - 我们抄：MVP 做 6 个预设 combo 作为**快捷模板**（Calm / Prosperity / Love / Protection / Focus / Confidence），用户点一键填入后仍可逐颗改。combo 降级为快捷模板按钮，**非主入口**（主入口是逐珠 builder）。
- **不抄**：expert-serve 人工 curated 模式（Curated/Premium/Concierge 服务分级）—— 我们走 self-serve 直接计价加购，不做人工服务溢价。

### Azuro Republic — 抄"多线材 + 东方 charm"

- **多线材**（与水晶正交的独立维度）：elastic / leather / bead string 三选一。
  - 我们抄：线材作为独立部件（弹力线 / 编织线 / 银线），影响总价。
- **东方 charm 库**：evil eye beads / buddha head charm / spiritual animal designs。
  - 我们抄：东方 charm（莲花 / 佛头 / Om）+ 通用 charm（心 / 十字 / 锚 / 星），无 IP。
- **加购桥模式**（Shopify 主站 + 独立 SPA 子域，配完 POST 回主站 `/pages/addtocart`）：这是 Shopify 站做高度定制 builder 的标准模式。WooCommerce 等价方案见第 2 篇 §技术方案。

### Annflora — 抄"右侧选品结构 + 拖拽交互 + 实时手围校验"

- **右侧选品结构（决定性）**——抄 Annflora 的"顶部类目/左侧分类/右侧商品卡"结构，但不照抄其尺寸拆卡：
  ```json
  {
    "categories": [
      {
        "id": "white_crystal",
        "name": "白水晶", "name_en": "Clear Quartz",
        "icon": "/assets/products/prod_052.png",
        "beads": [
          {"id":"clear_quartz_pure","name":"净体白水晶","name_en":"Clear Quartz (Pure)",
           "prices":{"6":3,"8":5,"10":8,"12":12},"thumbnail":"...","preview":"..."},
          {"id":"milky_quartz","name":"奶白水晶","name_en":"Milky Quartz",
           "prices":{"6":3,"8":5,"10":8,"12":12}}
        ]
      }
    ]
  }
  ```
  - Annflora 把同一款水晶按 6/8/10/12mm 拆成多张卡；我们不抄这个交互。
  - 我们的商品卡只代表"水晶款式/石种"，尺寸由左侧设计区顶部的全局珠径 step 控制。
  - price 仍可按 size 非线性累进，但写在同一个 bead 的 `prices` 对象里，切换全局尺寸时卡片价格、3D 珠径、目标槽位和总价一起更新。
- **拖拽交互规范（HTML 帮助页原文，最完整）**：
  - 沿手串圆环拖动珠子 → 改变位置（重排）
  - 拖空白处 → 旋转整串
  - 外拖离环 → 触发删除提示 → 松手删除（drag-to-delete）
  - 桌面端悬停珠子 → 红色 × 快捷删除
  - 自动存草稿（localStorage，关页再来继续）
- **实时手围校验**：珠径(mm) × 珠数 ≈ 手围周长(cm)；13–25cm = Good fit；< 13cm Too small；> 25cm Too big（超限阻止添加）。
- **技术栈**：Vue 3 + Vite + Three.js（PerspectiveCamera + DirectionalLight + AmbientLight + StandardMaterial + WebGLRenderer）+ SVG/canvas 拖拽 UI + 原生 PointerEvent。
  - 我们抄 3D 场景配置（光源 + 相机 + 材质），但用 Vanilla JS 而非 Vue（WP 集成最简）。

### MoonArq — 抄"尺寸独立选（与水晶种类解耦）"

- **5 步流程**：① 选石方向（color/texture/symbolism）→ ② 选尺寸（bead size + wrist fit，**独立 step**）→ ③ 复核 composition → ④ 加购 → ⑤ Shopify 结账。
- **尺寸独立**：bead size（珠径）+ wrist fit（手围）都是独立 step，与选石完全解耦。
  - 我们抄：尺寸（珠径 6/8/10/12mm 独立 step + 手围数值输入为主、XS-XL 快捷档备选），不绑死水晶种类。
- **反 kit 定位文案**："creative part of DIY without managing loose beads"（在线设计 + 工作室成品，不是 loose bead kit）。
  - 我们抄：goearthward 是工作室模式，文案同样切割"DIY kit"，强调"你设计，我们串好发货"。

## 1.3 Lucid 3D 实现细节 + 我们为何选 Three.js（非 Babylon）

**Lucid 3D 实测（决定性证据）**：
- 库：Babylon.js（`vendor/babylon.d0084c3c0e1c.js` + `compose.js` 内 195 处 BABYLON 引用）
- 模型格式：每颗独立 .glb，按 `bead.id + '.glb'` 动态加载（`SceneLoader.ImportMeshAsync('', base, bead.id + '.glb', scene)`）
- 相机：ArcRotateCamera（球面坐标，自带旋转/缩放）
- 几何：MeshBuilder.CreateCylinder / CreateLathe / CreateDecal / CreateText（charm 文字雕刻）
- 材质：shadowOnlyMaterial（地面只投影不显示）

**我们选 Three.js 共享球体 + 纹理贴图（不抄 Lucid 逐颗 .glb；珠子零建模，charm 用 AI 3D 生成 .glb）**，理由：

| 方案 | 实现成本 | 视觉 | 推荐 |
|------|---------|------|------|
| **Three.js + 共享 SphereGeometry + 纹理贴图**（珠子采用） | 低（仅需纹理图，零建模） | 中高 | ★★★★★ |
| Three.js + 每颗独立 .glb（抄 Lucid） | 极高（20+ 颗建模） | 极高 | ★★ |
| Babylon.js + 每颗 .glb（完全抄 Lucid） | 极高 + 学新库 | 极高 | ★ |

**珠子方案 A 详解（零成本，不需 ChatGPT/Meshy）**：
- 几何：1 个共享 SphereGeometry（6/8/10/12mm 共享拓扑，仅 scale 不同）
- 材质：每颗珠子 1 个 MeshStandardMaterial，加载该石种纹理贴图（**优先还原竞对单珠视觉，可用竞对参考图或站点现有水晶产品图 / crystal-profile 图沉淀为本地纹理，无需建模**）
- 光源：1 DirectionalLight（主光带阴影）+ 1 AmbientLight（去死黑）= Annflora 同款
- 相机：PerspectiveCamera（Annflora 同款真 3D）+ OrbitControls（旋转/缩放/平移）
- 渲染器：WebGLRenderer({antialias:true, alpha:true}) + ACESFilmicToneMapping
- 环串逻辑：珠子沿圆环排列，每颗 rotation 对齐圆心，Group 包裹整体旋转

**charm 方案（异形需建模，用 AI 3D 工具）**：
- 珠子是规则球体用 SphereGeometry + 纹理零成本；charm 是异形（莲花/佛头/Om 等），必须用 .glb 模型
- 生成流程：charm 设计图（image-studio 生图）→ **Meshy AI 或 Tripo3D** image-to-3D → .glb 文件（每个几毛钱，免费 credits 够 MVP 7 个 charm）
- Three.js 用 `GLTFLoader` 加载 charm .glb，挂在圆环指定槽位
- ChatGPT 本身不能生成 3D 模型，但能写 Three.js 代码（builder 主体逻辑由 ChatGPT/Claude 写）

**为何不用 Babylon**：
1. Three.js 生态更大（Azuro / Annflora 双案例背书，可参考代码多）
2. Lucid 逐颗 .glb 工程成本极高（要为每颗石种建模/贴图/导出 .glb），不适合 MVP 20 颗 + 后续扩量
3. "共享球体 + 纹理贴图"只需图（我们已有），无需建模

**体积预估**：Three.js ~150KB gzipped + 20 张纹理（每张 ~50KB webp）+ 7 个 charm .glb（每个 ~50-150KB）≈ 1.2MB；首屏只加载选中珠子，懒加载其他。

## 1.4 Annflora 数据结构（直接抄 categories[].beads[] 结构）

Annflora 用静态 JSON `/data/crystals.json` 托管珠子库（非后端 API），对我们更友好：
- 数据结构：`categories[].beads[]`，每个 bead = `{id, name, name_en, prices, thumbnail, preview}`；我们抄分类/卡片结构，不抄"同款按尺寸拆多卡"
- 无 Lucid Django API + CSRF 复杂度
- 前后端共用真源（价格表独立 `prices.json`，后端重算也读这个）
- 部署：输出到 `/wp-content/uploads/builder/parts.json`（静态托管）

## 1.5 SERP 关键结论（精简，方法论不复述）

# 第 2 篇：PRD 需求文档

## 2.1 产品定位

**逐珠 DIY 水晶手链 Builder** —— 一个 3D 可视化的 per-bead 配置器：用户选珠子（按颗）+ 选配饰/吊坠 + 选包装 + 选手绳，实时累加计价，直接加购下单。

- **本质**：WooCommerce 里 bead / charm / cord 三类"产品部件"的配置器，水晶版的 charm builder。
- **不是**：水晶知识图谱 / 推荐引擎 / quiz。**无算法匹配**（用户主动筛选 OK，算法推兼容不做）。
- **角色**：T17 工具线的**转化收口**（引流工具：塔罗/玄学/水晶 quiz → T17 定制 → Shop）。
- **品牌化**：学 Kendra Color Bar®，给工具起名（如 "Earthward Composer"）。

## 2.2 目标用户

- **意图驱动型**：想要特定功效（love / protection / calm / wealth），按意图选石。
- **审美驱动型**：按颜色 / 石种外观选，追求个性化。
- **礼物型**：生辰石 / 星座石定制送礼。
- **痛点**：成品手链固定搭配不够个性化；loose bead kit 要自己穿珠麻烦（我们"工作室代做"切割此痛点）。

## 2.3 核心功能清单（8 条）

### F1. 3D 手串可视化
- **描述**：Three.js 实时渲染手串圆环（珠子用共享 SphereGeometry + 纹理贴图，charm 用 .glb 模型，沿圆环等距排布），支持旋转 / 缩放 / 拖拽重排。
- **抄哪家**：Lucid（3D 范式）+ Annflora（Three.js 场景配置：PerspectiveCamera + DirectionalLight + AmbientLight + StandardMaterial + WebGLRenderer）。
- **验收标准**：
  - 底部信息区显示 Design Summary、Price Breakdown、Checkout；Total 位于 Price Breakdown 胶囊右侧独立单列
  - 首屏 3D 场景 ≤ 2s 加载完成（Three.js + 20 张纹理懒加载）
  - 单指拖拽旋转 / 双指 pinch 缩放（移动端 OrbitControls 原生支持）
  - 沿圆环拖动珠子可换位（Annflora 重排交互）
  - 外拖珠子离环触发删除提示（Annflora drag-to-delete）
  - 桌面端悬停珠子显示红色 × 快捷删除
  - 默认预填 1 颗主石（Rose Quartz）+ Alternating 排列起步（Lucid Compose 模式，非空白）

### F2. 珠子选择器
- **描述**：20 颗 MVP 珠子库，支持按品种 / 颜色筛选 + 搜索；右侧商品卡只代表水晶款式/石种，不按 6/8/10/12mm 拆成重复卡；尺寸独立选（6/8/10/12mm，与石种解耦）。
- **抄哪家**：Annflora（数据结构 + 分类目录 + 搜索）+ MoonArq（尺寸独立 step，不绑石种）+ Lucid（多维筛选 select2）。
- **验收标准**：
  - 水晶 Tab 左侧默认用**品种分类**（20 个 MVP 石种/款式组），可切换到**颜色分类**（7 色）；不做五行属性
  - 珠子面板按分类浏览（20 颗，附单珠图 + 中文名 + 英文名 + 当前珠径单价 + 月搜索量 + 含义 hover）
  - 搜索可叠加当前分类；意图筛选降为后续增强，不作为 MVP 左侧固定入口
  - 尺寸（珠径）作为左侧设计区顶部的独立全局配置，切换尺寸时所有珠子统一切换（不绑死石种，不在商品卡内拆 4 个尺寸）
  - 点珠卡 +1 颗到圆环（可重复加，可逐颗删）
  - 实时显示"还差 X 颗填满" / "已超 Y 颗"（基于手围算目标珠数）

### Tab 与左侧分类规则（按商品类型裁剪）

> 不再固定套"五行属性 / 颜色分类 / 品种分类"三件套。五行属性从 MVP 删除；颜色分类只服务水晶，不服务所有 Tab。

| 一级 Tab | 左侧分类 | MVP 分类 |
|---------|---------|---------|
| 水晶 | 默认品种分类，可切换颜色分类 | 品种分类：20 个 MVP 石种/款式组；颜色分类：7 色 |
| 配饰 | 只保留品种分类 | 全部 / 隔片 / 花类 / 生肖 / 装饰 / 花托 / 随型水晶 / 双尖水晶 |
| 吊坠 | 当前只显示"全部"占位 | 全部；后续吊坠 SKU 多了再扩展东方 / 符号 / 星月 / 心形 |
| 包装 | 当前只显示"全部"占位，或不显示左侧分类 | 全部；基础包装 / 礼盒包装 / 高级礼盒 |
| 手绳 | 当前只显示"全部"占位 | 全部；后续手绳 SKU 多了再扩展弹力线 / 编织线 / 银线 |

**实现要求**：切换一级 Tab 时，左侧分类列表必须随 Tab 替换，不保留上一个 Tab 的筛选状态。例如从水晶切到配饰后，不应继续显示"颜色分类/白水晶/紫水晶"。

### 右侧商品卡 UI 约束

- 水晶商品卡属于 UI 视觉设计重点，不只是数据渲染；当前实现里的水晶图偏粗糙、图片偏大、字体偏大，需要重做。
- 单珠图优先使用清晰、居中、带透明/浅色背景的圆珠图；不能用模糊、低分辨率、过度拉伸的图。
- 桌面端右侧卡片以紧凑网格为主，单卡图片建议约 52-64px，不要超过卡片高度的一半；文字层级要轻，名称不超过 2 行，价格/尺寸用更小字号。
- 商品名、尺寸、价格不能挤成大号正文；默认视觉应接近竞对的精致商品面板，而不是后台列表。
- 水晶卡只显示当前全局珠径下的价格和尺寸，不重复展示 6/8/10/12mm 多张卡。

### F3. 吊坠库
- **描述**：无 IP 吊坠库（莲花 / 佛头 / Om / 心 / 十字 / 锚 / 星），每个固定单价，占据圆环 1 槽位。
- **抄哪家**：Azuro（东方 charm：evil eye / buddha head）+ Djuna（symbolic charm 叠加）。
- **3D 资产生成方式**（异形需建模）：
  - charm 设计图（image-studio 生图）→ **Meshy AI 或 Tripo3D** image-to-3D → 输出 `.glb` 文件（每个几毛钱，免费 credits 够 MVP 7 个）
  - Three.js 用 `GLTFLoader` 加载 charm .glb，挂到圆环槽位
  - ChatGPT 不能生成 3D 模型，但写 Three.js 代码（builder 主体逻辑）
  - 与珠子的区别：珠子是规则球体用 SphereGeometry + 纹理（零建模零成本）；charm 是异形必须 .glb
- **验收标准**：
  - 吊坠 Tab 左侧当前只显示**全部**占位；后续吊坠 SKU 多了再扩展东方 / 符号 / 星月 / 心形
  - MVP ≥ 7 个 charm（全无 IP，HelloKitty 弃用因 Sanrio 版权）
  - 东方 charm（莲花 / 佛头 / Om）+ 通用 charm（心 / 十字 / 锚 / 星）
  - 每个 charm 有 3D .glb 模型 + 缩略图 + 名 + 单价（$5-15 按复杂度）
  - 点 charm 卡 +1 个到圆环（0-5 个可选）
  - charm 在圆环上视觉占 1 槽位（比珠子略大）

### F4. 手绳选择
- **描述**：3 种手绳/线材（弹力线 / 编织线 / 银线），3 选 1，影响总价。
- **抄哪家**：Azuro（elastic / leather / bead string 多线材）。
- **验收标准**：
  - 手绳 Tab 左侧当前只显示**全部**占位；后续手绳 SKU 多了再扩展弹力线 / 编织线 / 银线
  - 弹力线 $0 / 编织线 +$3 / 银线 +$8（MVP 定价，可调）
  - 切换手绳实时更新总价 + 圆环视觉（线材颜色/质感变化）
  - 选 1 种，不可多选

### F4A. 包装选择（可选加购）
- **描述**：包装是可选加购项，不进入 3D 圆环，不占槽位，只影响总价和订单履约。
- **抄哪家**：用户提供的 MineStone 参考（包装 Tab）。
- **验收标准**：
  - 包装 Tab 左侧当前只显示**全部**占位，或不显示左侧分类
  - MVP 包装：基础包装 $0 / 礼盒包装 +$6 / 高级礼盒 +$15（价格可调）
  - 选 0 或 1 种；切换包装实时更新总价
  - 订单 meta 显示包装名称，便于履约

### F5. 实时计价
- **描述**：总价 = Σ(珠子 × 单价) + Σ(配饰/吊坠) + 包装 + 手绳，按颗累加，Standard/Premium 分层；前端实时算 + 后端重算防篡改。
- **抄哪家**：Lucid / Annflora（前端实时 + 后端重算）。
- **验收标准**：
  - Price 面板实时累加（每加/删珠子、配饰、吊坠、切换包装/手绳即更新）
  - 价格分层清晰显示（Standard 珠 $X/颗 / Premium 珠 $Y/颗 / 配饰 $Z / 吊坠 $Z / 包装 $W / 手绳 $W = 总价）
  - 后端 `woocommerce_before_calculate_totals` hook 按 slug 查部件单价表重算，覆盖前端价（防篡改）
  - 整串参考价区间显示（如 "$40-70 典型定制"）

### F6. 预设 Combo（快捷模板，非核心流程）
- **描述**：5-6 个快捷模板（Calm / Prosperity / Love / Protection / Focus / Confidence），点击一键填入珠子，用户可任意改。**降级为快捷模板按钮**，主入口是逐珠 builder（Lucid Compose 默认预填珠子起步模式）。
- **抄哪家**：Crystalis（Calm / Prosperity / Protection 预设启动）作为快捷模板。
- **数据来源**：简单意图-石映射表（**手工定义**，非算法匹配，非 390 库兼容引擎）。
- **验收标准**：
  - **6 个快捷模板**（基于 20 颗库内石种手工映射）：
    - Calm：Amethyst + Clear Quartz + Sodalite
    - Prosperity：Citrine + Tiger Eye + Pyrite
    - Love：Rose Quartz + Moonstone
    - Protection：Obsidian + Bloodstone
    - Focus：Fluorite + Sodalite + Clear Quartz
    - Confidence：Carnelian + Tiger Eye + Pyrite
  - 点 combo → 一键填入对应珠子（基于手围算 N，平均分配）
  - 填入后用户可逐颗增删改（预设不锁定）
  - combo 是"快捷模板"按钮，**非主入口**；主入口是逐珠 builder

### F7. 加购（WooCommerce 虚拟产品）
- **描述**：配置序列化存 cart_item_data，加到单一虚拟产品 "Custom Crystal Bracelet"，后端重算价；订单 meta 完整存配置供履约。
- **抄哪家**：Azuro（Shopify line item properties 模式 → WooCommerce cart_item_data 等价）。
- **验收标准**：
  - 只需要一个隐藏承载产品 "Custom Crystal Bracelet"（price=$0 占位，type=simple，不在 Shop 显示）；**不为每个用户定制结果新建 Woo 产品**
  - 加购时传 cart_item_data（含完整配置 JSON + 摘要）
  - 付款后订单必须拿到：预览图 URL、手围、珠径、完整 sequence、每种珠子数量、配饰/吊坠、包装、手绳、前端总价、后端重算总价、用户可读摘要
  - 4 个 hook 全覆盖：`woocommerce_get_item_data`（购物车显示）/ `woocommerce_checkout_create_order_line_item`（订单 meta）/ 邮件模板 hook / 后台订单详情 hook
  - 后端重算价（F5）
  - 预览图：3D 场景 `toDataURL()` → 上传 media library → meta 存 URL（不存 base64，避免数据库膨胀）
  - 加购后跳 Woo `/cart`

### F8. 尺寸选择（首次弹窗 + 左侧顶部全局控制）
- **描述**：首次进入 builder 弹出手围设置弹窗；用户可通过身高/体重估算推荐手围，也可直接选常用手围。进入后，**手围数值输入框**（cm/inch 切换）+ 滑块微调作为主输入；**珠径独立全局 step**（6/8/10/12mm，与手围/石种解耦，学 MoonArq）；尺寸控制放在左侧设计区顶部，和 Hand Dimension 并列；右侧商品卡只响应当前全局尺寸，不承载尺寸入口。
- **抄哪家**：MoonArq（珠径独立 step，与手围解耦）+ Annflora（实时手围校验 13-25cm）；自定义数值输入 UX（用户提供的图设计）。
- **验收标准**：
  - 首次进入显示"定制手围"弹窗；确认后关闭并写入 localStorage
  - 弹窗包含身高、体重、推荐周长（可修改）、常用尺寸快捷卡；桌面端做居中 modal，移动端做底部/居中弹窗
  - 用户关闭弹窗但未确认时，使用默认 M 码 16.5cm + 8mm，并在 3D 状态条提示可修改
  - **主输入：手围数值输入框**（cm/inch 单位切换）+ 滑块微调（13-25cm 范围）
  - **测量教学图示**：软尺 / 纸条量手腕骨的线稿图（教学用户正确测量手围）
  - **珠径独立全局 step**：6/8/10/12mm 独立选择（与手围解耦，不绑石种），MVP 默认 8mm；MVP 一串手链只允许一个全局珠径
  - **混合尺寸不进 MVP**：不支持同一串里同时使用 6mm/8mm/10mm；该能力会引入逐槽尺寸、拖拽换位、价格、目标槽位、履约拣货的复杂度，后续如做"高级模式"再评估
  - 系统按手围 + 珠径自动算目标珠数 N = floor(手腕周长mm / 珠径mm)（如 16cm 手围 × 8mm ≈ 20 颗）
  - **保留 XS-XL 快捷档**（XS 13cm / S 15cm / M 16.5cm / L 18cm / XL 20cm）作为"懒得量"用户备选（非主输入，次级按钮）
  - 超 max_beads 禁止添加（Annflora "Too big" 逻辑）
  - 不足时提示"还需 X 颗填满"

## 2.6 定价规则

### 按颗累加公式
```
总价 = Σ(bead.prices[当前珠径] × qty)  +  Σ(charm.price × qty)  +  cord.price
```

### Standard / Premium 分层（20 颗）
| 层级 | 数量 | 包含 | 单价（8mm 参考） |
|------|------|------|-----------------|
| **Standard** | 17 颗 | 大部分主流石种（Volume≥5000 适珠，Amethyst/Rose Quartz/Citrine/Tiger Eye 等） | $2.0/颗 |
| **Premium** | 3 颗 | Malachite（Lucid 列为 premium，拉高客单价）+ Jade + Labradorite（视觉稀缺溢价） | $3.5/颗 |

### 整串参考价（M 码 ~22 珠 8mm）
| 配方 | 计算 | 售价 |
|------|------|------|
| 全 Standard（如 Amethyst×22） | 22 × $2.0 + 弹力线 $0 | **$44** |
| Standard + 4 Premium（18 Amethyst + 4 Malachite） | 18×$2 + 4×$3.5 + 编织线 $3 | **$53** |
| 全 Premium（Malachite×22） | 22 × $3.5 + 银线 $8 | **$85** |
| + 1 charm（莲花） | + $8 | 任一配方 +$8 |

> **定价心理学**：按颗累加（非整体一口价）是 charm builder 核心心理机制 —— 每加一颗看价格涨 $2，用户主动控制预算，消费体验像"购物"而非"报价"。客单价 $44-85，比成品手链 $32-48 高 35-90% 且无"被报价"抗拒感。

## 2.7 技术方案（WordPress + WoodMart + WooCommerce 部署红线）

> **部署红线**：T17 **必须在 WordPress + WoodMart + WooCommerce 内搭建**，不能搞成独立 Shopify/Next.js 站。5 竞品无一用 WooCommerce，必须自建"3D 配置 → Woo 订单"的桥。

### 可视化（珠子零建模 + charm AI 3D）
- **Three.js（r160+）** + OrbitControls + MeshStandardMaterial + 共享 SphereGeometry + 每颗纹理贴图（珠子，零成本）+ GLTFLoader 加载 charm .glb（Meshy/Tripo3D 生成）
- 光源：1 DirectionalLight + 1 AmbientLight（Annflora 同款）
- 相机：PerspectiveCamera + OrbitControls（enableDamping + min/maxDistance + enablePan=false）
- 渲染器：WebGLRenderer({antialias:true, alpha:true}) + ACESFilmicToneMapping

### 前端框架
- **Vanilla JS + ES Modules**（**不引入 Vue/React**，Annflora 用 Vue 3 是因独立 SPA，我们嵌 WP 选最简）+ Three.js
- **部署陷阱（必遵 memory `wp-html-block-js-base64`）**：复杂 JS（含 &&/比较）+ 大 JSON 必须用 Base64 包装规避 wp_kses_post 实体转义
  - 可执行 JS：`<script type="text/plain" id="t17-app-b64">` + 微型 loader `atob→eval`
  - 部件 JSON：`<script type="application/json">` + asciiJSON（非 ASCII 转 \uXXXX）
  - loader 须在数据块之后
  - 复用站点 18 个工具的 generate.js 模式（参考 crystal-tarot-draw 等现有工具部署）

### WordPress 部署（工具页用 wp:html 块）
- 工具页用 `<!-- wp:html -->` 块嵌 Three.js + 应用 JS（base64 包装）
- 部件数据静态 JSON 嵌 `<script type="application/json">`（asciiJSON 转义非 ASCII）
- WoodMart 主题兼容（参考 crystal-tarot-draw 等 18 个现有工具部署模式）
- 部署到 `/tools/crystal-bracelet-builder/`（tools 父页 43101 的子页），用 `_shared/scripts/create-page.js`

### WooCommerce 加购（单一虚拟产品 + cart_item_data + 后端重算）
1. 新建虚拟产品 "Custom Crystal Bracelet"（price=$0 占位，type=simple，不在 Shop 显示）
2. 配置序列化（存 cart_item_data）：
   ```js
   {
     bead_size: 8, wrist: 16.5, cord: 'elastic_black',
     beads: [{slug:'rose_quartz', qty:8}, {slug:'amethyst', qty:4}, {slug:'malachite', qty:2}],
     charms: [{slug:'lotus', qty:1}],
     sequence: ['rose_quartz','rose_quartz',...,'lotus',...],
     totalPrice: 53.0,  // 后端必重算
     previewImg: 'media_library_url'
   }
   ```
3. 4 个 hook 全覆盖（cart 显示 / 订单 meta / 邮件 / 后台订单详情）
4. **后端 PHP 重算总价**（`woocommerce_before_calculate_totals`）：按 slug 查部件单价表（`prices.json` 真源），重算覆盖 line item 价格。**防前端篡改的关键**。
5. 预览图：Three.js `renderer.domElement.toDataURL()` → 上传 media library → meta 存 URL

> **风险**：WoodMart 主题可能有自己的价格计算逻辑，需在 staging 验证 hook 优先级（不冲突）。

### 部件库
- **静态 JSON 文件**（抄 Annflora `/data/crystals.json`，categories[].beads[] 结构），托管在 `/wp-content/uploads/builder/parts.json`
- 价格表独立 `prices.json`（前后端共用真源，后端重算也读这个）
- 三表结构（beads / charms / cords），字段见 §2.5

## 2.11 开发前锁定与 Bug 风险清单（必须先处理）

> 结论：主方向不用推翻，但开发前必须补齐以下执行约束。否则最容易在 WordPress 嵌入、WooCommerce 加购、尺寸算法、预览图上传四处返工。

### R1. Base64 + ES Modules 不能直接 `eval import`

- 文档写了 Vanilla JS + ES Modules + `atob→eval`，但普通 `eval()` 里不能写静态 `import`。
- **开发锁定**：Three.js 采用以下二选一：
  - A. 用 `<script type="module">` loader 动态 `import()` Three.js / OrbitControls / GLTFLoader，再启动 base64 解码后的 app 工厂函数；
  - B. loader 解码后创建 `Blob` URL，并插入 `<script type="module" src="blob:...">`。
- **验收**：WP 页面发布后浏览器控制台不得出现 `Cannot use import statement outside a module`。

### R2. 预览图上传不能假设匿名用户可写 Media Library

- 前端 `renderer.domElement.toDataURL()` 后直接上传 media library 在匿名访客场景通常不可行；WP Media REST 需要权限，自建公开上传接口又有滥用风险。
- **MVP 锁定**：加购/订单履约以 `sequence + parts summary + total breakdown` 为真源；预览图上传降级为可选增强。
- **若要做预览图上传**：必须走自定义受限 endpoint，至少包含 nonce、文件大小上限、MIME 校验、频率限制、订单/购物车绑定；失败时不阻断加购。
- **验收**：关闭预览图上传时，购物车、订单、邮件、后台仍能完整展示配置摘要并可履约。

### R3. 尺寸算法要从“手围”改成“目标内周长”

- 当前 `N = floor(手腕周长mm / 珠径mm)` 会偏紧，也和“XS-XL / M 码约 22 颗 8mm”的参考价不完全一致。
- **开发锁定**：
  - 用户输入仍是手围；
  - 系统内部先算目标内周长：`targetCm = wristCm + fitAllowanceCm`；
  - MVP 默认 `fitAllowanceCm = 1.5`，后续可加 Snug / Comfort / Loose；
  - 目标槽位：`targetSlots = round(targetCm * 10 / beadSizeMm)`；
  - charm 占槽位，但可配置 `slotWeight`，MVP 先按 1 槽处理。
- **验收**：16cm 手围、8mm 珠径默认约 22 槽；切换珠径后目标槽位实时重算。

### R4. `sequence` 必须成为后端唯一计价输入

- 前端同时传 `beads[]`、`charms[]`、`sequence[]`、`totalPrice` 容易出现数量不一致或被篡改。
- **开发锁定**：后端只信任 `sequence[] + bead_size + cord`；`beads[]` / `charms[]` / `totalPrice` 仅用于前端显示，后端重算时忽略。
- `sequence` item 结构固定为：
  ```json
  {"type":"bead","id":"rose_quartz"}
  ```
  或：
  ```json
  {"type":"charm","id":"lotus","slotWeight":1}
  ```
- **验收**：手动篡改 `totalPrice` 或 `beads[]` 数量不会影响 Woo line item 最终价格。

### R5. `parts.json` 与 `prices.json` 不能双写漂移

- 文档现在同时提 `parts.json` 和独立 `prices.json`，如果人工维护两份，很容易价格不同步。
- **开发锁定**：维护一个源文件 `parts-source.json`；构建脚本生成：
  - 前端用 `parts.json`；
  - 后端重算用 `prices.json`；
  - 生成时做 schema 校验和价格完整性校验。
- **验收**：20 颗珠子 × 4 个尺寸、7 个 charm、3 个 cord 都有价格；任一缺价构建失败。

### R6. 3D 拖拽重排与 OrbitControls 会抢事件

- 移动端单指拖拽既可能旋转场景，也可能拖珠子换位；不定义事件仲裁会出现“想转动却拿起珠子”。
- **开发锁定**：
  - 默认单指/鼠标拖动空白区域 = 旋转；
  - 点中珠子后短按选中，进入 item 操作态；
  - 操作态内拖动才重排/外拖删除；
  - 移动端必须有显式删除按钮作为 drag-to-delete 兜底。
- **验收**：桌面和手机都能稳定完成旋转、缩放、选中、删除，不因拖拽误删。

### R7. charm `.glb` 不能阻塞主流程

- 7 个 charm 3D 模型依赖 Meshy/Tripo3D，资产未齐时会卡住整个开发。
- **开发锁定**：MVP 先支持 charm fallback：无 `.glb` 时用简单金属圆牌/图标材质占位；资产到位后自动切到 `GLTFLoader`。
- **验收**：缺少任一 charm `.glb` 时页面不报错，仍可选择、计价、加购。

### R8. WooCommerce PHP 承载位置要先定

- 普通 `wp:html` 工具页只能承载前端；加购重算、订单 meta、邮件/后台展示必须有 PHP hook。
- **开发锁定**：PHP 部分独立成 `t17-crystal-bracelet-woocommerce.php`，部署到 Code Snippets 或 child theme，和前端 HTML 分开版本管理。
- **验收**：至少覆盖 `woocommerce_add_cart_item_data`、`woocommerce_before_calculate_totals`、`woocommerce_get_item_data`、`woocommerce_checkout_create_order_line_item`。

### R9. localStorage 草稿必须带版本号

- 部件库、价格、字段结构更新后，旧草稿可能恢复出不存在的 id 或旧价格。
- **开发锁定**：草稿结构包含 `schemaVersion`、`partsVersion`、`savedAt`；版本不一致时只恢复可识别 item，丢弃失效项并提示用户。
- **验收**：删除或改名某个 bead/charm 后，旧草稿不会让页面崩溃。

### R10. 开发顺序建议

1. 先做本地静态 MVP：parts schema、尺寸槽位、sequence、计价、3D 圆环、fallback charm。
2. 再做 WordPress `wp:html` 嵌入：Base64/module loader、移动端适配、localStorage。
3. 再做 WooCommerce PHP：虚拟产品、cart item data、后端重算、订单 meta。
4. 最后接预览图上传和 `.glb` charm 资产，不阻断前 3 步。
