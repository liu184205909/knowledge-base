# 1B 数据处理工具手册

> RLM 步骤 1B（多源数据采集与处理）的工具操作手册 | 最后更新: 2026-05-26

---

## RLM 流程对应

```
步骤1A: 竞品发现（人工）→ 竞品清单（Google Sheets）
    │
步骤1B: 多源数据采集（多条轨道可并行）
    ├── 轨道A: 竞品总表数据 → SEMrush AS/流量/截图 + traffic.cv 月访问量/渠道拆分
    ├── 轨道B: 竞品SEO明细 → SEMrush Top Pages + Top Keywords + 关键词排名明细
    │     └── 轨道B产出后：Competitor-Sheet-Map 配置 + Topic-Discovery 主题发现
    ├── 轨道C: 种子词关键词主表 → SEMrush/Ahrefs Seed Keywords 导入与清洗
    └── 轨道D: Sitemap解析 → sitemap-mcp-server(MCP) → 竞品页面报告
```

## 轨道A：竞品总表数据

```bash
# 前置：Tabbit 带 --remote-debugging-port=9222 --remote-allow-origins=* 启动，并登录 semrush.com

python semrush_to_sheets.py              # 全量查询
python semrush_to_sheets.py --start 4    # 从第4行开始
python semrush_to_sheets.py --limit 5    # 只查前5个
python semrush_to_sheets.py --dry-run    # 只提取不写回 Sheets
```

**功能：** 从 Google Sheets「竞品总表」读取域名 → SEMrush Domain Overview 提取对应字段 → 写回竞品总表对应列。traffic.cv 渠道数据也填入同一张竞品总表。

| 写入字段 | 列号 | 说明 |
|---------|------|------|
| AS | C | Authority Score（0-100） |
| Organic Traffic | D | 自然搜索流量 |
| SEMrush 截图 | E | Distribution 区域截图 |

### traffic.cv（手动）

浏览器访问 `https://traffic.cv/{domain}`，获取月访问量和流量渠道拆分。

| 字段 | 列号 | 格式 |
|------|------|------|
| 月访问量 | F | 如 233K、2.71M |
| 搜索% | G | 如 53.80% |
| 直接% | H | 如 14.63% |
| 社交% | I | 如 16.26% |
| 引荐% | J | 如 0.17% |
| 邮件% | K | 如 1.75% |
| 付费% | L | 如 0.17% |

## 轨道B：竞品SEO明细数据

Top Pages、Top Keywords 等大表数据各写入单独的电子表格，不塞进竞品总表。

采集规则：

- 在 `SEMrush-Top-Pages` 表格中，每个竞争对手创建一个独立工作表。
- 在 `SEMrush-Top-Keywords` 表格中，每个竞争对手创建一个独立工作表。
- 工作表数量由竞品数量决定，不能写死。
- 1B 只负责原始数据采集和标准列精简；`TopPages_All`、`TopKeywords_All`、`Keyword-Page-Proof` 的生成、0流量过滤和 URL 关联方法见 `01-AI营销/00-营销方法论基础/01-关键词研究方法论.md`。

```python
EXCLUDED_SHEETS = {'README', '字段说明', 'TopPages_All', 'TopKeywords_All', 'Keyword-Page-Proof'}
```

> **注意**：此黑名单仅用于旧版脚本兼容。新版本脚本应改为读取 `Competitor-Sheet-Map`（位于 SEMrush-Top-Keywords Google Sheet）中 `include_in_top_keywords = TRUE` 或 `include_in_top_pages = TRUE` 的工作表。不要用 EXCLUDED_SHEETS 黑名单，因为输出表、空白表、临时表、说明表都不会出现在 Competitor-Sheet-Map 中，天然不会被读入。

| 电子表格 | 来源模块 | 用途 | 标准列（保留） | Semrush 导出时删除的列 |
|------|---------|------|--------------|----------------------|
| `SEMrush-Top-Pages` | Organic Research / Pages | 判断哪些页面真正承担 SEO 获客 | URL / Traffic(%) / Traffic / Top Keyword / Primary Intent / LLM Prompts | Adwords Positions / Number of Keywords / 5×Positions by Intent / 5×Traffic by Intent / Traffic Change / Answer Engines |
| `SEMrush-Top-Keywords` | Organic Research / Positions | 记录竞品排名关键词与排名URL | Keyword / Search Volume / Keyword Difficulty / CPC / URL / Traffic / Traffic(%) / Number of Results / Keyword Intents | Position / Competition / Previous Position / Traffic Cost / Trends / Timestamp / SERP Features by Keyword / Position Type |

> **新增工作表时**：往 `SEMrush-Top-Pages` 或 `SEMrush-Top-Keywords` 新增竞品工作表时，Semrush 导出的原始数据必须按上方"标准列"精简后再导入，删除无用列。

### 数据瘦身规则（防止表格膨胀）

Google Sheets 单个电子表格上限 **10,000,000 格**。SEMrush 导出的竞品数据中 95%+ 的行 Traffic=0，直接导入会导致格数迅速爆满、无法新增数据。

**适用范围**：`SEMrush-Top-Keywords` 和 `SEMrush-Top-Pages` 中以竞品域名命名的工作表（如 `tinyrituals.co`、`crystalvaults`）。汇总表（`TopKeywords_All`、`TopPages_All`、`Keyword-Page-Proof`）不适用此规则。

**执行时机**：每导入一个新竞品工作表后立即执行。

**新增竞品工作表处理流程**：

当用户在 `SEMrush-Top-Keywords` 或 `SEMrush-Top-Pages` 中新增竞品工作表时，由 Claude 执行以下标准化处理：

1. **数据瘦身** — 按下方清理规则删除零流量行
2. **调整网格** — resize grid 至实际数据行列数，释放多余格数
3. **冻结首行** — 冻结第 1 行（表头），便于滚动查看
4. **补录 Competitor-Sheet-Map** — 在 SEMrush-Top-Keywords 的 `Competitor-Sheet-Map` 工作表末尾新增一行：
   - `source_sheet_name`：工作表名称
   - `competitor_domain`：从工作表名称推导域名
   - `competitor_type`：默认 `Direct`
   - `primary_topic_pillar` / `secondary_topic_pillar`：留空，用户审核时填写
   - `business_relevance`：留空，用户审核时填写
   - `include_in_top_keywords` / `include_in_top_pages`：根据所在表格自动设为 TRUE
   - `status`：默认 `Review`（用户审核后改为 Active）
   - `notes`：留空

> 用户只需：导入数据 → 告诉 Claude 新增了哪些工作表 → Claude 完成上述 4 步 → 用户最后审核 Sheet-Map 的 status 和 topic_pillar。

**清理规则**：

| 表格 | 保留列 | 删除行条件 |
|------|--------|-----------|
| `SEMrush-Top-Keywords` 竞品工作表 | Keyword / Search Volume / Keyword Difficulty / CPC / URL / Traffic / Traffic(%) / **Number of Results** / Keyword Intents | Traffic=0 **或** Traffic(%)=0/空 |
| `SEMrush-Top-Pages` 竞品工作表 | URL / Traffic(%) / Traffic / Top Keyword / Primary Intent / LLM Prompts | Traffic=0 **或** Traffic(%)=0/空 |

**执行后**：调整工作表网格尺寸（Grid Properties）至实际数据行列数，释放多余格数。

> **后续处理**：1C 关键词研究会遍历这些竞品工作表，生成过滤后的 All 证据索引和竞品验证视图。All 表不是原始数据备份，低信号行保留在竞品原始工作表中，不提升到 All 表；默认页面阈值为 Traffic >=100，Top Keywords 排除单英文词关键词；为控制规模，每个竞品最多保留300个 Top Pages 和2000个 Top Keywords；Proof 视图只保留 Seed Keywords 命中证据和非主表 Strong/Medium 增量机会。1B 不在本轨道展开该处理逻辑。

### 维护脚本

| 脚本 | 用途 | 运行时机 |
|------|------|---------|
| `generate_all_tables.py --top-pages` | 生成/刷新 TopPages_All | 新增竞品 Top Pages 工作表后 |
| `generate_all_tables.py --top-keywords` | 生成/刷新 TopKeywords_All | 新增竞品 Top Keywords 工作表后 |
| `generate_topic_discovery.py` | 生成/刷新 Topic-Discovery | TopKeywords_All 或 Competitor-Sheet-Map 刷新后 |
| `generate_keyword_page_proof.py` | 生成/刷新 Keyword-Page-Proof | TopKeywords_All 或 TopPages_All 刷新后 |
| `backfill_seed_keywords.py` | 回填 Seed-Master 竞品验证 | Keyword-Page-Proof 刷新后 |

## 轨道C：种子词关键词主表

用产品词、类目词、场景词、用户问题等种子词在 Semrush Keyword Magic / Keyword Overview 或 Ahrefs 中导出关键词列表，写入 `SEMrush-Seed-Keywords` 电子表格。该表是关键词库/关键词主表的承接表，也是步骤1中最重要的关键词数据源；进入 1C 清洗后就是项目关键词主表。

| 电子表格 | 来源模块 | 用途 | 标准列（保留） | Semrush 导出时删除的列 |
|------|---------|------|--------------|----------------------|
| `SEMrush-Seed-Keywords` | Keyword Magic / Keyword Overview 导出 | 承接种子词扩展导入，清洗后作为关键词主表 | Keyword / Volume / KD / CPC / Intent / Topic Cluster / Recommended Page Type / Priority / Source Type / Source Detail / Competitor Proof / Proof URL | Competitive Density / SERP Number |

> 日常内容选题读取清洗后的 `SEMrush-Seed-Keywords` 主表；判断某个词是否值得做时，再参考 `Keyword-Page-Proof` 的竞品验证结果。竞品验证中发现的增量词，补回该主表并标记来源。

### 竞品配置与主题发现

**Competitor-Sheet-Map**（位于 SEMrush-Top-Keywords Google Sheet）是竞品工作表的运行配置。它同时服务于 Top Keywords 和 Top Pages 两个表格。

字段：`source_sheet_name` / `competitor_domain` / `competitor_type` / `primary_topic_pillar` / `secondary_topic_pillar` / `business_relevance` / `include_in_top_keywords` / `include_in_top_pages` / `top_pages_sheet_name` / `status` / `notes`

- `include_in_top_keywords/pages` 是技术读取开关，不是业务筛选字段
- `business_relevance` 是读入后的业务标签
- `status`（Active / Review / Exclude）控制这条竞品配置是否已审核

**Topic-Discovery** 由独立脚本 `generate_topic_discovery.py` 生成。读取 TopKeywords_All + Competitor-Sheet-Map，输出 Topic-Discovery 工作表。

Topic-Discovery 字段：`topic_pillar` / `source_sheet_count` / `keyword_count` / `proof_candidate_count` / `top_competitors` / `example_keywords` / `suggested_action` / `confirmed_action` / `notes`

脚本只读取 `confirmed_action`，未确认的主题不进入后续流程。

### Seed-Master 架构

`Seed-Master`（位于 SEMrush-Seed-Keywords Google Sheet）是项目唯一关键词主表。其他 Seed-* 工作表是导入暂存表。

Seed-Master 19 列标准：Keyword / Normalized Keyword / Volume / KD / CPC / Intent / Topic Pillar / Subtopic / Business Relevance / Recommended Page Type / Priority / Status / Source Type / Source Detail / Competitor Proof / Proof URL / Proof Traffic / Proof Competitor / Notes

Status 生命周期：Candidate → Approved → In Content Plan / Skip

### 数据流顺序

```
竞品原始工作表
  → TopKeywords_All / TopPages_All（通过 Competitor-Sheet-Map 白名单读取）
  → Topic-Discovery（人工确认主题）
  → 确认主题后进入 Seed-Master
  → Keyword-Page-Proof（竞品验证）
  → 回填 Seed-Master 最佳证据
```

**关键原则**：All 层负责发现全貌，Seed-Master 负责决策，Proof 负责验证。不要让 Proof 承担主题发现。

## 轨道D：Sitemap解析

**数据来源**：竞品 URL 列表从**项目 Google Sheets 竞对清单导出**，不使用本目录下的静态 `competitor_urls.txt`（该文件仅为格式示例）。

**结果存放**：Sitemap 解析报告存放在**项目文件夹 `01-竞品分析/1B-sitemap解析/`**（如 `1B-sitemap解析报告.md`），不存放在工具目录。只有 URL 清单很大、需要频繁筛选/排序时，才可另建 `Sitemap-Pages` 电子表格。

**产出内容**：每个成功竞品的页面清单、URL 路径模式、内容类型分布（博客/产品/分类/静态页面）。

### 工具

| 工具 | 类型 | 状态 |
|------|------|------|
| **sitemap-mcp-server** (MCP) | Sitemap 快速解析 → 提取竞品博客/产品/页面URL清单 | **已可用** |

---

### sitemap-mcp-server

> 来源：[mugoosse/sitemap-mcp-server](https://github.com/mugoosse/sitemap-mcp-server)（MIT 开源）
> 底层库：ultimate-sitemap-parser（自动通过 robots.txt 发现 sitemap）

**为什么选 sitemap-mcp-server**：
- 核心需求是**解析竞品 sitemap → 提取博客/产品等页面 URL → 了解竞品内容策略**
- 直接解析 sitemap.xml，秒级返回结果
- 支持按子sitemap过滤（`sitemap_url`参数）和按路径过滤（`route`参数）
- 返回精确页面数量和完整 URL 列表（含 lastmod 时间戳）

**已安装路径**：

| 项目 | 路径 |
|------|------|
| venv Python | `D:\Tools\sitemap-mcp-server\.venv\Scripts\python.exe` |
| MCP 配置 | `~/.claude.json` 的 `mcpServers.sitemap` |

> **注意**：sitemap-mcp-server 0.1.3 与 mcp>=1.25 不兼容（`FastMCP.__init__` 移除了 `version` 参数），已在安装目录 `server.py` 第156行修补（移除 `version=` 和 `description=` 两个参数）。**勿用 uvx 重建环境**，否则修补会丢失。

**MCP 配置内容**（已写入 `~/.claude.json` → `mcpServers.sitemap`）：
```json
"sitemap": {
  "type": "stdio",
  "command": "D:\\Tools\\sitemap-mcp-server\\.venv\\Scripts\\python.exe",
  "args": ["-m", "sitemap_mcp_server"],
  "env": { "TRANSPORT": "stdio" }
}
```

**可用工具**：

| 工具 | 功能 | 典型用法 |
|------|------|---------|
| `get_sitemap_tree` | 获取完整 sitemap 树结构 | 了解竞品有哪些子sitemap |
| `get_sitemap_pages` | 获取页面列表（支持过滤/分页） | 用 `sitemap_url` 指定子sitemap，用 `route` 过滤路径 |
| `get_sitemap_stats` | 获取 sitemap 统计信息 | 快速了解竞品页面总量、子sitemap 数 |
| `parse_sitemap_content` | 直接解析 sitemap XML 内容 | 已有 sitemap 文本时直接解析 |

**典型工作流**：

```
1. get_sitemap_stats(url="https://competitor.com")                    → 快速获取页面总量
2. get_sitemap_tree(url="https://competitor.com")                     → 查看子sitemap结构
3. get_sitemap_pages(url="https://competitor.com",                    → 指定blog子sitemap
     sitemap_url=".../sitemap_blogs_1.xml",
     include_metadata=true, limit=0)
4. 如有分页：用返回的 nextCursor 继续拉取
5. 将 URL 列表按主题分类（水晶含义/天使号码/星座/脉轮/颜色/功效等）
6. 汇总写入竞品分析报告
```
