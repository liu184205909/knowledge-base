# 1B 数据处理工具手册

> RLM 步骤 1B（多源数据采集与处理）的工具操作手册 | 最后更新: 2026-06-02

---

## RLM 流程对应

```
步骤1A: 竞品发现（人工）→ 竞品清单（Google Sheets）
    │
步骤1B: 多源数据采集（多条轨道可并行）
    ├── 轨道A: 竞品总表数据 → SEMrush AS/流量/截图 + traffic.cv 月访问量/渠道拆分
    ├── 轨道B: Seed-* 主题关键词池 → 导入、清洗并构建 Seed-Master v1
    ├── 轨道C: 竞品SEO明细 → SEMrush Top Pages + Top Keywords + All 表
    ├── 轨道D: Seed-Master 竞品证据增强 → TopKeywords_All + TopPages_All 补证据
    └── 轨道E: Sitemap解析 → sitemap-mcp-server(MCP) → 竞品页面报告
```

## 轨道A：竞品总表数据

```bash
# 前置：Tabbit 带 --remote-debugging-port=9222 --remote-allow-origins=* 启动，并登录 semrush.com

python semrush_to_sheets.py              # 全量查询
python semrush_to_sheets.py --start 4    # 从第4行开始
python semrush_to_sheets.py --limit 5    # 只查前5个
python semrush_to_sheets.py --dry-run    # 只提取不写回 Sheets
```

从 Google Sheets「竞品总表」读取域名 → SEMrush Domain Overview 提取+截图 → 写回竞品总表。traffic.cv 手动填同一张表。

| 字段 | 列号 | 来源 |
|------|------|------|
| AS | C | SEMrush Authority Score（0-100） |
| Organic Traffic | D | SEMrush 自然搜索流量 |
| SEMrush 截图 | E | SEMrush Distribution 区域截图 |
| 月访问量 | F | traffic.cv，如 233K、2.71M |
| 搜索% / 直接% / 社交% / 引荐% / 邮件% / 付费% | G–L | traffic.cv 渠道拆分 |

所有目标竞品均完成后视为结束；新增竞品随时回来补充。

## 轨道B：Seed-* 主题关键词池清洗与 Seed-Master v1 构建

所有 `Seed-*` 工作表存放在 `SEMrush-Seed-Keywords` 电子表格中，每个 `Seed-*` 代表一个 Topic Pillar（如 `Seed-Crystals`、`Seed-Chakra`、`Seed-Feng-Shui`），后续新增主题时继续使用 `Seed-{Topic}` 命名。

**Seed-* 标准字段**：Keyword / 中文 / Entity / Subtopic / Content Role / Volume / KD / CPC / Number of Results / Intent。后三列（Entity / Subtopic / Content Role）Semrush 默认不提供，由 AI 辅助生成并人工抽查。

`Subtopic` 字段名通用，但分类值不通用。每个 `Seed-*` 维护自己的 Subtopic 词表（如 `Seed-Crystals`: Meaning / Benefits / Properties），不得跨主题硬套。

清洗完成后的 `Seed-*` 应默认可进入 `Seed-Master`；`Delete` / `Review` 行不得进入。未清洗完的不导入 `Seed-Master`。

### 清洗流水线（B-1）

**规则：先标记，人工确认后再物理删除；只把 Keep 行导入 Seed-Master。**

字段补全优先使用本地脚本（`update_seed_structure.py`、`translate_seed_keywords.py`）；高级聚类和 SERP overlap 工具只用于抽查验证，不是必跑步骤。

#### 步骤 1：结构性噪音（不依赖语义，纯规则可判断）

| 类型 | 标记 | 说明 |
|------|------|------|
| 平台词 / 社媒词 | Delete | 明确平台黑名单：Etsy、Amazon、TikTok、Pinterest 等 |
| 纯乱码 / 碎片词 | Delete | 无实际含义或明显拼写残缺 |
| 本地意图词 | Keep/Review | `near me` 等不误删，放到内容策略判断 |
| Volume 为空 / = 0 | Review | 空值≠零搜索量；=0 可标记但不自动删 |

#### 步骤 2：补语义字段

| 字段 | 方法 |
|------|------|
| `中文` | `translate_seed_keywords.py` 批量字面翻译 |
| `Entity` | `update_seed_structure.py` 按主题规则批量填；AI 辅助 + 人工抽查 |
| `Subtopic` | 人工/AI 按各 Seed-* 专属词表判断，不得跨主题硬套 |
| `Content Role` | `update_seed_structure.py` 批量填初步承接建议（Guide / Main Article / Separate Article / Product Page / Local SEO / Topic Keyword） |
| `Intent` | 保留 Semrush 原始 Intent；缺失或明显异常时再用 AI 辅助补充 |

#### 步骤 3：语义筛选（依赖 Entity / Intent / Content Role 判断）

| 类型 | 标记 | 说明 |
|------|------|------|
| 第三方品牌词 | Delete/Review | 需语义识别品牌名与普通词重合（如含"crystal"的品牌名） |
| 实体错位词 | Delete/Review | 表面含主题词但实际指向餐厅/地名/影视实体（需 Entity） |
| 近重复 | 保留一个 | 大小写/词序/单复数差异；保留 Volume 更高更短的变体；Intent 明显不同则同时保留 |
| 超长尾低量 | Review | 词数≥5 且 Volume<200（需 Content Role 判断性价比） |
| 高 KD 低量 | Review | KD>80 且 Volume<500（同上） |
| 业务无关 | Review/Delete | 需 Entity + Content Role 判断 |
| 集群支撑词 | Keep | 需 Subtopic 判断主题完整性 |

#### 步骤 4：AI 标记 → 人工确认

将判断写入审核辅助列（B-2）：`Suggested Action`（Keep / Delete / Review）、`Reason`、`Confidence`（High / Medium / Low）、`Reviewed`。

人工确认 `Delete` 行后物理删除；`Review` 行必须先改为 Keep 或 Delete 再处理。`Seed-*` 保留辅助列用于复盘，但 `Seed-Master` 只导入核心字段和 Keep 行。

### 审核辅助列（B-2）

脚手架列，不导入 Seed-Master：

| 字段 | 含义 |
|------|------|
| Suggested Action | Keep / Delete / Review |
| Reason | 建议原因 |
| Confidence | High / Medium / Low |
| Reviewed | 人工复核状态 |

每个 `Seed-*` 应开启筛选、冻结首行；`Suggested Action` 和 `Reviewed` 建议用下拉菜单。

### Seed-Master v1

**导入字段**：Keyword / 中文 / Topic Pillar（工作表名继承） / Entity / Subtopic / Content Role / Volume / KD / CPC / Number of Results / Intent

**准入规则**：只导入已清洗 Seed-* 中 Keep 行；可阶段性构建。

**跨主题去重**：按 normalized keyword 去重；跨主题重复时保留一条主记录并标记 Topic 冲突，由人工确认 Topic Pillar 归属，不自动按 Volume 决定。

### 维护脚本

| 脚本 | 用途 |
|------|------|
| `translate_seed_keywords.py` | 通用，批量翻译所有 Seed-* 中文列 |
| `update_seed_structure.py --topic <name>` | 按 `--topic`（chakra / feng-shui / angel-numbers / palmistry）填 Entity / Content Role；`--all` 处理所有已注册主题 |
| `build_seed_master_v1.py` | 通用，合并已清洗 Seed-* 为 Seed-Master v1 |
| `audit_seed_sheets.py` | 检查所有 Seed-* 字段完整性 |

新增 Seed-* 主题时，在 `update_seed_structure.py` 的 `TOPICS` 字典中添加 sheet_name、sheet_id 和正则规则即可。脚本位于 `02-竞品研究工具/scripts/`，认证按 [02-Google-Cloud凭证创建指南.md](./../../../00-基础能力/02-Google-Cloud凭证创建指南.md) 执行。

## 轨道C：竞品SEO明细数据

来自 SEMrush Top Keywords 和 Top Pages。每个竞品在 `SEMrush-Top-Keywords` / `SEMrush-Top-Pages` 中有独立原始工作表，由 `generate_all_tables.py` 汇总生成 `_All` 表。

**字段名统一**：SEMrush 原始导出的 `Keyword Intents` / `Primary Intent`，在 `_All` 表和 `Seed-Master` 中统一为 `Intent`。

### 字段与处理总表

| 对象 | 保留字段 | 派生字段 | 删除行条件 | 可重建 |
|------|---------|---------|-----------|--------|
| TopKeywords 竞品原始表 | Keyword / Search Volume / Keyword Difficulty / CPC / URL / Traffic / Traffic(%) / Number of Results / Keyword Intents | — | Traffic=0 或 Traffic(%)=0/空 | 否 |
| TopKeywords_All | 同原始表（→ Intent） | normalized_url / source_sheet | 同原始表 + Traffic≥1 且≥2词 | 是 |
| TopPages 竞品原始表 | URL / Traffic(%) / Traffic / Top Keyword / Primary Intent / LLM Prompts | — | Traffic(%)=0/空 | 否 |
| TopPages_All | 同原始表（→ Intent） | normalized_url / source_sheet | 同原始表 + Traffic≥100 | 是 |

- 派生字段只在 `_All` 表生成，不回写原始竞品工作表。
- 每个 `_All` 表按流量降序排列，TopPages 最多 300 行，TopKeywords 最多 2000 行。

新增竞品：导入数据 → 告诉 Claude → 执行数据瘦身 + 调整网格 + 冻结首行。

### 维护脚本

| 脚本 | 用途 |
|------|------|
| `generate_all_tables.py --both` | 生成/刷新 TopKeywords_All、TopPages_All |

脚本只读取已确认的竞品原始工作表（排除 `_All`、`Competitor-Sheet-Map`、`Topic-Discovery`、`Keyword-Page-Proof`、`README`、`字段说明` 及所有临时/说明表）；临时表、说明表、派生表不得进入读取范围。新增非竞品工作表必须加入排除清单。

## 轨道D：Seed-Master 竞品证据增强

用 `TopKeywords_All` 和 `TopPages_All` 给 `Seed-Master v1` 补充竞品证据。只在 `Seed-Master` 有实际数据后执行。

### D2：TopKeywords_All → Seed-Master

匹配键：标准化后的 Keyword（小写、去首尾空格、合并连续空格）。不做词形/同义词归并。命中多行取 Traffic 最高的。

| 未命中率 | 动作 |
|----------|------|
| < 20% | 仅记录，不回补 |
| 20%–40% 且集中 1-2 主题 | 回轨道B 补充/新建 Seed-*，重建 Seed-Master 后重跑轨道D |
| > 40% 或高度分散 | 人工筛选候选视图再决定 |

### D3：TopPages_All → Seed-Master

对 D2 已命中的 URL，用 `normalized_url` 去 `TopPages_All` 补页面级证据。找不到则保留 D2 关键词证据，页面字段留空。

`normalized_url` 规则：全小写 → 去协议 → 去 www → 删查询参数 → 删锚点 → 去尾斜杠（`https://WWW.Example.com/Crystals/?x=1` → `example.com/crystals`）。

### D4：回填字段

| 字段 | 来源 |
|------|------|
| Competitor Keyword Matched | TopKeywords_All（Yes/No） |
| Competitor Match Count / Source Count | 命中明细行数 / 来源竞品数 |
| Best Competitor Source / URL / Keyword Traffic / Traffic% | 最佳关键词证据 |
| Best Page Traffic / Traffic% / Top Keyword / Intent | 最佳页面证据 |
| Evidence Updated At | 脚本运行时间 |

**不覆盖**：Keyword、中文、Topic Pillar、Entity、Subtopic、Content Role、Volume、KD、CPC、Intent 及后续人工添加的研究字段。

## 轨道E：Sitemap解析

**数据来源**：项目 Google Sheets 竞对清单读取域名，逐个调用 sitemap-mcp-server。

**结果存放**：项目文件夹 `01-竞品分析/1B-sitemap解析/`。

**工具**：[sitemap-mcp-server](./sitemap-mcp-server工具说明.md)。

```
1. get_sitemap_stats(url="https://competitor.com")        → 页面总量
2. get_sitemap_tree(url="https://competitor.com")         → 子sitemap结构
3. get_sitemap_pages(url="https://competitor.com",         → 指定子sitemap，支持分页
     sitemap_url=".../sitemap_blogs_1.xml", limit=0)
4. 将 URL 按主题分类 → 汇总写入竞品分析报告
```
