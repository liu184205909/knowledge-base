# 1B 数据处理工具手册

> RLM 步骤 1B（多源数据采集与处理）的工具操作手册 | 最后更新: 2026-05-26

---

## RLM 流程对应

```
步骤1A: 竞品发现（人工）→ 竞品清单（Google Sheets）
    │
步骤1B: 多源数据采集（多条轨道可并行）
    ├── 轨道A: 竞品总表数据 → SEMrush AS/流量/截图 + traffic.cv 月访问量/渠道拆分
    ├── 轨道B: Seed-* 主题关键词池 → 导入、清洗并构建 Seed-Master v1
    ├── 轨道C: 竞品SEO明细 → SEMrush Top Pages + Top Keywords + All 表
    ├── 轨道D: Seed-Master 竞品证据增强 → TopKeywords_All + TopPages_All 补证据
    └── 轨道E: Sitemap解析 → sitemap-mcp-server(MCP) → 竞品页面报告
```

## 轨道A：竞品总表数据

```bash
# 前置：Tabbit 带 --remote-debugging-port=9222 --remote-allow-origins=* 启动，并登录 semrush.com

python semrush_to_sheets.py              # 全量查询
python semrush_to_sheets.py --start 4    # 从第4行开始
python semrush_to_sheets.py --limit 5    # 只查前5个
python semrush_to_sheets.py --dry-run    # 只提取不写回 Sheets
```

**功能：** 从 Google Sheets「竞品总表」读取域名 → SEMrush Domain Overview 提取对应字段 → 写回竞品总表对应列。traffic.cv 渠道数据也填入同一张竞品总表。

| 写入字段 | 列号 | 说明 |
|---------|------|------|
| AS | C | Authority Score（0-100） |
| Organic Traffic | D | 自然搜索流量 |
| SEMrush 截图 | E | Distribution 区域截图 |

### traffic.cv（手动）

浏览器访问 `https://traffic.cv/{domain}`，获取月访问量和流量渠道拆分。

| 字段 | 列号 | 格式 |
|------|------|------|
| 月访问量 | F | 如 233K、2.71M |
| 搜索% | G | 如 53.80% |
| 直接% | H | 如 14.63% |
| 社交% | I | 如 16.26% |
| 引荐% | J | 如 0.17% |
| 邮件% | K | 如 1.75% |
| 付费% | L | 如 0.17% |

**结束条件**：轨道A 在竞品总表中所有目标竞品均完成 SEMrush 数据采集和 traffic.cv 手动填写后视为完成，即可进入后续轨道。如果新增竞品，随时回来补充。

## 轨道B：Seed-* 主题关键词池清洗与 Seed-Master v1 构建

`Seed-*` 工作表承接的是按主题从 Semrush Keyword Magic / Keyword Overview 或 Ahrefs 大范围导出的相关关键词池，不是已经筛好的最终关键词库。每个 `Seed-*` 工作表代表一个 Topic Pillar，例如 `Seed-Crystals`、`Seed-Chakra`、`Seed-Dreams`、`Seed-Angel-Numbers`、`Seed-Tarot`、`Seed-Astrology`、`Seed-Zodiac`、`Seed-Numerology`、`Seed-Feng-Shui`。后续新增主题时继续使用 `Seed-{Topic}` 命名。

空的 `Seed-*` 工作表也应保留。它们是后续手动导入对应主题关键词的占位主题池，不得因为暂无数据而删除。

`Seed-*` 内先完成 AI + 人工清洗：删除不相关词、无效词、重复词和明显低价值词。清洗流程为 **AI 标记（B-2 审核辅助列）→ 人工确认 → 删除无效词 → 得到干净的 Seed-\***。清洗完成后的 `Seed-*` 工作表应默认 100% 可进入后续关键词研究，不把无效词留在正式 `Seed-*` 中再依赖状态字段过滤。

| 电子表格 | 来源模块 | 用途 | Seed-* 标准字段 |
|------|---------|------|--------------|
| `SEMrush-Seed-Keywords` | Keyword Magic / Keyword Overview / Ahrefs 导出 | `Seed-*` 承接主题关键词池导入与清洗，并构建 `Seed-Master v1` | Keyword / 中文 / Entity / Subtopic / Content Role / Volume / KD / CPC / Number of Results / Intent |

`Number of Results` 必须保留，不得在 Seed-* 清洗时无故删除。`中文` 是关键词字面翻译列，用于人工快速理解左侧关键词；`Entity` 是关键词明确指向的对象/水晶/主题实体；`Subtopic` 是主题内部内容角度；`Content Role` 是该关键词在内容体系中的承接角色。后三列 Semrush 默认不提供，可由 AI 辅助生成并由人工抽查，不作为进入下一步的硬门槛。

`Subtopic` 的字段名通用，但分类值不通用。每个 `Seed-*` 工作表可以维护自己的 Subtopic 词表，例如 `Seed-Crystals` 可使用 Meaning / Benefits / Properties，而 `Seed-Chakra` 可使用 Chakra Basics / Specific Chakra / Chakra Healing & Practice；不得把某一主题的分类体系硬套到其他主题。

### 轨道B-1：Seed-* 清洗流水线

每个有数据的 `Seed-*` 工作表，在进入 `Seed-Master v1` 前必须先清成“小而准”的主题关键词池。清洗可以分批执行，但必须遵守：**先标记，人工确认后再物理删除；只把 Keep 行导入 Seed-Master**。

#### 阶段 1：硬噪音标记

先用规则把明显无效词标记为 `Delete`，默认不直接物理删除。

| 类型 | 标记 | 说明 |
|------|------|------|
| 平台词 / 社媒词 | Delete | Etsy、Amazon、Alibaba、Temu、TikTok、Instagram、Pinterest、YouTube 等 |
| 第三方品牌词 | Delete | C 端电商项目默认不做竞品品牌词和 `Brand A vs Brand B` 内容，除非负责人明确批准 |
| 纯乱码 / 碎片词 | Delete | 无实际英文含义、明显拼写残缺且无法形成搜索意图 |
| 实体错位词 | Delete/Review | 关键词表面含主题词，但实际指向餐厅、地名、品牌、影视/娱乐实体等 |
| 本地意图词 | Keep/Review | `near me`、`local shop` 等不要误删，承接方式放到内容策略判断 |
| Volume 为空 | Review | 空值可能是数据缺失，不等于零搜索量，不直接 Delete |
| Volume = 0 | Delete/Review | 可标记低价值，但若有主题集群或转化价值，人工可保留 |

#### 阶段 2：补可读化与语义字段

- `中文`：只做关键词字面翻译，不写删除建议、不写解释。
- `Entity`：关键词指向的核心实体，例如 Amethyst、Root Chakra、Angel Number 333；泛主题词可填默认实体。
- `Subtopic`：主题内部内容角度。字段名通用，但分类值由各 `Seed-*` 自己定义。
- `Content Role`：初步承接建议，例如 Guide Index / Hub、Main Article、Separate Article Candidate、Product / Category Page、Local SEO Candidate、Topic Keyword。

`Content Role` 只是初步建议，不等于最终页面决策。是否独立成文，还要结合搜索量、竞品页面、SERP 意图和内链结构判断。

#### 阶段 3：近重复处理

识别大小写、词序、单复数、冠词介词差异造成的近重复词。默认保留搜索量更高、更短、更通用的变体；如果两个词 Intent 明显不同，可同时保留。

#### 阶段 4：质量筛选

| 条件 | 标记 | 说明 |
|------|------|------|
| 超长尾低量 | Review | 如词数 >= 5 且 Volume < 200，先复核，不自动删 |
| 高 KD 低量 | Review | 如 KD > 80 且 Volume < 500，性价比低但不直接删 |
| 业务无关 | Review/Delete | 完全无产品、内容或品牌价值时删除；不确定先 Review |
| 内容集群支撑词 | Keep/Review | 即使搜索量低，也可能支撑 E-E-A-T 或主题完整性 |

#### 阶段 5：AI 标记审核辅助列

将判断写入 `Suggested Action / Reason / Confidence / Reviewed`：

- `Keep`：已通过筛选，可进入 `Seed-Master v1`
- `Delete`：候选删除词，必须人工确认后才能删除
- `Review`：需要人工判断，不能自动删除

#### 阶段 6：人工确认与准入

- 筛选 `Delete`：人工确认后再物理删除。
- 筛选 `Review`：人工改为 Keep 或删除。
- 清洗完成后，`Seed-*` 可以保留审核辅助列用于复盘，但 `Seed-Master` 只导入核心字段和 Keep 行。
- 如果一个 `Seed-*` 仍需要大量状态过滤，说明它还没有完成清洗，不能合并进 `Seed-Master v1`。

高级聚类或 SERP-overlap 工具可用于抽查验证，但不是 `Seed-*` 清洗的必需步骤；默认以 `update_seed_*_structure.py` 的主题专属规则 + 人工复核为主。

### 轨道B-2：Seed-* 审核辅助列

`Seed-*` 工作表可以在清洗阶段临时或长期保留审核辅助列，用于人工筛选和复核。不要为每个 `Seed-*` 单独创建 `Review` 工作表；`Seed-*` 本身就是该主题的导入、清洗和最终暂存工作表。

建议审核辅助列：

| 字段 | 含义 | 是否进入 Seed-Master |
|------|------|----------------------|
| Suggested Action | AI 对该关键词的处理建议：Keep / Delete / Review | 否 |
| Reason | 给出建议的原因，例如 Local SEO opportunity、Brand/platform keyword、Idiomatic noise | 否 |
| Confidence | AI 判断置信度：High / Medium / Low | 否 |
| Reviewed | 人工复核状态，例如 Reviewed / Need Follow-up | 否 |

使用规则：

- `Keep`：默认保留，清洗完成后可进入 `Seed-Master v1`。
- `Delete`：候选删除词，必须人工确认后才能从 `Seed-*` 删除。
- `Review`：需要人工判断，不应自动删除。
- `中文` 只做翻译：例如 `chakra chakra chakra` 写为 `脉轮 脉轮 脉轮`，不能写成 `建议删除：...`，也不能写成 `待复核：chakr`。
- 清洗判断只写入 `Suggested Action / Reason / Confidence / Reviewed`，不要再额外新增中文解释列，避免正式 `Seed-*` 表膨胀。
- `Reason`、`Confidence`、`Reviewed` 是清洗阶段脚手架，保留在 `Seed-*` 中方便后续复盘，但不导入 `Seed-Master`。
- 每个 `Seed-*` 应开启筛选并冻结首行；`Suggested Action` 和 `Reviewed` 建议使用下拉菜单，方便人工过滤和确认。

Seed-Master v1 初始字段：

| 字段 | 来源 |
|------|------|
| Keyword | Seed-* |
| 中文 | Seed-* |
| Topic Pillar | 由 Seed-* 工作表名继承 |
| Entity | Seed-* |
| Subtopic | Seed-* |
| Content Role | Seed-* |
| Volume | Seed-* |
| KD | Seed-* |
| CPC | Seed-* |
| Number of Results | Seed-* |
| Intent | Seed-* |

Seed-Master v1 合并规则：

- 已清洗完成的 `Seed-*` 才能进入 `Seed-Master v1`；当前可先合并 `Seed-Crystals` 与 `Seed-Chakra` 作为阶段性主表底座。
- 导入字段为：Keyword / 中文 / Topic Pillar / Entity / Subtopic / Content Role / Volume / KD / CPC / Number of Results / Intent。
- 不导入清洗辅助列：Suggested Action / Reason / Confidence / Reviewed。
- 只导入 `Suggested Action = Keep` 或空状态但已确认可保留的行；`Delete / Review` 不进入 Seed-Master。
- 按 normalized keyword 去重；同一关键词跨主题重复时，优先保留搜索量更高或人工确认主意图更明确的记录。
- `Seed-Master v1` 可以阶段性构建，并在项目简报中标明当前覆盖的 Topic Pillar；不必等待所有 `Seed-*` 全部清洗完成。

### 当前表格整理规则

- 空的 `Seed-*` 工作表保留。它们是后续手动导入主题关键词的占位主题池，不列入删除清单。
- `Seed-Master` 若为空壳，应按轨道B规则重建，不直接沿用旧表头或旧字段。
- `Seed-*` 字段名应标准化：`Keyword Difficulty` → `KD`、`CPC (USD)` → `CPC`、带尾随空格的工作表名应去空格。

## 轨道C：竞品SEO明细数据

轨道C数据来自 SEMrush Top Keywords 和 Top Pages。它们不是关键词主表，而是竞品已经拿到流量的关键词证据和页面证据。

采集规则：

- 在 `SEMrush-Top-Pages` 表格中，每个竞争对手创建一个独立工作表。
- 在 `SEMrush-Top-Keywords` 表格中，每个竞争对手创建一个独立工作表。
- 工作表数量由竞品数量决定，不能写死。
- 1B 负责原始工作表清洗，并生成 `TopKeywords_All`、`TopPages_All` 这两个竞品SEO明细汇总表。

| 电子表格 | 来源模块 | 用途 | 标准列（保留） | Semrush 导出时删除的列 |
|------|---------|------|--------------|----------------------|
| `SEMrush-Top-Pages` | Organic Research / Pages | 判断哪些页面真正承担 SEO 获客 | URL / Traffic(%) / Traffic / Top Keyword / Intent / LLM Prompts | Adwords Positions / Number of Keywords / 5×Positions by Intent / 5×Traffic by Intent / Traffic Change / Answer Engines |
| `SEMrush-Top-Keywords` | Organic Research / Positions | 记录竞品排名关键词与排名URL | Keyword / Search Volume / Keyword Difficulty / CPC / URL / Traffic / Traffic(%) / Number of Results / Intent | Position / Competition / Previous Position / Traffic Cost / Trends / Timestamp / SERP Features by Keyword / Position Type |

**字段标准化映射：** SEMrush 原始导出中的意图字段名因导出模块不同而异，在文档和 `_All` 汇总表中统一为 `Intent`。

| SEMrush 原始字段名 | 标准字段名 | 出现在 |
|------|------|------|
| `Keyword Intents` | `Intent` | Top Keywords 原始导出 |
| `Primary Intent` | `Intent` | Top Pages 原始导出 |

> 原始竞品工作表保留 SEMrush 导出时的原始字段名；汇总表（`TopKeywords_All`、`TopPages_All`）和 `Seed-Master` 统一使用 `Intent`。

### 轨道C-1：逐个竞品原始工作表清洗

先处理每个竞品的原始工作表，不能直接汇总。

Top Keywords 原始工作表保留字段：

| 字段 | 说明 |
|------|------|
| Keyword | 竞品拿到流量的关键词 |
| Search Volume | 搜索量 |
| Keyword Difficulty | 关键词难度 |
| CPC | 点击价格 |
| URL | 该关键词对应的竞品排名页面 |
| Traffic | 该关键词带来的估算流量 |
| Traffic(%) | 该关键词流量占比 |
| Number of Results | 结果数，必须保留 |
| Keyword Intents | 关键词意图（原始字段名，汇总表统一为 Intent） |

Top Pages 原始工作表保留字段：

| 字段 | 说明 |
|------|------|
| URL | 竞品拿到流量的页面 |
| Traffic(%) | 页面流量占比 |
| Traffic | 页面估算流量 |
| Top Keyword | 页面最主要关键词 |
| Primary Intent | 页面主要意图（原始字段名，汇总表统一为 Intent） |
| LLM Prompts | Semrush 导出的提示字段 |

清洗规则：

- 删除无流量行和明显无效行。
- 保留原始必要字段，不在原始工作表中新增 `competitor_domain`、`normalized_url` 等分析字段。
- 缩小网格，冻结首行，避免 Google Sheets 格数膨胀。

### 数据瘦身规则（防止表格膨胀）

Google Sheets 单个电子表格上限 **10,000,000 格**。SEMrush 导出的竞品数据中 95%+ 的行 Traffic=0，直接导入会导致格数迅速爆满、无法新增数据。

**适用范围**：`SEMrush-Top-Keywords` 和 `SEMrush-Top-Pages` 中以竞品域名命名的工作表（如 `tinyrituals.co`、`crystalvaults`）。

**执行时机**：每导入一个新竞品工作表后立即执行。

**新增竞品工作表处理流程**：

当用户在 `SEMrush-Top-Keywords` 或 `SEMrush-Top-Pages` 中新增竞品工作表时，由 Claude 执行以下标准化处理：

1. **数据瘦身** — 按下方清理规则删除零流量行
2. **调整网格** — resize grid 至实际数据行列数，释放多余格数
3. **冻结首行** — 冻结第 1 行（表头），便于滚动查看

> 用户只需：导入数据 → 告诉 Claude 新增了哪些工作表 → Claude 完成上述 3 步。竞品工作表后续如何汇总和筛选，待 1C 关键词研究流程重新梳理后再定。

**清理规则**：

| 表格 | 保留列 | 删除行条件 |
|------|--------|-----------|
| `SEMrush-Top-Keywords` 竞品工作表 | Keyword / Search Volume / Keyword Difficulty / CPC / URL / Traffic / Traffic(%) / **Number of Results** / Keyword Intents（汇总表统一为 Intent） | Traffic=0 **或** Traffic(%)=0/空 |
| `SEMrush-Top-Pages` 竞品工作表 | URL / Traffic(%) / Traffic / Top Keyword / Primary Intent（汇总表统一为 Intent） / LLM Prompts | Traffic(%)=0/空 |

**执行后**：调整工作表网格尺寸（Grid Properties）至实际数据行列数，释放多余格数。

### 轨道C-2：生成 TopKeywords_All

从清洗后的 Top Keywords 原始工作表汇总生成 `TopKeywords_All`，存放在 `SEMrush-Top-Keywords` 电子表格内。它是派生汇总工作表，可删除重建，不是原始数据。

`TopKeywords_All` 代表竞品拿到流量的关键词集合。每一行表示：

```
某个竞品的某个关键词，通过某个 URL 拿到了流量
```

建议字段：

| 字段 | 说明 |
|------|------|
| Keyword | 竞品关键词 |
| Search Volume | 搜索量 |
| Keyword Difficulty | 关键词难度 |
| CPC | 点击价格 |
| URL | 排名页面 |
| Traffic | 关键词流量 |
| Traffic(%) | 关键词流量占比 |
| Number of Results | 结果数，必须保留 |
| Intent | 关键词意图（原始导出为 Keyword Intents，汇总表统一为 Intent） |
| normalized_url | 如后续需要与 TopPages_All 关联，可在汇总表中生成 |
| source_sheet | 可选溯源字段，用于排错和人工追踪 |

`competitor_domain` 暂不单独作为字段；如 `normalized_url` 保留完整域名和路径，已经足够判断来源站点。

### 轨道C-3：生成 TopPages_All

从清洗后的 Top Pages 原始工作表汇总生成 `TopPages_All`，存放在 `SEMrush-Top-Pages` 电子表格内。它是派生汇总工作表，可删除重建，不是原始数据。

`TopPages_All` 代表竞品拿到流量的页面集合。它不直接作为关键词主表，也不直接按关键词匹配 Seed-Master。

建议字段：

| 字段 | 说明 |
|------|------|
| URL | 竞品页面 |
| Traffic(%) | 页面流量占比 |
| Traffic | 页面估算流量 |
| Top Keyword | 页面最主要关键词 |
| Intent | 页面主要意图（原始导出为 Primary Intent，汇总表统一为 Intent） |
| LLM Prompts | Semrush 导出的提示字段 |
| normalized_url | 如后续需要与 TopKeywords_All 关联，可在汇总表中生成 |
| source_sheet | 可选溯源字段，用于排错和人工追踪 |

重复 URL 处理规则：

- `TopPages_All` 是页面证据明细表，不在生成阶段强制删除重复 `normalized_url` 行。
- 后续回填 `Seed-Master` 时，以 `normalized_url` 作为页面证据查找键。
- 同一个 `normalized_url` 命中多行时，优先取 `Traffic` 最高的行；`Traffic` 相同再取 `Traffic(%)` 最高的行。
- 若最高行的 `Intent` 或 `Top Keyword` 为空，可从同 URL 的其他行补充非空值，但不得把多个不同 URL 的信息混合成一条证据。
- `_All` 表保留完整明细；去重只发生在回填摘要时，不反向改写 `TopPages_All`。

### 维护脚本

| 脚本 | 用途 | 运行时机 |
|------|------|---------|
| `generate_all_tables.py --both` | 生成/刷新 TopKeywords_All、TopPages_All | 轨道C原始表清洗后执行 |

**`generate_all_tables.py` 行为说明：**

- **输入**：自动扫描 `SEMrush-Top-Keywords` / `SEMrush-Top-Pages` 电子表格中所有非排除工作表（排除 `TopKeywords_All`、`TopPages_All`、`Competitor-Sheet-Map`、`Topic-Discovery`、`Keyword-Page-Proof`、`README`、`字段说明`）。
- **过滤**：按最低流量阈值过滤低信噪比行（Top Pages 要求 Traffic ≥ 100，Top Keywords 要求 Traffic ≥ 1 且至少 2 个单词）。
- **排序与截断**：每个竞品按流量降序排列，Top Pages 最多 300 行，Top Keywords 最多 2000 行。
- **新增字段**：`source_sheet`（来源工作表名）和 `normalized_url`（标准化 URL）只在 `_All` 汇总表中生成，不回写到原始竞品工作表。
- **输出**：`TopKeywords_All` 写入 `SEMrush-Top-Keywords` 电子表格，`TopPages_All` 写入 `SEMrush-Top-Pages` 电子表格。
- **覆盖逻辑**：如果 `_All` 工作表已存在，先删除旧工作表再新建同名工作表写入（删除重建），不影响原始竞品工作表。
- **意图字段标准化**：脚本表头中 `Keyword Intents` 和 `Primary Intent` 统一输出为 `Intent`。

### 当前表格整理规则

以 `SEMrush-Top-Keywords` 和 `SEMrush-Top-Pages` 当前结构为准，整理时遵守以下规则：

- 每个竞品原始工作表保留，不因行数大而删除；行数过大时应按轨道C-1重新清洗/瘦身。
- `TopKeywords_All`、`TopPages_All` 是派生汇总表，可按新规则重建；重建后以新表头和新行数为准。
- `Competitor-Sheet-Map`、`Topic-Discovery`、`Keyword-Page-Proof` 属于旧流程遗留；当前重构流程不再依赖这些工作表。项目执行中确认后可删除，删除后不得再把它们作为当前流程依据。
- `competitor_domain` 不作为新流程必需字段；`source_sheet` 仅用于溯源，`normalized_url` 仅在 URL 匹配需要时使用。
- **字段边界**：`source_sheet` 和 `normalized_url` 是 `_All` 汇总表的专属字段，不回写到原始竞品工作表。原始竞品工作表保持 SEMrush 导出 + 清洗后的结构，不新增分析字段。

## 轨道D：Seed-Master 竞品证据增强

轨道D处理轨道B和轨道C的结果：用 `TopKeywords_All` 和 `TopPages_All` 给 `Seed-Master v1` 补充竞品关键词证据与页面价值证据。

### D2：TopKeywords_All → Seed-Master

用 `TopKeywords_All.Keyword` 匹配 `Seed-Master.Keyword`。

目标：

- 找出 Seed-Master 中哪些关键词已经被竞品实际拿到流量。
- 记录该关键词对应的竞品 URL、关键词流量、命中竞品数量等证据摘要。
- 一个关键词可能对应多个竞品、多个 URL；Seed-Master 只保留摘要或最佳证据，完整明细保留在 TopKeywords_All 或后续证据视图中。

匹配规则：

- 匹配键为标准化后的 Keyword：小写、去首尾空格、合并连续空格。
- 不做词形归并和同义词归并，`crystal` 与 `crystals`、`rose quartz` 与 `pink quartz` 分别保留。
- 命中多行时，最佳关键词证据优先取 `Traffic` 最高的行；`Traffic` 相同再取 `Traffic(%)` 最高的行。
- 同一个关键词的完整命中明细留在 `TopKeywords_All`，不全部写入 `Seed-Master`。

未命中 Seed-Master 的 TopKeywords_All 关键词，先不直接写入 Seed-Master。

未命中处理规则（按未命中率判断）：

| 未命中率 | 动作 |
|----------|------|
| < 20% | 仅观察，记录到未命中日志，不回补、不新建候选表 |
| 20%–40% 且集中在 1-2 个主题 | 回轨道B，补充或新建对应 `Seed-*`（如 `Seed-Dreams`、`Seed-Tarot`），清洗后重建 `Seed-Master v1`，再重新执行轨道D |
| > 40% 或主题高度分散 | 先人工筛选候选视图，再决定是否回补或新建 Seed-*；候选视图是临时工具，不是标准必需工作表 |

> 未命中率 = TopKeywords_All 中未命中 Seed-Master 的去重关键词数 / TopKeywords_All 去重关键词总数。

### D3：TopPages_All → Seed-Master

对 D2 中已命中的竞品 URL，再用 URL 去 `TopPages_All` 中补页面级流量证据。

目标：

- 判断该关键词对应的竞品 URL 是否也是高流量页面。
- 补充页面流量、页面主要意图、页面类型线索等证据。
- TopPages_All 不直接按关键词进入 Seed-Master，它通过 TopKeywords_All 命中的 URL 提供页面价值补充。

URL 关联规则：

- 关联键为 `normalized_url`：D2 命中的 `TopKeywords_All.URL` 经标准化后，匹配 `TopPages_All.normalized_url`。
- `normalized_url` 生成规则（与 `generate_all_tables.py` 保持一致）：
  1. 全部转小写
  2. 去协议前缀（`http://` / `https://`）
  3. 去 `www.` 前缀
  4. 删除 `?` 后查询参数
  5. 删除 `#` 后锚点
  6. 去尾斜杠（`/`）
  - 示例：`https://WWW.Example.com/Crystals/?x=1` → `example.com/crystals`

页面证据规则：

- 只处理 D2 已命中的 URL，不从 `TopPages_All` 直接反推关键词进入 `Seed-Master`。
- 若某个 D2 命中 URL 在 `TopPages_All` 中找不到，仍保留 D2 的关键词证据，页面证据字段留空。
- 若多个 D2 命中 URL 都有页面证据，最佳页面证据优先取 `TopPages_All.Traffic` 最高的 URL。
- `TopPages_All.Top Keyword` 只作为页面主关键词参考，不替代 `Seed-Master.Keyword`。

### D4：Seed-Master 回填字段规则

`Seed-Master v1` 的基础字段不变。轨道D只追加或刷新竞品证据字段，不覆盖人工清洗和研究字段。

建议追加字段：

| 字段 | 来源 | 说明 |
|------|------|------|
| Competitor Keyword Matched | TopKeywords_All | Yes/No，Seed 词是否被竞品实际拿到流量 |
| Competitor Match Count | TopKeywords_All | 命中的竞品关键词明细行数 |
| Competitor Source Count | TopKeywords_All.source_sheet | 命中的竞品来源数量 |
| Best Competitor Source | TopKeywords_All.source_sheet | 最佳关键词证据来源工作表 |
| Best Competitor URL | TopKeywords_All.URL | 最佳关键词证据 URL |
| Best Keyword Traffic | TopKeywords_All.Traffic | 最佳关键词证据带来的关键词流量 |
| Best Keyword Traffic % | TopKeywords_All.Traffic(%) | 最佳关键词证据流量占比 |
| Best Page Traffic | TopPages_All.Traffic | 最佳 URL 对应页面流量；没有页面命中则留空 |
| Best Page Traffic % | TopPages_All.Traffic(%) | 最佳 URL 对应页面流量占比 |
| Best Page Top Keyword | TopPages_All.Top Keyword | 页面级 Top Keyword，仅作参考 |
| Best Page Intent | TopPages_All.Intent | 页面级主要意图 |
| Evidence Updated At | 脚本运行时间 | 本次证据字段刷新时间 |

回填边界：

- 可覆盖：上述竞品证据字段。
- 不覆盖：Keyword、中文、Topic Pillar、Entity、Subtopic、Content Role、Volume、KD、CPC、Number of Results、Intent，以及后续人工添加的 Priority、Recommended Page Type、Status、Notes 等研究字段。
- `Seed-Master` 只存摘要和最佳证据，不存所有竞品 URL 列表；需要完整明细时回查 `TopKeywords_All` 和 `TopPages_All`。
- 轨道D必须在 `Seed-Master v1` 有实际关键词数据后执行；空 `Seed-Master` 不做证据回填。

## 轨道E：Sitemap解析

**数据来源**：竞品 URL 列表从**项目 Google Sheets 竞对清单**导出。读取竞对清单中的竞品域名/URL 字段，逐个调用 sitemap-mcp-server 解析。不使用本目录下的静态 `competitor_urls.txt`（该文件仅为格式示例）。

**结果存放**：Sitemap 解析报告存放在**项目文件夹 `01-竞品分析/1B-sitemap解析/`**（如 `1B-sitemap解析报告.md`），不存放在工具目录。只有 URL 清单很大、需要频繁筛选/排序时，才可另建 `Sitemap-Pages` 电子表格。

**产出内容**：每个成功竞品的页面清单、URL 路径模式、内容类型分布（博客/产品/分类/静态页面）。

### sitemap-mcp-server

> 来源：[mugoosse/sitemap-mcp-server](https://github.com/mugoosse/sitemap-mcp-server)（MIT 开源）
> 底层库：ultimate-sitemap-parser（自动通过 robots.txt 发现 sitemap）

**为什么选 sitemap-mcp-server**：
- 核心需求是**解析竞品 sitemap → 提取博客/产品等页面 URL → 了解竞品内容策略**
- 直接解析 sitemap.xml，秒级返回结果
- 支持按子sitemap过滤（`sitemap_url`参数）和按路径过滤（`route`参数）
- 返回精确页面数量和完整 URL 列表（含 lastmod 时间戳）

**已安装路径**：

| 项目 | 路径 |
|------|------|
| venv Python | `D:\Tools\sitemap-mcp-server\.venv\Scripts\python.exe` |
| MCP 配置 | `~/.claude.json` 的 `mcpServers.sitemap` |

> **注意**：sitemap-mcp-server 0.1.3 与 mcp>=1.25 不兼容（`FastMCP.__init__` 移除了 `version` 参数），已在安装目录 `server.py` 第156行修补（移除 `version=` 和 `description=` 两个参数）。**勿用 uvx 重建环境**，否则修补会丢失。

**MCP 配置内容**（已写入 `~/.claude.json` → `mcpServers.sitemap`）：
```json
"sitemap": {
  "type": "stdio",
  "command": "D:\\Tools\\sitemap-mcp-server\\.venv\\Scripts\\python.exe",
  "args": ["-m", "sitemap_mcp_server"],
  "env": { "TRANSPORT": "stdio" }
}
```

**可用工具**：

| 工具 | 功能 | 典型用法 |
|------|------|---------|
| `get_sitemap_tree` | 获取完整 sitemap 树结构 | 了解竞品有哪些子sitemap |
| `get_sitemap_pages` | 获取页面列表（支持过滤/分页） | 用 `sitemap_url` 指定子sitemap，用 `route` 过滤路径 |
| `get_sitemap_stats` | 获取 sitemap 统计信息 | 快速了解竞品页面总量、子sitemap 数 |
| `parse_sitemap_content` | 直接解析 sitemap XML 内容 | 已有 sitemap 文本时直接解析 |

**典型工作流**：

```
1. get_sitemap_stats(url="https://competitor.com")                    → 快速获取页面总量
2. get_sitemap_tree(url="https://competitor.com")                     → 查看子sitemap结构
3. get_sitemap_pages(url="https://competitor.com",                    → 指定blog子sitemap
     sitemap_url=".../sitemap_blogs_1.xml",
     include_metadata=true, limit=0)
4. 如有分页：用返回的 nextCursor 继续拉取
5. 将 URL 列表按主题分类（水晶含义/天使号码/星座/脉轮/颜色/功效等）
6. 汇总写入竞品分析报告
```
