# 深度文章审查器 - 魔改设计方案

> **版本**：v1.0
> **日期**：2026-07-07
> **适用范围**：先 Earthward 水晶站（goearthward.com），引擎通用、ruleset 可换
> **核心哲学**：判断比产出难。用 AI 做内容不难，难的是判断 AI 做的对不对。
> **定位**：本审查器 = RLM §3.6 发布前质检路由的**实现引擎**，不是 3.6 之外的另一套。3.6 矩阵是规则源，本器是执行层。

---

## 0. 核心设计哲学——为什么是三层不是一层

整个设计最难的不是"用 AI 做"，是"判断 AI 做的对不对"。审查器本身是 LLM 会幻觉，加一个 L2 验证、L2 也幻觉，无限递归。本设计用**四机制叠加**把幻觉压进一个收敛边界——不是消除幻觉，是把它关进笼子。

### 机制 1：硬规则最大化（把软判断不断硬化）

能确定性判断的绝不交给 LLM。L0 硬规则不幻觉（base64 编了就是编了，内链 404 就是 404，化学式大小写 includes 命中就是命中）。ruleset 持续迭代的方向，就是**把 L1 的软判断一条条沉淀成 L0 的硬规则**。

> **活例子**：去 AI 化一开始只能 LLM 判断"这段像不像 AI 写的"。跑多了发现 `delve / leverage / harness / unlock the power of / transformative` 这批词是 AI 高频标志 → 沉淀成 L0 的 `includes` 风险词表（确定性检查）。剩下"可预测节奏 / 虚假共情 / 结构同质"才继续留给 L1。软判断越跑越少，幻觉面越缩越小。

### 机制 2：L1 输出强制带可验证证据链（禁止只给结论）

L1 禁止只说"这段去 AI 化不够"。必须给：

```
原文位置: 第3段第2句
原文引用: "unlock the transformative power of amethyst"
命中模式: AI痕迹模式 P3（AI专属词汇 transform/unlock 组合）
置信度:   0.85
规则依据: ruleset/crystal-site/v1 §去AI化.标志词表
```

这样 L2 / 人的判断成本极低——**只验证证据是否成立**（这句话真在原文里吗？这个词真在标志词表里吗？），不是重新审一遍文章。证据链把 L2 从"重新判断"降级成"事实核查"，幻觉面大幅收窄。

### 机制 3：置信度门控 + 对抗验证

L1 每条结论带置信度 0-1，按阈值分流后续动作：

| 置信度 | 处理 | 走不走 L2 |
|--------|------|-----------|
| >0.85 且锚定硬规则 | 直接采纳 | 跳过 |
| 0.5 – 0.85 | L2 对抗验证 | 走 |
| <0.5 | 强制人审，不进自动队列 | 走人 |

**对抗验证**：开一个独立 agent，prompt = "试图推翻这条结论"，**默认推翻**。两个独立 agent 从相反方向看同一条结论，比单个 agent 自审可靠——幻觉是单向的（自信地编一个方向），对抗结构逼出反方向。

### 机制 4：人审反馈回流（唯一不幻觉的锚点）

人是唯一不幻觉的判断者，但不可规模化。所以人只审「抽样 + 低置信 + critical」三类，审完的判断回流 ruleset：人判"这条 L1 错了" → 分析为什么错 → 修规则。**人是规则的最终锚点，规则是规模化的载体**——人的判断通过规则放大到全量，而不是每篇都靠人。

### 四机制合起来

硬规则不幻觉（M1），软判断带证据可低成本核查（M2），关键决策有对抗+人审兜底（M3+M4），系统随反馈越来越准（M1+M4 把软判断不断硬化）。这不是消灭幻觉，是把它关进笼子。

## 1. 三层架构

```
┌─────────────────────────────────────────────────────────────────┐
│  ruleset/crystal-site/v1.{md,json}   ← 规则源（外置/版本化/双写同源）│
│  风险词表 · CTA降级表 · 品牌语调检查项 · schema映射 · 类型路由表 ·  │
│  AI痕迹模式表 · 化学式白名单                                       │
└──────────────────────────┬──────────────────────────────────────┘
                           │ 读规则
        ┌──────────────────┴──────────────────┐
        ▼                                       ▼
┌─────────────────────────┐        ┌────────────────────────────┐
│ L0 硬规则脚本层          │        │ L1 LLM 语义层               │
│ （确定性，必跑，全量）    │        │ （软判断，全量跑，带证据链） │
│ regex/curl/WP REST      │        │ 输出: 维度×结论×原文位置×    │
│ 零幻觉 零成本 秒出       │        │ 命中模式×置信度×规则依据     │
└──────────┬──────────────┘        └──────────────┬─────────────┘
           │                                       │
           │            ┌──────────────────────────┘
           ▼            ▼
    ┌──────────────────────────┐
    │ L2 判断层（验证 L1）      │
    │ 高置信→采纳              │
    │ 中置信→对抗验证agent      │
    │ 低置信→强制人审           │
    └──────────────┬───────────┘
                   ▼
    ┌──────────────────────────────────────────┐
    │ 综合报告                                  │
    │ 维度×结论(pass/fail/warn)×证据原文引用×   │
    │ 修复建议×严重度×置信度×L2验证状态          │
    └──────────────┬───────────────────────────┘
                   ▼
    ┌──────────────────────────────────────────┐
    │ 审查-修复-复审闭环                        │
    │ 审查agent ≠ 修复agent · 批级触发 ·        │

    └──────────────────────────────────────────┘
```

| 层 | 职责 | 工具 | 输入 | 输出 | 为什么单独一层 |
|----|------|------|------|------|---------------|
| L0 | 确定性检查 | regex/curl/WP REST/Python | 文章HTML/文本+元数据 | 结构化JSON | 零幻觉零成本，能确定的不交LLM |
| L1 | 语义软判断 | LLM（按内容类型合并agent） | L0结果+文章+ruleset | 带证据链+置信度的发现 | 语义维度无法形式化，必须LLM |
| L2 | 验证L1 | 对抗agent / 人 | L1发现 | 验证状态（采纳/推翻/人审） | LLM会幻觉，关键决策需对抗+人审兜底 |

## 3. 差距分析

| 项目需要的深度 | 现有基础 | Gap |
|--------------|---------|-----|
| 内容证据防幻觉（meta层溯源） | 无 | 完全缺失，且是命门 |
| 风险词/化学式校验（大小写敏感） | 无 | SiO₂ 易被 /i regex 误判为 sio₂ |
| 竞品依据前置校验 | 无 | 生产阶段抓了竞品但发布前没人验"是否真引用" |
| 内链完整性（future/draft） | seo-technical 通用 | 不懂 status=any，curl -L 假阳性 |
| Shop CTA 不死链 | 无 | 三级降级链是项目专属 |
| WP 部署就绪（base64/asciiJSON） | 无 | wp_kses_post 破坏 &&/</> 是踩过的大坑 |
| 去 AI 化 | seo-content 笼统 | 07 工作流有七模式+标志词表但没接入审查 |
| DREAM 模型 | 无（Brief 阶段用） | 发布前没人验 DREAM 五角度是否真覆盖 |
| 品牌语调合规 | 无 | §4 禁用词 / §5 销售页禁用没自动化检查 |
| GEO/AI citation | seo-geo 通用 | 不懂水晶内容段落级可引用性 |
| Schema 匹配 | seo-schema 通用 | 不懂水晶内容类型→schema 映射 |
| L2 判断层 | 无 | 命门缺失——LLM 审查无人验证 |
| 审查-修复闭环 | 无 | 现在审查和生产混在一起 |

**结论**：seo-* 是"各自为战的通用兵"，EEAT 是"单点裁判"，都缺**项目专属规则 + 证据链 + 判断层 + 闭环**。魔改的核心不是再叠维度，是把这些维度装进一个**带判断层和反馈闭环**的框架。

## 4. 魔改维度详设

每个维度标注：**检查什么 / 数据源·工具 / pass-fail 标准 / 输出格式 / 属 L0 还是 L1 / 证据链要求**。

### 4.1 内容证据防幻觉（L1 · meta 层）

- **检查什么**：文章里的"事实宣称"是否标注了可信来源（竞品结构 / 知识库 / 化学数据库 / 一手经验）。**不是查"内容对不对"，是查"有没有证据"**。
- **数据源**：文章自身 + Brief 里的 evidence source 字段。
- **pass/fail**：每个事实宣称（水晶功效 / 化学式 / 矿物属性 / 历史传统）能溯源到 Brief 的 evidence source 或竞品 URL → pass；裸断言 → fail。
- **输出**：`{维度:"证据溯源", 层:"L1", 结论:"warn", 原文位置:"§2¶3", 原文引用:"Amethyst is scientifically proven to reduce anxiety", 命中模式:"裸断言+虚假科学声明", 置信度:0.9, 规则依据:"ruleset§证据溯源.虚假科学声明", 修复建议:"改为 traditionally associated with calming, many people report..."}`
- **证据链**：必须给原文位置 + 命中模式 + 规则依据。
- **防幻觉要点**：审查器自己不判断"amethyst 能不能治焦虑"（它自己会幻觉出"能"），只判断"这句话有没有标来源 / 有没有用虚假科学声明词（scientifically proven）"。meta 层。

### 4.2 风险词 / 化学式校验（L0 · 确定性）

- **检查什么**：品牌语调 §4 禁用词（heal/cure/treat/attract wealth/100% guaranteed/magic 等）+ 化学式大小写正确性（SiO₂ 不能写成 sio₂/SIO2/sio2）。
- **数据源**：ruleset `§风险词表` + `§化学式白名单`。
- **工具**：Python `includes()` 小写匹配查真违规；化学式用精确 regex（`SiO₂` 带 Unicode 下标，不 `/i`）。
- **pass/fail**：禁用词命中 = fail；化学式不匹配白名单 = fail。
- **输出**：`{维度:"风险词", 层:"L0", 结论:"fail", 原文位置:"§1¶1", 原文引用:"This crystal will cure anxiety", 命中:"cure(禁用词-医疗化)", 严重度:"critical", 置信度:1.0}`
- **红线**：**绝不用 `/i` regex 查化学式**——`/i` 会把正确的 `SiO₂` 误判为 `sio₂`，触发批量重写毁好内容（血泪教训）。大小写敏感 `includes('sio₂')` 只查真违规。

### 4.3 竞品依据前置校验（L0+L1 · meta 层）

- **检查什么**：文章是否标注了 evidence source / 是否引用了竞品结构。**不去抓竞品**（那是生产阶段 07 的事），只查"有没有标注"。
- **数据源**：文章 + Brief 的竞品 URL 字段。
- **L0**：Brief 里竞品 URL 字段非空 → pass；空 → fail。
- **L1**：文章结构是否与 Brief 标注的竞品结构有可识别的差异化（不是照抄也不是完全脱离）。
- **输出**：L0 结构化 + L1 带证据链。
- **原则**：审查器不抓竞品、不改写。竞品依据是 07 工作流生产时前置完成的（见 `seo-content-research-competitor-first` memory），审查器只校验事实。

### 4.4 内链完整性（L0 · 确定性）

- **检查什么**：文章里所有内部 `<a href>` 指向的页面是否存在（含 future/draft 状态）。
- **工具**：WP REST `status=any` 查目标页。**不用 `curl -L`**（curl -L 对 future 文章的 `?p=<id>` 动态链接假阳性，见 `wp-future-link-p-format` memory）。
- **pass/fail**：目标页 status=any 命中 → pass；真 404 → fail。规划中页（schedule future / 表登记 / 已建框架）接受临时 404 → warn（见 `internal-link-planned-pages-ok-404` memory）。
- **输出**：每条内链 `{url, 目标状态, 结论, 严重度}`。

### 4.5 Shop CTA 不死链（L0 · 确定性）

- **检查什么**：文章里 Shop CTA 跳特定石类目（`/product-category/{stone}-crystals/`）前验证 200。
- **三级降级链**（见 `shop-cta-no-deadlink-rule` memory）：
  1. `/product-category/{stone}-crystals/` → 验证 200
  2. 404 → 降级 `/shop/?s={stone}`（产品搜索页）
  3. 搜索也空 → 降级总 `healing-jewelry` 类目
- **工具**：WP REST / `wc/store` 验证产品类目存在。
- **pass/fail**：CTA 落在三级里某级 200 → pass；否则 fail。
- **输出**：`{cta_url, 实际落点, 降级级数, 结论}`。

### 4.6 WP 部署就绪（L0 · 确定性）

- **检查什么**：`<!-- wp:html -->` 块内的 JS 是否 Base64 包装（防 `wp_kses_post` 把 `&&`→`&#038;&#038;`、`</`→`&lt;/` 转义致 IIFE 不执行）；JSON 是否用 asciiJSON（非 ASCII 转 `\uXXXX`）。
- **工具**：解析 HTML 块，regex 查 `<script type="text/plain" id=>` + atob loader 模式；查 JSON 块非 ASCII 字符。
- **pass/fail**：复杂 JS（含 `&&`/比较/`<`）+ 大 JSON（CJK）必须 Base64 包装 → pass；裸 `<script>` 含上述字符 → fail。简单工具（恰好没触发 `&&`/`<`）可豁免 → warn。
- **输出**：`{html块ID, JS是否base64, JSON是否ascii, 结论, 严重度:"critical"}`。
- **依据**：`wp-html-block-js-base64` memory（塔罗 v2/v3 连续踩坑，ECTaro=undefined 牌背=0）。

### 4.7 去 AI 化（L0+L1 · 天然两层）

这是机制 1（硬化）的最佳示范——维度天然分两层：

**L0 硬规则（已硬化的软判断）**：
- AI 标志词表（来自 07 工作流）：`delve, leverage, foster, ignite, empower, uncover, unleash, underscore, optimize, streamline, navigate, demystify, elevate, spearhead, catalyze, orchestrate, harness, augment, synergize, encapsulate, elucidate, contextualize, amalgamate, tapestry, landscape, seamless, robust, realm`
- 通用过渡词表：`Furthermore, Moreover, Additionally, In conclusion`
- `includes` 命中 = warn（置信度 1.0）

**L1 LLM（无法形式化的）**：
- 模式2 可预测节奏（句子长度一致 15-20 词，缺 burstiness）
- 模式5 虚假共情（"Picture this...", "As a crystal lover, you know..."）
- 模式1 否定/对比结构（"It's not just X, it's Y"）
- 模式6 过度修饰
- 输出带证据链 + 置信度。

- **pass/fail**：L0 命中标志词 = warn；L1 命中模式 + 置信度 ≥0.7 = warn/fail。
- **输出**：每条 `{层, 命中模式, 原文位置, 原文引用, 置信度, 修复建议}`。

### 4.8 DREAM 模型（L1）

- **检查什么**：五角度是否覆盖——Demand（需求）/ Reason（理由）/ Evidence（证据）/ Advantages（优势）/ Motivation（动机）。
- **数据源**：文章结构 + Brief 的 DREAM 标注（Brief 锁定时已做 DREAM 检查，见 RLM §3.4）。
- **pass/fail**：五角度均有对应段落且非空 → pass；缺角度 = warn。
- **输出**：`{角度, 是否覆盖, 原文位置, 置信度}`。

### 4.9 品牌语调合规（L0+L1）

**L0**（来自品牌语调配置.md §4/§5，确定性）：
- §4 合规禁用词：heal/cure/treat（疾病语境）/attract wealth/guarantee fortune/change destiny/scientifically proven/100% guaranteed/replace medical treatment/purify negative energy/magic/supernatural power
- §5 销售页禁用：虚构数据（"4.9/5 Stars"/"10,000+ Happy Customers"）/绝对承诺（"100% Natural"）/能量治愈（"work with your energy"/"shield your energy"）/虚构评价（"Verified Buyer — Name"）/文化挪用（"rooted in Native American tradition"）
- `includes` 命中 = fail（critical）

**L1**（语义）：
- §1 语调四维：温和可信 / 灵性不玄乎 / 先结论后三视角 / 不硬推
- §4.2 三句话原则是否落实（传统语境 + 个人体验 + 证据边界）
- 输出带证据链。

### 4.10 GEO / AI citation（L0+L1）

**L0**：
- llms.txt 存在性（`/llms.txt`、`/llms-full.txt`）
- AI crawler robots 处理（GPTBot/ClaudeBot/PerplexityBot/Google-Extended/CCBot 是否误封或误放）
- 工具：seo-geo 子 agent + `aeo_llms_txt_check` / `aeo_ai_bots_robots_audit` MCP

**L1**：
- 段落级可引用性：关键段落是否有"可被 AI 引用的自包含陈述"（事实 + 上下文 + 来源，不依赖前后文）
- 输出每段 citability 评分 + 置信度。

### 4.11 Schema 与内容类型匹配（L0 · 确定性）

- **检查什么**：JSON-LD schema 类型是否与内容类型匹配。
- **映射表**（ruleset `§schema映射`）：

| 内容类型 | 必备 schema |
|---------|------------|
| 百科/meaning | Article + BreadcrumbList |
| 教程/how-to | HowTo + FAQPage + BreadcrumbList |
| 工具页 | Article + FAQPage + BreadcrumbList（+ItemList 若列表） |
| 转化/product | Product + Offer + BreadcrumbList |
| 对比/combinations | Article + ItemList + BreadcrumbList |
| FAQ | FAQPage + BreadcrumbList |

- **工具**：抓 JSON-LD + `schema_validate_url` MCP。
- **pass/fail**：必备类型齐全且属性非空 → pass；缺类型 / 关键属性空（Article 无 headline / Product 无 offers）= fail。
- **输出**：`{内容类型, 应有schema, 实有schema, 缺失, 结论}`。

## 5. L2 判断层详设

### 5.1 四机制落地

| 机制 | 落地 | 触发条件 |
|------|------|---------|
| M1 硬规则最大化 | L0 已是；持续从 L1 沉淀 | 见 §13.3 硬化阈值 |
| M2 证据链 | L1 输出格式强制 | 所有 L1 输出 |
| M3 置信度门控 | 分流器 | 所有 L1 结论 |
| M4 人审回流 | 人审队列 + ruleset changelog | 抽样+低置信+critical |

### 5.2 对抗验证 agent prompt 骨架

```
你是审查结果验证员。下面是一条 L1 审查发现。你的任务是【试图推翻它】——
默认它有问题，找出 L1 可能错的地方。只有证据确实支持 L1 时才确认。

L1 发现：
- 维度: {dimension}
- 原文位置: {location}
- 原文引用: {quote}
- 命中模式: {pattern}
- L1 结论: {verdict}
- L1 置信度: {confidence}
- 规则依据: {rule_ref}

验证维度（逐条查）：
1. 原文引用是否真的在文章该位置？（事实核查，不是重新判断）
2. 命中模式是否真的命中？（词在词表里吗？结构真匹配吗？）
3. 规则依据是否真存在于 ruleset 当前版本？
4. 有没有合理豁免？（如 "unlock your phone" 是合理用法，非 AI 痕迹）

输出：
- 验证结论: 确认 / 推翻 / 存疑
- 推翻理由: （若推翻）
- 验证置信度: 0-1
- 建议: 采纳 / 人审
```

### 5.3 置信度门控阈值表

| L1 置信度 | L2 动作 | 自动进修复队列？ |
|-----------|---------|-----------------|
| >0.85 且锚定 L0 硬规则 | 跳过 L2，直接采纳 | critical 才进 |
| 0.7 – 0.85 | 对抗验证 | 验证确认才进 |
| 0.5 – 0.7 | 对抗验证 + 人审抽样 | 不自动 |
| <0.5 | 强制人审 | 不自动 |

### 5.4 人审抽样策略

- **全量人审**：所有 critical + 置信度 <0.5 的发现。
- **抽样人审**：每批抽 10% 的 pass 项 + 20% 的 warn 项，验 L1 有没有漏判 / 误判。
- **专项人审**：新规则上线首周全审（见 §13.5 灰度）。

## 6. 审查-修复-复审闭环

```
审查器(L0+L1+L2) → 综合报告
        │
        ▼
修复队列（只挑 critical + true positive + 置信度达标）
        │
        ▼
修复 agent（独立，职责单一：执行修复建议，不改判）
        │
        ▼
复审（再跑审查器，验修复是否有效）
        │
   ┌────┴────┐
   ▼         ▼
 通过      不通过 → 回队列（迭代计数+1）
   │
   ▼
 发布
 最大迭代 2 次 → 仍不通过 → 强制人审
```

### 三条铁律

1. **审查 agent ≠ 修复 agent**。审查的不能改，改的不能判。自己审自己改，错误会被掩盖。
2. **触发时机是批级不是篇级**。一批审完出报告 → 调度层（人/调度 agent）看报告决定哪些进修复队列 → 修复 agent 批量跑。不审一篇改一篇（审查-修复耦合）。
3. **每条 critical 必须带修复建议**（改哪句、改成什么方向）。修复建议置信度低 → 标"需人审"，不进自动队列。

### 修复建议格式契约

```json
{
  "finding_id": "F-007",
  "维度": "风险词",
  "严重度": "critical",
  "原文位置": "§2¶3 第2句",
  "原文": "This crystal will cure anxiety",
  "问题": "cure 是 §4 禁用词（医疗化声明）",
  "修复建议": "改为 'This crystal can support a sense of calm as part of your self-care routine'",
  "建议置信度": 0.92,
  "是否自动修复": true
}
```

## 8. 实现方案对比

| 方案 | 描述 | 优点 | 缺点 | 智谱并发 |
|------|------|------|------|---------|
| A 单复合 agent | 一个 prompt 跑全维度 | 简单 | 硬规则也交LLM不稳、prompt爆炸、不可批量、幻觉面大 | 1 agent |
| B workflow 编排 | L0脚本 + L1按类型合并 + L2按需 | 两层分离、抗过载、可批量 | 需写编排脚本 | ≤3 并行 |
| C seo-* + 补丁层 | 复用 seo-* + 项目补丁 | 复用现有 | seo-* 不带证据链/置信度/判断层，补丁层难加 | 依调用数 |

**推荐 B（workflow 编排）**。理由：两层分离是命门（硬规则不能交 LLM），智谱并发下 L1 按内容类型合并成 2-3 个 agent、L2 只对中置信段触发，不超 5 并发。

### 脚本骨架（伪代码）

```python
# L0 硬规则层（一次跑完所有确定性检查）
def run_l0(article, ruleset):
    results = []
    results += check_risk_words(article, ruleset)        # 风险词 includes
    results += check_chem_formulas(article, ruleset)     # 化学式大小写精确
    results += check_internal_links(article, wp_rest)    # status=any
    results += check_shop_cta(article, ruleset)          # 三级降级
    results += check_wp_deploy(article)                  # base64/asciiJSON
    results += check_schema(article, ruleset)            # 类型映射
    results += check_ai_flag_words(article, ruleset)     # AI标志词(已硬化)
    results += check_brand_forbidden(article, ruleset)   # 品牌禁用词
    return results  # 全置信度1.0，直接进报告

# L1 LLM 层（按内容类型合并 agent，智谱 ≤3 并行）
def run_l1(article, l0_results, content_type, ruleset):
    # 合并维度到一个 agent，按类型路由
    prompt = build_l1_prompt(article, l0_results, content_type, ruleset)
    findings = call_llm_agent(prompt, schema=L1_FINDINGS_SCHEMA)
    # 强制带证据链（schema 校验）
    return findings

# L2 判断层（按置信度分流）
def run_l2(l1_findings):
    verified = []
    for f in l1_findings:
        if f.confidence > 0.85 and f.anchored_l0:
            f.l2_status = "采纳"
        elif f.confidence >= 0.5:
            v = adversarial_verify(f)  # 独立 agent 试图推翻
            f.l2_status = "确认" if v.confirmed else "推翻"
        else:
            f.l2_status = "需人审"
        verified.append(f)
    return verified

def main(article, metadata):
    ruleset = load_ruleset(metadata.site)  # crystal-site/v1
    l0 = run_l0(article, ruleset)
    l1 = run_l1(article, l0, metadata.content_type, ruleset)
    l2 = run_l2(l1)
    report = compile_report(l0, l2)  # 综合报告
    return report  → 进修复队列
```

## 13. 持续迭代机制（命门章节，不是附录）

> 这章是整个设计的命门。审查器会不会越用越准，全看这章能不能落地。

### 13.1 误判上报

- 审查报告每条 finding 有"误判反馈"按钮（人或调度 agent 标记：这是误判 / 漏判）。
- 误判记录入库：`{finding, 反馈类型, 反馈理由, 关联文章, 时间}`。
- 关键：**false positive（误判合格为不合格）比 false negative（漏判）更优先修**——误判会触发批量重写毁好内容。

### 13.2 反馈回流（机制 4 落地）

```
人审标记误判 → 入误判库 → 周期复盘(每周/每批) 
    → 分析误判模式 → 修 ruleset 规则 → bump 版本 
    → 新规则灰度验证(§13.5) → 上线
```

人是规则最终锚点，规则是规模化载体。

### 13.3 软判断硬化阈值（机制 1 落地）

什么时候把 L1 软判断沉淀成 L0 硬规则？**三条件 AND**：

| 条件 | 阈值建议 | 含义 |
|------|---------|------|
| A 频率 | 最近 50 篇同模式命中 ≥10 次，或命中率 ≥20% | 够频繁才值得硬化 |
| B 确认率 | 人审确认率 ≥85% **且样本量 ≥8 次** | 够准且不是运气（只审2次都对不算） |
| C 可形式化 | 判据能抽象成词表/regex/结构/数值 | 能确定性检查（语义感受不能硬化） |

- **C 是最易被漏的**：EEAT"是否有独特一手经验"永远不能硬化（纯语义）；"unlock/transformative 是 AI 标志词"能硬化（词表）。
- 三条件全满足 → 候选硬化。
- **阈值校准**：季度复盘两件事驱动调整——漏判多（该硬化没硬化）→ 阈值偏松；误判回潮（硬化后 L0 误判率升）→ 阈值偏紧或规则退回 L1。

### 13.4 硬化后灰度验证（防形式化抓错特征）

ABC 全满足也不直接全量启用。新硬规则先进 **shadow mode**：
- L0 跑新规则但**只记录不阻断**，L1 同时继续跑该维度。
- 对比 L0 新规则 vs L1 判断的一致率。
- 一致率 >90% → 正式启用。
- 一致率低 → 形式化抓错特征（如把 `unlock` 全禁，误伤 `unlock your phone`）→ 退回 L1 或调整规则边界。

### 13.5 规则变更回溯 + A/B

- **回溯**：改规则后，bump 版本，对已发布内容跑新规则（shadow），评估"若用新规则会有多少新发现/误判"，决定是否回审已发布内容。
- **A/B 灰度**：新规则首周只对 10% 内容生效，全审人审，监控误判率/漏判率。指标达标 → 全量；不达标 → 回滚或调整。
