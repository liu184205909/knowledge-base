## 第一阶段：竞品数据汇总与主题发现

### 1.1 竞品数据汇总

从 1B 的 SEMrush 明细表中读取竞品已经被验证有排名/流量的关键词。

**数据源**：

| 来源表 | 作用 | 标准列 |
|------|------|--------|
| SEMrush-Top-Pages | 找出竞品哪些页面真正带来SEO流量 | 每个竞品一个原始工作表；URL / Traffic(%) / Traffic / Top Keyword / Primary Intent / LLM Prompts |
| SEMrush-Top-Keywords | 找出竞品排名关键词和排名URL | 每个竞品一个原始工作表；Keyword / Search Volume / Keyword Difficulty / CPC / URL / Traffic / Traffic(%) / Number of Results / Keyword Intents |

**Competitor-Sheet-Map**：

脚本不使用 EXCLUDED_SHEETS 黑名单，而是读取 `Competitor-Sheet-Map`（位于 SEMrush-Top-Keywords Google Sheet）中 `status = Active` 且对应 `include_in_top_keywords = TRUE` 或 `include_in_top_pages = TRUE` 的工作表。

Competitor-Sheet-Map 字段：
- `source_sheet_name`：竞品工作表名
- `competitor_domain`：竞品域名
- `competitor_type`：竞品类型
- `known_topic_hint`：可选先验提示（如 `Crystals; Tarot; Spirituality`），不参与强制分组，没把握就留空
- `business_relevance`：High / Medium / Low
- `include_in_top_keywords`：是否读取 Top Keywords 工作表（技术读取开关，不做业务筛选）
- `include_in_top_pages`：是否读取 Top Pages 工作表
- `top_pages_sheet_name`：Top Pages 里对应的工作表名（不同时填写）
- `status`：Active / Review / Exclude（竞品数据读取权限。Active = 数据质量和读取权限通过，不表示主题已确认）
- `notes`：备注

> **核心原则**：主题是关键词级属性，不是竞品级属性。竞品可以有主题倾向（通过 `known_topic_hint` 提示），但关键词才有最终主题归属。真正的主题发现由主题候选视图（工具实现名：Topic-Discovery）在 TopKeywords_All 之后完成。

`include_in_top_keywords` 是技术读取开关，不是业务筛选字段。`business_relevance` 回答的是"读进来之后，这些数据和我们业务关系有多近"。空表、测试表、输出表、误建表才设成 `FALSE`。

**加工结果**：

| 加工结果 | 来源 | 用途 |
|------|------|------|
| `TopPages_All` | 读取 `Competitor-Sheet-Map` 中 `status=Active` 且 `include_in_top_pages=TRUE` 的工作表，只保留页面 `Traffic >= 100` 的行，并按每个竞品最多300页截断 | 汇总所有竞品有明确流量证据的页面 |
| `TopKeywords_All` | 读取 `Competitor-Sheet-Map` 中 `status=Active` 且 `include_in_top_keywords=TRUE` 的工作表，只保留关键词 `Traffic > 0` 或 `Traffic(%) > 0`、有排名URL、且英文词数不少于2的行，并按每个竞品最多2000词截断 | 汇总所有竞品有流量证据的关键词排名与排名URL |

All 表必须过滤低信号行：原始数据仍保留在各竞品工作表中，All 表只承接可用于判断 SEO 价值的证据行。All 表追加 `competitor_domain`、`source_sheet`、`normalized_url` 三个额外列。

`normalized_url` 统一小写，去掉 `http/https`、`www.`、查询参数、锚点和末尾 `/`。

**关键原则**：All 层是 topic-agnostic 的证据索引，不依赖 Seed 进行主题筛选。它回答的问题是"竞品靠哪些关键词和页面获得了流量"，而不是"我们应该做哪些词"。

### 1.2 主题发现

从 TopKeywords_All 中分析关键词的主题分布，生成主题候选视图（工具实现名：Topic-Discovery）报告。

主题候选视图由独立脚本 `generate_topic_discovery.py` 生成，读取 TopKeywords_All + Competitor-Sheet-Map。

**主题候选视图字段**：

| 字段 | 说明 |
|------|------|
| topic_pillar | 主题名称 |
| source_sheet_count | 涉及多少个竞品工作表 |
| keyword_count | 该主题有多少关键词 |
| proof_candidate_count | 有多少关键词同时有页面流量证据 |
| top_competitors | 该主题排名前 3-5 的竞品 |
| example_keywords | 该主题代表性关键词（最多 5 个） |
| suggested_action | 脚本建议：Add to Seed-Master / Keep as Candidate / Exclude / Need Review |
| confirmed_action | 人工确认（空 = 未确认） |
| notes | 备注 |

**人工确认规则**：脚本只读取 `confirmed_action`，不读取 `suggested_action`。未确认的主题不进入后续流程。

**核心原则**：主题发现发生在 TopKeywords_All 层，不应该等到竞品验证视图（工具实现名：Keyword-Page-Proof）。因为验证视图已经被 Seed 范围和 Strong/Medium 规则筛过了，它不是全量观察层。用验证视图做主题发现，会漏掉那些还没有 Seed 承接、但未来可能值得做的主题。

## 第二阶段：主题确认与关键词主表建立

> 目标：根据主题候选视图确认哪些主题进入 Seed-Master，建立项目关键词主表。

### 2.1 Seed-Master 关键词主表

`SEMrush-Seed-Keywords` Google Sheet 中的 `Seed-Master` 工作表是项目唯一关键词主表。其他 `Seed-Crystals`、`Seed-Chakra` 等工作表是**导入暂存表**，不是最终主表。

字段标准见附录A。本节只说明 Seed-Master 的核心原则和字段标注时机。

**字段标注时机**：

| 标注时机 | 字段 | 说明 |
|---------|------|------|
| **1B 轨道B 初步标注** | Entity / Subtopic / Content Role / 中文 | 清洗辅助字段，服务于底座构建质量，不代表最终研究结论 |
| **1B 轨道B 导入保留** | Volume / KD / CPC / Intent / Number of Results | SEMrush 导出原始数据 |
| **1B 轨道B 机械继承** | Topic Pillar / Source Type / Source Detail | 从 Seed-* 工作表名和导入来源继承 |
| **1B 轨道D 证据回填** | Competitor Proof / Proof URL / Proof Traffic / Proof Competitor | 竞品关键词/页面证据 |
| **1C 决策标注** | Business Relevance / Recommended Page Type / Priority / Status | 研究验证和策略决策字段 |

**去重规则**：`Normalized Keyword` 是唯一键。Seed-Master 保持"一词一行"。交叉词（如 "chakra crystal bracelet"）不按 Topic Pillar + Keyword 生成多行，而是选一个主归属 Topic Pillar，交叉属性记在 Subtopic 或 Notes。

**Status 生命周期**：

```
暂存表导入 / 竞品增量词 → Candidate
人工确认值得做 → Approved
进入内容清单 / 页面规划 / Brief → In Content Plan
确认不做 → Skip
```

### 2.2 暂存表到 Seed-Master 的合并规则

1. 暂存表优先作为 Source Type = Seed，TopKeywords_All 增量词标记为 Source Type = Competitor_Top_Keyword
2. 同 keyword 已存在于 Seed-Master 时，不新增重复行，只补来源和 proof
3. 暂存表之间 Topic Pillar 冲突时，不自动覆盖，进入人工审核
4. TopKeywords_All 发现的增量词，只能进 Seed-Master 的 Candidate 状态，不能直接变成 Approved 或 In Content Plan

### 2.3 Proof scope 切换规则

- Seed-Master 为空时，Proof 脚本不得切换到只读 Seed-Master
- 切换条件：Seed-Master 首轮合并完成（有实际数据行）
- 切换前，Proof 脚本仍读取所有 Seed-* 暂存表作为 scope

## 第三阶段：竞品验证与证据回填

> 目标：用竞品数据验证 Seed-Master 中的关键词是否已有成功案例，并回填证据。

### 3.1 竞品验证视图（工具实现名：Keyword-Page-Proof）

`TopKeywords_All` 与 `TopPages_All` 通过 `competitor_domain + normalized_url` 关联，生成竞品验证视图（工具实现名：Keyword-Page-Proof）。

验证视图只保留"关键词本身有流量 + 对应页面也有流量"的记录；关键词流量为 0 的排名行不能因为页面有流量就被当成成功案例。

验证视图控制范围：优先保留 Seed-Master 中已经存在关键词的竞品证据；对主表没有的关键词，只保留 Strong / Medium 级别的增量机会。

### 3.2 回填 Seed-Master

从竞品验证视图回填到 Seed-Master，只填充**最佳证据**（Proof Level 最强的那条记录）。多个竞品证据保留在验证视图中，Seed-Master 只需要知道"这个词有没有强证据，最强证据是谁"。

验证视图 = 一对多证据表（同一个 keyword 可能有多个竞品的证据）
Seed-Master = 每个 keyword 一行，只回填最佳 proof

### 3.3 增量词发现

从 TopKeywords_All 中发现 Seed-Master 没有但有关键词流量和页面流量双重证据的增量词，补回 Seed-Master，Status 设为 Candidate，Source Type 标记为 `competitor_top_keyword`。

## 关键词扩展方法库

### 2.1 长尾关键词截流策略

> 来源：哥飞 SEO 实战

同一个需求有不同的搜索表达（"A B C"、"A B"、"B A"、"B C A"……），大厂只盯搜索量最大的表达，其余变体表达竞争小。用变体词注册域名、优化首页 TDH，先拿小词排名，随权重积累逐步蚕食大词流量。

### 2.2 KGR 黄金关键词法

> 专为新站/弱域名设计的低竞争快速排名打法。

```
KGR = allintitle 结果数 ÷ 月均搜索量（月搜索量 ≤ 250）
```

| KGR 值 | 含义 |
|--------|------|
| < 0.25 | 黄金词，数周内有望进首页 |
| 0.25 ~ 1.0 | 尚可，值得尝试 |
| > 1.0 | 过热，放弃 |

局限性：单篇流量天花板低，需靠数量取胜；allintitle 结果数不完全稳定，需结合 SERP 质量二次判断。

### 2.3 蓝海关键词挖掘法

> 来源：子木聊出海（bysocket.com）

不直接挖词，从低权重竞品反推关键词。用 Ahrefs 找 DR 低但流量不错的小站，分析其 Top Pages，记录关键词/流量/排名，再从 SERP 中发现更多低权重竞品，形成持续扩展的竞品库。蓝海词指标：有搜索量、KD < 20~30、SERP 首页存在低 DR 页面。

### 2.4 竞品关键词差分析（Keyword Gap Analysis）

对比你和竞品的排名词库，找出竞品有排名但你未覆盖的关键词。用 Ahrefs Content Gap 或 Semrush Keyword Gap 输入多个竞品域名，工具列出遗漏词列表，优先选择搜索意图匹配、竞品排名靠前的词。

### 2.5 零搜索量关键词（Zero-Volume Keywords）

工具显示搜索量为0不代表没人搜，可能工具数据未覆盖或属于新兴需求。竞争几乎为零，抢先布局可获得先发优势。适合关注行业趋势变化，捕捉工具尚未收录的新词。

### 2.6 AI 辅助关键词发现

输入行业种子词，让 AI 批量生成变体表达、用户问题、场景化长尾词。还可以用 AI 做语义聚类——把语义相近的关键词归为一组，每组对应一篇内容，避免内部页面互相竞争。

### 2.7 Splinter Method（关键词裂变法）

> 来源：SEO社区分享（2026），适用于从宽泛种子词裂变为细分受众关键词群

**核心思路**：把一个宽泛的关键词拆分为不同受众群体的细分变体，每个变体对应一个独立的搜索意图和内容页面。

**方法**：
1. 选定一个宽泛种子词（如 "crystal healing"）
2. 用 AI（如 Claude）列出所有可能的受众细分维度：
   - 按人群（beginners / men / women / kids / seniors / therapists）
   - 按场景（sleep / anxiety / love / wealth / protection）
   - 按形式（bracelets / necklaces / raw stones / grids）
   - 按信仰/文化背景（Buddhist / New Age / Christian / secular）
3. 每个细分维度生成对应的关键词变体（如 "crystal healing for sleep"、"crystal healing for beginners"）
4. 每个变体在 SEMrush 中验证搜索量和竞争度
5. 每个通过验证的变体对应一个独立内容页面

**示例**：

| 种子词 | 裂变维度 | 裂变关键词 | 对应内容 |
|--------|---------|-----------|---------|
| crystal healing | 按人群 | crystal healing for beginners | 入门指南 |
| crystal healing | 按场景 | crystal healing for anxiety | 焦虑场景应用 |
| crystal healing | 按形式 | crystal healing grids | 水晶网格教程 |
| crystal healing | 按文化 | crystal healing spiritual | 灵性实践 |

**与其他方法的协同**：
- Splinter Method 的裂变结果可以作为 2.6 AI辅助关键词发现的输入
- 裂变出的关键词进入第二阶段的 Seed-Master 主表进行竞品验证
- 特别适合多品类内容策略（如水晶+星座+情感），每个品类独立裂变后交叉组合

## 第四阶段：关键词过滤与评估

> 目标：从大量候选词中筛出值得投入的关键词。先过滤再做竞品分析，避免浪费时间。

### 4.1 搜索意图分类（Search Intent）

将关键词按用户意图分类，决定内容类型，并过滤掉不该做的词：

| 意图类型 | 用户在做什么 | 内容策略 |
|---------|------------|---------|
| **信息型** | 想了解/学习，"how to..."、"what is..." | 教程、指南、博客 |
| **导航型** | 找特定网站/品牌 | 不是你的品牌就跳过 |
| **商业考察型** | 在对比/研究，"best..."、"vs"、"review" | 对比评测、推荐清单 |
| **交易型** | 准备购买，"buy..."、"discount"、"free" | 产品页、落地页 |

**优先做商业考察型 + 交易型**，它们离钱最近。

### 4.2 SERP 竞争度手动分析

不迷信工具 KD 分数，直接打开 Google 搜索结果判断真实竞争：
- 排名前10的 DR 高不高？有没有低 DR 小站？
- 排名页面是专题页还是首页/分类页？（专题页说明竞争更激烈）
- 有没有 Featured Snippet、People Also Ask 可抢占？
- 竞品内容质量如何？你能写得更好吗？

KD 高但 SERP 首页全是弱内容页，反而是机会。

### 4.3 关键词优先级评分模型

对候选关键词按以下5个维度打分（每项 1-5 分），总分决定优先级：

| 维度 | 评分标准 | 高分特征 |
|------|---------|---------|
| 搜索意图 | 交易型/商业考察型优先 | 用户明确想买或正在比较 |
| 商业价值 | CPC高、有广告投放 | 竞品愿意为此付费 |
| 竞争机会 | SERP首页存在弱页面 | 低DR站、薄内容、内容空白 |
| 内容适配 | 适合当前网站生产 | 有能力写出更高质量内容 |
| 转化路径 | 能导向产品页/询盘/注册 | 离转化动作近 |

**总分映射**：
- 20-25 分：**P0**，优先做
- 15-19 分：**P1**，进入内容计划
- 10-14 分：**P2**，作为补充内容
- <10 分：**P3**，暂不投入

### 4.4 商业价值评估（10个方法）

> 来源：哥飞 SEO 实战经验总结

**快速判断（必做）**：

| # | 方法 | 判断标准 |
|---|------|---------|
| 1 | 看 CPC | CPC 越高，商业价值越大 |
| 2 | 看搜索量 | 有搜索量才有流量基础 |
| 3 | 看搜索意图 | 用户想买 > 想学 > 想白嫖 |

**竞品变现验证**：

| # | 方法 | 判断标准 |
|---|------|---------|
| 4 | 看有没有人投广告 | 有广告 = 有商业价值 |
| 5 | 看竞品是否收费 | 付费说明能赚钱 |
| 6 | 看价格套餐 | 价格越高、卖点越多 = 市场越大 |
| 7 | 看竞品外链投入 | 竞品花钱 = 验证了该词能赚钱 |

**数据深度验证**：

| # | 方法 | 判断标准 |
|---|------|---------|
| 8 | Similarweb + Stripe | 有支付流量 = 有真实交易 |
| 9 | Similarweb + 其他支付平台 | 多支付渠道交叉验证 |
| 10 | AI 搜索竞品营收 | 有公开营收 = 市场规模可观 |

**判断矩阵**：

```
高CPC + 高搜索量 + 商业意图 + 竞品收费 → ✅ 强烈建议做
高CPC + 低搜索量 + 商业意图 + 竞品收费 → ⚠️ 小众但精准，值得做
低CPC + 高搜索量 + 信息意图 + 竞品免费 → ❌ 流量大但难变现，不做
```

## 第五阶段：关键词组织与内容规划

> 目标：把筛选后的关键词变成可执行的内容计划。

### 5.1 话题集群策略（Topic Cluster / Pillar Page）

不围绕单个关键词写文章，而是按**主题**组织内容。选一个核心主题写支柱页面（Pillar Page），围绕该主题的所有子关键词分别写集群页面（Cluster Pages），支柱页与集群页互相链接。这种"Hub & Spoke"结构能帮 Google 理解你在该领域的权威性，所有子关键词排名整体提升。

#### Topical Map 构建：从关键词列表到主题语义网络

Topic Cluster 是单个集群的 Hub & Spoke。**Topical Map 是全站级别的主题语义网络**——多个集群通过实体关系和内链连接，形成搜索引擎认可的"主题权威"。

**构建步骤**：

1. **确定知识领域**：不是选关键词，是先定位网站属于哪个领域。研究该领域 SERP 的内容类型偏好（百科？博客？电商？PAA 多不多？FS 多不多？）、专家站与普通站的质量差异
2. **按实体维度拆解子主题**：不从关键词工具出发，从实体和概念关系出发。同一领域可按多个维度拆解（如水晶领域：按矿物实体 amethyst/rose quartz、按功效 protection/love/sleep、按场景 meditation/jewelry/cleansing、按文化 chakra/zodiac/tarot），每个维度是一个集群
3. **收集问题**：PAA + Related Searches + 竞品 H2/H3 + Reddit/Quora 讨论
4. **意图聚类**：同一意图的问题合并到同一页面；不同意图拆成独立页面；子问题放进父页面的 FAQ
5. **排发布顺序**：先写低竞争、高相关性的子主题建立 thematic foundation → 等搜索引擎建立"主题理解"后 → 再冲高搜索量核心词

**核心原则**：Topical Map 不是按搜索量排序的关键词列表，而是按语义关系组织的页面网络。**发布顺序比关键词搜索量更重要**——先建覆盖，再冲核心词。

### 5.2 语义 SEO（Semantic SEO）

搜索引擎不只匹配关键词，而是理解实体和关系。写内容时围绕主题覆盖相关实体和术语，不堆砌同一个关键词。如写 "best running shoes"，自然融入 cushioning、pronation、arch support、heel drop 等术语，让 Google 认为你的页面是对该话题的全面覆盖。

### 5.3 关键词→页面类型映射

关键词研究最终必须服务于页面。根据关键词类型决定承接页面：

| 关键词类型 | 推荐页面 | 示例方向 |
|----------|----------|---------|
| 核心产品词 | 产品页 / 分类页 | 产品介绍、参数、应用 |
| 应用场景词 | 解决方案页 | 行业、场景、痛点 |
| How-to 问题词 | 博客 / 指南 | 教程、步骤、注意事项 |
| Best / Top 词 | 榜单页 / 对比页 | 推荐、评测、选择指南 |
| Vs / Alternative 词 | 对比页 | 替代方案、优劣分析 |
| Price / Cost 词 | 定价页 / 商业页 | 成本、报价、套餐 |
### 5.4 倒漏斗策略（Inverted Funnel）：放弃关键词思维

> 核心理念：**不再从搜索量最大的关键词开始，而是从离钱最近的词开始。**

传统 SEO 策略遵循线性漏斗：先做大量 TOFU（顶部漏斗）内容获取流量，再逐步引导到 BOFU（底部漏斗）转化。但这对新站/小团队是致命陷阱——TOFU 竞争最激烈、排名周期最长、离收入最远。

**倒漏斗策略反转这个顺序**：

```
传统漏斗（从上到下）         倒漏斗（从下到上）
─────────────────         ─────────────────
① TOFU: "什么是 CRM"       ① BOFU: "[竞品] alternatives"
② MOFU: "CRM 选购指南"      ② MOFU: "best CRM for small business"
③ BOFU: "[竞品] vs [你]"    ③ TOFU: "what is CRM"
   ↓                           ↓
流量大，转化慢               先赚钱，再扩大流量
排名周期 6-12 月              排名周期 1-3 月
```

#### 为什么倒漏斗有效

| 维度 | 传统漏斗 | 倒漏斗 |
|------|---------|--------|
| **首笔收入时间** | 6-12 个月 | 1-3 个月 |
| **竞争强度** | TOFU 词竞争最激烈 | BOFU 词竞争相对低 |
| **流量质量** | 大量无效流量 | 高意图、高转化率 |
| **内容 ROI** | 投入大、见效慢 | 投入小、见效快 |
| **主题权威** | 需要长期积累 | BOFU 内容天然含产品提及，同时建立权威 |

核心逻辑：BOFU 内容（对比页、替代方案页、定价页）天然适合植入产品信息，转化路径最短。当 BOFU 页面开始排名并产生收入后，积累的主题权威会帮助后续 MOFU 和 TOFU 内容排名更快。

#### 三阶段执行路径

**阶段 1（月 1-3）：BOFU 阵地建设**

| 优先做 | 关键词模式 | 页面类型 |
|--------|-----------|---------|
| 竞品替代页 | "[竞品] alternative", "[竞品] vs [你]" | 对比页 |
| 定价/方案页 | "[品类] pricing", "[品类] cost" | 定价页 |
| 评测/推荐页 | "best [品类] for [场景]" | 榜单页 |

目标：3-5 个高转化 BOFU 页面上线，开始产生询盘/注册。

**阶段 2（月 3-6）：MOFU 扩展**

基于 BOFU 页面积累的权威，扩展到考虑阶段内容：
- 解决方案页（按行业/场景/痛点）
- 案例研究
- 深度指南（"如何选择 X"、"X 选购清单"）

每个 MOFU 页面内链到 BOFU 页面，形成转化闭环。

**阶段 3（月 6+）：TOFU 规模化**

此时网站已有主题权威和稳定的收入流，可以安全地投资高搜索量 TOFU 内容：
- 教程和 How-to 文章
- 行业趋势分析
- 概念解释文章

#### 倒漏斗 + Topic Cluster 的组合策略

倒漏斗不是独立策略，而是 Topic Cluster 的**优先级排序方法**：

```
Topic Cluster 主题选择：
  不要选搜索量最大的主题
  而是选离收入最近的主题

  例：CRM 工具站
  ✗ 第一个 Cluster："什么是客户关系管理"（TOFU，搜索量 10K，但离钱远）
  ✓ 第一个 Cluster："Salesforce alternatives for small business"（BOFU，搜索量 2K，但每个访客价值 $50+）
```

**执行原则**：
1. 每个 Topic Cluster 都从 BOFU 子话题开始建设
2. 逐步扩展到 MOFU 子话题
3. 最后补充 TOFU 子话题
4. 支柱页面（Pillar Page）作为整个 Cluster 的中枢，链接所有子页面

#### 倒漏斗策略的适用场景

| 场景 | 适用度 | 原因 |
|------|--------|------|
| SaaS / 工具站 | ★★★★★ | 竞品替代词搜索量大、转化意图明确 |
| B2B 企业站 | ★★★★☆ | 采购决策周期长，BOFU 内容加速决策 |
| 电商独立站 | ★★★☆☆ | 产品页本身就是 BOFU，需要搭配 Collection 策略 |
| 内容/Affiliate 站 | ★★★☆☆ | 收入依赖流量规模，TOFU 仍重要 |

> 来源：John（英文SEO实战派分享会 2026.06）、Search Atlas（Upside Down Funnel）、Content Harmony（BOFU Keywords）、Semrush（TOFU-MOFU-BOFU Guide）

## 补充：AI Mode 任务地图方法

> **背景**：Google I/O 2026 披露，AI Mode 平均查询长度是传统搜索的 3 倍，用户开始用自然语言描述完整任务而非输入短关键词。关键词研究需要从"词量分析"升级为"任务链拆解"。
>
> **来源**：Google I/O 2026（2026-05-19）+ Lois/海外SEO与ASO 分析
>
> **适用场景**：当目标用户的搜索行为从"best X"类短查询转向"我是 Y，需要 Z，预算 W，帮我比较并告诉我怎么落地"类长查询时，使用本方法补充传统关键词研究。

### 核心思路

传统关键词研究看到的是用户被搜索框训练后的压缩表达。AI Mode 让用户可以说完整需求，关键词表需要升级为**任务地图**——不只是覆盖搜索词，而是覆盖用户从提出问题到完成行动的完整链路。

### 五层任务拆解框架

对每个核心主题/话题集群，补充以下五层分析：

| 层次 | 要回答的问题 | 对应内容方向 |
|------|------------|------------|
| **① 用户任务** | 用户要完成什么决定、解决什么问题？ | 不是搜什么词，而是要做什么 |
| **② 约束条件** | 用户有什么限制？预算、地区、团队规模、行业、已有工具、合规要求 | 场景化限定内容 |
| **③ 比较标准** | 用户用什么维度比较方案？价格、学习成本、集成能力、迁移成本、售后 | 对比页、评测页 |
| **④ 下一步动作** | 用户确定方向后会做什么？下载、试用、咨询、购买、部署 | CTA 页面、工具页、模板页 |
| **⑤ 追问链** | 用户可能继续追问什么？替代方案、失败原因、最佳实践、真实案例 | FAQ、案例页、教程页 |

### 与传统关键词研究的关系

任务地图**不替代**传统关键词研究，而是**补充**一个维度：

```
传统方法：关键词 → 搜索量/KD → 意图分类 → 页面类型 → 内容生产
任务地图：核心主题 → 五层拆解 → 识别内容缺口 → 补充到内容计划
```

- 传统方法回答"哪些词值得做"
- 任务地图回答"围绕这个词，用户需要什么完整的信息链"

**实操建议**：在第五阶段（关键词组织与内容规划）中，对 P0/P1 核心主题执行任务地图拆解，检查现有内容计划是否覆盖了五层需求中的缺口。

### 任务地图模板

每个核心主题的任务地图可用以下格式记录：

```markdown
## 任务地图：[核心主题]

### ① 用户任务
- 核心任务：[用户要完成什么]
- 典型完整查询示例：[模仿 AI Mode 用户的自然语言表达]

### ② 约束条件
- 常见约束：[预算/地区/规模/行业等]

### ③ 比较标准
- 决策维度：[用户用什么标准做选择]

### ④ 下一步动作
- 转化路径：[用户确定方向后做什么]

### ⑤ 追问链
- 后续问题：[用户可能继续追问的方向]

### 内容缺口
- [ ] 现有计划中缺少的页面/内容类型
```

## 附录A：关键词主表字段标准

> `SEMrush-Seed-Keywords` Google Sheet 中的 `Seed-Master` 工作表是项目唯一关键词主表。`Entity / Subtopic / Content Role` 是通用字段，`Subtopic` 的具体分类值由每个 Topic Pillar 自己定义，不得跨主题硬套。

| 字段 | 用途 |
|------|------|
| Keyword | 关键词原文 |
| Normalized Keyword | 去重键（全小写、去首尾空格、多空格合并为单空格，不做词形归并和同义词归并） |
| Volume | 搜索量 |
| KD | 关键词难度 |
| CPC | 商业价值参考 |
| Intent | 搜索意图 |
| Topic Pillar | 主题归属（Crystals / Tarot / Dreams / Angel Numbers / Astrology / Spirituality / Numerology 等） |
| Entity | 关键词明确指向的对象/实体，例如 Amethyst、Root Chakra、Tarot Spread、Dream Symbol |
| Subtopic | 主题内部内容角度；字段名通用，但分类值由各 Topic Pillar 自定义 |
| Content Role | 内容承接角色，例如 Main Article、H2 Section、Child Article Candidate、Product / Category Page、Local SEO Candidate |
| Business Relevance | 业务相关性（High / Medium / Low） |
| Recommended Page Type | 推荐承接页面类型 |
| Priority | 关键词优先级（高 / 中 / 低 / 暂存） |
| Status | 状态（Candidate / Approved / In Content Plan / Skip） |
| Source Type | 来源类型（seed_expansion / competitor_top_keyword / manual/AI_candidate） |
| Source Detail | 来源说明 |
| Competitor Proof | 最佳竞品证据域名 |
| Proof URL | 最佳竞品证据 URL |
| Proof Traffic | 最佳竞品证据流量 |
| Proof Competitor | 最佳竞品证据竞品名 |
| Notes | 备注 |

### 附录A-1：竞品验证视图字段标准

> 竞品验证视图（工具实现名：Keyword-Page-Proof）是验证视图，不是关键词主表。它用于判断某个关键词是否已有竞品页面实际拿到流量，并给关键词主表补证据。

| 字段 | 来源 | 说明 |
|------|------|------|
| Keyword | `TopKeywords_All` | 竞品排名关键词 |
| Competitor Domain | `TopKeywords_All` / `TopPages_All` | 关联键之一 |
| Ranking URL | `TopKeywords_All` | 关键词排名URL |
| Normalized URL | 两个 All 表 | 关联键之一 |
| Search Volume | `TopKeywords_All` | 搜索量 |
| Keyword Traffic | `TopKeywords_All` | 该关键词估算流量 |
| Page Traffic | `TopPages_All` | 对应页面估算流量 |
| Proof Level | 计算字段 | strong / medium / weak / no_match |
