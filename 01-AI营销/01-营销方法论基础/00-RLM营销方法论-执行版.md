## 1. RLM是什么

**RLM营销方法论 = 用AI的分而治之思想 + 从网站架构到内容全面超越竞品的策略，实现营销全流程自动化**

### 核心思路：模仿 + 超越 + 差异化

核心逻辑：先看市场上谁做得好 → 判断他们为什么好 → 找到他们没做好的地方 → 设计更快、更系统、更低成本的方案。


默认服务于**海外市场**（B2B外贸、B2C独立站、SaaS/工具站、内容站/Affiliate）。

### 使用原则：让AI泼冷水

向AI提问时，不要问"这想法怎么样"，要问"这想法最可能在哪里崩"。让AI泼冷水，直到泼不出来。AI最大的危险不是犯错，而是用同样的热情帮一个烂想法做到产品级。


## 2. AI执行纪律

**这是执行版中最重要的章节。违反以下任何一条都算输出不合格。**

1. **不编造数据**：没有来源的数据必须标注"待验证"，不得虚构搜索量、流量、KD、CPC。
2. **不凭空做用户画像**：用户画像必须来自竞品页面、关键词意图、FAQ、评论、产品属性或行业场景。依据不足时标注"初步推测"。
3. **不做无依据预测**：除非用户提供历史数据或计算模型，否则不输出流量、收入、排名、转化率的具体预测，不输出伪精确的发布时间表。
4. **不输出空泛策略**：模仿、超越、差异化必须具体到竞品、页面、内容、结构、关键词或转化路径。禁止"提升品牌力""优化用户体验"等无动作表述。RLM主流程围绕SEO内容展开，付费广告、社媒、邮件等渠道按需扩展。
5. **每次输出必须有明确文件名和路径**：输出不仅要解决问题，还要明确写到哪个文件。模糊的"输出一份分析"不算合格。
6. **输出即动作**：每阶段结束时的交付物必须能作为下一步输入；若仅停留在分析，补任务清单或验收项。
7. **素材有源，语气有参照**：进入 Brief、页面文案或正式内容生成前，必须声明素材证据来源和语气参考；具体字段写入模板库或单页依据包。
8. **不一次性输出大而全**：先写 spec（规格/框架），拆成小块，逐步实现，持续测试，每步 review。AI 越强，流程反而越要保守。（Addy Osmani）

**分析数据表**（进入 1B 时创建，导入时按标准列精简）：

| 表格名称 | 用途 | 标准列（保留） | Semrush 导出时删除的列 |
|------|------|--------------|----------------------|
| SEMrush-Seed-Keywords | Seed-* 主题关键词池 | Keyword / 中文 / Entity / Subtopic / Content Role / Volume / KD / CPC / Number of Results / Intent | Competitive Density / SERP Features / Updated / Priority / Recommended Page Type |
| SEMrush-Top-Pages | 页面级SEO价值 | URL / Traffic(%) / Traffic / Top Keyword / Primary Intent / LLM Prompts | Adwords Positions / Number of Keywords / 5×Positions by Intent / 5×Traffic by Intent / Traffic Change / Answer Engines |
| SEMrush-Top-Keywords | 关键词排名与排名URL | Keyword / Search Volume / Keyword Difficulty / CPC / URL / Traffic / Traffic(%) / Number of Results / Keyword Intents | Position / Competition / Previous Position / Traffic Cost / Trends / Timestamp / SERP Features by Keyword / Position Type |

> `SEMrush-Seed-Keywords` 中每个 `Seed-*` 工作表代表一个 Topic Pillar。`Entity / Subtopic / Content Role` 是通用字段，但 `Subtopic` 的可选值由各 Topic Pillar 自己定义，不能把 Crystals 的 Meaning/Benefits/Properties 等分类硬套到 Chakra、Tarot、Dreams 等主题。`Seed-Master v1` 由 1B 轨道B 构建，后续竞品证据增强由 1B 轨道D 承接。

**数据规则**：
- 项目简报记录所有 Google Sheets 链接，包含标准列和删除列说明
- **AI 接手时先通过 Google Workspace MCP 搜索谷歌对应项目文件夹、遍历本地对应项目文件夹，与项目简报索引比对，不一致时以实际为准更新索引**
- 竞品 Top Pages / Top Keywords 的汇总、筛选和验证细则见 `02-自动化工具库/02-竞品研究工具/1B数据处理工具手册.md` 的轨道C/D
- **`1A-竞品清单.md` 只放链接和筛选口径**，不重复搬运表格数据

## 3.5 项目规模分级与执行深度闸门

> **核心原则**：深度档不是错误，轻量档也不是偷懒。它们是不同项目阶段的合理配置。先判断项目档位，再决定哪些分析必须完整、哪些只保留证据索引、哪些延后到对应批次。

### 四变量打档

项目启动时，用以下四个变量判断执行深度：

| 变量 | 选项 | 影响 |
|------|------|------|
| **市场竞争强度** | 低（蓝海/新站） / 中（垂直站） / 高（红海/大行业） | 竞品拆解深度、关键词清洗范围 |
| **网站类型** | B2C独立站 / B2B外贸站 / SaaS工具站 / 内容站/Affiliate | 内容重点、页面结构复杂度 |
| **首次上线目标** | 快速验证 / SEO起量 / 品牌正式发布 | 分析和内容的优先级 |
| **资源约束** | 单人AI执行 / 小团队 / 专项团队 | 步骤合并、文档粒度 |

### 三档执行深度

根据四变量组合，选择轻量/标准/深度三档之一。下表定义每档在各模块的具体配置：

| 模块 | 轻量档 | 标准档 | 深度档 |
|------|--------|--------|--------|
| **竞品深拆** | 5-10 家核心竞品完整 1D；参考竞品只做 Top Pages/Sitemap 摘要；灵感竞品只提取可复用策略进策略灵感库 | 10-20 家完整 1D | 20-30+ 家全部完整 1D |
| **关键词清洗** | 3-5 个核心 Pillar 完整清洗+证据增强+Page Type映射；其余Pillar保留原始导出，执行到对应批次时再清洗 | 8-10 个 Pillar | 全量清洗 |
| **Step 1 文档** | 1B→1D→1E+1F合并→1G+1H合并 = 4-5个输出 | 保持6个左右 | 完整1A-1H |
| **页面依据包** | 模板级依据包+实例页最小差异记录（canonical/主关键词/参考URL异常/图片独立/特殊风险）；单页型页面（Home/About/FAQ）仍用完整依据包 | 重点页完整依据包 | 全部完整依据包 |
| **内容清单** | 首批50篇完整字段 + 候选池（关键词/Volume/意图/Topic，不补执行字段） | 首批100篇完整 | 全量字段 |
| **竞品段落采集** | 仅高竞争内容类型（对比/教程/Best/Alternative）；信息型百科页跳过，用差异化模板+少量SERP结构复核（详细分级见步骤 3.4 + §3.5 触发表） | 按内容类型分级 | 全量执行 |

### 竞品三层分级

1A 发现的所有竞品在进入 1D 之前必须分级，不同层级决定拆解深度：

| 竞品层级 | 判定标准 | 拆解深度 | 输出 |
|---------|---------|---------|------|
| **核心竞品** | 直接竞品（卖同类产品）+ 高流量内容竞品 | 完整 1D 深拆 | `1D-竞品深度拆解/` 正式文档 |
| **参考竞品** | 同大类不同细分、有可参考结构但非直接竞争 | Top Pages + Sitemap + 结构摘要 | 不写 1D 文档，关键发现直接进入 1E/1F |
| **灵感竞品** | 跨界竞品，只有个别策略值得借鉴 | 只提取可复用策略点 | 进入 `1H-策略清单.md` 的灵感来源标注，不写拆解文档 |

> **闸门规则**：1D 只为核心竞品写正式拆解文档。参考竞品和灵感竞品的分析结果直接汇入 1E/1F/1H，不单独成文。

### Pillar Priority Gate

1B 轨道B 清洗 Seed-* 时，按以下优先级分批处理：

| 优先级 | 判定标准 | 处理方式 |
|--------|---------|---------|
| **P0 Pillar** | 核心产品/服务直接相关的主题（如水晶站的 Crystals、Chakra） | 立即完整清洗 + 证据增强 + Page Type 映射 |
| **P1 Pillar** | 上线后 1-3 月内计划覆盖的主题（如 Angel Numbers、Moon Phases） | 完整清洗但证据增强可延后到执行批次前 |
| **P2/P3 Pillar** | 远期或差异化主题（如 Palmistry、Feng Shui） | 保留 SEMrush 原始导出，不做清洗和证据增强，等执行到对应批次时再处理 |

### 模板级依据包 vs 页面级依据包

2C 依据包按页面类型分两种：

| 类型 | 适用场景 | 内容 |
|------|---------|------|
| **页面级依据包** | 单页型页面（Home/About/FAQ/Contact/Ethical Sourcing） | 完整依据包：核心参考竞品、真实参考URL、上游来源索引、采用/规避/差异化点、素材证据声明 |
| **模板级依据包** | 模板矩阵页（Crystal Single ×15、Intention Category ×6 等） | 先定义模板 canonical：字段结构、参考竞品、可复用语气、图片规则。实例页只填最小差异记录：canonical URL、主关键词、参考URL异常、图片是否独立、特殊风险 |

> **实例页最小差异记录不可省略**：即使使用模板级依据包，每个实例页仍必须有差异记录，不能回到无依据批量生成。

### 内容清单分两层

| 层级 | 数量 | 字段深度 |
|------|------|---------|
| **执行清单** | 首批 50 篇（轻量档）/ 100 篇（标准档）/ 全量（深度档） | 完整字段：目标关键词、Volume、KD、CPC、意图、内容类型、对应页面/URL、竞品验证、发布节奏、Brief 状态 |
| **候选池** | 其余所有选题 | 轻量字段：关键词、Volume、意图、Page Type、Topic Pillar。不补竞品验证、URL、内链等执行字段 |

### 竞品段落采集分级触发

竞品段落素材采集不是所有内容都做，按内容类型和竞争强度分级：

| 内容类型 | 竞争强度 | 是否执行采集 | 替代方案 |
|---------|---------|-------------|---------|
| 百科/科普（xxx meaning, what is xxx） | 低 | **跳过** | 用差异化模板（如三视角模型）+ 少量 SERP 结构复核（只看 H2 大纲，不做段落评分） |
| 场景应用（crystals for xxx, best xxx for yyy） | 中-高 | **执行** | 完整 Top 10-20 段落采集 + 评分 |
| 对比/选购（xxx vs yyy, how to choose xxx） | 高 | **执行** | 完整段落采集 + 评分 |
| 教程/How-to（how to cleanse xxx, how to use xxx） | 中 | **可选** | 视竞品内容同质化程度决定 |

> **判断标准不是"项目大小"，而是"该内容类型的竞争强度"。** 小项目里也可能有高竞争内容需要段落采集，大项目里也有低竞争百科页可以跳过。具体执行位置见步骤 3.4。

### 档位判定示例

| 项目类型 | 推荐档位 | 四变量 |
|---------|---------|--------|
| B2C 小站（水晶手链、小众饰品） | **轻量档** | 低竞争 × B2C独立站 × 快速验证 × 单人AI |
| B2B 外贸站（机械配件、化工原料） | **深度档** | 高竞争 × B2B外贸 × SEO起量 × 小团队 |

| 内容站/Affiliate（测评、榜单） | **标准档** | 中-高竞争 × 内容站 × SEO起量 × 单人AI |

### 步骤1：竞品查找与分析（知道对手是谁 + 知道内容怎么做）

> **执行深度**：项目启动时按 §3.5 判定档位，决定竞品拆解深度和关键词清洗范围。不要默认全量深拆。

**目标**：找到所有相关竞争对手，按 §3.5 竞品三层分级决定拆解深度，建立数据基础，为网站规划和内容策略提供依据。

步骤1是整个流程的数据基础，内部分8个子阶段：

```
1A: 竞品发现（人工）
    ├── 轨道A: Google 搜索 + 排除语法 → 筛选竞品
    └── 轨道B: SEMrush Organic Competitors → 补充遗漏竞品
1B: 多源数据采集（多条轨道可并行）
    ├── 轨道A: 竞品清单数据 → SEMrush AS/Organic Traffic/截图 + traffic.cv月访问量/渠道拆分
    ├── 轨道B: Seed-* 主题关键词池 → SEMrush 相关关键词导入与清洗 → Seed-Master v1
    ├── 轨道C: 竞品SEO明细数据 → SEMrush Top Pages（页面级流量证据）+ SEMrush Top Keywords（关键词级排名证据）→ All 表
    ├── 轨道D: Seed-Master 竞品证据增强 → TopKeywords_All + TopPages_All 补充关键词/页面证据
    └── 轨道E: Sitemap解析 → 页面清单/URL结构/内容类型分布 → 竞品页面报告
1C: 关键词研究       → 读取 Seed-Master 和竞品证据结果，做主题集群/意图/推荐页面类型/优先级分析
1D: 竞品深度拆解     → 核心输入（Top Pages + Top Keywords + Sitemap + Seed-Master + 1B 轨道D 竞品证据增强结果）+ 辅助输入（竞品清单汇总）
                     → 每个竞品逐个深度拆解，回答：X靠什么拿流量、X建了什么、X覆盖/遗漏了什么
1E: 网站结构分析（综合1B+1C+1D做结构对比）→ 可进入步骤2
1F: 内容策略分析     → 可进入步骤3
1G: 用户画像
1H: 策略清单         → 汇入步骤2和步骤3
```

#### 1A 竞品发现（人工）

**目标**：通过人工搜索和SEMrush双轨道发现竞品，将竞品填入启动阶段已创建的竞品清单表。

##### 轨道A：Google 搜索

**核心任务**：
- 使用 Google 搜索种子关键词，通过排除语法过滤大平台
- 人工浏览搜索结果，筛选竞品域名

**搜索排除语法**（直接复制使用）：
```
关键词 -amazon.com -ebay.com -etsy.com -walmart.com -pinterest.com -reddit.com -youtube.com -facebook.com -tiktok.com -wikipedia.org -quora.com -target.com -aliexpress.com -alibaba.com
```

##### 轨道B：SEMrush Organic Competitors（人工操作）

**核心任务**：
- 打开 SEMrush，查看 Organic Competitors 报告

> 1A 完成后可以暂停，不强制立即进入 1B。

#### 1B 多源数据采集（多条轨道可并行）

从 1A 竞品清单出发，分多条轨道并行采集数据。详细操作见 `02-自动化工具库/02-竞品研究工具/1B数据处理工具手册.md`。

##### 轨道A：竞品清单数据

逐个竞品填充 3.1 竞品清单中的对应列：

- SEMrush → AS（C 列）、Organic Traffic（D 列）、Distribution 截图（E 列）
- traffic.cv → 月访问量（F 列）、渠道拆分（G-L 列）
- Wappalyzer / 人工 → 建站平台（M 列）


##### 轨道B：Seed-* 主题关键词池与 Seed-Master v1

按主题从 SEMrush 导出相关关键词，写入 `SEMrush-Seed-Keywords` 中对应的 `Seed-*` 工作表。

- 每个 `Seed-*` 工作表代表一个 Topic Pillar，例如 `Seed-Crystals`、`Seed-Chakra`、`Seed-Dreams`、`Seed-Angel-Numbers`
- 空的 `Seed-*` 工作表也要保留，用于后续手动导入对应主题关键词，不得因为暂无数据而删除
- **Pillar Priority Gate（§3.5）**：清洗 Seed-* 前先给所有 Pillar 分级（P0/P1/P2-P3）。P0 Pillar 立即完整清洗 + 证据增强 + Page Type 映射；P1 Pillar 完整清洗但证据增强可延后；P2/P3 Pillar 保留 SEMrush 原始导出，不清洗不增强，等执行到对应批次时再处理。不要对所有 Pillar 默认做完整清洗。
- 清洗四步法：① 结构性噪音（平台词/社媒词/乱码/Volume 异常，纯规则可判断）→ ② 补全部语义字段（中文/Entity/Subtopic/Content Role/Intent）→ ③ 语义筛选（品牌词/实体错位/近重复/性价比，依赖 Entity/Intent/Content Role 判断）→ ④ AI 标记 → 人工确认；详细规则见 1B 手册
- 平台词/社媒词可按规则删除；第三方品牌词进入语义筛选步骤，默认不做竞品品牌词和 `vs` 内容，除非负责人批准
- `Seed-*` 应保持小而准；清洗后的工作表应默认整表可进入 `Seed-Master v1`，不把无效词留到主表里再依赖状态字段过滤
- `Suggested Action / Reason / Confidence / Reviewed` 可作为清洗辅助列保留在 `Seed-*`，但不进入 `Seed-Master`；不要额外新增中文解释列，`中文` 列只做关键词字面翻译
- `near me` 等本地意图词不要在数据清洗阶段误删；具体承接策略属于项目内容策略，不在 1B 展开
- 标准字段：Keyword / 中文 / Entity / Subtopic / Content Role / Volume / KD / CPC / Number of Results / Intent
- `Number of Results` 必须保留
- 清洗后的 `Seed-*` 去重并继承 Topic Pillar 后，构建 `Seed-Master v1`

##### 轨道C：竞品SEO明细数据

从 SEMrush 导出竞品的 Top Pages 和 Top Keywords，分别写入 `SEMrush-Top-Pages` 和 `SEMrush-Top-Keywords`，每个竞品一个工作表。

- 本轨道只定义原始竞品数据如何采集和标准列保留
- 本轨道生成 `TopKeywords_All` 和 `TopPages_All`
- 轨道C数据被轨道D、1C、1D、1E 复用，用于判断竞品关键词、页面流量和页面类型

> 4张核心表格的结构定义（标准列和删除列）见 §3.1"Google Sheets 表格结构"。

##### 轨道D：Seed-Master 竞品证据增强

用 `TopKeywords_All` 先匹配 `Seed-Master v1`，找出已有关键词的竞品关键词证据；再用命中的 URL 去 `TopPages_All` 补页面级价值证据。

- TopKeywords_All 补关键词证据：竞品 URL、关键词流量、命中竞品数量等
- TopPages_All 补页面证据：页面流量、页面主要意图、页面类型线索等
- 未命中 Seed-Master 的竞品关键词按未命中率处理：<20% 仅记录；20%-40% 且集中 1-2 主题则回轨道B 补 Seed-*；>40% 或高度分散则人工筛选候选视图再决定

##### 轨道E：Sitemap解析

调用 `sitemap-mcp-server` MCP 解析每个竞品的 sitemap.xml，获取页面清单、URL 路径模式和内容类型分布，结果存放在项目 `1B-sitemap解析/` 目录下。详细工具说明见 `02-自动化工具库/02-竞品研究工具/1B数据处理工具手册.md` 的“轨道E：Sitemap解析”。

> Sitemap 默认输出到 `1B-sitemap解析/` 和 `1B-sitemap解析报告.md`；只有 URL 清单很大、需要频繁筛选/排序时，才可另建 `Sitemap-Pages` 电子表格。

**关键输出（1B 整体）**：更新后的竞品清单 + `Seed-Master v1` + SEMrush竞品SEO明细（Top Pages/Top Keywords，每个竞品一个工作表）+ `TopKeywords_All` + `TopPages_All` + Seed-Master 竞品证据增强结果 + Sitemap解析报告/页面清单

#### 1C 关键词研究

**核心任务**：
- 读取 1B 产出的 `Seed-Master` 及其竞品证据增强结果
- 不能把工具原始导出直接当作最终关键词主表；1C 从 1B 数据底座开始做研究分析
- 对关键词进行主题集群、搜索意图、商业价值、难度和推荐页面类型标注
- 识别哪些关键词应由产品类目页、产品页、博客、指南页、工具页或信任页承接
- **Pillar Priority Gate（§3.5）**：1C 分析范围与 1B 轨道B 一致——P0 Pillar 立即完整分析；P1 Pillar 做基础分组但详细映射可延后；P2/P3 Pillar 只保留原始数据索引，不做主题集群和页面类型映射。优先服务首批执行批次的关键词需求。
- 详细流程参见 `01-关键词研究方法论.md`

**关键输出**：`Seed-Master` 关键词主表（1B 轨道B/D 提供，1C 增强标注）+ `关键词分组与优先级.md`。竞品证据由 1B 轨道D 提供。

**工具链**：
- 数据获取: SEMrush 导出 CSV → AI 分析

> 1C 不是最终内容策略，只是读取 1B 产出的 Seed-Master 和竞品证据结果，把关键词机会变成可筛选、可分组、可映射页面类型的数据底座。

#### 1D 竞品深度拆解

**目标**：对每个竞品进行深度拆解，回答三件事：X 靠什么拿流量、X 站内建了什么、X 相对市场需求覆盖了什么和遗漏了什么。

**数据输入**（每个竞品）：

1D 数据输入不是按 1B/1C 分层使用，而是按分析问题组合使用。每个竞品深度拆解必须同时使用竞品自身表现数据、站点结构数据、市场关键词参照数据和竞品验证数据。站点级概览数据只用于判断竞品权重和背景，不作为内容/页面机会判断的核心依据。

**核心输入**：
- 该竞品的 SEMrush Top Pages（1B 轨道C）：判断高流量页面、页面类型、流量集中度
- 该竞品的 SEMrush Top Keywords（1B 轨道C）：判断排名关键词、排名 URL、关键词意图和页面承接关系
- 该竞品的 Sitemap 解析结果（1B 轨道E）：判断网站结构、页面清单、URL 规则、内容类型分布
- Seed-Master 关键词主表（1B 轨道B/D 构建，1C 进一步标注和研究）
- 1B 轨道D 的竞品证据增强结果（TopKeywords_All / TopPages_All 回填的竞品关键词/页面证据）

**辅助输入**：
- 竞品清单表中的 AS、Organic Traffic、月访问量、竞品类型等汇总信息（1B 轨道A）：仅用于判断竞品权重、参考优先级和背景定位
- 页面截图、渠道拆分、站点级流量概览：仅在分析商业模式、转化路径或内容体验时按需引用，不作为 1D 的必需核心数据

**1D 只回答**：
- 竞品 X 做了哪些内容/页面？
- 哪些页面和关键词已经拿到流量？
- X 的内容类型、页面结构、关键词覆盖有什么强项和缺口？
- 哪些做法值得进入 1H 候选策略池？

**1D 不回答**：
- 我们最终网站怎么建
- 我们最终写哪些内容
- 我们最终模仿/超越哪些策略

**核心任务**：
- 按上述数据输入组合分析，每个竞品逐个深度拆解，按类型侧重不同：
  - **直接竞品**：全面拆解（产品、定价、页面、转化路径、网站结构）
  - **内容竞品**：侧重内容策略和关键词覆盖（他们写了什么、排名了什么、流量从哪来）
  - **间接竞品**：侧重差异化参考（他们的定位和打法与你有什么不同）
- **竞品分级闸门（§3.5）**：只有核心竞品写完整 1D 拆解文档。参考竞品只做 Top Pages/Sitemap 结构摘要（关键发现直接进入 1E/1F）。灵感竞品只提取可复用策略点（进入 1H 灵感来源标注）。不要对所有发现竞品都写拆解文档。

**关键输出**：每个竞品的深度拆解文档（存放在 `1D-竞品深度拆解/` 目录）

> 1D 完成后进入 1E 网站结构分析、1F 内容策略分析。

#### 1E 网站结构分析（知道竞品网站怎么建）

> **定位**：1E 是综合分析层，基于 1B 采集的 sitemap/SEMrush 页面数据 + 1C 关键词研究 + 1D 深度拆解结论，进行跨竞品的结构对比分析。它只回答"竞品怎么建、哪些结构被验证、哪些页面类型高频/高价值"，不决定我们最终网站怎么建。

**核心任务**：

1. **网站结构分析** — 导航结构、页面层级、布局优劣（综合 1B sitemap/Top Pages 数据 + 1D 深度拆解结论）
2. **产品分类模式识别** — 评估竞品分类方式优缺点

   | 竞品 | 分类方式 | 分类维度 | 优点 | 缺点 |
   |------|---------|---------|------|------|
   | 竞品X | [实际分类] | [按什么维度分] | [哪里好] | [哪里不足] |

3. **URL结构分析** — 命名规则、层级深度、关键词出现
4. **类目关键词确认** — 从竞品URL和类目名中提取核心词，确认搜索趋势（仅类目级别，不做完整关键词研究）

**关键输出**：竞品结构对比 + 类目关键词方向 + **架构层候选模仿点**（最终策略归入 1H，最终网站方案归入 02）

> **边界约束**：1E 只能输出竞品结构事实、页面类型频率、SEO页面价值和候选模仿点，不能输出"我们的网站结构方案"。网站结构、URL规则、页面清单和执行批次只能在步骤2 `02-网站规划/` 中决定。

**工具链**：
- Skills: `frontend-design` — 分析竞品目录结构的 UX 优劣
- MCP: `webReader` — 按需复核竞品具体页面结构（轻量复核，不重新抓取全站 sitemap）

> 1E + 1H 完成后可进入步骤2网站规划与建设。

#### 1F 内容策略分析

**输入来源**：1D 单竞品深度拆解文档（内容类型、内容优势、内容缺口、§9 候选策略点）+ Seed-Master 关键词主表（1B 轨道B/D 提供）+ 1B 轨道D 竞品证据增强结果 + 必要的核心页面轻量复核。

**核心任务**：
- 内容类型分析（博客/案例/教程/指南）
- 竞品内容优势与缺口识别
- 竞品共同弱点识别（找到 90%+ 竞品都没做的事情 = 最大差异化机会）

**关键输出**：内容策略分析（每个竞品的内容类型、优势、缺口、共同弱点）

> **边界约束**：1F 输出内容缺口和候选机会时使用"候选方向"语气（如"候选方向（→ 1H 策略清单决策）"），不使用"我们的策略""我们将"等决策语气。最终内容策略决策归入步骤3。

#### 1G 用户画像

**输入来源**：Seed-Master 关键词主表中的意图、问题和主题集群（1B 轨道B/D 提供）+ 1B 轨道D 竞品证据增强结果 + Top Pages/Top Keywords 暴露的页面需求 + 1D/1F 中反复出现的痛点、场景、FAQ、评论和转化路径观察。

**核心任务**（基于上述输入推导，不凭空想象）：
- 目标用户画像
- 核心痛点
- 用户需求

**关键输出**：用户画像（每个画像可追溯到关键词意图、竞品页面、FAQ 或评论）

#### 1H 策略清单

**核心任务** — 必须独立成章：
- 汇总来自 1D §9、1E、1F 的候选策略点，并结合 1G 用户画像去重、排序、映射到页面或内容
- **模仿清单**：格式 "模仿X: [竞品]的[具体做法]"
- **超越清单**：格式 "超越X: [竞品不足的具体描述]"
- **差异化清单**：格式 "差异化X: [竞品空白的具体机会]"

**关键输出**：策略清单 → 汇入步骤2（网站规划与建设）和步骤3（内容策略）

> **边界约束**：1H 不重复搬运竞品原始数据（导航、截图、URL等），只吸收来自 1D §9、1E、1F 的候选策略点和 1G 用户画像，进行去重、排序、映射。1H 是唯一完整的策略中枢。

### 步骤2：网站规划与建设（把网站建起来）

**目标**：基于竞品结构分析和关键词方向，规划并搭建 WordPress 网站的核心页面

步骤2内部分4个子阶段：

```
2A: 架构规划     → 5个规划文档 + 竞品页面频率矩阵（页面候选池）
2B: 页面决策     → 决策表：做/不做/批次 + 竞品参考来源索引
2C: 逐页生成     → 每个页面：页面内容指南 → 单页依据包（复用1B/1C/1D/1E/1F/1H + 轻量复核URL）→ 写内容 → 图片资产清单/prompt → 构建脚本
2D: 上传与验收   → 逐页：图片生成/上传/URL回写 → 脚本上传 → 前端验收
```

> **步骤2核心规则**：2B 是 gate，未通过决策的页面不能进入 2C。2C 不重新深度拆解竞品，只读取上游结论并对核心参考 URL 做轻量复核。

#### 2A 架构规划

**核心任务**：

1. **网站结构设计** — 参考(模仿)竞品好的结构，优化(超越)竞品不足
2. **URL结构与内部链接** — 基于类目关键词方向设计
3. **产品类目规划** — 基于关键词命名，比竞品更清晰合理
4. **核心页面内容规划** — 每个页面标注：参考哪家竞品、模仿什么、超越什么、差异化创新
5. **产品页面规划** — 产品标题、描述、属性
6. **竞品页面频率矩阵**（2B 的直接输入） — 从 1E 竞品结构分析中提炼页面候选池，格式如下：

   | 页面类型 | 出现竞品数/总竞品数 | 普及率 | 出现的竞品 | 不出现的竞品 | 1E来源索引 | 备注 |
   |---------|-------------------|-------|-----------|------------|----------|------|

   > **边界**：原始证据（竞品导航截图、完整URL列表、详细结构图）留在 1E。2A 的频率矩阵只保留决策所需的摘要和引用（如"详见 1E §2.3"），不复制 1E 原始数据。

**关键输出**：`02-网站规划/` 下5个规划文档 + **竞品页面频率矩阵**（作为 2B 页面决策的直接输入）

#### 2B 页面决策（gate）

**目标**：基于 2A 的竞品频率矩阵 + 1H 策略清单，决定实际要做哪些页面、什么顺序。

**2B 是步骤2的 gate。** 没有通过 2B 决策的页面不能进入 2C 生成。

**强制输出 — 页面决策表**：

| 页面名称 | 页面类型 | 竞品覆盖率 | 对应竞品页面URL | 用户意图 | 1H策略匹配 | SEO/商业价值 | 实现复杂度 | 决策状态 | 执行批次 | 候选批次 | 不做/延后原因 |
|---------|---------|-----------|---------------|---------|-----------|------------|-----------|---------|---------|---------|------------|

> **字段规则**：
> - `决策状态` 只允许使用：`✅ 批准执行`、`⏸ 候选延后`、`❌ 不做`。
> - 只有 `✅ 批准执行` 的独立页面才能填写 B1a/B1b/B2/B3 等 `执行批次` 并进入 2C；平台模板/动态组件承接项只能填写承接方式，不进入独立页面上传数量。
> - `⏸ 候选延后` 页面只能填写 `候选批次`，不能进入 2C；后续必须经过 2B 复审才能激活。
> - `对应竞品页面URL` 可以在 2B 阶段使用 URL 模板（如 `/products/[product]`），但进入 2C 前必须替换为实际可抓取 URL。
> - `核心参考竞品` 由 2B 选定：业务必要/低复杂度页面至少 2 家，标准核心页面建议 3-5 家，高价值SEO模板或SERP竞争强页面可扩展到 5-10 家。这里的数量指"单页参考来源"，不是要求在 2C 重新深度拆解。
> - `平台模板/动态组件承接` 必须单独标注：如 WooCommerce 产品详情页、博客归档页、产品列表区、文章列表区等由主题、CMS、短代码、query 或 REST 动态组件承接的对象，不计入独立页面上传数量；处理方式写为"模板承接/动态组件承接"，不能混入 B1a/B1b 的独立 page 计数。
> - 每个核心参考竞品必须能追溯到 1D 深度拆解文档、1E/1F 汇总结论或 1H 策略条目；没有上游依据的临时参考不能直接支撑页面结构和文案。

**决策规则**：

| 条件 | 决策 | 说明 |
|------|------|------|
| 竞品覆盖率 75%+ | **默认必须做** | 大多数竞品都有，不做等于缺失基础；除非与项目定位明显冲突 |
| 竞品覆盖率 25-75% | **进入判断** | 看 1H 策略价值和资源成本，决定"应该做 / 差异化做 / 延后做" |
| 竞品覆盖率 <25% | **默认不做/远期** | 除非它是 1H 明确要求的差异化页面（如 Crystal Quiz 只有 25% 竞品有，但 1H 明确要求做） |
| About / Contact / FAQ / Shipping / Returns | **业务必要页** | 不完全受频率阈值支配，属于网站信任基础，即使部分竞品没有也必须做 |

**执行批次排序原则**：

- B1a：上线骨架页（首页、About、FAQ、Contact、产品详情模板/承接规则、核心分类入口等）— 网站存在与信任基础
- B1b：核心 SEO 模板（百科单品、按意图分类等可模板化矩阵）— 流量基础，但不阻塞 B1a 上线
- B2：上线后 1 个月内扩展页（更多分类、博客、轻量互动或转化页）— 扩展转化和内容覆盖
- B3：3 个月内差异化页（工具、资源、周期性内容）— 竞争优势验证
- B4：远期或模板工厂页（大规模矩阵、供应链依赖、蓝海验证不足页面）— 需要复审后再执行
- 每批内的页面按重要性、依赖关系和实现复杂度排序

**关键输出**：`02-网站规划/页面决策表.md` — 确定批准执行的页面清单 + 候选延后/不做页面及原因 + 执行批次 + 候选批次

#### 2C 逐页生成

**目标**：按 2B 确定的页面清单和执行批次，逐个页面完成内容撰写和构建脚本。

**流程**：调用页面内容指南 → 建立单页依据包 → 写内容 → 图片资产清单 → 构建脚本

**开工门槛**：独立页面 `决策状态=✅ 批准执行`，`对应竞品页面URL` 已替换为实际可访问 URL（不含 `[product]` 等模板占位符）。平台模板/动态组件承接项不按独立页面进入 2C，只验收模板或组件是否接入真实数据。

**核心规则**：
- **禁止占位文本**：不使用 Lorem ipsum、placeholder、"Coming Soon"
- **禁止图片复用**：关键视觉位（Hero、CTA、产品展示、信任背书）每张图必须有独立 prompt
- **依据可追溯**：内容基于依据包 + 1H 策略 + 业务事实，不凭通用知识编造
- **素材证据声明**：依据包必须包含"本页可用素材证据"，用于约束页面文案、图片 prompt 和差异化表达
- **先依据后构建**：每个页面先完成依据包再进入构建，不能跳过
- **依据包分级（§3.5）**：
  - **单页型页面**（Home/About/FAQ/Contact/Ethical Sourcing）：使用完整页面级依据包——核心参考竞品、真实参考URL、上游来源索引、采用/规避/差异化点
  - **模板矩阵页**（Crystal Single ×15、Intention Category ×6 等同类结构页面）：先定义模板级依据包（字段结构、参考竞品、可复用语气、图片规则），每个实例页只填最小差异记录（canonical URL、主关键词、参考URL异常、图片是否独立、特殊风险）。实例页最小差异记录不可省略
- **动态区数据源在构建时决定**：产品区、博客区、相关文章/相关产品、分类导航等动态区域，必须在构建阶段就选定数据接入方式（shortcode / WP Query / REST API / 空容器人工占位），不得先用静态假卡片占位再等 2D 处理

**关键输出**：`pages/*.js` + 单页依据包 + 图片资产清单/prompt + `assets/site-images.js`

> **详细操作（按需读取）**：
> - 页面内容框架、Section 规范：`02-自动化工具库/03-WordPress建站/页面内容指南/`
> - 布局规则、Widget 选择、踩坑清单：`02-自动化工具库/03-WordPress建站/Elementor REST API 操作手册.md`
> - 代码模式参考：`02-自动化工具库/03-WordPress建站/elementor-upload.js`
> - 图片资产状态追踪（`needs_generation → generated → uploaded → url_written`）见项目 `assets/site-images.js`

#### 2D 上传与前端验收

**目标**：逐个页面完成图片生成/上传 → 页面上传 → 前端验收。

**核心规则**：
- **单页试点先行**：第一个页面必须走完全闭环（生成→图片→上传→前端验收），验证通过后再批量推进其余页面
- 逐页闭环，不做完当前页面不开始下一个
- **前端验收必须打开 preview URL 确认**，不能只看 API 返回 200
- **动态内容必须接真实数据源**：产品区、博客区、相关文章、相关水晶、分类导航等动态模块必须调用真实产品、文章、query、REST 数据或分类集合；不得用重复图片、虚拟产品卡、虚拟文章卡或 Lorem ipsum 冒充完成
- 图片必须先生成/验收/上传/URL回写，再运行页面上传脚本

**关键输出**：核心页面已上线（draft 或 published），内容质量和前端效果已通过验收

**里程碑**：核心页面上传完成 → 网站具备基本框架 → 可进入步骤3的 Brief 锁定、内链确认和内容发布准备

> **详细操作（按需读取）**：同 2C 的参考文档，另见项目 `00-项目简报.md` 中的"正确的步骤 2C/2D 执行标准"。

#### 3.0 内容增长目标确认

> 对应 ACE 模型的闭环设计。ACE 是战略框架，不是决策器。本节用决策表代替直觉判断。

内容营销的目标不是"写更多"，而是服务具体的增长目标。项目启动时按当前状态选择默认优先目标：

| 项目状态 | 内容优先目标 | 判断依据 | 不优先做 |
|---------|------------|---------|---------|
| 新站/冷启动 | 收录 + 排名基础 | 关键词覆盖不足、内容资产少 | 转化型复杂内容 |
| 有流量无转化 | 转化 + 信任建设 | 页面有流量但跳出/转化弱 | 盲目扩更多选题 |
| 品类竞争激烈 | 差异化 + E-E-A-T | 竞品内容同质化严重 | 普通泛科普 |
| 已有权威内容 | AI Citation + 外部分发 | 内容已可被引用但曝光不足 | 只做站内新增 |

> 详细增长闭环设计见 [08-AI内容增长引擎-ACE模型.md](08-AI内容增长引擎-ACE模型.md)。

#### 3.1 内容机会输入与数据完整性检查

**输入来源**：1C 关键词库 + 1D 竞品深拆 + 1F 内容策略分析 + 1G 用户画像 + 1H 策略清单。

**用户旅程阶段框架**：

| 旅程阶段 | 核心内容类型 | 常见选题模式 | 对应关键词意图 |
|---------|------------|-------------|-------------|
| **认知** | 百科/科普/指南 | "什么是X"、术语解释、行业入门指南 | 信息型 |
| **考虑** | 对比/评测/教程 | A vs B 对比、产品评测、"最佳 X for Y" | 商业考察型 |
| **决策** | 选购指南/案例/替代方案 | 选购指南、客户案例、竞品替代方案 | 交易型（低意图） |
| **转化** | 行动页/工具/FAQ | 试用/演示、购买指南、FAQ | 交易型（高意图） |

> 实际阶段和内容类型根据竞品数据 + 产品特性调整，不硬套固定模板。关键是确保用户旅程的每个阶段都有内容承接，不出现"断层"。

**数据完整性门槛**（进入 3.2 的前置条件）：

| 必须完成 | 最低要求 |
|---------|---------|
| 核心 Topic Pillar 的关键词研究（1C） | 至少 P0 Pillar 已完成清洗和竞品证据增强 |
| 核心竞品深度拆解（1D） | 至少核心竞品已完成正式拆解文档 |
| 用户痛点或场景证据 | 来自 1G 用户画像、1D §9 或 1F 内容缺口 |

> **数据不完整时的处理**：仍可生成候选池（轻量字段），但不能进入执行清单。等数据补齐后再升级为执行清单。

**品类覆盖检查**：如果内容策略覆盖多品类（如水晶 + 星座 + 塔罗），每个品类需确认 1C 中是否有对应关键词数据和竞品验证证据，缺失的需补充关键词研究。

#### 3.2 内容清单生成与执行池分层

**核心原则：一次性做草案，分批进入正式 Brief**

- **内容清单分两层（§3.5）**：执行清单 + 候选池
  - **执行清单**：首批要做的内容（轻量档 50 篇、标准档 100 篇、深度档全量），补齐完整字段后进入 Brief
  - **候选池**：其余选题，只保留轻量字段（关键词/Volume/意图/Page Type/Topic Pillar），等执行到对应批次时再补执行字段
- **草案可一次性生成，Brief 分批生成** — 规划阶段先锁定候选选题、关键词和发布节奏；正式 Brief 必须在 KD/CPC、竞品验证、URL、承接页面和内链规则补齐后生成或复核
- **数据齐全是进入 Brief 的条件，不是生成草案的前提**
- **"优先级"改为"发布节奏"** — 给每篇内容标注"建议发布顺序"
- **关键词主表可能不完整** — 多品类项目每个品类需单独补充种子词导出、清洗和竞品验证

**内容清单字段**：目标关键词 / Volume / KD / CPC / 意图 / 内容类型 / 对应页面或 URL / 竞品验证状态 / 建议发布节奏 / Brief 状态。

> 详细字段定义见 [04-内容清单字段模板.md](03-模板库/04-内容清单字段模板.md)。

**关键输出**：内容类型规划 + 选题方向 + 内容清单草案（Google Sheets）+ 待补字段清单。

#### 3.3 批次规划原则

> 对应 ACE 的"小额验证→放大"逻辑：先验证小批次选题模式，再放大已验证的模式。不是把所有内容一次性排满，而是按验证节奏分批推进。

**规划原则**：

| 原则 | 说明 |
|------|------|
| 先核心后扩展 | 首批覆盖核心 Topic Pillar 的高价值关键词，不追求全品类覆盖 |
| 按意图分层 | 每批包含不同意图类型（信息型 + 商业型 + 交易型），验证意图→内容的匹配效果 |
| 批次大小受档位约束 | 轻量档每批 10-15 篇、标准档 20-30 篇、深度档 30-50 篇（§3.5） |
| 验证门槛 | 首批发布后观察 2-4 周，至少获得收录和初步排名信号后再启动下一批 |
| 主题集中 | 单批不跨太多 Topic Pillar，优先在一个 Pillar 内做深，再横向扩展 |

**批次与 Brief 的关系**：批次规划确定"这批做哪些选题"，3.4 的 Brief 锁定确定"这篇内容可以开始写了吗"。批次规划先于 Brief 锁定。

> 批次内的具体排期、分工和操作由 [07-英文SEO批量写作与去AI化工作流.md](07-英文SEO批量写作与去AI化工作流.md) 执行，本节只定义规划原则。

#### 3.4 Brief 锁定与竞品素材采集

> **定位**：Brief 锁定是内容从"规划"进入"生产"的 **gate**。未通过锁定的内容不能进入生产。
>
> 竞品段落素材是 Brief 的直接输入，因此采集触发规则合并在此节。

**Brief 锁定门槛**（全部满足才能锁定）：

| 门槛项 | 要求 | 检查方式 |
|--------|------|---------|
| 关键词数据完整 | 目标关键词已有 Volume/KD/CPC/意图标注 | 1C 关键词库可查 |
| 竞品验证完成 | 至少 2-3 篇竞品内容已复核，覆盖结构和关键论点 | 1D/1F 结论可引用 |
| URL 和承接页面确定 | 内容有明确的发布 URL 和承接页面（步骤 2B 已决策） | 页面决策表可查 |
| 内链规则明确 | 与已有内容的内链关系已规划 | 内链矩阵或清单可查 |
| 选题角度差异化 | 与竞品至少有一个结构/视角/信息增量差异 | DREAM 检查（见下） |

**DREAM 模型 — Brief 角度检查工具**：

锁定 Brief 时用 DREAM 框架检查内容是否覆盖用户决策全链路（详细说明见 [08-AI内容增长引擎-ACE模型.md](08-AI内容增长引擎-ACE模型.md)）：

| DREAM 维度 | Brief 中对应的检查 | 未覆盖时的风险 |
|------------|------------------|-------------|
| D — Demand（需求） | H2 是否说明了"为什么需要" | 用户看不到相关性 |
| R — Reason（理由） | 是否给出了"为什么选这个" | 用户无法做比较 |
| E — Evidence（证据） | 是否有数据/案例/来源支撑 | 信任不足，AI 不引用 |
| A — Advantages（优势） | 是否展示了具体差异化 | 淹没在竞品同质内容中 |
| M — Motivation（动机） | 是否给了"现在行动"的理由 | 流量不转化 |

> DREAM 可直接嵌入 [05-内容Brief模板.md](03-模板库/05-内容Brief模板.md) 的大纲结构：H2 需求场景（D）→ H3 理由（R）→ H3 证据（E）→ H2 优势（A）→ H2 行动动机（M）。

**品牌语调配置**（项目级前置资产）：

- 品牌语调在首次 Brief 锁定前完成定义，存放在项目 `00-项目简报.md` 或独立品牌语调文档中
- 语调定义包括：语气正式度、人称选择、专业术语使用范围、情感表达尺度
- Brief 模板中引用品牌语调配置，不重复定义

**竞品段落素材采集**（分级触发）：

> 核心逻辑：AI 引用的是"最好的段落"，不是"最好的页面"。采集最优段落作为写作骨架，再在骨架上重写并注入原创内容。适用范围：博客文章（信息型/对比型/教程型）。Landing Page、产品页等转化型页面不适用。

**触发规则**（完整定义见前文"项目规模分级与执行深度闸门 / 竞品段落采集分级触发"）：

| 内容类型 | 竞争强度 | 是否采集 | 替代方案 |
|---------|---------|---------|---------|
| 百科/科普 | 低 | **跳过** | 差异化模板 + SERP 结构复核 |
| 场景应用 | 中-高 | **执行** | Top 10-20 段落采集 + 评分 |
| 对比/选购 | 高 | **执行** | 完整段落采集 + 评分 |
| 教程/How-to | 中 | **可选** | 视同质化程度决定 |

- 判断标准是"该内容类型的竞争强度"，不是"项目大小"
- 采集结果写入 [05-内容Brief模板.md](03-模板库/05-内容Brief模板.md) 的 `竞品段落素材` 字段（附评分和来源），AI 在骨架上重写 + 注入原创观点/数据，不是照搬
- 写作质量标准见 `03-SEO与GEO/01-内容质量标准.md`（BLUF、实体密度、原创数据）
- 采集操作由竞品内容分析工具或人工流程承接：当前 `02-自动化工具库/05-竞品内容分析工具/content_analyzer.py` 可用于 SERP 结构与内容类型分析，段落评分和写入 Brief 需在 Brief 生成阶段人工/半自动完成；07 只负责把已入库素材纳入生产流程

**关键输出**：已锁定的内容 Brief（含选题角度、竞品素材、差异化声明）+ 品牌语调配置文档。

#### 3.5 内容生产边界

> **定位**：定义哪些内容可以进入生产、生产过程中的质量底线、以及内容生产与社媒分发的边界。

**进入生产的条件**：Brief 已通过 3.4 锁定门槛。

**内容质量地板**（4 项，任一不满足即打回）：

| 质量项 | 要求 | 不通过时的处理 |
|--------|------|-------------|
| 一手观察 | 包含来自竞品复核、用户评论或原始数据的独特视角 | 补充观察或标注"待补充一手信息" |

| 未答之问 | 回应了 SERP 中用户明确提出但竞品未回答的问题 | 补充 FAQ 或专门段落 |

**生产边界约束**：

- **本节只管站内文本内容生产**。博客文章、百科页、指南页等 SEO 内容的生产流程由 [07-英文SEO批量写作与去AI化工作流.md](07-英文SEO批量写作与去AI化工作流.md) 执行
- **社媒再加工不在步骤 3 范围内**：博客文章发布后，社媒再加工、多平台分发归入步骤 4（`04-社媒内容分发工作流.md`）

#### 3.6 发布前质检路由

> **定位**：内容从"生产完成"到"站内发布"之间的最后一道 gate。

**质检路由矩阵**（内容类型 × 检测项）：

| 检测项 \ 内容类型 | 百科/科普 | 对比/选购 | 教程/How-to | 转化型页面 |
|-----------------|----------|----------|------------|----------|
| **EEAT 质量评估** | ✅ 标准 | ✅ 加强 | ✅ 标准 | ✅ 轻量 |
| **站外原创性检测** | ⬜ 可选 | ✅ 必做 | ⬜ 可选 | ⬜ 跳过 |
| **站内重复检测** | ✅ 必做 | ✅ 必做 | ✅ 必做 | ⬜ 轻量/可选¹ |
| **GEO/AI Citation 结构** | ✅ 必做 | ⬜ 可选 | ⬜ 可选 | ⬜ 跳过 |
| **Schema 类型检查** | ⬜ 可选 | ⬜ 可选 | ⬜ 可选 | ✅ 必做 |

**检测项说明与工具**：

| 检测项 | 工具位置 | 检查内容 |
|--------|---------|---------|
| EEAT 质量评估 | `02-自动化工具库/06-内容质检工具/EEAT-内容质量评估.md` | E-E-A-T 信号、事实准确性、实体密度、结构完整性 |
| 站外原创性检测 | `02-自动化工具库/05-竞品内容分析工具/content_originality_checker.py` | 与竞品段落的 N-gram 重合度，高重合段落需重写 |
| 站内重复检测 | `02-自动化工具库/05-竞品内容分析工具/content_duplicate_checker.py` | 站内页面间的 SimHash 相似度，避免内部内容蚕食 |
| GEO/AI Citation | `03-SEO与GEO/` 相关标准 | AI 可引用结构（明确引用块、数据标注、来源声明） |
| Schema 类型检查 | 步骤 2C 构建时配置 | Schema 类型与内容类型匹配（Article / HowTo / FAQ / Product） |

**质检通过标准**：
- 质量地板 4 项全部满足（3.5）
- 该内容类型标记为"必做"的检测项全部通过
- 无 AI 检测高风险段落（或已通过去 AI 化处理）
- 内链指向的页面已存在（不存在则标注待建）
- Schema 标记已配置（如适用）

> 详细质检 SOP 由 07 执行，本节只定义路由规则和通过标准。

**条件覆盖规则**：
- **GEO 与 3.0 目标联动**：若 3.0 的优先目标包含"AI Citation + 外部分发"，则所有站内内容至少执行轻量 GEO/AI Citation 结构检查（不限于百科/科普）
- **¹ 转化型页面站内重复**：少量 Landing Page 可选；模板化产品页、集合页、pSEO 批量页面**必做**

#### 3.7 站内发布与内链承接

> **定位**：只管站内内容的发布和内链。社媒分发在步骤 4。

**发布前检查清单**：

| 检查项 | 要求 |
|--------|------|
| URL 规则一致 | 与 2B 页面决策表中规划的 URL 一致 |
| Meta 信息完整 | Title / Description / H1 / canonical 已设置 |
| 内链双向确认 | 新内容链接到已有内容，已有内容是否需要回链到新内容 |
| 图片已上传 | 所有图片已生成/上传/URL 回写 |
| 分类/标签正确 | 所属分类、Topic Pillar 标签与内容清单一致 |

**内链承接规则**：

| 内链类型 | 规则 | 示例 |
|---------|------|------|
| 上下层级 | 百科页 → 分类页 → 产品页，各层级互相链接 | Amethyst 百科 → Crystals 分类 → Amethyst 产品 |
| 主题关联 | 同 Topic Pillar 下的内容互相链接 | Amethyst 百科 ↔ Amethyst 冥想指南 |
| 意图递进 | 信息型 → 商业型 → 交易型的用户旅程链接 | "什么是紫水晶" → "最好的紫水晶手链" → 产品页 |
| 锚文本规范 | 使用描述性锚文本，不使用"点击这里" | "紫水晶的疗愈功效"而非"了解更多" |

**关键输出**：已发布的站内内容 + 内链矩阵更新 + 待回链清单。

> **视频型内容分叉**：站内内容发布完成后，部分内容的视频素材可进入 CAST 短视频流水线（见 [08-AI内容增长引擎-ACE模型.md](08-AI内容增长引擎-ACE模型.md)），具体分发节奏由步骤 4 决定。

#### 3.8 复盘回写与资产沉淀

> **定位**：内容发布不是终点。复盘的目的是把经验和数据回写到知识资产中，驱动下一轮内容循环。

**触发条件**：

| 触发时机 | 条件 | 复盘范围 |
|---------|------|---------|
| 首批发布后 4 周 | 首批内容已有收录和初步排名数据 | 整批复盘：选题模式、意图匹配、批次节奏 |
| 单篇内容异常 | 排名 30+ 天未进入前 100，或跳出率 > 80% | 单篇复盘：内容质量、关键词匹配、竞争环境 |
| 批次完成时 | 一个执行批次全部内容发布完成 | 批次复盘：批次规划原则是否有效 |
| 季度复盘 | 每 3 个月 | 全局复盘：内容资产盘点、增长目标进展 |

**回写目标与字段**：

| 回写目标 | 回写字段 | 示例 |
|---------|---------|------|
| 内容清单（Google Sheets） | 发布状态、实际排名、实际流量、Brief 状态更新 | `Brief 状态: 已发布 → 已复盘` |
| 1C 关键词库 | 关键词实际表现（排名/流量）、意图验证结果 | 新发现的关联关键词补充进 Seed-Master |
| 用户画像（1G） | 用户实际行为数据（搜索词、停留时长、转化路径） | 修正用户画像中的痛点/场景假设 |
| 策略清单（1H） | 验证结果（哪些策略有效、哪些需要调整） | "模仿X"策略效果标注 |
| 品牌语调配置 | 用户互动反馈（评论、社媒反应） | 语调微调建议 |

**状态变更规则**：

```
Brief 状态流转：
草案 → 已锁定 → 生产中 → 待质检 → 已发布 → 已复盘
  ↑                                          │
  └──────── 打回重做（质检不通过）←────────────┘
                              ↓
                        待优化（复盘发现异常）
```

**资产沉淀**：

| 沉淀资产 | 存放位置 | 用途 |
|---------|---------|------|
| 高表现内容模式 | 项目复盘文档 | 驱动下一批选题和 Brief |
| 竞品段落素材库（已验证有效的） | Brief 文档 | 新内容可复用的写作骨架 |
| 用户原话库（评论/反馈） | 1G 用户画像文档 | 后续内容的素材来源 |
| 关键词表现数据 | 1C/Seed-Master | 优化后续关键词策略 |

> 复盘的具体数据分析操作见 [08-AI内容增长引擎-ACE模型.md](08-AI内容增长引擎-ACE模型.md) 的"验证与放大"环节和 `02-SEO全链路工作流.md`。
