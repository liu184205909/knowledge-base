## 1. RLM是什么

**RLM营销方法论 = 用AI的分而治之思想 + 从网站架构到内容全面超越竞品的策略，实现营销全流程自动化**

### 核心思路：模仿 + 超越 + 差异化

核心逻辑：先看市场上谁做得好 → 判断他们为什么好 → 找到他们没做好的地方 → 设计更快、更系统、更低成本的方案。


默认服务于**海外市场**（B2B外贸、B2C独立站、SaaS/工具站、内容站/Affiliate）。

### 使用原则：让AI泼冷水

向AI提问时，不要问"这想法怎么样"，要问"这想法最可能在哪里崩"。让AI泼冷水，直到泼不出来。AI最大的危险不是犯错，而是用同样的热情帮一个烂想法做到产品级。


## 2. AI执行纪律

**这是执行版中最重要的章节。违反以下任何一条都算输出不合格。**

1. **不编造、不预测**：没有来源的数据必须标注"待验证"，不得虚构搜索量、流量、KD、CPC。用户画像必须来自竞品页面、关键词意图、FAQ、评论或行业场景，依据不足时标注"初步推测"。除非用户提供历史数据或计算模型，否则不输出流量、收入、排名、转化率的具体预测。
2. **不输出空泛策略**：模仿、超越、差异化必须具体到竞品、页面、内容、结构、关键词或转化路径。禁止"提升品牌力""优化用户体验"等无动作表述。RLM主流程围绕SEO内容展开，付费广告、社媒、邮件等渠道按需扩展。
3. **输出必须可落地**：每次输出必须有明确文件名和路径，模糊的"输出一份分析"不算合格。每阶段交付物必须能作为下一步输入；若仅停留在分析，补任务清单或验收项。
4. **素材有源，语气有参照**：进入 Brief、页面文案或正式内容生成前，必须声明素材证据来源和语气参考；具体字段写入模板库或单页依据包。
5. **不一次性输出大而全**：先写 spec（规格/框架），拆成小块，逐步实现，持续测试，每步 review。AI 越强，流程反而越要保守。（Addy Osmani）

| B | 网址 | 人工 Google 搜索 / SEMrush | 1A | 人工 |
| C | AS | SEMrush 脚本 | 1B | 自动 |

| F | 月访问量 | traffic.cv | 1B | 手动 |
| G-L | 流量渠道拆分（G搜索 / H直接 / I社交 / J引荐 / K邮件 / L付费） | traffic.cv | 1B | 手动 |
| M | 建站平台 | 人工/Wappalyzer | 1A | 手动 |

| O | 商业模式 | AI辅助判断 | 1D | AI |
| P | 深度拆解文档 | — | 1D | — |

> 1A 填入 A、B、M、N 列，其余在 1B/1D 逐步填充。N 列分类：直接竞品（卖同类产品）/ 内容竞品（不卖货但覆盖关键词）/ 间接竞品（同大类不同细分）。

**分析数据表**（进入 1B 时创建，导入时按标准列精简）：

| 表格名称 | 用途 | 标准列（保留） | Semrush 导出时删除的列 |
|------|------|--------------|----------------------|
| SEMrush-Seed-Keywords | Seed-* 主题关键词池 | Keyword / 中文 / Entity / Subtopic / Content Role / Volume / KD / CPC / Number of Results / Intent | Competitive Density / SERP Features / Updated / Priority / Recommended Page Type |
| SEMrush-Top-Pages | 页面级SEO价值 | URL / Traffic(%) / Traffic / Top Keyword / Primary Intent / LLM Prompts | Adwords Positions / Number of Keywords / 5×Positions by Intent / 5×Traffic by Intent / Traffic Change / Answer Engines |
| SEMrush-Top-Keywords | 关键词排名与排名URL | Keyword / Search Volume / Keyword Difficulty / CPC / URL / Traffic / Traffic(%) / Number of Results / Keyword Intents | Position / Competition / Previous Position / Traffic Cost / Trends / Timestamp / SERP Features by Keyword / Position Type |

> `SEMrush-Seed-Keywords` 中每个 `Seed-*` 工作表代表一个 Topic Pillar。`Entity / Subtopic / Content Role` 是通用字段，但 `Subtopic` 的可选值由各 Topic Pillar 自己定义，不能把一个 Pillar 的分类硬套到其他主题（如水晶站的 Crystals→Meaning/Benefits/Properties 不应硬套到 Chakra、Tarot 等主题）。`Seed-Master v1` 由 1B 轨道B 构建，后续竞品证据增强由 1B 轨道D 承接。上表中 Priority / Recommended Page Type 作为 SEMrush 原始导出列删除，1C 后续重新生成这两个决策字段。

**数据规则**：
- 项目简报记录所有 Google Sheets 链接，包含标准列和删除列说明
- **AI 接手时先通过 Google Workspace MCP 搜索谷歌对应项目文件夹、遍历本地对应项目文件夹，与项目简报索引比对，不一致时以实际为准更新索引**
- 竞品 Top Pages / Top Keywords 的汇总、筛选和验证细则见 `02-自动化工具库/02-竞品研究工具/1B数据处理工具手册.md` 的轨道C/D
- **`1A-竞品清单.md` 只放链接和筛选口径**，不重复搬运表格数据

## 4. RLM 五步执行流程

### 步骤1：竞品查找与分析（知道对手是谁 + 知道内容怎么做）

**目标**：找到所有相关竞争对手，建立数据基础，为网站规划和内容策略提供依据。

步骤1是整个流程的数据基础，内部分8个子阶段：

```
1A: 竞品发现（人工）
    ├── 轨道A: Google 搜索 + 排除语法 → 筛选竞品
    └── 轨道B: SEMrush Organic Competitors → 补充遗漏竞品
1B: 多源数据采集（多条轨道可并行）
    ├── 轨道A: 竞品清单数据 → SEMrush AS/Organic Traffic/截图 + traffic.cv月访问量/渠道拆分
    ├── 轨道B: Seed-* 主题关键词池 → SEMrush 相关关键词导入与清洗 → Seed-Master v1
    ├── 轨道C: 竞品SEO明细数据 → SEMrush Top Pages（页面级流量证据）+ SEMrush Top Keywords（关键词级排名证据）→ All 表
    ├── 轨道D: Seed-Master 竞品证据增强 → TopKeywords_All + TopPages_All 补充关键词/页面证据
    └── 轨道E: Sitemap解析 → 页面清单/URL结构/内容类型分布 → 竞品页面报告
1C: 关键词研究验证与决策标注 → 验证 1B 初步标注，补充 Business Relevance / Priority / Page Type，输出关键词分组与优先级
1D: 竞品深度拆解     → 核心输入（Top Pages + Top Keywords + Sitemap + Seed-Master + 1B 轨道D 竞品证据增强结果）+ 辅助输入（竞品清单汇总）
                     → 每个竞品逐个深度拆解，回答：X靠什么拿流量、X建了什么、X覆盖/遗漏了什么
1E: 网站结构分析（综合1B+1C+1D做结构对比）→ 可进入步骤2
1F: 内容策略分析     → 可进入步骤3
1G: 用户画像
1H: 策略清单         → 汇入步骤2和步骤3
```

#### 1A 竞品发现（人工）

**目标**：通过人工搜索和SEMrush双轨道发现竞品，将竞品填入启动阶段已创建的竞品清单表。

##### 轨道A：Google 搜索

**核心任务**：
- 使用 Google 搜索种子关键词，通过排除语法过滤大平台
- 人工浏览搜索结果，筛选竞品域名

**搜索排除语法**（直接复制使用）：
```
关键词 -amazon.com -ebay.com -etsy.com -walmart.com -pinterest.com -reddit.com -youtube.com -facebook.com -tiktok.com -wikipedia.org -quora.com -target.com -aliexpress.com -alibaba.com
```

##### 轨道B：SEMrush Organic Competitors（人工操作）

**核心任务**：
- 打开 SEMrush，查看 Organic Competitors 报告

> 1A 完成后可以暂停，不强制立即进入 1B。

#### 1B 多源数据采集（多条轨道可并行）

从 1A 竞品清单出发，分多条轨道并行采集数据。详细操作见 `02-自动化工具库/02-竞品研究工具/1B数据处理工具手册.md`。

##### 轨道A：竞品清单数据

逐个竞品填充 3.1 竞品清单中的对应列（AS、Organic Traffic、月访问量、渠道拆分、建站平台）。

##### 轨道B：Seed-* 主题关键词池 → 构建 Seed-Master v1

按主题从 SEMrush 导出相关关键词，清洗后构建 `Seed-Master v1`。Entity / Subtopic / Content Role 在本阶段作为**初步标注/清洗辅助字段**，服务于底座构建质量，不代表最终关键词研究结论（1C 负责验证和深化）。详细操作见 `1B数据处理工具手册.md` 轨道B。

> **知识体系增强**：如果已构建行业 Taxonomy/Ontology（见 [03-SEO与GEO/04-实体与知识图谱理论.md](../03-SEO与GEO/04-实体与知识图谱理论.md) 的「知识体系构建 4 步法」），可以用它作为 Entity/Subtopic 分类的上游框架，减少主观标注偏差。

##### 轨道C：竞品SEO明细数据

从 SEMrush 导出竞品的 Top Pages 和 Top Keywords，生成 `TopKeywords_All` 和 `TopPages_All`。结构定义见 §3.1。

##### 轨道D：Seed-Master 竞品证据增强

用 `TopKeywords_All` 匹配 `Seed-Master v1` 补关键词证据，再用命中 URL 去 `TopPages_All` 补页面证据。未命中处理阈值见 1B 手册。

> **轨道 B vs D 的职责**：轨道 B 构建关键词底座（Seed-Master v1），轨道 D 为底座补充竞品证据。两者共同构成 1C 和 1D 的权威数据源。

##### 轨道E：Sitemap解析

解析每个竞品的 sitemap.xml，获取页面清单、URL 路径模式和内容类型分布。详细工具说明见 1B 手册。

**关键输出（1B 整体）**：更新后的竞品清单 + Seed-Master 竞品证据增强结果 + `TopKeywords_All` + `TopPages_All` + Sitemap 解析报告

#### 1C 关键词研究验证与决策标注

> **总规则**：1B 负责生成可用关键词底座；1C 负责把底座转成可决策关键词表。

**输入**：Seed-Master 竞品证据增强结果（1B 轨道B/D 产出）
**方法参考**：`01-关键词研究方法论.md` 阶段4-5
**执行操作**：`1C关键词研究工具手册.md`

**核心任务**：
- 验证 1B 初步标注：审查 Entity / Subtopic / Content Role / 搜索意图是否准确，修正错误归类、跨主题冲突和泛 Entity
- 补充决策字段：标注 Business Relevance / Recommended Page Type / 关键词优先级（高/中/低/暂存）/ Status（Candidate → Approved/Skip）
- 高价值词复核：对当前批次候选词做商业价值评估和 SERP 竞争度检查
- 输出 `关键词分组与优先级.md`，按执行节奏排序，支撑 1E / 1F / 2B / 步骤3

**关键输出**：Seed-Master（1C 策略标注后）+ `关键词分组与优先级.md`

> 1C 不是重新采集关键词，而是把 1B 产出的 Seed-Master 从"可用数据表"变成"可支撑页面、内容和批次决策的关键词决策表"。

#### 1D 竞品深度拆解

**目标**：对每个竞品进行深度拆解，回答三件事：X 靠什么拿流量、X 站内建了什么、X 相对市场需求覆盖了什么和遗漏了什么。

**数据输入**（每个竞品）：

1D 数据输入按分析问题组合使用，不按 1B/1C 分层。**核心输入**（判断页面/关键词/结构机会）：该竞品的 Top Pages + Top Keywords（1B 轨道C）+ Sitemap（1B 轨道E）+ Seed-Master + 轨道 D 竞品证据增强结果。**辅助输入**（仅用于竞品权重、背景和转化体验判断）：竞品清单汇总、页面截图、渠道拆分。核心输入用于判断机会，辅助输入用于判断背景——两者不能混用。

**1D 只回答**：
- 竞品 X 做了哪些内容/页面？
- 哪些页面和关键词已经拿到流量？
- X 的内容类型、页面结构、关键词覆盖有什么强项和缺口？
- 哪些做法值得进入 1H 候选策略池？

**1D 不回答**：
- 我们最终网站怎么建
- 我们最终写哪些内容
- 我们最终模仿/超越哪些策略

**核心任务**：
- 所有入选竞品逐个深度分析，按竞品清单表格顺序执行，按类型侧重不同：
  - **直接竞品**：全面拆解（产品、定价、页面、转化路径、网站结构）
  - **内容竞品**：侧重内容策略和关键词覆盖（他们写了什么、排名了什么、流量从哪来）
  - **间接竞品**：侧重差异化参考（他们的定位和打法与你有什么不同）
- **竞品增长侧分析**（直接竞品必做）：流量构成拆解 + 冷启动策略推断 + 增长节奏（页面/外链曲线），用于回答"我们应该模仿哪条增长路径"

**关键输出**：每个竞品的深度拆解文档（存放在 `1D-竞品深度拆解/` 目录）

> 1D 完成后进入 1E 网站结构分析、1F 内容策略分析。

#### 1E 网站结构分析（知道竞品网站怎么建）

> **定位**：1E 是跨竞品结构对比分析层，只回答"竞品怎么建、哪些结构被验证、哪些页面类型高频/高价值"，不决定我们最终网站怎么建。
>
> **输入来源**：1B 轨道E Sitemap 解析报告 + 1B 轨道C `TopPages_All` / `TopKeywords_All` + 1D 竞品深度拆解文档 + Seed-Master / 1C 结果（如已完成则使用）

**核心任务**：

1. **网站结构分析** — 导航结构、页面层级、布局优劣（综合 Sitemap 解析报告 + TopPages 数据 + 1D 深度拆解结论）
2. **产品/内容分类模式识别** — 评估竞品分类方式优缺点

   | 竞品 | 分类方式 | 分类维度 | 优点 | 缺点 |
   |------|---------|---------|------|------|
   | 竞品X | [实际分类] | [按什么维度分] | [哪里好] | [哪里不足] |

3. **URL 结构分析** — 命名规则、层级深度、关键词出现模式
4. **类目 / URL 命名观察** — 从竞品 URL 和类目名中提取关键词命名模式，观察搜索趋势（仅类目级别，不做完整关键词研究）

**关键输出**：竞品结构对比 + 分类/URL 命名观察 + 页面类型频率/SEO价值观察 + **架构层候选模仿点**（最终策略归入 1H，最终网站方案归入 02）

> **边界约束**：1E 只能输出竞品结构事实、页面类型频率、SEO页面价值和候选模仿点，不能输出"我们的网站结构方案"。网站结构、URL规则、页面清单和执行批次只能在步骤2 `02-网站规划/` 中决定。

> **已移除**：~~SEO 增长目标倒推~~ 属于项目级增长规划，不属于竞品结构分析。如需使用，可独立执行或汇入步骤3增长复盘模块。

**工具链**：
- Skills: `frontend-design` — 分析竞品目录结构的 UX 优劣
- MCP: `webReader` — 按需复核竞品具体页面结构（轻量复核，不重新抓取全站 sitemap）

> **知识体系增强**：1E 的"分类模式识别"可以借助行业 Taxonomy/Ontology（见 [03-SEO与GEO/04-实体与知识图谱理论.md](../03-SEO与GEO/04-实体与知识图谱理论.md) 的「知识体系构建 4 步法」）来理解竞品分类背后的概念层级——不仅看竞品"怎么分"，还看"为什么这样分"，以及"哪些概念被遗漏了"。

> 1E + 1H 完成后可进入步骤2网站规划与建设。

#### 1F 内容策略分析

**输入来源**：1D 单竞品深度拆解文档（内容类型、内容优势、内容缺口、§9 候选策略点）+ Seed-Master 关键词主表（1B 轨道B/D 提供）+ 1B 轨道D 竞品证据增强结果 + 必要的核心页面轻量复核。

**核心任务**：
- 内容类型分析（博客/案例/教程/指南）
- 竞品内容优势与缺口识别
- 竞品共同弱点识别（找到 90%+ 竞品都没做的事情 = 最大差异化机会）

**关键输出**：内容策略分析（每个竞品的内容类型、优势、缺口、共同弱点）

> **边界约束**：1F 输出内容缺口和候选机会时使用"候选方向"语气（如"候选方向（→ 1H 策略清单决策）"），不使用"我们的策略""我们将"等决策语气。最终内容策略决策归入步骤3。

#### 1G 用户画像

**输入来源**：Seed-Master 关键词主表中的意图、问题和主题集群（1B 轨道B/D 提供）+ 1B 轨道D 竞品证据增强结果 + Top Pages/Top Keywords 暴露的页面需求 + 1D/1F 中反复出现的痛点、场景、FAQ、评论和转化路径观察。

**核心任务**（基于上述输入推导，不凭空想象）：
- 目标用户画像
- 核心痛点
- 用户需求

**关键输出**：用户画像（每个画像可追溯到关键词意图、竞品页面、FAQ 或评论）

#### 1H 策略清单

**核心任务** — 必须独立成章：
- 汇总来自 1D §9、1E、1F 的候选策略点，并结合 1G 用户画像去重、排序、映射到页面或内容
- **模仿清单**：格式 "模仿X: [竞品]的[具体做法]"
- **超越清单**：格式 "超越X: [竞品不足的具体描述]"
- **差异化清单**：格式 "差异化X: [竞品空白的具体机会]"

**关键输出**：策略清单 → 汇入步骤2（网站规划与建设）和步骤3（内容策略与内容营销总控）

> **边界约束**：1H 不重复搬运竞品原始数据（导航、截图、URL等），只吸收来自 1D §9、1E、1F 的候选策略点和 1G 用户画像，进行去重、排序、映射。1H 是唯一完整的策略中枢。

### 步骤2：网站规划与建设（把网站建起来）

**目标**：基于竞品结构分析和关键词方向，规划并搭建 WordPress 网站的核心页面

步骤2内部分4个子阶段：

```
2A: 架构规划     → 5个规划文档 + 竞品页面频率矩阵（页面候选池）
2B: 页面决策     → 决策表：做/不做/批次 + 竞品参考来源索引
2C: 逐页生成     → 每个页面：页面内容指南 → 单页依据包（复用1B/1C/1D/1E/1F/1H + 轻量复核URL）→ 写内容 → 图片资产清单/prompt → 构建脚本
2D: 上传与验收   → 逐页：图片生成/上传/URL回写 → 脚本上传 → 前端验收
```

> **步骤2核心规则**：2B 是 gate，未通过决策的页面不能进入 2C。2C 不重新深度拆解竞品，只读取上游结论并对核心参考 URL 做轻量复核。

#### 2A 架构规划

**核心任务**：

1. **网站结构设计** — 参考(模仿)竞品好的结构，优化(超越)竞品不足
2. **URL结构与内部链接** — 基于类目关键词方向设计
3. **产品类目规划** — 基于关键词命名，比竞品更清晰合理
4. **核心页面内容规划** — 每个页面标注：参考哪家竞品、模仿什么、超越什么、差异化创新
5. **产品页面规划** — 产品标题、描述、属性
6. **竞品页面频率矩阵**（2B 的直接输入） — 从 1E 竞品结构分析中提炼页面候选池，格式如下：

   | 页面类型 | 出现竞品数/总竞品数 | 普及率 | 出现的竞品 | 不出现的竞品 | 1E来源索引 | 备注 |
   |---------|-------------------|-------|-----------|------------|----------|------|

   > **边界**：原始证据（竞品导航截图、完整URL列表、详细结构图）留在 1E。2A 的频率矩阵只保留决策所需的摘要和引用（如"详见 1E §2.3"），不复制 1E 原始数据。

**关键输出**：`02-网站规划/` 下5个规划文档 + **竞品页面频率矩阵**（作为 2B 页面决策的直接输入）

#### 2B 页面决策（gate）

**目标**：基于 2A 的竞品频率矩阵 + 1H 策略清单，决定实际要做哪些页面、什么顺序。

**2B 是步骤2的 gate。** 没有通过 2B 决策的页面不能进入 2C 生成。

**强制输出 — 页面决策表**：

| 页面名称 | 页面类型 | 竞品覆盖率 | 对应竞品页面URL | 用户意图 | 1H策略匹配 | SEO/商业价值 | 实现复杂度 | 决策状态 | 执行批次 | 候选批次 | 不做/延后原因 |
|---------|---------|-----------|---------------|---------|-----------|------------|-----------|---------|---------|---------|------------|

> **字段规则**：
> - `决策状态` 只允许使用：`✅ 批准执行`、`⏸ 候选延后`、`❌ 不做`。
> - 只有 `✅ 批准执行` 的独立页面才能填写 B1a/B1b/B2/B3 等 `执行批次` 并进入 2C；平台模板/动态组件承接项只能填写承接方式，不进入独立页面上传数量。
> - `⏸ 候选延后` 页面只能填写 `候选批次`，不能进入 2C；后续必须经过 2B 复审才能激活。
> - `对应竞品页面URL` 可以在 2B 阶段使用 URL 模板（如 `/products/[product]`），但进入 2C 前必须替换为实际可抓取 URL。
> - `本页参考竞品` 由 2B 选定：业务必要/低复杂度页面至少 2 家，标准核心页面建议 3-5 家，高价值SEO模板或SERP竞争强页面可扩展到 5-10 家。这里的数量指"单页参考来源"，不是要求在 2C 重新深度拆解。
> - `平台模板/动态组件承接` 必须单独标注：如 WooCommerce 产品详情页、博客归档页、产品列表区、文章列表区等由主题、CMS、短代码、query 或 REST 动态组件承接的对象，不计入独立页面上传数量；处理方式写为"模板承接/动态组件承接"，不能混入 B1a/B1b 的独立 page 计数。
> - 每个本页参考竞品必须能追溯到 1D 深度拆解文档、1E/1F 汇总结论或 1H 策略条目；没有上游依据的临时参考不能直接支撑页面结构和文案。

**决策规则**：

| 条件 | 决策 | 说明 |
|------|------|------|
| 竞品覆盖率 75%+ | **默认必须做** | 大多数竞品都有，不做等于缺失基础；除非与项目定位明显冲突 |
| 竞品覆盖率 25-75% | **进入判断** | 看 1H 策略价值和资源成本，决定"应该做 / 差异化做 / 延后做" |
| 竞品覆盖率 <25% | **默认不做/远期** | 除非它是 1H 明确要求的差异化页面（如差异化互动工具/内容即使覆盖率低也值得做） |
| About / Contact / FAQ / Shipping / Returns | **业务必要页** | 不完全受频率阈值支配，属于网站信任基础，即使部分竞品没有也必须做 |

> **水晶站案例**：Crystal Quiz 只有 25% 竞品有，但 1H 明确要求做 → 批准执行

**执行批次排序原则**：

- B1a：上线骨架页（首页、About、FAQ、Contact、产品详情模板/承接规则、核心分类入口等）— 网站存在与信任基础
- B1b：核心 SEO 模板（百科单品、按意图分类等可模板化矩阵）— 流量基础，但不阻塞 B1a 上线
- B2：上线后 1 个月内扩展页（更多分类、博客、轻量互动或转化页）— 扩展转化和内容覆盖
- B3：3 个月内差异化页（工具、资源、周期性内容）— 竞争优势验证
- B4：远期或模板工厂页（大规模矩阵、供应链依赖、蓝海验证不足页面）— 需要复审后再执行
- 每批内的页面按重要性、依赖关系和实现复杂度排序

**关键输出**：`02-网站规划/页面决策表.md` — 确定批准执行的页面清单 + 候选延后/不做页面及原因 + 执行批次 + 候选批次

#### 2C 逐页生成

**目标**：按 2B 确定的页面清单和执行批次，逐个页面完成内容撰写和构建脚本。

**流程**：调用页面内容指南 → 建立单页依据包 → 写内容 → 图片资产清单 → 构建脚本

**开工门槛**：独立页面 `决策状态=✅ 批准执行`，`对应竞品页面URL` 已替换为实际可访问 URL（不含 `[product]` 等模板占位符）。平台模板/动态组件承接项不按独立页面进入 2C，只验收模板或组件是否接入真实数据。

**核心规则**：
- **禁止占位文本**：不使用 Lorem ipsum、placeholder、"Coming Soon"
- **禁止图片复用**：关键视觉位（Hero、CTA、产品展示、信任背书）每张图必须有独立 prompt
- **依据可追溯**：内容基于依据包 + 1H 策略 + 业务事实，不凭通用知识编造
- **素材证据声明**：依据包必须包含"本页可用素材证据"，用于约束页面文案、图片 prompt 和差异化表达
- **先依据后构建**：每个页面先完成依据包再进入构建，不能跳过
- **依据包分级**：
  - **单页型页面**（Home/About/FAQ/Contact/Ethical Sourcing）：使用完整页面级依据包——本页参考竞品、真实参考URL、上游来源索引、采用/规避/差异化点
  - **模板矩阵页**（同类结构批量页面，如产品单品页 ×N、场景分类页 ×M）：先定义模板级依据包（字段结构、参考竞品、可复用语气、图片规则），每个实例页只填最小差异记录（canonical URL、主关键词、参考URL异常、图片是否独立、特殊风险）。实例页最小差异记录不可省略（水晶站：Crystal Single ×15、Intention Category ×6）
- **动态区数据源在构建时决定**：产品区、博客区、相关文章/相关产品、分类导航等动态区域，必须在构建阶段就选定数据接入方式（shortcode / WP Query / REST API / 空容器人工占位），不得先用静态假卡片占位再等 2D 处理

**关键输出**：`pages/*.js` + 单页依据包 + 图片资产清单/prompt + `assets/site-images.js`

> **详细操作（按需读取）**：
> - 页面内容框架、Section 规范：`02-自动化工具库/03-WordPress建站/页面内容指南/`
> - 布局规则、Widget 选择、踩坑清单：`02-自动化工具库/03-WordPress建站/Elementor REST API 操作手册.md`
> - 代码模式参考：`02-自动化工具库/03-WordPress建站/elementor-upload.js`
> - 图片资产状态追踪（`needs_generation → generated → uploaded → url_written`）见项目 `assets/site-images.js`

#### 2D 上传与前端验收

**目标**：逐个页面完成图片生成/上传 → 页面上传 → 前端验收。

**核心规则**：
- **单页试点先行**：第一个页面必须走完全闭环（生成→图片→上传→前端验收），验证通过后再批量推进其余页面
- 逐页闭环，不做完当前页面不开始下一个
- **前端验收必须打开 preview URL 确认**，不能只看 API 返回 200
- **动态内容必须接真实数据源**：产品区、博客区、相关文章/相关产品、分类导航等动态模块必须调用真实产品、文章、query、REST 数据或分类集合；不得用重复图片、虚拟产品卡、虚拟文章卡或 Lorem ipsum 冒充完成
- 图片必须先生成/验收/上传/URL回写，再运行页面上传脚本

**关键输出**：核心页面已上线（draft 或 published），内容质量和前端效果已通过验收

**里程碑**：核心页面上传完成 → 网站具备基本框架 → 可进入步骤3的 Brief 锁定、内链确认和内容发布准备

> **详细操作（按需读取）**：同 2C 的参考文档，另见项目 `00-项目简报.md` 中的"正确的步骤 2C/2D 执行标准"。

### 步骤3：内容策略与内容营销总控

**目标**：基于步骤 1 的数据基础，规划、生产、质检、发布和复盘内容，形成完整的内容营销闭环。

> **定位**：本节是**控制面**（目标、门槛、路由、产出物归属），不是执行 SOP。分层关系：**步骤 3** = 控制面 → **08-ACE** = 增长战略层 → **07-批量写作** = 执行层。具体批量生产操作由 07 执行。

**跨步骤依赖**：内容清单草案可先启动；最终 URL、承接页面、内链规则和内容 Brief 需等步骤 2（2A+2B）完成后锁定或复核。

#### 3.0 内容增长目标确认

> 对应 ACE 的闭环设计。ACE 是战略框架，本节用决策表代替直觉判断。

项目启动时按当前状态选择默认优先目标：

| 项目状态 | 内容优先目标 | 判断依据 | 不优先做 |
|---------|------------|---------|---------|
| 新站/冷启动 | 收录 + 排名基础 | 关键词覆盖不足、内容资产少 | 转化型复杂内容 |
| 有流量无转化 | 转化 + 信任建设 | 页面有流量但跳出/转化弱 | 盲目扩更多选题 |
| 品类竞争激烈 | 差异化 + E-E-A-T | 竞品内容同质化严重 | 普通泛科普 |
| 已有权威内容 | AI Citation + 外部分发 | 内容已可被引用但曝光不足 | 只做站内新增 |

> 详细增长闭环设计见 [08-AI内容增长引擎-ACE模型.md](08-AI内容增长引擎-ACE模型.md)。

#### 3.1 内容机会输入与数据完整性检查

**输入来源**：1C 关键词库 + 1D 竞品深拆 + 1F 内容策略分析 + 1G 用户画像 + 1H 策略清单。

**用户旅程阶段框架**：

| 旅程阶段 | 核心内容类型 | 选题模式 | 关键词意图 |
|---------|------------|---------|----------|
| **认知** | 百科/科普/指南 | "什么是X"、术语解释 | 信息型 |
| **考虑** | 对比/评测/教程 | A vs B、"最佳 X for Y" | 商业考察型 |
| **决策** | 选购指南/案例/替代方案 | 选购指南、客户案例 | 交易型（低意图） |
| **转化** | 行动页/工具/FAQ | 试用/演示、购买指南 | 交易型（高意图） |

> 确保每个阶段都有内容承接，不出现"断层"。实际内容类型根据竞品数据 + 产品特性调整。

**数据完整性门槛**（进入 3.2 的前置条件）：

| 必须完成 | 最低要求 |
|---------|---------|
| 核心 Topic Pillar 的关键词研究（1C） | 至少当前批次 Pillar 已完成清洗和竞品证据增强 |
| 竞品深度拆解（1D） | 至少直接竞品已完成正式拆解文档 |
| 用户痛点或场景证据 | 来自 1G 用户画像、1D §9 或 1F 内容缺口 |

> **数据不完整时的处理**：仍可生成候选池（轻量字段），但不能进入执行清单。等数据补齐后再升级。

**品类覆盖检查**：多品类项目（如核心产品 + 周边场景 + 延伸知识）每个品类需独立确认 1C 关键词数据和竞品验证证据。

#### 3.2 内容清单生成与执行池分层

**核心原则：一次性做草案，分批进入正式 Brief**

- 内容清单分两层：执行清单（首批完整字段）+ 候选池（轻量字段，等批次轮到时再补）
- 草案可一次性生成，正式 Brief 必须在数据补齐后锁定（KD/CPC、竞品验证、URL、内链规则）
- 用"发布节奏"代替"优先级"，给每篇内容标注建议发布顺序
- 多品类项目每个品类需单独补充种子词导出、清洗和竞品验证

**内容清单字段**：目标关键词 / Volume / KD / CPC / 搜索意图 / 内容类型 / 对应页面或 URL / 竞品验证状态 / 建议发布节奏 / Brief 状态。

> 详细字段定义见 [05-内容清单字段模板.md](03-模板库/05-内容清单字段模板.md)。

**关键输出**：内容类型规划 + 选题方向 + 内容清单草案（Google Sheets）+ 待补字段清单。

#### 3.3 批次规划原则

> 对应 ACE 的"小额验证→放大"逻辑。批次规划确定"这批做哪些选题"，3.4 的 Brief 锁定确定"这篇可以开始写了吗"。

**规划原则**：

| 原则 | 说明 |
|------|------|
| 先核心后扩展 | 首批覆盖核心 Topic Pillar 的高价值关键词 |
| 按意图分层 | 每批包含不同意图类型，验证意图→内容匹配 |
| 批次大小 | 每批 10-30 篇，按当前批次执行能力确定 |
| 验证门槛 | 首批发布后观察 2-4 周，有收录和排名信号后再启动下一批 |
| 主题集中 | 单批不跨太多 Pillar，先做深再横向扩展 |

> 批次内的具体排期和操作由 07 执行。

#### 3.4 Brief 锁定与竞品素材采集

> Brief 锁定是内容从"规划"进入"生产"的 **gate**。竞品段落素材采集触发规则合并在此节。

**Brief 锁定门槛**（全部满足才能锁定）：

| 门槛项 | 要求 | 检查方式 |
|--------|------|---------|
| 关键词数据完整 | 目标关键词已有 Volume/KD/CPC/意图标注 | 1C 关键词库可查 |
| 竞品验证完成 | 至少 2-3 篇竞品内容已复核，覆盖结构和关键论点 | 1D/1F 结论可引用 |
| URL 和承接页面确定 | 内容有明确的发布 URL 和承接页面（步骤 2B 已决策） | 页面决策表可查 |
| 内链规则明确 | 与已有内容的内链关系已规划 | 内链矩阵或清单可查 |
| 选题角度差异化 | 与竞品至少有一个结构/视角/信息增量差异 | DREAM 检查（见下） |

**DREAM 模型 — Brief 角度检查工具**：

锁定 Brief 时用 DREAM 框架检查内容是否覆盖用户决策全链路（详细说明见 [08-AI内容增长引擎-ACE模型.md](08-AI内容增长引擎-ACE模型.md)）：

| DREAM 维度 | Brief 中对应的检查 | 未覆盖时的风险 |
|------------|------------------|-------------|
| D — Demand（需求） | H2 是否说明了"为什么需要" | 用户看不到相关性 |
| R — Reason（理由） | 是否给出了"为什么选这个" | 用户无法做比较 |
| E — Evidence（证据） | 是否有数据/案例/来源支撑 | 信任不足，AI 不引用 |
| A — Advantages（优势） | 是否展示了具体差异化 | 淹没在竞品同质内容中 |
| M — Motivation（动机） | 是否给了"现在行动"的理由 | 流量不转化 |

> DREAM 可直接嵌入 [06-内容Brief模板.md](03-模板库/06-内容Brief模板.md) 的大纲结构：H2 需求场景（D）→ H3 理由（R）→ H3 证据（E）→ H2 优势（A）→ H2 行动动机（M）。

**品牌语调配置**（项目级前置资产）：

首次 Brief 锁定前完成定义（存放在项目 `03-内容策略/品牌语调配置.md`），Brief 模板引用配置不重复定义；`00-项目简报.md` 只保留入口链接或状态说明。

**竞品段落素材采集**（分级触发）：

> 核心逻辑：AI 引用的是"最好的段落"，不是"最好的页面"。采集最优段落作为写作骨架，再在骨架上重写并注入原创内容。适用范围：博客文章（信息型/对比型/教程型），转化型页面不适用。

**触发规则**（完整分级表在此定义）：

| 内容类型 | 竞争强度 | 是否采集 | 替代方案 |
|---------|---------|---------|---------|
| 百科/科普 | 低 | **跳过** | 差异化模板 + SERP 结构复核 |
| 场景应用 | 中-高 | **执行** | Top 10-20 段落采集 + 评分 |
| 对比/选购 | 高 | **执行** | 完整段落采集 + 评分 |
| 教程/How-to | 中 | **可选** | 视同质化程度决定 |

- 判断标准是"该内容类型的竞争强度"，不是"项目大小"
- 采集结果写入 Brief 的 `竞品段落素材` 字段，AI 在骨架上重写 + 注入原创观点/数据
- 采集操作由竞品内容分析工具或人工流程承接；07 只负责把已入库素材纳入生产流程

**提纲生成方法参考 — Fan-out 驱动三步法**：

> 来源：「SEO技术流」zhidaow.com 自研工具的设计逻辑。将 SEO（竞手差距）和 GEO（查询扩展/PAA）统一到提纲生成环节。

```
步骤1：关键词扇出（Query Fan-out）
  → 模拟 AI 搜索的查询扩展，把目标关键词拆为用户的主要/次要/隐含子需求
  → 确保提纲覆盖用户全意图谱，而非仅围绕单一关键词

步骤2：竞手内容差距分析
  → 拉取 Google 首页排名页面，分析竞手已覆盖 vs 遗漏的话题
  → 原则："你有我有，你无我也有" — 用户关注的主要需求一定要覆盖 + 竞手遗漏的需求一定要覆盖

步骤3：PAA → FAQ 生成
  → 挖掘 People Also Ask 问题，生成文章 FAQ 段落
  → 同时提升用户需求覆盖率和 GEO 引用率
```

**与 RLM 流程的映射**：步骤1 对应 1C 关键词研究的意图扩展，步骤2 对应 1D/1F 竞品缺口分析的提纲化应用，步骤3 对应 3.6 质检路由中的 GEO 检查前置化。三步结果汇总后写入 Brief 的推荐大纲字段。

**提纲生成后的人工增值**（不可省略）：补充真实经验和行业认知（E-E-A-T）、加入 CTA、补充语料（产品列表/用户画像/使用场景）、给出内链需求。

**工具参考**：zhidaow.com/tools/article-outline/（免费，英文关键词 → 中文提纲输出，含建议字数和段落简介）

**关键输出**：已锁定的内容 Brief（含选题角度、竞品素材、差异化声明）+ 品牌语调配置文档。

**进入条件**：Brief 已通过 3.4 锁定门槛。

**内容质量地板**（4 项，任一不满足即打回）：

| 质量项 | 要求 | 不通过时的处理 |
|--------|------|-------------|

| 未答之问 | 回应了 SERP 中用户明确提出但竞品未回答的问题 | 补充 FAQ 或专门段落 |

**生产边界**：本节只管站内文本内容，生产 SOP 由 07 执行。社媒再加工归入步骤 4。

## 6. 项目类型适配

不同项目执行重点不同，先判断类型再决定结构和内容优先级：

| 项目类型 | 竞品重点 | 网站重点 | 内容重点 |

| 内容/Affiliate | 关键词集群、评测结构 | Topic Cluster、榜单页 | Best、Review、Vs、Guide |
