# 实体与知识图谱理论

> AI 搜索优化的底层认知框架。AI 不读关键词，读实体——被识别"概念"之间的关联网络决定了你是否被引用。基于 Princeton GEO 论文、Frase 实操指南、Search Engine Land 等研究。

---

## 核心认知：从关键词到实体

> 传统 SEO 优化关键词，AI 搜索优化实体——从字符串匹配到概念关联网络的根本转变。SEO vs GEO 的完整对比见 [01-内容质量标准](./01-内容质量标准.md)。

### 什么是实体

**实体（Entity）** = 一个有身份的"事物"。品牌是实体，产品是实体，概念也是实体。

每个实体有：
- **属性**：名称、类别、特征
- **关系**：与其他实体的关联
- **规范标识符**：Wikidata Q-ID、Google Knowledge Graph MID、或自定义 schema `@id`

```
关键词是字符串 → "content optimization"
实体是字符串指代的事物 → "Frase 是一个内容优化平台"
```

### 实体如何影响 AI 引用

AI 检索流程：
1. **查询扇出**：用户问题 → 多个子查询
2. **段落检索**：每个子查询命中段落索引，按实体清晰度/事实密度/权威性打分
3. **Grounded 生成**：选中的段落输入 LLM，合成答案。模糊/实体贫乏/矛盾的段落被过滤
4. **引用选择**：模型依赖最多的段落被展示为引用来源

> **LLM 不读整页——只读个别段落，按段落优化而非按页面。**（BrightonSEO April 2026，AirOps × Kevin Indig）

**实体密度与引用率**（BrightonSEO April 2026 补充数据）：
- **高实体密度**是被引用内容的核心特征之一
- 实体在**多个引用中出现** = 最可能被 AI 答案包含
- 覆盖 26-50% 子查询的页面优于覆盖 100% 的页面——精准聚焦 > 大而全

> **一个 2000 词堆满"内容优化"关键词的页面，会输给一个 400 词但清晰定义了品牌实体的段落。**

## 从实体优化到知识体系构建

> 上面的认知解决了「单个实体怎么让 AI 识别」的问题。但在 AI 搜索的当下，仅做好关键词调研、竞对调研、SERP 调研是不够的。更上游的问题是：**你是否用结构化的方式，让搜索引擎理解你覆盖了整个行业的实体、概念和关系？**

### 三层演进

| 层级 | 方法 | 典型操作 |
|------|------|---------|
| **传统 SEO** | 关键词驱动 | 关键词调研 → 竞对分析 → SERP 分析 → 内容生产 |
| **AI 搜索优化** | 实体驱动 | 实体识别 → 定义性开头 → Schema → 共现信号（上文 6 步工作流） |
| **知识体系驱动** | 结构化知识驱动 | 先构建行业知识体系，再用这个体系指导网站结构和内容生产 |

**核心思路**：先理解一个行业的概念体系，再用这个概念体系指导网站结构和内容生产——让各类搜索引擎（传统 + AI）更容易理解网站覆盖了哪些实体、概念、关系。

### 知识体系构建 4 步法

#### Step 1：收集行业受控词表（Controlled Vocabularies）

受控词表 = 一个行业里通用的、被广泛认可的标准化术语集合。

- 行业标准术语（如 SEO 中的 "SERP"、"Backlink"、"Crawl Budget"）
- 产品/服务类目标准（如 Google Product Category、Schema.org 类型）
- 竞品网站的导航结构和分类体系
- 用户搜索问题中反复出现的概念

> 受控词表的价值：确保你使用的术语与搜索引擎已理解的实体一致，而非自造词汇。

#### Step 2：从重要资源抓取结构化数据

| 数据源 | 获取方式 | 用途 |
|--------|---------|------|
| **Wikipedia** | 页面结构（目录、分类、infobox） | 行业概念的权威定义和层级 |
| **Wikidata** | SPARQL 查询实体和关系 | 可机器读取的实体关系网络 |
| **竞品网站** | 导航结构、URL 结构、H1/H2 层级 | 行业内容覆盖的边界 |
| **用户问题** | People Also Ask、AnswerThePublic、社区问答 | 用户视角的概念关联 |

#### Step 3：建立 Taxonomy / Ontology / Knowledge Base

| 层级 | 含义 | 产出 |
|------|------|------|
| **Taxonomy（分类体系）** | 概念的层级分类 | 分类树、导航结构 |
| **Ontology（本体）** | 概念之间的关系类型 | "属于"、"相关于"、"替代"、"前置条件" |
| **Knowledge Base（知识库）** | 实体 + 属性 + 关系 + 来源的完整库 | 可查询、可关联的结构化知识 |

> 这三层的产出直接映射到网站的信息架构（IA）和内部链接策略。

#### Step 4：用知识体系指导内容生产

知识体系的产出不是文档，而是**内容决策框架**：

| 知识体系产出 | 对应内容决策 |
|-------------|------------|
| 每个概念节点 | 至少一个覆盖页面（Pillar 或 Spoke） |
| 概念之间的层级关系 | 导航结构和 URL 层级 |
| 概念之间的关系 | 内部链接策略（锚文本 = 关系描述） |
| 缺失的概念节点 | 内容缺口（Content Gap）→ 新内容 Brief |
| 重复的概念节点 | 内容合并或差异化定位 |

### AI 辅助知识体系构建

构建行业知识体系是高认知负荷工作，AI 可以大幅提效：

- **概念提取**：从行业文档/Wikipedia 中批量提取实体和关系
- **Taxonomy 生成**：自动聚类和层级化
- **内容缺口识别**：对比知识体系与现有网站覆盖，自动发现缺口
- **Brief 生成**：基于知识体系中的实体关系，自动生成内容 Brief

#### 为什么 Obsidian 特别适合做知识体系构建

Obsidian 不是唯一的工具，但它在 SEO 知识体系构建场景有几个独特优势：

| 特性 | 作用 | 与 SEO 的对应 |
|------|------|-------------|
| **Markdown 纯文本** | AI agent 原生可读，无需 API 集成或格式转换 | 等同于搜索引擎偏好 HTML 而非 JS 渲染 |
| **双向链接 `[[ ]]`** | 概念之间的关联天然可视化 | 等同于内部链接 + 锚文本 |
| **YAML frontmatter** | 每个笔记的结构化元数据 | 等同于 Schema.org 结构化数据 |
| **Graph View** | 实体关系的可视化验证 | 等同于搜索引擎的实体图 |
| **Bases（数据库视图）** | 类似 Notion 的表格化管理 | 等同于内容计划矩阵 |
| **Obsidian MCP + Claude** | AI agent 直接读写 Vault | 实现批量概念提取、缺口分析、Brief 生成 |

> 关键洞察（Pablo Oliva, 2026）：Obsidian 的知识组织进化路径——文件夹→标签→关系属性→语义搜索→知识图谱→RAG——与 SEO 的进化路径（关键词→实体→知识体系驱动）是**同一个模式在不同尺度上的体现**。

#### "提前结构化"的复利效应

> "在 AI agent 到来之前做的所有手动打标签、建立关系属性、结构化笔记属性工作，正是让笔记现在能被 AI agent 消费的原因。你在组织信息上的投资，回报方式是你当时无法预测的。" — Pablo Oliva

这意味着：**现在为行业知识体系做的受控词表整理、Taxonomy 构建、实体关系梳理，在 AI 搜索进一步普及时会产生复利回报**。即使当前 AI 引擎尚未充分利用这些结构化数据，先构建好就是竞争优势。

#### SEO 场景的具体工作流

基于 SEOTISTICS 的实践和知识体系构建 4 步法：

```
1. 建 Vault → 创建行业知识库
   └─ 文件夹结构映射 Taxonomy 层级
   └─ 每个概念 = 一个 Note（YAML 属性记录：实体类型、关系、数据来源）

2. 采集数据 → 用 Obsidian Clipper / trafilatura 抓取
   └─ Wikipedia 页面 → 概念定义
   └─ 竞品页面 → 存入 Vault 做对比
   └─ 用户问题 → 存为独立 Note 标记为"需求侧"

3. AI 处理 → Obsidian MCP + Claude
   └─ 批量提取实体和关系 → 生成 Ontology
   └─ 对比知识体系与现有网站覆盖 → 发现缺口
   └─ 基于实体关系 → 生成内容 Brief

4. 持续维护 → 定期 Sweep
   └─ 新概念自动提醒（对比已有 Taxonomy）
   └─ 关系更新（概念间的关联变化）
   └─ 质量审计（检查孤立节点 = 未被链接的概念 = 内容缺口）
```

**Obsidian MCP + Claude 典型 Prompt 示例**（SEOTISTICS）：

```markdown
检查我的知识库 Content 文件夹，准备：
- 已覆盖概念的主题回顾
- 推荐下一个应该覆盖的概念/实体
- 现有内容如何复用到不同格式或渠道
结果按我网站的价值主张结构化呈现。
```

## AI 识别实体的 6 大信号

### 信号 1：规范标识符（Canonical Identifier）

实体需要一个 AI 引擎能解析的稳定 ID：

| 标识符 | 作用 |
|--------|------|
| Wikidata Q-ID | 连接公共知识图谱 |
| Google Knowledge Graph MID | 连接 Google 索引 |
| Schema `@id` | 自定义页面级标识 |
| `sameAs` 属性 | 将品牌链接到 Wikipedia、LinkedIn、Crunchbase 等权威来源 |

> Google Knowledge Graph 包含 5000 亿条事实，覆盖 50 亿实体。没有规范 ID = 对 AI 检索路径不可见。

### 信号 2：定义性开头句（Definitional Opener）

AI 检索系统偏好以清晰定义开头的段落。

**标准格式**：`[实体] 是一个 [类别]，[差异化描述]`

```
✅ "Frase 是一个 AI 驱动的 SEO 和 GEO 平台，帮助内容团队研究、写作和优化内容。"
❌ "在当今数字营销的世界中，内容优化变得越来越重要..."
```

> 这个句子往往占据引用的**不成比例的份额**，因为它是 AI 检索引擎最先提取的段落。

### 信号 3：事实密度 + 行内引用（Fact Density）

| 方法 | 效果 |
|------|------|
| 添加统计/引用 | +40% AI 可见度（Princeton GEO） |
| 每 150-200 词至少一个有来源的数据 | 高引用概率 |
| 模糊声明 | 被过滤 |

```
✅ "2025 年 1-5 月，AI 推荐流量增长 527%（Previsible 2025 AI Traffic Report）"
❌ "AI 推荐流量显著增长"
```

### 信号 4：实体共现（Entity Co-occurrence）

AI 引擎通过观察实体周围出现哪些其他实体来建立"类别画像"。

一个关于"内容优化工具"的页面如果从不提及 Google、搜索引擎、SERP、排名、AI 引用，在检索系统中看起来不像是内容优化页面。

> 页面中讨论的**相邻实体**（人物、方法、竞品、类别）越多，信号越强。

**品牌实体与AI引用的交叉验证**（BrightonSEO April 2026 补充）：
- **53%** 的非品牌 prompt 会触发至少提及一个品牌——品牌实体在 AI 检索中无处不在
- AI Search 已是品牌认知的 **Top 5 触点**
- 实体在多个来源（官网、G2、媒体、Reddit）中被一致描述 = 最可能被 AI 答案包含（与 04 文档中"共识层"概念一致）

**共现的框架：VP 与 USP**（鸭老师SEO 2026.05）：

共现需要回答两个问题：**共现什么** 和 **在哪里共现**。VP 和 USP 为"共现什么"提供了框架。

- **VP（Value Proposition，价值主张）**：你给用户带来什么具体价值。针对特定人群，可以有多个。
- **USP（Unique Selling Proposition，独特卖点）**：你与竞争对手的本质区别。一个品牌在一个市场中聚焦一个核心 USP。

> USP 一定是 VP，VP 不一定是 USP。VP/USP 是告诉 Google 你这个 Entity "代表什么"的语义线索。

**没有清晰 VP/USP 的 Entity 是模糊的**。一个网站今天说做 SEO，明天说做建站，后天说做广告——Google 无法判断它应该在哪类查询里被优先召回。模糊的 Entity = 可替代的 Entity。

**跨平台共现实操**：

| 维度 | 在哪里 | 共现什么 |
|------|--------|---------|
| **站内** | Title、H1、首屏文案、Schema description | 品牌名 + 核心服务 |
| **站外** | 目录 profile、社交媒体 bio、PR 稿、guest post author bio | 品牌名 + 差异化属性 |
| **第三方提及** | 别人引用你、评价你时自然带出属性 | 品牌名 + 目标市场 + 证据 |

共现不是堆砌关键词，是在所有 Google 能爬到的触点上，用一致的语言反复强化同一组属性，直到 Entity 从模糊变成清晰。

**LLM 的角色比喻**（BrightonSEO April 2026）：
> **LLM 是法官。来源是证人。你是律师** —— 你需要准备并影响证人。
>
> 你影响 grounding/RAG 的机会远大于影响模型训练数据的机会。你的战场是**触发网络搜索的 prompt**。

### 信号 5：结构化数据连接（Structured Data）

| Schema 类型 | 实体关系 |
|------------|---------|
| `Organization` + `sameAs` | 品牌实体连接到公共知识图谱 |
| `BlogPosting` / `Article` | 页面级实体定义 |
| `author` | 连接到 Person 实体 |
| `mainEntityOfPage` | 声明页面主要实体 |
| `mentions` | 连接到相关实体 |
| `FAQPage` | 问答型查询优化 |

> 只有 6-27% 的最常被提及品牌同时也是 AI 引用的 Top 来源——差距几乎总是**结构化数据连接**。

### 信号 6：新鲜度 + 交叉验证（Freshness & Corroboration）

| 信号 | 说明 |
|------|------|
| Perplexity 30 天内内容 | 82% 引用率 |
| Perplexity 12 个月以上内容 | 37% 引用率 |
| Claude 训练窗口内一致描述 | 跨源一致是关键 |
| 与权威来源矛盾 | 被过滤 |

> 新鲜、事实密集、与其他权威来源描述一致的内容被选中。矛盾的被排除。

## 实体优化 6 步工作流

### Step 1：每页确定一个规范实体

最常见错误：一个页面试图覆盖五个事物。

**做法**：在写任何内容之前，在 Brief 顶部写下这个页面关于**唯一一个**实体。

**测试**：一个陌生人读你的页面，能说出它关于什么吗？

### Step 2：写定义性开头句

在写引言之前，先写一句 `[实体] 是一个 [类别]，[差异化描述]`。

**测试**：AI 引擎能否将这句话原样提取，作为通用问题的独立回答？能 → 保留。不能 → 重写。

### Step 3：映射相邻实体

列出规范实体类别中自然归属的所有实体：
- 竞品、工具、方法、人物、概念

这些成为：
- 内部链接目标
- 正文中的关键词汇
- 向 AI 引擎发出"主题权威"的信号

**测试**：相邻实体是否出现在正文中？

### Step 4：添加连接实体身份的结构化数据

- `BlogPosting` / `Article` schema（页面级）
- `Organization` schema（品牌级）
- `sameAs` 链接到所有权威外部档案
- `mainEntityOfPage` + `author` 属性
- 适合时添加 `FAQPage` schema
- 发布前通过 Schema.org Validator 验证

**Schema 嵌套链示例**（层级关系声明）：

```
Article
  ├─ author → Person
  │    └─ worksFor → Organization
  └─ hasPart → 子内容/子产品页面
       └─ isPartOf → 父级内容（反向声明）
```

- `hasPart` / `isPartOf`：建立页面间从属关系，帮助 Google 和 AI 理解内容层级
- `author → Person → worksFor → Organization`：形成完整的实体身份链
- `knowsAbout`（声明核心专业领域）、`areaServed`（标注服务范围）：帮助 Google 理解"你是谁、你做什么、你服务谁"

**Organization schema 检查清单**：
1. `name` 和 `url` 与规范域名一致
2. `sameAs` 包含 Wikipedia、LinkedIn、Crunchbase、X/Twitter、YouTube
3. `logo` 引用真实 ImageObject
4. `@id` 使用稳定、规范的 URL
5. Schema.org Validator 无警告

**测试**：Schema 是否通过验证？

### Step 5：构建实体枢纽（Pillar + Spokes）

一个页面无法承载一个实体。Topic Cluster 模型：
- **Pillar 页面**：规范实体的综合页面
- **Spoke 页面**：更窄子实体的支持页面
- 每个 Spoke 用描述性锚文本链接回 Pillar

**测试**：Spoke 是否链接到 Pillar？

### Step 6：监控跨平台引用覆盖率

实体优化不是发布就结束的维护纪律。每月检查 ChatGPT、Perplexity、Claude、Gemini 等是否在目标查询中引用你的实体。

**测试**：你知道本周的引用份额吗？

## 实体覆盖评分

| 分数 | 状态 | 常见问题 |
|------|------|---------|
| **70+** | 健康 | — |
| **50-70** | 需改进 | 1-2 个信号缺失 |
| **<50** | 高风险 | 通常失败在：通用开头句、缺失 schema、无相邻实体、零行内引用、孤立页面 |

### 常见实体优化错误

| 错误 | 修复 |
|------|------|
| 一个页面五个实体 | 拆分页面，或选一个规范实体 |
| 关键词伪装成实体 | 围绕实体重写，让关键词自然跟随 |
| Schema 没有 `sameAs` | 添加 Wikipedia、LinkedIn、Crunchbase 等链接 |
| 关键词驱动的内部链接 | 围绕实体枢纽重建内部链接 |
| 发布就忘 | 建立引用监控 + 滚动刷新节奏 |

## 实体优化是维护纪律，不是项目

> AI 模型重训练 → 爬虫重新抓取 → 竞品发布更新版本 → 你的内容不变但引用消失了，因为**检索上下文变了**。

| 节奏 | 操作 |
|------|------|
| 高价值页面 | 每周监控 |
| 普通页面 | 每月监控 |
| 内容刷新 | 每 2-4 周（Perplexity）/ 每月（Google）/ 每 2-3 周（ChatGPT） |
| Schema 审计 | 每季度 |
