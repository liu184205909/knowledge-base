## 排名架构

```
Crawling → Indexing → Query Processing → Core Ranking(T*×Q*×P*) → Post-Ranking(Twiddler) → SERP(含AIO)
```

- **Core Ranking** 决定你是不是候选池
- **Post-Ranking Twiddler** 决定你在候选池里排第几
- AIO、Featured Snippet、PAA 都是 Twiddler 层的产物

### 排名的数学基础：检索、排序与信号组合

> 基于信息检索（Information Retrieval）文献整合。

搜索引擎是数学系统——每一个排名决策都是数学函数处理可量化输入后的输出。如果一个信号不能被系统观测、编码、建模为数值，它就不能进入排名函数，就不能影响排名。许多听起来抽象的概念（"内容质量""品牌信任""专业性"）每天都在通过**代理变量**被量化：语言模型评分、链接模式、实体识别、行为信号。

#### 第一阶段：检索——进入候选池（存亡级）

页面必须先被检索到才能参与排名。检索是多路径过程，主要依赖两个通道：

**BM25：词法关卡**

BM25 是词法检索的经典模型，考虑三个要素：
1. 查询词在文档中的出现频率
2. 该词在整个索引中的稀有程度（IDF）
3. 文档相对于平均长度的长短

输出是一个实数，越高表示词法匹配越好。BM25 具有**饱和特性**：关键词第 10 次出现的增益远小于第 1 次——这就是关键词堆砌收益递减的数学原因。

**稠密检索：语义关卡**

查询和文档各自被转化为高维向量（意义的数值表示），两个向量之间的余弦相似度衡量语义接近程度（0-1）。即使使用的词完全不同，只要语义接近就能匹配。RankBrain、BERT、MUM 等系统涉及交叉编码、上下文语言理解和多任务架构，远超基本向量比较，但基础思想不变：语言变成数字，数字被比较。

> **存亡约束**：页面必须通过至少一个关卡才能进入候选池。两个都没通过，再多的权威性或用户互动都救不了你——你根本不在池子里。

#### 第二阶段：排名——多层管道，非线性组合

被检索到之后，文档不是拿到一个总分直接排位。现代排名是多阶段管道：

```
轻量级初排 → 深度重排序模型 → 质量/垃圾分类过滤 → 多样性/时效性调整 → 展示层决策
```

**"为什么我的页面明明相关却不排名？"**——很多时候不是相关性的问题，而是在下游某一层闸门被过滤或降权了。

#### 信号组合：Learning to Rank（LTR）

现代搜索引擎使用学习排序（Learning to Rank）在海量标注数据上训练 ML 模型预测最优排序，而非固定线性公式。LTR 与固定公式的根本区别：

| 维度 | 固定公式（过时） | Learning to Rank（当前） |
|------|----------------|----------------------|
| 特征关系 | `score = w₁x₁ + w₂x₂ + w₃x₃` 线性 | **非线性**——特征间交互由模型学习 |
| 权重 | 静态不变 | **动态**——随查询类型、意图、垂直领域变化 |
| 上下文 | 不考虑 | 高权威性在商业查询中权重极高，在信息查询中权重下降 |

然而约束不变：进入 LTR 模型的每一个特征仍然是可计算的数值。数字进去，分数出来，按分数排序。

#### 快因素与慢因素（ Glenn Gabe）

Google 排名机制分两层：**快因素**（相关性等可快速变化的信号）和**慢因素**（Q/整站质量，包含大量组合元素）。短期爬升的数据一般都会经历 tank。

**慢因素的核心**：13个月用户数据（NavBoost）影响整站的天花板。当 Q（慢因素）追上快因素时，所有异常数据会如同萤火虫在黑夜一般闪亮——HCU 会在此时盯上你，进而打击整个域名。

**Q 天花板**：当整站到达你的 Q 极限时，无论产出多少内容、页面，流量会趋于直线。只有突破 Q 的底层信号（外链质量、用户行为、品牌权威积累）之后才能有进一步提升。这不是靠多发内容能解决的。

#### 实操简化公式

绝大多数 SEO 操作可以归入两个核心变量：

```
Authority × Relevance = Ranking
```

| 优化方向 | 具体操作 |
|---------|---------|
| **优化相关性** | slug/H1 精准反映查询意图、内容与查询高语义相似度、术语与排名页面对齐 |
| **建设权威性** | 主题相关高权威外链、内链结构合理分配权重、链接画像自然 |
| **获取正向用户信号** | 赢得点击的标题、满足查询需求的内容（NavBoost 校准） |

#### 评估 SEO 概念的三层过滤器

遇到任何 SEO 概念时，问三个问题：

1. **能被计算吗？** 不能直接计算，能被拆解成可计算的代理特征吗？如果都不能，它不参与排名函数
2. **怎么产生的？** 统计函数（BM25）、图计算（PageRank）、神经模型（嵌入相似度）、行为聚合（NavBoost）？理解机制才能知道优化什么
3. **在哪个阶段起作用？** 检索（存亡级）→ 初排（大致位置）→ 重排序（更深评分）→ 质量分类（过滤/降权）→ 展示层调整。**阶段决定优先级：检索阶段的信号是存亡级的**

> 许多流行概念是可计算信号的通俗描述："匹配搜索意图"= 词法+语义检索高分；"主题权威性"= 实体覆盖度+内链结构+入站链接相关性的组合。描述没错，但不精确。底层可计算特征才是搜索引擎实际处理的东西。

### Google I/O 2026：搜索进入 Agentic AI 时代

Google 明确宣告搜索从**生成式 AI 阶段**正式跨入**智能体 AI 时代（Agentic AI Era）**。未来的 Google 搜索 = 动态 UI + 自主智能体 + 原生交易。

#### 生成式 UI 与 Mini Apps（杀伤力 5/5）

**Google Antigravity 技术**：当用户搜索复杂问题或需要工具时，SERP 不再呈现网页，而是直接在搜索框里动态编写代码，生成定制交互式"微型应用（Mini App）"或动态仪表盘。

- **工具类网站和 Programmatic SEO 灭顶之灾**——计算器、文件格式转换器、大量模板页面直接在 SERP 被复刻
- 用户在搜索框里完成所有交互，根本不需要点击进入你的网站
- **自救路径**：融入独家私域数据（Proprietary Data）、用户社区互动、核心价值前置 + 会员锁（Gated Experience），将搜索流量沉淀为私域资产

#### 智能搜索框与多模态查询（杀伤力 4/5）

搜索框被彻底重构：随用户输入动态扩展，支持文本/图片/长视频/Chrome 标签页作为上下文，进行多轮自然语言对话。

- **短尾关键词加速失效**——用户不再浓缩为 2-3 个词，而是用自然语言描述复杂需求
- 用户会丢一张图片 + 一个工作软件标签页，直接问"我的电脑跑这个软件卡，怎么升级？性价比最高"
- **影响**：传统关键词研究工具的数据价值持续下降，语义理解和意图覆盖变得更加重要

#### 信息智能体 7×24 后台运行（杀伤力 3.5/5）

搜索引入 7×24 在后台持续运行的 AI 智能体。用户设定一次参数（如"监控西雅图某街区新出的、租金低于 $2000 且允许养狗的公寓"），智能体持续扫描全网，满足条件时弹窗通知。

- **静态/过期内容被无情抛弃**——智能体高频扫描时，信息过时会被瞬间踢出筛选清单
- 要求内容有持续刷新机制，尤其是价格、库存、行业报告、政策变动等时效性数据

#### Gemini 3.5 Flash 成为默认引擎

AI Mode 全球默认引擎升级为 Gemini 3.5 Flash，4 倍速度提升 + 极强多步骤推理能力。

- **可引证性（Citability）门槛暴增**——毫秒级对比几十个网页，洗稿/无独家观点内容被轻易剔除
- Google 加大 SynthID（内容水印）和 C2PA（来源追踪）普及 → 纯 AI 生成无背书站点引用权重跌入谷底
- SynthID 已打标超 **1000亿** 张图片和视频 + **6万年** 音频；C2PA 即将进入 Search 和 Chrome

### Firefly：规模化内容滥用检测引擎

> 基于 Content Warehouse API Leak 中的 `QualityCopiaFireflySiteSignal` 模块（Shaun Anderson 取证分析）。Firefly 是**站点级信号**，直接影响排名。

**模块命名解读**：Quality（质量生态）+ Copia（拉丁语"过量"，对应 scaled）+ Firefly（萤火虫算法，在海量数据中发现微弱操纵信号）+ SiteSignal（站点级聚合）

**追踪的关键属性**：

| 属性 | 机制 | 意义 |
|------|------|------|
| **内容生产速度** | 连续 30 天内新发现 URL 数量突变即触发审查 | 从每天 1 篇变 50 篇 = 立刻跳变 |
| **高质量内容占比** | 内部评分 ≥0.8 的页面数量 / 总页面数 | 发 1000 篇但只有 5 篇达标 = 强信号 |
| **dailyClicks vs dailyGoodClicks** | 来自 NavBoost，goodClicks = 用户未立即返回搜索 | **大量点击 + 低 goodClicks = 低质量的数学证据** |
| **临时排名提升追踪** | `impressionsInBoostedPeriod` + `firstBoostedTimeSec` | 新内容"测试窗口"表现不佳 → 后续机会递减 |
| **展示量突增预警** | `recentImpForQuotaSystem` 触发资源配额审核 | 突然几千新页面 = 潜在垃圾来源 |
| **AI 生成内容评分** | `racterScores`（AGC 分类评分，带版本历史） | 追踪 AI 内容比例趋势，突变即跳变 |

**关键影响**：Firefly 是站点级信号——一个站上的规模化滥用行为不只影响被标记的页面，而是影响**整个站点**的排名。（Patrick Stox / Ahrefs 确认：Ahrefs 把有风险的 programmatic 内容搬到独立域名 ahrefstop.com，避免站点级信号污染主站。）

**规模化策略的时间差本质**：Relevance Systems 快速找到相关内容，Quality Systems（含 Firefly）需要几周到几个月完成评估。黑帽策略赌的就是这个时间差——流量是真实的，但排名崩溃是必然的。这也解释了为什么需要不断注册新域名——每个站都是消耗品。

### SegIndexer：索引分层机制

> 页面不是"被索引了就有机会排名"。Google 内部的 SegIndexer 系统将已抓取页面分配到不同 serving tier，不同层级获得的排名机会天差地别。（来源：2004 年 Google Index Partitioning 专利 + 2024 年 Content Warehouse API Leak 中的 SegIndexer 模块。2026.4 Google Search Central Live Toronto 官方确认：AI 降低了内容创作门槛，迫使 Google 提高了索引标准。）

**分层逻辑**：

| 层级 | 质量定位 | 排名机会 |
|------|---------|---------|
| **高 tier** | 高质量页面 | 用户查询时**优先搜索**，绝大多数搜索结果来源 |
| **中 tier** | 中等质量页面 | 仅当高 tier 结果不够时才扩展搜索，机会大幅减少 |
| **低 tier** | 低质量/重复/thin content | **排名过程开始前就被跳过**，等同"已抓取-未编入索引" |

**核心影响**：

- **站点级信号决定新页面初始分配**：整站 static rank 低 → 新页面默认进低 tier → 没有曝光 → 没有用户信号 → 继续低 tier（自我强化的死循环）
- **"Crawl Budget 不够"是常见误诊**：真正瓶颈在 serving 层而非 crawling 层。Google 抓了你的页面但分配到低 tier，优化 crawl budget 无济于事
- **HCU 的本质就是站点级 tier 重新分配**：整站被标记为"unhelpful" → 所有页面的 tier 被整体压低（Glenn Gabe 追踪数据：被 HCU 打击的站近两年大部分未恢复）
- **翻盘路径**：唯一有效的是改变 static rank 的底层信号——外链质量、用户行为、站点 authority 积累。对已在低 tier 的单个页面反复修改，ROI 通常不如把精力花在新页面上

> ⚠️ 注：具体分为几层、内部名称是什么，是外部基于专利和 API 文档的推演。分层机制本身可靠，但精确层级划分不应被视为 Google 官方确认。

### 收录问题的本质：权威性不够

> 来源：David Quaid（Primary Position）实验

很多人以为收录问题是技术问题——检查 noindex、robots.txt、服务器、孤岛页面。排除明显技术错误之后，**收录问题本质上只有一个原因：权威性不够**。

**David Quaid 实验**：一个高权威域名上的页面，哪怕几乎没有什么实质内容，也能被 Google 快速收录。反过来，一个新站上精心撰写的文章可能迟迟不被索引。区别就在于域名积累的权威性（PageRank）。

**与 SegIndexer 的对应**：这正是分层机制的体现——新站 static rank 低 → 新页面默认进低 tier → 没有曝光 → 没有用户信号 → 继续低 tier。**解决收录问题的根本路径不是反复提交 URL，而是建设网站权威。**

## 排名公式：T* × Q* × P*

### T*（Topicality 主题相关）— ABC Signals

| 信号 | 含义 |
|------|------|
| A = Anchors | 外链锚文本 |
| B = Body | 页面内容与查询匹配 |
| C = Clicks | 用户点击行为 |

### Q*（Quality 站点质量）— 三源交叉验证

- **subdomain 级别**评分（www.example.com 和 help.example.com 是不同分数）
- **0-1 区间，0.4 是 SERP features 硬门槛**（Featured Snippet / PAA）

Q* 的三个核心输入：

| 输入 | 含义 |
|------|------|
| Brand Visibility | 有多少查询直接包含品牌名 |
| **Selection Rate** | 用户在 SERP 主动点选你的比例，**特别是你不在第一位时** |
| Anchor Text Brand Prevalence | 锚文本中包含品牌名/站名的比例 |

**Selection Rate 是最反直觉但最重要的信号**：
- 排第 5 但用户跳过前 4 个直接点你 → Selection Rate 高 → Q* 增加
- 排第 1 但用户划过不点 → Selection Rate 低 → Q* 下降

### P*（Popularity 受欢迎度）— NavBoost

**NavBoost 是最重要排名信号之一**（Pandu Nayak 宣誓作证，2019年内部邮件称"独自可能比其他因素加起来更强"）：

| 变量 | 含义 |
|------|------|
| goodClicks | 点击后满意（停留、无bounce） |
| badClicks | 点击后立刻返回 SERP |
| lastLongestClicks | 最长停留的那次点击（"终极满意"信号） |

- **13 个月滚动窗口** → 短期 CTR 操纵基本无效
- **国家/语言/设备分层** → 各市场独立评分
- **累积优势** → 持续好的点击信号形成竞争对手短期无法复制的壁垒

**Glue 系统**：NavBoost 之外，Google 还有 **Glue 系统**进一步聚合 Chrome 跳转、搜索点击等多渠道数据。借助 Chrome 全球过半份额，**用户在非 Google 站点上的访问路径同样被纳入排名信号**。API Leak 中的 `chrome_trans_clicks`、`uniqueChromeViews` 字段证实了 Chrome 数据的直接参与——意味着用户行为信号的采集范围远超搜索结果页内的点击。（来源：DOJ 反垄断审判文件 + API Leak）

## Site Quality Score：新站"先飞后崩"

Google 有"predicting site quality"专利，新站初始分继承自向量相似的已有站：

1. AI 内容基于互联网最好内容训练 → 向量像高质量站 → 初始分 0.8-0.9 → 排名飞升
2. 6-12 个月后真实用户信号跟不上 → 分数下调 → 排名崩盘

> 核心判断："如果因为任何原因你失去可见度，Google 发现没人主动搜你——你就不存在——那你的站就完了。"

### 2026年5月核心更新实证：先飞后崩模式的再次验证

> 来源：Glenn Gabe 追踪数据。2026年5月核心更新：5月21日开始，6月2日完成，历时11天，比3月更新更猛烈。

**更新特征**：
- 48小时内开始见效（通常大规模核心更新都是如此，但3月更新反常地温和）
- 两次"震荡"（更新过程中其他系统同步更新导致剧烈波动），第二次震荡在6月2日最后一天尤其猛烈，部分网站从 Google Discover 中消失
- YMYL 领域（健康、金融、赌博）再次受到早期且强烈的冲击

**Grokipedia 案例**（100% AI 生成内容网站）：
1. 初始阶段：内容基于互联网最好内容训练，向量像高质量站，排名飞升
2. 2026年2月：Google 人工干预或算法惩罚，排名急剧崩盘，ChatGPT 引用量同步下跌（AI 搜索通过 Google 索引获取答案）
3. 2026年5月核心更新后：排名再次大幅增长
4. Glenn Gabe 判断："非长久之计，完全预料到会再次因算法原因下降"

**关键洞察**：Google 在 Google 搜索中的排名变化会**级联影响 AI 搜索**——ChatGPT/Perplexity 通过 Google 搜索获取答案，Google 排名下降 → AI 搜索引用同步下降。这验证了"对 SEO 有害 = 对 AI 搜索也有害"的判断。

### siteQualityStddev：一致性比最高质量更重要

`siteQualityStddev` 是网站页面级 PQ 评分（pqData）的**标准差估计值**——衡量站内页面质量波动程度。标准差越高，说明站内页面质量越参差不齐。

**核心逻辑**：

```
低质量内容 = 低 pqData 整数值
"影响整个网站" = 高 siteQualityStddev
"删除、合并或迁移" = 降低标准差的唯一统计学方法
```

**实操意义**：

- **你的排名不仅取决于你最好的作品，还取决于你与卓越水平的偏差**
- 一个站上有 10 篇优秀文章和 100 篇垃圾 → siteQualityStddev 很高 → 那 10 篇优秀的排名潜力会被拖垮
- 删除/合并低质量页面、将内容单薄页面合并为更有用页面、将低质量页面迁移到不同域名 → 这些都是降低标准差的手段
- Google 2011 年 Panda 指南就已明确说了这个逻辑，Leak 只是揭示了具体执行属性

## HCU 真相：实体未定义

Google 官方说 HCU 评估"内容是否为人写的"，三源真相：

- HCU 是 **site-wide 信号**，不是 page-level
- 2024年3月已合并进 core ranking
- 机制是 **demotion-first**（只降权，不提升）
- **真正触发降权的不是"内容不够好"，是"实体未定义"**

### Disconnected Entity Hypothesis（根因链条）

```
实体未定义 → Google无法评估"你为什么存在" → 被归类为"Unhelpful" → HCU site-wide demotion
```

**恢复路径**：不是优化内容，不是改技术 SEO，是**定义实体**：
- About 页、作者信息、schema、sameAs、真实业务证据
- Google Quality Rater Guidelines Section 2.5.2

### HCU 传导机制（API Leak 证据）

```
HCU classifier → CompressedQualitySignals → scaledSelectionTierRank → 页面被分配到低 tier（如 Landfills）
```

- HCU 打击是**断崖式**的——不是慢慢下降，而是从索引层面直接被过滤掉
- 放进 Landfills 后，检索阶段不会调出，NavBoost 数据归零，排名消失
- Shaun Anderson（hobo-web.co.uk）对数十个 HCU 受害站分析证实：核心问题是 Entity Health 和 Trust 缺失，不是某篇文章写得好不好
- 2024年10月 Google Creator Summit 上，搜索 VP Pandu Nayak 承认受影响站长的内容本身没有问题

### Synthetic Gap（Tom Capper / Moz）

HCU 受害者共性：**Domain Authority 远高于 Brand Authority**（DA:BA = 2:1+）。
- 你外链攒得快，但没人搜你的品牌 = 高风险
- HCU losers 平均 Brand Authority 37，winners 50-52

## 三层链接索引机制

| 层级 | 条件 | 效果 |
|------|------|------|
| **高质量** | 页面上的链接获得真实用户点击 | 正常传递 PageRank 和锚文本信号 |
| **中质量** | 少量互动 | 部分传递 |
| **低质量** | 页面上链接 **TotalClicks 为零** | **排名算法实际上忽略该链接**，不传递 PageRank，不传递锚文本信号 |

**核心机制**：零点击页面上的所有内链和外链都不传递任何价值。你以为在通过内链把权重导向 money page，实际上这些链接被归入了低质量层，什么都没传递。

**anchorMismatchDemotion**：在完全不相关的页面上放置精确匹配锚文本的链接不只是浪费——**可能对排名造成实际损害**。

**实操结论**：获取链接不再是终点。让链接所在的页面获得真实用户互动验证，才是让链接产生价值的前提条件。批量发布几千个无人访问的页面，页面上的所有内链和外链全部失效。

**外链评估三标准**（优先级从高到低）：

| 优先级 | 标准 | 说明 |
|--------|------|------|
| 1 | **相关性** | 来自同行业、同主题网站的链接价值远高于不相关的高权重网站 |
| 2 | **域名权威性** | 链接来源网站本身的 PageRank 和整体权威度 |
| 3 | **链接页面的流量和排名** | 有真实流量和排名的页面上的链接，远比无人访问的页面有价值——前者意味着页面本身有 Google 的信任 |

> 相关性 > 域名权威 > 页面流量。一个同主题低权威站点的链接，可能比一个不相关高权威站点的链接更有价值。

## 内链置信度：信心即信号

### 核心发现：用户信心 = Google 置信度

UIE 的 Lincoln 研究发现：**前三次点击的信心值就能预测用户最终能否找到目标内容**。区分好网站和差网站的不是导航结构，不是页面层级，而是每一次点击时用户对链接的信心。

Google 专利 US 7,590,628 描述了完全相同的判断逻辑——只不过不叫"信心"，叫**置信度（confidence）**。

Bill Slawski 在 2022 年明确将两者联系起来：用户寻找的是锚文本与目标页面内容之间的匹配关系（confidence），Google 在实体识别中使用的也是同样的匹配关系。**底层逻辑完全相同。**

### 触发词（Trigger Words）

信心高低的关键因素是锚文本中是否包含**触发词**——直接对应用户当前意图的词语。

| 用户意图 | 高信心锚文本（触发词命中） | 低信心锚文本（无触发词） |
|---------|------------------------|----------------------|
| 查新车价格 | "Price with Options" | "Learn More" |
| 了解设备类型 | "Types of Squat Machines" | "Our Products" |
| 对比两种设备 | "Smith Machine vs Power Rack" | "Click Here" |

### Google 置信度三层累加模型

当 Google 判断"这个页面的主题是否是某实体"时，检查三层信号，**每多一层一致信号，置信度往上走一格**：

| 层级 | 信号 | 含义 |
|------|------|------|
| **第1层** | 页面标题包含实体名称 | 页面自身声明 |
| **第2层** | 同域名相关页面（peer documents） | 同站其他页面是否围绕同一实体 |
| **第3层** | 站内锚文本使用实体名称 | 其他页面怎么描述这个页面 |

三层叠加 = 页面标题含实体 + 同站有相关主题页面 + 站内锚文本用实体名称 → 置信度最高。

### 意图收窄：层次来自用户旅程，不是站点结构图

层次不是从上到下的页面层级，不是 pillar-cluster 架构图。**层次来自用户意图的收窄过程**：

```
模糊意图（"谷歌SEO"）→ 收窄（"谷歌SEO教程"）→ 再收窄（"如何提高排名"）→ 行动（"获取报价"）
```

每条内链的锚文本应基于"用户读到此处自然会想知道什么"来设计，而非"这两篇属于同一个 cluster 所以链接它们"。

**同一个目标页面，从不同文章链过去时，锚文本应该不同**——因为用户走到那篇文章时的语境不同，脑子里的"下一步"也不同。

### 常见做法的有效性判断

| 做法 | 判断 | 原因 |
|------|------|------|
| **统一 CTA banner（"立即咨询""了解更多"）** | 无效 | 不携带主题信号，置信度为零；用户还在探索阶段就推送行动，具体性倒退 |
| **"点击这里""here"** | 最差 | Google 实体提取时直接删除这类文本，置信度为零，无触发词 |
| **图片 alt 作为锚文本** | 有效 | alt 精确描述目标页面主题时，效果等同文字锚文本（三层信号全部对齐） |
| **CTA 放在意图收窄到行动层** | 有效 | 从"了解参数"到"获取针对这个型号的报价"，具体性继续累加 |

**核心原则**：CTA 不是不该用，而是应放在用户意图已收窄到"我要行动"的那一层。提前推送 = 信心断裂。

### 实操框架

```
内链不是 PageRank 管道，是意图收窄的路标。
每一步锚文本比上一步更具体 → 用户信心累加 → Google 置信度同步提升。
```

| 原则 | 做法 |
|------|------|
| 锚文本 = 目标页面的精确描述 | 让锚文本准确描述链接另一端的内容 |
| 同一目的地，不同路标 | 从不同文章链到同一页面时，锚文本因语境不同而不同 |
| 每一步比上一步更具体 | 从泛到窄，不能倒退（具体性倒退 = 信心断裂） |
| 优先保证锚文本携带主题信号 | "点击这里"是零信号，统一 CTA 是零信号 |

**内链权威传递衰减规律**（基于 PageRank 原理）：

- 每一跳削减本体 **85%** 的权威
- 控制内链在 **三跳之内**
- **超过 4 跳，权威传递几乎为零**

这意味着：money page（核心产品页/转化页）必须尽量靠近首页或高权重页面，信息型内容页面通过内链"喂"权重给 money page 时，中间的跳转层数越少越好。

## 虚伪惩罚：声称与实际不匹配

Google 的 Quality Raters 被训练去寻找**网站声明和实际行为之间的不匹配**。Section 7.0 明确指出：声明目的与实际内容之间存在不匹配的页面应被评为最低质量。

**触发条件**：About 页面写着"资深专家团队，每篇内容严格审核"，实际每天发 50 篇未编辑 AI 内容 = Deceptive Page Purpose。

**Leak 属性对应**：可能触发 `scamness`（欺骗性评分）和 `unauthoritativeScore`（非权威评分）。

**核心原则**：内容生产价值必须与政策声明对齐。不能用精心制作的 About 页面伪造 E-E-A-T，如果实际产出与声明严重不符。

## 四种系统性失败模式

| 模式 | 表现 | 根因 |
|------|------|------|
| **语境过度延伸**（Context Overreach） | 个人博客排名"心脏病治疗方案"等 YMYL 查询 | 网站试图在需要更多信任/权威的查询中排名，超出了合理范围 |
| **实体膨胀**（Entity Inflation） | 没有外部证据却以权威机构身份出现 | 通过 EntityAnnotations 和 Knowledge Graph 验证身份，无证据 = 信任摩擦 + 排名阻力 |
| **意图漂移**（Intent Drift） | 页面对不满足的查询暂时排名靠前，最终下降 | NavBoost 的 `lastLongestClicks` 机制——用户搜索任务未在你的页面完成 → 排名随时间衰减 |
| **信任信号债务**（Trust Signal Debt） | UX 问题、激进盈利、信息不透明侵蚀信任 | `clutterScore`（杂乱度评分）等属性追踪，悄无声息地累积 |

**共同特点**：都不是内容质量问题，是系统性和背景性问题，单靠写更好的内容或获取更多外链无法解决。需要从站点定位、实体定义、用户验证等层面系统性地修复。

## 关键误区修正

### Sandbox：不是新站专属

hostAge 只 sandbox "fresh spam"，不影响干净新站。**PageRank 是跳过 sandbox 的 VIP 通行证。**

触发场景：
- 新域名 + 突然大量内容（内容农场）
- 老域名 + 突然改主题 + 批量内容
- **不要为了"域名年龄"买老域名——买来改主题比新域名还惨**

### Freshness：只对 QDF 查询生效

QDF 只对 3 类查询启动：突发新闻、周期性事件、频繁变动话题。**三者同时满足才激活。**

| 指标 | 2017 | 2025 | 趋势 |
|------|------|------|------|
| Top 10 中 3年以上页面 | 59% | 72.9% | 老内容占比更高 |
| 新页面 1年内进 Top 10 | 5.7% | 1.74% | 大幅下降 |
| #1 页面平均年龄 | 2年 | 5年 | 翻倍 |

**结论**：2026年 SERP 比以往任何时候都更被老内容主导。

### lastmod：信任是 binary 的

Gary Illyes 原话："It's binary. We either trust it or we don't."

| 改动类型 | Google 反应 |
|----------|-------------|
| 大改动 + 更新 lastmod | 正向信号 |
| 小改动 + 更新 lastmod | 负面，多次后失效 |
| **完全不改 + 更新 lastmod** | **最负面，直接"骗子"判定** |

### 301 重定向：不丢排名

> 基于 Google 官方《Site moves with URL changes》《Redirects and Google Search》文档 + 去重专利 US 8,868,559 / US 9,779,072 + API Leak PerDocData 容器

**"301 丢排名"是范畴错误**——把排名当成了存在 URL 字符串上的存量，实际上排名是查询时现算的。真正要问的是：产生排名的信号能不能合并过去。

| 信号 | 301 后能否转移 | 依据 |
|------|--------------|------|
| **权威（PageRank）** | ✅ 完全转移 | 2016 年前最多和一条链接一样有微小阻尼；2016 年起 Gary Illyes 确认 30x 不再丢 PageRank；官方文档明确"301 不会导致 PageRank 损失" |
| **相关性** | ✅ 自带（绑内容不绑 URL） | 目标页内容等价就一样 |
| **交互信号（NavBoost）** | ✅ 随规范化归集 | 301 本身就是合并——源 URL 被收为规范页的"别名"，信号向规范页归集（去重专利：索引单位是内容指纹定义的代表性文档，不是字面 URL） |

**乘数都在（Authority × Relevance），积自然还在。URL 字符串不是第三个乘数。**

**大家"看到"的掉量是处理窗口，不是信号损失**：Google 要重抓旧 URL、识别 301、合并信号、传播索引，期间旧的在退、新的没接上，排名就晃。官方定性为"暂时波动，会稳定下来"。

**301 真正导致永久掉量的情形**（全是混淆变量，没有一条是"301 本身少传了信号"）：目标页不等价、重定向链/环、同时改内容/结构、蚕食、窗口期撞算法更新。

**状态码（301/302/307）本身不是排名因素**：Google 把重定向、canonical 归在「抓取与索引」下，与「排名与搜索呈现」分属两块。重定向只是"目标应为规范"的信号，决定哪个 URL 入索引、信号往哪并，不进排名打分。

### Parasite SEO：已死

2024.3 Site Reputation Abuse 政策 → 2024.11 人工处罚 Forbes/WSJ/Time/CNN → 2025.8 算法化执行 → 2026.3 Core Update 强化。

### Crawl Budget：多数情况下是误诊

页面没排名，多数人归因于"crawl budget 不够"。但真正瓶颈在 serving tier（见上方 SegIndexer），不在 crawling。Google 抓了你的页面但分到低 tier，优化 robots.txt / sitemap / 服务器速度都无济于事。**把 serving 层的分区问题错误诊断成 crawling 层的资源分配问题，是治错了病。**

### 站点地图：不直接影响排名和抓取速度

> 来源：John Mueller 多次公开表态

**XML Sitemap** 不会直接影响排名或抓取速度，数量和大小也不影响抓取过程，Google 更不会对每个网站都去读取它。站点地图只是**辅助发现 URL 的信号之一**，不是保证。对于内链结构清晰的网站，它的主要作用是方便在 GSC 中按板块监控索引状态。

**关键区分：XML sitemap vs HTML sitemap**：

| 类型 | 作用 | 是否传递权重 |
|------|------|------------|
| **XML sitemap** | 给爬虫的 URL 清单，辅助发现 URL | **不传递权重** |
| **HTML sitemap** | 用真实链接列出全站重要页面的页面，走内链通道 | **传递权重**（走内链通道） |

对于页面量大、层级较深的网站，可以额外做 HTML sitemap：既帮爬虫发现深层页，也能把权重导向埋得过深的页面，弥补 XML sitemap 不传权重的短板。小站层级本来就浅，通常不需要。

### Core Web Vitals：是排名信号，但是弱信号

CWV 是确认的排名信号之一，但**不是强信号**——它不会把你从第 10 页推到第 1 页。在其他条件接近的竞争中，CWV 更好的页面会有优势。

**优先级判断**：CWV 优化应在实体健康、站级质量、内容相关性等核心层之后做，不是 SEO 的起点。提速核心（图片 WebP、CDN、缓存、压缩 JS/CSS）做好基础即可，过度优化 CWV 的边际收益递减。

### llms.txt 和 Markdown 转换：对 SEO 无用

> 2026.4 Google Search Central Live Toronto 官方确认：创建 llms.txt 文件对 SEO 没有任何好处。将网站转换为 Markdown 格式对于 LLM 或 SEO 也无任何好处。

这两个是 GEO 领域流传甚广的做法，Google 官方直接否定了。**不要在这两件事上浪费时间。**

### data-nosnippet：阻止 AI 引用的唯一方法

屏蔽 Google 扩展搜索机器人（Extended Search Bot）**不会阻止 AI 在 AIO/AI Mode 中使用你的内容**——因为你的网站已经在搜索索引中，Google 仍然可以在"分页搜索"中使用这些数据生成 AI 答案。

**唯一真正有效的方法是 `data-nosnippet`**。但这把双刃剑：阻止了 AI 引用，同时也会降低传统 SEO 效果（SERP 中不显示摘要）。

> 实操判断：如果你的流量主要依赖传统搜索 → 不建议用。如果你的内容被 AI 大量引用但不带来点击 → 可考虑对特定段落使用。

### 规模化内容滥用 vs AI 使用

Google **不反对 AI 本身**。AI 内容流量下降的根因是**规模化内容滥用算法**（Scaled Content Abuse），不是"你用了 AI"。

区别：用 AI 写 1 篇高质量内容 = 没问题。用 AI 批量生成 1000 篇内容 = 触发规模化滥用检测。

### AI 翻译规模化：2026年5月核心更新的新案例

> 来源：Glenn Gabe 追踪（2026年5月核心更新）

某网站先规模化大量低质量内容，再通过 AI 翻译扩展到多种语言/国家组合。Google 之前解释过"只要内容质量高，AI 规模化翻译可以接受"，但问题在于**并非所有网站所有者都能准确判断内容质量**。

2026年5月核心更新后，该网站遭受重创。这构成"滥用规模化内容"的典型案例。

**风险提示**：Reddit 在多国规模化数千万 URL 的 AI 翻译是成功的，但这与 Reddit 的域名权威、真实用户生成内容和品牌认知有关。普通网站复制此策略风险极高。

### AI 搜索操纵：Google 即将出手

> 来源：Glenn Gabe 追踪（2026年5月核心更新）

部分网站使用高风险 SEO 策略操纵 AI 搜索排名（AI Overviews、AI Mode、Gemini）。5月核心更新后，部分此类网站排名下降，部分保持稳定，部分反而上升——表明 Google **尚未通过这次更新专门解决此问题**。

但 Google 已发布辟谣文档揭露这些高风险策略，并更新了 AI 搜索相关指南。Glenn Gabe 预测：Google 会采取**人工干预 + 针对垃圾邮件的多次更新**相结合的方式。

**警告**：一旦被 Google 惩罚，网站会从"十大蓝链接"、"AI Overview"、"AI Mode"和"Gemini"等所有搜索结果中消失。且因为 ChatGPT 等平台通过 Google 搜索获取答案，Google 排名下降 → AI 搜索排名进一步下降（**级联惩罚效应**）。

**另一个案例**：某网站因操纵新闻排名被人工处罚并从索引移除，调整后撤销了人工处罚。但5月核心更新后，算法层面的改动导致该网站在"头条新闻"等功能的排名再次暴跌。结论：**人工处罚撤销 ≠ 算法层面的安全**，Google 会在后续核心更新中通过算法"补刀"。

## Schema 和实体建设

Schema 不是直接排名因素，但是**实体建设的加速器**：

```
Schema → 实体消歧加速 → 实体权威建立 → Knowledge Graph 认可 → LLM 引用概率提升（间接）
```

**分清主次**：

| Schema 类型 | 价值 | 状态 |
|------------|------|------|
| **实体消歧类**（Person、Organization、Service + areaServed/serviceType/provider + sameAs） | **真正普适的价值**——明确告诉 Google"页面主体是谁、做什么、服务谁、由谁负责" | 长期有效 |
| **富媒体类**（如电商 Product，展示价格、星级） | 特定场景能提升 CTR | 有效 |
| **FAQPage** | 曾用于富媒体展示 | **2026 年起 Google 已取消** |

> 多数人忽略的 Schema 真正价值是**实体消歧**，而非富媒体展示。用 Person/Organization/Service 配合 areaServed、serviceType、provider，再用 sameAs 把品牌实体关联到各平台账号，是加速品牌实体建立、让 Google 在知识图谱中识别你的关键。

### 2026 年实体建设 3 个核心条件

1. **Notability**：20-30 个独立权威来源提及
2. **Entity Home**：一个 URL 作为"真相来源"（通常是 About 页）
3. **Corroboration**：所有平台信息完全一致

### Google Search Profile：个人创作者获取知识面板的快捷方式（2026.6）

Google 于 2026 年在美国市场正式推出 **Search Profile（搜索名片）**——创作者专属聚合主页。URL 格式：`https://profile.google.com/@rustybrick`

**定位**：Search Profile 主要面向 **Publishers/Creators（个人创作者）**，不是面向企业的直接通道。企业无法以 Organization 身份申请 Search Profile。但企业如果有知名创始人/CEO 且其社媒粉丝达标，可以以 Person 身份申请，间接为企业建立实体信号。

**核心功能**：
- **社媒内容聚合**：展示最新文章、视频、社交帖子
- **一键关注**：用户关注后，Discover 信息流优先展示该创作者内容（类似 Preferred Sources）
- **个性化定制**：头像、简介、官网链接、关联社媒、置顶作品
- **知识图谱联动**：认领 Search Profile 可能**触发自动创建 Knowledge Panel**；已有面板则获得功能增强

**对 SEO/GEO 的影响**：
- **实体证明**：达到粉丝门槛 = Google 确认你是有效 Entity → 站点权重和信任度加成
- **创建知识面板的新方式**：之前需要 20-30 个权威来源提及 + Wikidata + 长期积累；现在认领 Search Profile 可直接触发
- **Discover 流量增长**：关注用户的 Discover 信息流优先展示内容（Discover 流量在 GSC 中显示为自然流量）
- **GEO 权威信号**：多平台活跃 + 数据互通 → AI 判定为领域"权威专家" → 引用概率提升

**准入门槛**（目前仅美国，且主要面向个人创作者）：
- YouTube ≥ 100K / Instagram ≥ 100K / X ≥ 100K / TikTok ≥ 300K 粉丝
- 需年满 18 岁
- 入口：https://profile.google.com/
- 重大修改需 Google 人工审核

**实操建议**：
1. 符合条件的个人/KOL 尽快用美国 IP 登录认领，抢注与品牌名一致的 Handle
2. Search Profile Insights 提供数据：Top 5 文章、点击量、展示次数、主要国家/地区
3. 对于未达门槛的个人：将"达到 Search Profile 准入门槛"列入社媒建设路线图
4. **对于企业**：走下方"企业知识面板创建四阶段法"，Search Profile 不是企业路径

### 企业知识面板创建指南：KPEB 框架 + 四阶段法

> 来源：Instant Press（2026-04）、Stay Digital Marketers（2026-06），基于数十个成功触发案例的实战经验

#### KPEB 框架：Google 决定谁获得知识面板的四个条件

| 条件 | 含义 | 典型失败原因 |
|------|------|------------|
| **K – Known（已知）** | 实体在 Google 信任的来源中被一致命名和描述 | 不同平台用不同名称/描述 |
| **P – Proven（已证实）** | 有独立、权威的来源证实实体存在 | 只有自发布的信息，缺乏第三方验证 |
| **E – Exact（精确）** | 实体无歧义，Google 能将其与同名实体区分 | 品牌名太通用（如"Summit Digital"） |
| **B – Bridged（桥接）** | 实体网站、结构化数据和外部档案之间有明确链接 | 没有 sameAs 桥接，Google 靠推断连接各信号 |

**大部分企业不是四项都失败，而是卡在一两项上。** 诊断弱项比全面优化更高效。

#### 四阶段法（从零到触发知识面板）

**Stage 1：建立结构化数据基础（1-2个月）**

在 Google 知识图谱读取的数据库中建立实体的规范记录：

| 操作 | 优先级 | 详情 |
|------|--------|------|
| **Wikidata 条目** | **最高** | 开放可编辑数据库，Google KG 直接读取。无需达到 Wikipedia 门槛。约30分钟创建，永久存续。详见下方 Wikidata 深度指南 |
| 网站 Organization Schema | 必须 | JSON-LD 格式，含 name/url/logo/foundingDate/address/**sameAs** |
| Crunchbase 档案 | 推荐 | 完整填写创始信息 |
| LinkedIn 公司页 | 必须 | 完整工作经历和公司信息 |
| Google Business Profile | 本地业务必须 | 每个可用字段都填写，描述含地理+行业关键词 |

**`sameAs` 是最被低估的属性**：显式桥接网站到 LinkedIn、Crunchbase、GBP、Wikidata。没有 sameAs，Google 靠推断连接各信号——更慢、更不可靠。

**Stage 2：构建权威引用密度（2-6个月）**

Google 不会仅凭结构化数据就触发面板，需要看到实体在**多个独立权威来源**中被一致描述。

| 实体类型 | 目标来源 |
|---------|---------|
| B2B 企业 | Clutch、G2、BBB、行业协会、行业媒体、地方商业期刊 |
| B2C 品牌 | 行业媒体、测评网站、地方新闻 |
| 个人/专家 | 行业出版物署名文章、会议演讲者介绍、大学网站、研究报告引用 |

**核心原则**：5 篇文章在 5 个不同权威出版物中，每篇以相同方式描述你的公司 > 50 篇文章在同一出版物。Google 实体系统寻找的是**独立佐证**，不是重复。

**公关建议**：每篇报道应一致使用同一实体名称，描述相同的核心属性（创始年份、行业、位置、核心产品）。Feature article（包含创始人名称、公司描述、引述）> press release 转载。

**Stage 3：Wikipedia 文章（可选，最强信号）**

Wikipedia 仍然是触发知识面板的**单一最强信号**——一篇 Wikipedia 文章通常能独立触发面板。

| 要点 | 说明 |
|------|------|
| 不是必须的 | 2026 年没有 Wikipedia 也能触发面板（Wikidata + Schema + 权威引用就够了） |
| **不能自己创建** | Wikipedia 严格的不知名性门槛，自创建文章会被标记为推广并删除 |
| 正确的顺序 | 先完成 Stage 1-2，积累足够的独立报道后再考虑 |
| **删除的文章比没有文章更糟** | 被标记为推广的删除文章会损害实体档案 |

**Stage 4：等待、监控、纠错（3-12个月）**

| 时间节点 | 预期 |
|---------|------|
| 1-2 个月 | 结构化数据设置完成 |
| 2-6 个月 | 公关活动开展，积累 8-15 篇权威报道 |
| 4-9 个月 | 知识图谱开始识别（搜索品牌名时出现早期信号） |
| 6-12 个月 | 面板触发（面板静默出现，无通知） |
| 12 个月+ | 持续监控纠错，新报道持续强化实体 |

**面板出现后**：使用面板底部"Suggest an edit"提交纠错，附权威来源引用。已认领的面板可以建议修改描述、更新社媒链接、上传图片。Google 审核后决定是否采纳。

**个人知识面板**比企业面板更难：Google 对人的不知名性门槛高于企业，通常需要 12-24 个月。

#### Wikidata 深度指南（单一最高杠杆操作）

创建 Wikidata 条目是整个过程中**投入产出比最高的动作**。

**企业核心属性**：

| 属性 | 说明 |
|------|------|
| instance of | business / organization / 更具体类型（SaaS company 等） |
| inception | 创立日期 |
| founded by | 创始人名称（链接到其 Wikidata 条目） |
| headquarters location | 城市、国家 |
| country | 运营国家 |
| official website | URL |
| industry | 具体行业代码 |
| number of employees | 如公开或可估算 |
| described at URL | 链接到 Crunchbase、LinkedIn、权威报道 |

**关键**：每个属性必须引用来源（新闻文章、Crunchbase、公司文件、About 页面）。**无引用的属性会被社区编辑删除，有引用的属性永久存续。** 一个有 15-20 个属性且全部引用来源的 Wikidata 条目，通常是触发面板的临界点。

#### 不起作用的做法

| 做法 | 为什么无效 |
|------|----------|
| 付费购买知识面板服务 | 多数提交垃圾 Wikipedia 文章被删除，或伪造引用，或收钱不做事 |
| 关键词堆砌公司描述 | 知识图谱关注实体级信号，不是关键词密度 |
| 购买 Wikipedia 文章 | 违反 Wikipedia 条款，被检测后删除的文章比没有文章更糟 |
| 仅靠社媒档案 | LinkedIn/Twitter 贡献微弱信号，不足以独立触发面板 |

#### 面板消失的常见原因

- Wikidata 条目被社区删除
- Google Business Profile 被停用
- 结构化数据引入重大不一致
- Google 对实体的置信度降至阈值以下

|------|------|------|

| 第 2 层 | 有 brand card 或 sitelinks | 中等 |

## SEO 优先级（正确顺序）

| 优先级 | 层级 | 内容 |
|--------|------|------|
| **第1层** | 实体健康 | About页 + schema + sameAs 完整性 |
| **第2层** | 站级质量 Q* | 搜品牌名官网排第一、Branded Search 量、Selection Rate |
| 第3层 | 站级权威 | 链接图质量、主题聚焦度（siteFocusScore） |

> **权威度是发散的"赦免状"**（B2B行业6站向量对比验证）：在权威度相近的站点之间，内容聚合程度几乎决定了流量效率；高权威站点可以用权威补偿发散的内容半径。对绝大多数中小站点，把内容点云收紧是唯一不依赖外部资源的效率杠杆。siteFocusScore（站点主题专注度）和 siteRadius（页面 embedding 偏离整站 embedding 的程度）相互印证：Google 在站点层面度量内容点云的紧凑程度，权威信号与主题信号在排名中**相乘而非相加**——任何一项接近零，另一项再大也救不回来。
| 第4层 | 用户信号 P* | NavBoost 13个月积累、goodClicks/badClicks 趋势 |
| 第5层 | 单页内容 T* | ABC signals、查询分类匹配、Schema |

**多数人从第5层往第1层做，顺序反了。**

## 27 条硬事实

1. Q* 真实存在，subdomain 级别，0.4 是 SERP features 硬门槛
2. Q* 核心输入 = Brand Visibility + Selection Rate + Anchor Text Brand Prevalence
3. NavBoost 是最重要排名信号之一，13个月窗口，国家/设备分层
4. HCU 是站级信号，**实体未定义是根因**
5. 降权机制是明确算法项，不是模糊的"Google 懂"
6. Sandbox 不是新站专属，是对"不可信+突然爆发"实体的 demotion
7. hostAge 只 sandbox fresh spam，不影响干净新站
8. lastmod 信任是 binary，假更新 = 永久失效
9. **Freshness 只对 QDF 查询生效，SERP 主体被老内容统治**
10. Parasite SEO 通路已基本关闭
11. Schema 对实体消歧有加速作用，对 LLM 引用是间接影响
12. 8 种查询分类决定不同算法权重
13. **实用指标是"搜品牌名官网排第一"，不是追求 Knowledge Panel**
14. **llms.txt 对 SEO 无用**（Google 官方确认，2026.4 Toronto）
15. **网站转 Markdown 对 LLM/SEO 无用**（Google 官方确认，2026.4 Toronto）
16. **屏蔽扩展搜索机器人不会阻止 AI 使用你的内容，data-nosnippet 是唯一方法**（但伤害传统 SEO）
17. **Google 不反对 AI 本身，反对的是规模化内容滥用**
18. **"已抓取-未索引"很少是技术问题，通常是质量信号**（Google 官方确认，2026.4 Toronto）
19. **Firefly（QualityCopiaFireflySiteSignal）是站点级规模化滥用检测引擎**，追踪内容生产速度、高质量占比、dailyClicks/goodClicks 比率、AI 内容比例趋势
20. **siteQualityStddev 衡量站内页面质量一致性**，标准差越高排名越不利，一致性比最高质量更重要
21. **三层链接索引机制**：零点击页面上的链接不传递任何 PageRank 和锚文本信号
22. **anchorMismatchDemotion**：不相关页面上的精确匹配锚文本不只是浪费，可能造成实际损害
23. **虚伪惩罚**：网站声明（About 页）与实际内容生产行为不匹配，可触发 scamness + unauthoritativeScore
24. **四种系统性失败模式**（语境过度延伸、实体膨胀、意图漂移、信任信号债务）单靠改进内容无法解决
25. **内链置信度三层累加模型**（Google 专利 US 7,590,628）：页面标题含实体 + 同站 peer documents + 站内锚文本用实体名称 → 置信度最高
26. **内链不是 PageRank 管道，是意图收窄的路标**：锚文本精确描述目标页面 → 用户信心和 Google 置信度同步提升
27. **统一 CTA 和"点击这里"是零置信度锚文本**：不携带主题信号，Google 实体提取时直接删除
28. **搜索结果中 Vibe Coding 将冲击信息站和低门槛 Micro SaaS**：Google I/O 2026 发布 Antigravity 能力，搜索结果中直接生成定制工具 App（如健身计划管理器），用户无需离开搜索即可解决需求。信息类网站和依赖 SEO 的简单工具站流量将进一步被截流（Simon，赛门快跑）
29. **Gemini 检测 AI 内容能力已覆盖 SynthID + C2PA 双标准**：Google I/O 2026 确认 Gemini 可识别自家 SynthID 水印 + 行业 C2PA 标准，OpenAI 也已在图片中加入 SynthID。AIGC 内容检测门槛大幅提升，去 AI 化策略需更加重视（Simon，赛门快跑）
30. **Glue 系统聚合 Chrome 数据参与排名**：NavBoost 之外，Glue 系统进一步聚合 Chrome 跳转、搜索点击等多渠道数据，API Leak 的 `chrome_trans_clicks`、`uniqueChromeViews` 字段证实 Chrome 浏览器数据直接参与排名信号（DOJ 反垄断审判文件 + API Leak）
31. **收录问题的本质是权威性不够**：排除技术错误后，收录失败的根本原因是域名权威性（PageRank）不足——高权威域名空内容秒收，新站精心撰写的内容可能迟迟不索引（David Quaid 实验）
32. **FAQPage Schema 已被 Google 取消**（2026 年起）：Schema 的真正普适价值是实体消歧（Person/Organization/Service + sameAs），而非富媒体展示
33. **XML sitemap 不传递权重，HTML sitemap 才传递**：XML sitemap 只是辅助发现 URL 的信号，不直接影响排名和抓取速度（John Mueller 确认）；HTML sitemap 走内链通道，能传递权重并帮爬虫发现深层页
34. **内链权威传递每跳衰减 85%**：基于 PageRank 原理，控制内链在三跳之内，超过 4 跳权威传递几乎为零——money page 必须靠近首页或高权重页面
35. **CWV 是排名信号但非强信号**：不会把你从第 10 页推到第 1 页，仅在竞争条件接近时提供边际优势，优先级在实体健康、站级质量、内容相关性之后
36. **外链评估三标准优先级：相关性 > 域名权威 > 页面流量**：同主题低权威站点的链接可能比不相关高权威站点的链接更有价值
37. **"301 丢排名"是范畴错误**：排名是查询时现算的，不是 URL 上存的存量。301 到等价目标，权威和相关性两个乘数都没动（PageRank 自 2016 年起不再因 30x 衰减，Gary Illyes 确认），排名不会因 URL 变了而永久掉
38. **状态码（301/302/307）本身不是排名因素**：Google 把重定向归在「抓取与索引」下，与「排名」分属两块；重定向只是"目标应为规范"的信号，不进排名打分
39. **索引单位是内容指纹定义的代表性文档，不是字面 URL**（去重专利 US 8,868,559 / US 9,779,072）：等价地址被并入代表文档，交互信号随之归集——所以 301 能转移 NavBoost 数据
40. **页面停留时长不是排名因素**（John Mueller 明确："Google 不知道用户在页面停留多久"）；真正起作用的是 SERP 侧 `lastLongestClick`——用户点进后是否返回 SERP，不是页面内停留追踪

## 2026 CTR 基准与流量预测方法论

> 数据来源：图帕先生 2026 Google SEO 基准报告（First Page Sage / Ahrefs / Backlinko）

### CTR 基准速查（干净 SERP vs 有 AIO）

| 排名 | 干净 SERP | 有 AIO | 差异 |
|------|----------|-------|------|
| 第1位 | 39.8% | ~19% | -52% |
| 第2位 | 18.7% | ~11% | -41% |
| 第3位 | 10.2% | ~7% | -31% |
| 前3位合计 | **68.7%** | ~37% | AIO 吃掉近半 |

**电商零售行业基准**：头部位置 CTR 仅 3%-6%，DA 门槛 Top10 需 DA 40-55，第1位需 DA 60+。

### 品牌词 vs 非品牌词：必须分开看

| 指标 | 品牌词 | 非品牌词 | 含义 |
|------|-------|---------|------|
| CTR | 30-40% | 5-10%（排第1位） | 差距 4-8 倍 |
| 转化率 | 5-8% | 2-3% | 品牌词是非品牌词的 2-3 倍 |
| 流量占比（成熟站） | 40-50% | 50-60% | 健康比例 |

**新站/初创站品牌词占比仅 15-20%**，80%+ 依赖非品牌词。混在一起看数据会严重失真。

### 流量预测方法论（含案例）

**公式**：`预估月流量 = 月搜索量 × 目标排名的 CTR × (1 - AIO 折扣)`

**案例：水晶手链项目预测 "healing crystal bracelet" 月搜索量 12,000**

| 场景 | 排名 | CTR | AIO 折扣 | 预估月点击 |
|------|------|-----|---------|-----------|
| 无 AIO，排第 3 | 3 | 10.2% | 0% | 12,000 × 10.2% = **1,224** |
| 有 AIO，排第 3 | 3 | 10.2% | -31% | 12,000 × 10.2% × 0.69 = **845** |
| 无 AIO，排第 5 | 5 | 5.1% | 0% | 12,000 × 5.1% = **612** |
| 有 AIO，排第 5 | 5 | 5.1% | -22% | 12,000 × 5.1% × 0.78 = **477** |

**再算转化**：以非品牌词转化率 2% 估算：
- 无 AIO 排第 3：1,224 × 2% = **24.5 单/月**
- 有 AIO 排第 3：845 × 2% = **16.9 单/月**

**加上 AI 搜索引荐**：AI 引荐转化率 3.49%（比自然搜索高 22%），如果 AI 引荐带来 200 月访问：
- AI 引荐转化：200 × 3.49% = **7 单/月**（额外增量）

**流量预测实操步骤**：
1. 从 GSC / Ahrefs 获取目标关键词的月搜索量
2. 根据当前 DA 和竞争分析预测可达排名
3. 用上方 CTR 表查对应 CTR
4. 判断该关键词是否触发 AIO（信息性查询 → 高概率；交易性/长尾 → 低概率）
5. 应用 AIO 折扣（信息性 -50%，交易性 -15%，长尾 0%）
6. 乘以转化率（电商非品牌 2%，品牌词 5-8%）
7. 叠加 AI 搜索引荐增量

### 品牌词 vs 非品牌词的网站阶段诊断

| 品牌/非品牌流量比 | 诊断 | 行动 |
|----------------|------|------|
| 品牌 < 20% | 新站/品牌弱 | 重点做内容集群扩大非品牌词覆盖 |
| 品牌 30-40% | 成长期 | 双轨并行：SEO 内容 + 品牌建设 |
| 品牌 40-50% | 健康成熟 | 维持平衡，注意关键词多样性 |
| 品牌 > 60% | 过度依赖品牌 | 非品牌词覆盖不足，SEO 内容有缺口 |

> 完整基准数据见 [references/2026-Google-SEO-Benchmark-Report.md](./references/2026-Google-SEO-Benchmark-Report.md)

## 实操检查清单

### 第1层：实体健康

| 检查项 | 做法 | 工具 |
|--------|------|------|
| 实体是否被识别 | 查询品牌名，是否有 @id: "kg:/m/..." | [audits.com KG Search](https://audits.com/tools/knowledge-graph-search) / [PlePer KG Tool](https://pleper.com/index.php?do=tools&sdo=google_knowledge_graph_raw) |
| 搜品牌名官网排第几 | Google 搜索品牌名 | 直接搜索 |
| 实体信息一致性 | 检查 About 页、LinkedIn、Crunchbase、Wikidata 信息是否完全一致 | 人工检查 |
| sameAs 部署 | Organization/Person schema 中 sameAs 字段是否连接了 Wikidata、LinkedIn 等 | 查看页面源码 schema |

**时间预期**：schema + sameAs 部署 → Google 处理 4-8 周 → KG 实体识别 3-6 个月

### 第2层：站级质量 Q*

| 检查项 | 做法 | 工具 |
|--------|------|------|
| Branded Search 量 | 品牌名有多少搜索量 | Google Search Console（搜索效果报告） |
| Selection Rate | 不在第1位时用户是否主动选你 | GSC → 效果报告 → 按查询查看 CTR vs 排名位置 |
| SERP features 资格 | 是否出现 Featured Snippet / PAA | 手动搜索目标关键词 |

> Q* 无公开直接检测工具（MWC 漏洞已修复）。间接判断：如果排第5-10位但 CTR 异常高 → Selection Rate 可能不错 → Q* 可能健康。如果排第1-3位但 CTR 低 → Q* 可能有问题。

### 第3层：站级权威

| 检查项 | 做法 | 工具 |
|--------|------|------|
| 锚文本品牌占比 | 外链锚文本中包含品牌名的比例 | Ahrefs / SEMrush 锚文本报告 |
| 主题聚焦度 | 站点内容是否围绕核心主题 | 人工判断 |
| DA:BA 比值 | Domain Authority vs Brand Authority 是否失衡 | Ahrefs（DA）+ 品牌搜索量 |

**高风险信号**：DA 远高于 BA（如 2:1+），说明外链攒得快但没人搜你的品牌 → HCU 高风险。

### 第4层：用户信号 P*

| 检查项 | 做法 | 工具 |
|--------|------|------|
| 页面停留时间 | ⚠️ Google **不知道**用户在页面停留多久（John Mueller 明确）。真正起作用的是 SERP 侧的 `lastLongestClick`——用户点进后是否返回 SERP | GSC → CTR vs 平均排名位对比（排第3但 CTR 接近第10位 = pogo-stick） |
| Pogo-sticking | 用户点击后是否立刻返回 SERP | GSC → CTR vs 平均排名位对比（排第3但 CTR 接近第10位 = pogo-stick） |
| 分市场表现 | 不同国家/语言的排名差异 | GSC 按国家筛选对比 |

### 站点一致性检查（siteQualityStddev）

| 检查项 | 做法 | 工具 |
|--------|------|------|
| 低质量页面占比 | 识别站内流量极低 + 停留时间短的页面 | GSC → 按页面查看低效页面 |
| 内容合并/删除 | 将内容单薄页面合并为更有用页面，或直接删除 | 人工判断 |
| 高风险内容隔离 | 把批量产出的有风险内容搬到独立域名 | — |

> 如果站内有大量低质量页面拉高 siteQualityStddev，优先处理这些比优化少数好页面 ROI 更高。

### 链接有效性检查（三层链接索引）

| 检查项 | 做法 | 工具 |
|--------|------|------|
| 内链页面是否有真实流量 | 内链指向的页面是否有用户访问 | GSC → 页面流量报告 |
| 锚文本相关性 | 内链锚文本是否与目标页面主题相关 | 人工检查 / Screaming Frog |
| 外链所在页面质量 | 外链来源页面是否有用户互动 | Ahrefs / SEMrush 外链报告 |

> 零点击页面上的链接不传递任何权重。重点不是建更多内链，而是让已有内链所在的页面获得真实用户访问。

### 内链置信度检查

| 检查项 | 做法 | 判断标准 |
|--------|------|---------|
| 锚文本是否包含触发词 | 从用户意图倒推——读到此处用户想了解什么 | 锚文本精确描述目标页面内容 = 高置信度 |
| 同一目标页面是否有不同锚文本 | 检查指向同一页面的所有内链锚文本 | 不同来源页面应使用不同锚文本（语境不同） |
| 内链路径是否逐层收窄 | 追踪用户从首页到目标页的点击路径 | 每一步锚文本应比上一步更具体 |
| CTA 位置是否在意图末端 | 检查"咨询""购买"等 CTA 出现的位置 | CTA 应出现在用户意图已收窄到行动层的页面 |
| "点击这里""Learn More"占比 | 统计站内零信息锚文本比例 | 越少越好，应为零 |

### 第5层：单页内容 T*

| 检查项 | 做法 | 工具 |
|--------|------|------|
| 查询分类 | 关键词属于8类中哪一类 | [MWC Refined Query Classifier](https://rqpredictor.streamlit.app)（免费，基于4.6M英文查询训练） |
| 内容与查询匹配 | 页面是否精准回答目标查询 | AI 搜索平台（ChatGPT / Perplexity）直接搜目标关键词 |
| lastmod 信任 | 是否在不改内容的情况下更新了日期 | 对比页面 Wayback Machine 快照 vs lastmod 日期 |
| Schema 实施 | 关键页面是否有正确 schema | [Schema Markup Validator](https://validator.schema.org) / GSC 增强体验报告 |

### 新站专项检查

| 检查项 | 做法 |
|--------|------|
| 是否被索引 | Google 搜索 `site:yourdomain.com` |
| 是否被 sandbox | 有索引但完全无排名 + 不是 spam 内容 → 可能只是缺少 PR 信号（非惩罚） |
| 初始评分预测 | 内容主题是否聚焦 + 有真实业务实体（About/schema/sameAs） → 避免继承低质量邻居分数 |
