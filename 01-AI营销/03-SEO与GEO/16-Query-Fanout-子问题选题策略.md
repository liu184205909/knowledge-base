# Query Fan-out：把选题单位从「关键词」升级为「子问题」

> **一句话定位**：AI 搜索把一个用户问题拆成 N 个子问题并行检索——内容策略的下一个单位不是"多排几个词"，而是"覆盖 AI 会拆出来的所有子问题"。
> **与已有文档差异**：03-AI Citation 讲"如何被引用"、14 讲"影响力者框架对比"。本篇只讲一件事——**如何用 Fan-out 思维做选题与覆盖度诊断**，是所有 C 端选品项目通用的内容生产 SOP。

---

## 一、为什么"关键词思维"在 2026 年开始失效

| 维度 | 传统 SEO（关键词） | GEO（子问题） |
|------|------------------|--------------|
| 优化单位 | 一个关键词 = 一篇文章 | 一个核心话题 = N 个子问题 = N 篇内容/章节 |
| 用户路径 | 输入词 → 看标题点 | 提问 → AI 后台拆解 → 多源验证 → 综合答案 |
| 命中条件 | 标题/正文有该词 + 排前 10 | 内容精确回答了 AI 拆出的某个子问题 |
| 内容形态 | 长文堆关键词 | **模块化、可被切片提取**的结构化段落 |

**关键认知差**：传统 SEO 你抢的是"位置"，GEO 你抢的是"**被切片**"——AI 从你页面里切走一段作为子问题答案。所以一篇文章堆 20 个关键词没用，**有没有独立回答某个子问题的清晰段落**才有用。

---

## 二、子问题挖掘：3 种实操路径

### 路径 A：直接问 AI（最快，5 分钟出清单）

把你的核心话题丢给 3 个引擎，让它"显示拆解过程"：

```
Prompt 模板（直接复制）：
我要写一篇关于「[核心话题]」的内容。
请你像 AI 搜索引擎那样，把这个问题拆解成你会并行检索的子问题清单。
要求：
1. 列出 8-15 个子问题
2. 按维度归类（如：定义/对比/价格/场景/风险/人群适配）
3. 标注哪些子问题现有内容很少覆盖（蓝海机会）
```

跑 3 个引擎：**Google AI Overview / ChatGPT / Perplexity** → 三份子问题清单去重合并 → 这就是你的**选题骨架**。

### 路径 B：People Also Ask + Autocomplete（数据源，免费）

- Google 搜核心话题 → 抓 PAA 展开 3 层（每个 PAA 点开后又冒新 PAA）
- Autocomplete 用 a-z 字母扫一遍（`关键词 a`、`关键词 b`...）
- 工具：`google_suggest_alphabet` MCP 工具一键扫

这两类数据本身就是 AI 拆问题的**训练信号源**——AI 拆出来的子问题大概率落在这个池子里。

### 路径 C：竞品反向工程（行业视角）

拿 3-5 个 SERP 前 10 的竞品页面，提取每个页面的 **H2/H3 大纲** → 合并去重 → 这是"Google 已认为合格的回答结构"。你对照自己的内容，缺口就是选题。

> **三条路径不是"交集 = 高置信度"那么干净**——它们是三个不同视角，各有盲区，彼此还会有矛盾：
>
> | 路径 | 视角 | 盲区 |
> |------|------|------|
> | A. 问 AI | AI 拆解逻辑 | AI "认为"会拆的，不等于真实用户在搜的；GPT 和 Perplexity 拆出来差异很大 |
> | B. PAA/Autocomplete | 真实用户行为（ground truth） | 是 Google 的扩展逻辑，不完全等于 AI 搜索的 Fan-out |
> | C. 竞品 H2/H3 | 行业惯例 | 过去式，预测不了 AI 关心的新维度 |
>
> **实操建议**：三份清单取**并集**，然后人工判断——B（真实用户）权重最高，A 用于发现"AI 关心但用户还没搜到"的蓝海，C 作为完整性兜底。出现矛盾时以 B 为准。

## 三、覆盖度诊断：一张表查清缺口

把挖掘出的子问题列表 × 你的现有内容，填一张矩阵：

| 子问题 | 现有内容是否覆盖 | 覆盖质量 | 行动 |
|--------|----------------|---------|------|
| 能不能一起戴？ | ✅ 有专门段落 | 充分 | 维护 |
| 价格区间？ | ❌ 没有 | — | **新建选题** |
| 适合什么人？ | ⚠️ 提了一句但没展开 | 不足 | **扩写** |
| 有什么副作用？ | ❌ 没有 | — | **新建选题** |

**判定标准**：
- ✅ 充分 = 有独立 H2 + 100 字以上针对性回答 + 可被 AI 直接切片
- ⚠️ 不足 = 只在别的段落里捎带提及，AI 无法单独提取
- ❌ 缺失 = 完全没覆盖

> **诊断频率**：核心话题每季度跑一次。AI 拆解逻辑会随模型更新变化，子问题骨架不是一次定终身。

## 四、8 个 C 端选品的子问题骨架（直接套用）

不同选品类型，AI 拆问题的维度权重不同。下面是按本知识库选品库归类的基础骨架，**起点模板，不是终点**：

### 类型 1：功效/能量型（疗愈水晶、睡眠改善）

```
核心话题示例：amethyst and citrine together
子问题维度：
├─ 适配性（能不能？为什么？）
├─ 功效叠加（一起戴有什么效果？）
├─ 能量冲突（会不会冲突？）
├─ 使用场景（什么情况下戴？）
├─ 人群适配（适合什么人？不适合什么人？）
├─ 禁忌与风险（哪些情况不能戴？）
├─ 净化保养（怎么净化？怎么保养？）
└─ 替代方案（不这样配还有什么选择？）
```

### 类型 2：选购决策型（靠垫、乐高积木、水枪）

```
核心话题示例：best cushion for lower back pain
子问题维度：
├─ 场景细分（办公/开车/孕妇/老人，AI 一定会按场景拆）
├─ 材质对比（记忆棉/乳胶/凝胶，优劣）
├─ 价格档位（预算 50 / 200 / 500 各推荐什么）
├─ 健康安全（有毒吗？孕妇能用吗？）
├─ 耐久性（用多久会塌？）
├─ 清洁保养（能不能洗？）
├─ 品牌对比（A vs B）
└─ 常见踩坑（哪些不能买？）
```

### 类型 3：内容/灵感型（地图艺术海报、中国旅游站）

```
核心话题示例：custom map poster gift
子问题维度：
├─ 定制流程（怎么下单？要提供什么？）
├─ 设计风格（极简/复古/手绘，哪种好看？）
├─ 尺寸选择（房间大小怎么选？）
├─ 材质工艺（海报/画布/相纸，质感差异）
├─ 礼物场景（送谁合适？结婚/搬家/纪念日）
├─ 价格与时效（多少钱？多久到？）
├─ 版权与商用（能商用吗？）
└─ 替代品对比（vs Etsy / vs 本地打印店）
```

### 类型 4：敏感/对比型（情趣内衣）

```
核心话题示例：best lingerie for [场景]
子问题维度：
├─ 尺码与合身（尺码怎么选？大胸/小胸适配）
├─ 材质肤感（扎吗？透气吗？）
├─ 场景适配（蜜月/纪念日/日常，AI 会按场景拆）
├─ 隐私（发货包装？退换政策？）
├─ 清洗保养（手洗？机洗？变形吗？）
├─ 伴侣视角（对方会喜欢什么风格？）
├─ 身材包容（plus size 有没有？）
└─ 避雷（哪些材质/款式别买？）
```

> **用法**：每个选品项目的内容生产前，先把对应骨架展开成具体子问题清单，再走第三节的覆盖度诊断。

## 五、最小手动流程（不进工具库）

> **为什么不做成脚本**：路径 A（问 AI）本身就是 Fan-out，不能脚本化；路径 B 已有 `google_suggest_alphabet` MCP 工具；路径 C 现成的 `audit_page` / DataForSEO 已能提取 H2/H3。强行自研脚本是用工程解决不了的问题。真正需要的是**一次手动 AI 访问 + 一张覆盖度表**。

```
[1. 种子词]
      ↓
[2. 三源挖掘]   AI 直问（手动） + PAA/Autocomplete（MCP 工具） + 竞品 H2（audit 工具）
      ↓
[3. 人工合并 → 子问题清单]   取并集，B 为主、A 找蓝海、C 兜底
      ↓
[4. 覆盖度诊断]   一张 Sheets/Notion 表，标 ✅⚠️❌
      ↓
[5. 缺口转选题]   ❌→新建、⚠️→扩写，进入 Brief
```

**产出物**：
- 步骤 3：子问题清单（1 份 Sheets，作为该话题的"选题母表"）
- 步骤 4：覆盖度矩阵（在母表上加几列，标覆盖状态）
- 步骤 5：本轮选题清单（直接进 Brief 模板）

**复用逻辑**：选题母表做完一次后，每季度增量更新——AI 新拆出来的子问题追加进去，重新标覆盖度。**不要追求一次定终身，也不要追求全自动**。

## 六、Fan-out 思维最容易踩的 3 个坑

| 坑 | 表现 | 破解 |
|----|------|------|
| **1. 子问题堆成一篇** | 把 10 个子问题塞进同一篇文章 | 每个子问题独立成段/成篇，AI 才能精确切片 |
| **2. 只覆盖"容易答的"** | 蓝海子问题（如禁忌、副作用）没人写，你也跳过 | 越是没人答的子问题，越是被 AI 当稀缺信号引用 |
| **3. 子问题骨架一次定终身** | 一年前挖的子问题清单一直用 | 模型在更新，子问题骨架每季度刷新一次 |
