# RLM营销方法论 - AI执行版

> 本文件是RLM项目的AI默认执行入口和当前唯一执行口径。AI执行RLM项目时，优先读取本文件，不默认读取参考版。`00-RLM营销方法论-完整参考版.md` 仅作历史参考；如与本文件冲突，以本文件为准。

---

## 1. RLM是什么

**RLM营销方法论 = 用AI的分而治之思想 + 从网站架构到内容全面超越竞品的策略，实现营销全流程自动化**

### 核心思路：模仿 + 超越 + 差异化

核心逻辑：先看市场上谁做得好 → 判断他们为什么好 → 找到他们没做好的地方 → 设计更快、更系统、更低成本的方案。

> **记住：优秀的策略，市场上已经有人验证过了。你要做的是学习它、拆解它、吸收它，然后结合自己的业务重新组合。剩下真正要拼的，还是业务理解、市场理解、供应链能力、流量获取、客户维护，以及你选的赛道到底能不能持续。**

默认服务于**海外市场**（B2B外贸、B2C独立站、SaaS/工具站、内容站/Affiliate）。分析对象优先选择海外本土品牌，中国出口站仅作参考。

## 2. AI防跑偏规则

**这是执行版中最重要的章节。违反以下任何一条都算输出不合格。**

1. **不编造数据**：没有来源的数据必须标注"待验证"，不得虚构搜索量、流量、KD、CPC、收入、转化率、竞品团队规模。
2. **不凭空做用户画像**：用户画像必须来自竞品页面、关键词意图、FAQ、评论、产品属性或行业场景。依据不足时标注"初步推测"。
3. **不做无依据预测**：除非用户提供历史数据或计算模型，否则不输出流量、收入、排名、转化率的具体预测，不输出伪精确的发布时间表。
4. **不把国内市场当目标市场**：RLM默认服务海外市场。分析对象优先选择海外本土品牌、垂直站点和独立站。
5. **不输出空泛策略**：模仿、超越、差异化必须具体到竞品、页面、内容、结构、关键词或转化路径。禁止"提升品牌力""优化用户体验"等无动作表述。RLM主流程围绕SEO内容展开，付费广告、社媒、邮件等渠道按需扩展。
6. **不一次性读取全部资料**：默认只读取执行版。需要模板、指令或专项方法论时再按需读取。
7. **不让AI当应声虫**：向AI提问时，不要问"这想法怎么样"，要问"这想法最可能在哪里崩"。让AI泼冷水，直到泼不出来。AI最大的危险不是犯错，而是用同样的热情帮一个烂想法做到产品级。（来源：刘小排r《AI唯一正确用法：泼冷水》）

**输出前检查清单**（每次输出前自检）：

- [ ] 是否有数据来源？无来源的数据是否标注"待验证"？
- [ ] 用户画像是否来自关键词、竞品页面、FAQ 或评论？
- [ ] 策略是否具体到页面、关键词、结构或内容？
- [ ] 是否明确输出文件名和路径？
- [ ] 当前结果是否能直接作为下一步的输入？

## 3. RLM五步流程

```
步骤1: 竞品查找与分析（一步到位）→ 步骤2: 网站规划（规划 + 决策 + 逐页生成 + 上传）→ 网站上线
                             → 步骤3: 内容策略 → 开始创作
步骤4: 社媒内容分发（可选）← 步骤3的博客文章再加工为多平台社媒内容
步骤5: 商业模式（可选）
```

> 步骤2和步骤3可并行。步骤2完成后网站即可先上线，步骤1的关键词库和策略清单汇入步骤3。步骤3的博客文章作为步骤4社媒分发的"母内容"，一次创作、多平台分发。

---

### 步骤1：竞品查找与分析（知道对手是谁 + 知道内容怎么做）

**目标**：找到所有相关竞争对手，全面分析后建立数据基础，为网站规划和内容策略提供依据。

步骤1是整个流程的数据基础，内部分8个子阶段：

```
1A: 竞品发现（人工）   → Google 搜索 + 排除语法 → 筛选到 ~40 个 → 人工标注竞品类型
1B: 数据采集（AI 辅助）→ SEMrush脚本(AS/流量/截图) + traffic.cv(月访问量/渠道拆分)
                         + Sitemap解析(页面清单/URL结构/内容类型分布)
                         → AI 自动填入 Sheets
                         → 纯数据采集，不做分析判断
1C: 竞品深度拆解     → 使用1B全部数据（SEMrush + traffic.cv + Sitemap）
                         → 每个竞品逐个深度拆解（按类型侧重不同，此时已有完整页面清单）
1D: 网站结构分析（综合1B+1C做结构对比）→ 可进入步骤2
1E: 内容策略分析
1F: 关键词研究       → 可进入步骤3（读取关键词研究方法论）
1G: 用户画像
1H: 策略清单         → 汇入步骤2和步骤3
```

#### 1A 竞品发现（人工）

**目标**：人工搜索并筛选竞品，建立竞品清单。

**核心任务**：
- 使用 Google 搜索种子关键词，通过排除语法过滤大平台
- 人工浏览搜索结果，筛选约 40 个竞品域名
- 访问每个网站后，填写竞品类型标签

**搜索排除语法**（直接复制使用）：
```
关键词 -amazon.com -ebay.com -etsy.com -walmart.com -pinterest.com -reddit.com -youtube.com -facebook.com -tiktok.com -wikipedia.org -quora.com -target.com -aliexpress.com -alibaba.com
```

**竞品类型分类**：

| 类型 | 定义 | 示例 |
|------|------|------|
| 直接竞品 | 卖同类产品，抢同一批客户 | 同品类独立站、品牌官网 |
| 内容竞品 | 不卖货但覆盖了你的关键词（博客/内容站/affiliate） | 博客、评测站、Buying Guide |
| 间接竞品 | 同大类但不同细分定位（高端 vs 平价、不同品类） | 相邻品类的品牌站 |

**关键输出**：竞品清单（Google Sheets）（筛选后约 40 家，每个竞品有人工标注的类型标签）— 每个项目使用独立的 Google Sheets，链接记录在项目文件夹 `01-竞品分析/` 中

> 1A 全程人工操作，不需要任何自动化工具。人在浏览网站时对"这是不是竞品"的判断比 AI 凭域名猜测准确得多。
> 1A 完成后可以暂停，不强制立即进入 1B。

#### 1B 数据采集（AI 辅助）

**目标**：为 1A 筛选后的竞品清单批量补充关键数据字段，包括 SEO 指标、流量数据和网站结构数据。

**核心任务**：
- 批量采集 SEO 数据（AS/Authority Score、自然搜索流量、Distribution 截图）
- 补充流量数据（月访问量、搜索/直接/社交/引荐/邮件/付费渠道占比）
- 批量解析竞品 Sitemap（页面清单、URL 结构、内容类型分布）

**采集字段与工具**：

| 字段 | 列号 | 工具 | 方式 |
|------|------|------|------|
| AS（Authority Score） | C | SEMrush 脚本 | 自动化 |
| Organic Traffic | D | SEMrush 脚本 | 自动化 |
| SEMrush 截图 | E | SEMrush 脚本 | 自动化 |
| 月访问量 | F | traffic.cv | 手动 |
| 流量渠道拆分（搜索/直接/社交/引荐/邮件/付费） | G-K | traffic.cv | 手动 |
| 页面清单 + URL 结构 + 内容类型分布 | - | Sitemap 解析 | 自动化 |

**Sitemap 解析的三级 fallback 方案**：

| 优先级 | 方案 | 适用场景 | 说明 |
|--------|------|---------|------|
| 1 | Python 脚本 | 主方案 | `sitemap_parser.py` 自动抓取并解析 sitemap.xml |
| 2 | MCP webReader | 脚本失败时 | 逐个手动抓取 sitemap URL，AI 解析 XML 内容 |
| 3 | 手动导出 | 工具都失败时 | 用户浏览器下载 sitemap.xml 放到项目目录，脚本读取本地文件 |

**工具链**：
```bash
# Sitemap 解析（竞品 URL 列表从项目 Google Sheets 竞对清单导出，不使用工具目录下的静态文件）
python 02-自动化工具库/02-竞品研究工具/sitemap_parser.py --file <项目竞品URL导出文件>
# 脚本失败时，可用 MCP webReader 逐个抓取 sitemap URL

# SEMrush 批量数据采集（AS + Organic Traffic + 截图 → 写入 Google Sheets）
python 02-自动化工具库/02-竞品研究工具/semrush_to_sheets.py
python 02-自动化工具库/02-竞品研究工具/semrush_to_sheets.py --start 4    # 从第4行开始
python 02-自动化工具库/02-竞品研究工具/semrush_to_sheets.py --limit 5    # 只查前5个
python 02-自动化工具库/02-竞品研究工具/semrush_to_sheets.py --dry-run    # 只提取不写回 Sheets
```
- 手动补充：`traffic.cv`（浏览器访问 `https://traffic.cv/{domain}`，获取月访问量和渠道拆分）
- **Sitemap 解析结果存放在项目文件夹 `01-竞品分析/`**，不存放在工具目录
- 详细操作手册：`02-自动化工具库/02-竞品研究工具/竞品研究工具使用指南.md`

**关键输出**：更新后的竞品清单（含 AS、流量、截图、渠道数据）+ Sitemap 解析报告（页面清单、URL 结构、内容类型分布）

> **Sitemap 数据存储规范**：
> - Sitemap 解析报告存放在项目文件夹 `01-竞品分析/1B-sitemap解析/` 目录下
> - 主报告 `1B-sitemap解析/1B-sitemap解析报告.md` 作为索引，包含总览、横向汇总、失败处理和关键洞察
> - 每个竞品的详细 sitemap 解析存为独立文件（如 `02-Energy-Muse.md`、`04-Crystal-Vaults.md`），文件名编号与1A竞品清单序号一致，与 `1C-竞品深度拆解/` 采用相同的按竞品拆分模式
> - 单竞品文件内容包含：建站平台、sitemap结构、页面分类统计、关键URL清单

> **Sitemap 数据来源规则**：sitemap_parser 的输入 URL 列表应从**项目 Google Sheets 竞对清单导出**，而不是使用工具目录下的静态 `competitor_urls.txt`。工具目录下的文件仅作为示例格式参考。每个项目的 `00-项目简报.md` 中记录 Google Sheets 链接。

#### 1C 竞品深度拆解

**目标**：基于 1B 采集的全部数据（SEMrush + traffic.cv + Sitemap），对每个竞品进行深度拆解。

**核心任务**：
- 使用 1B 全部数据（AS、流量、截图、渠道、**sitemap 页面清单**）作为分析基础
- 每个竞品逐个深度拆解，按类型侧重不同：
  - **直接竞品**：全面拆解（产品、定价、页面、转化路径、网站结构）
  - **内容竞品**：侧重内容策略和关键词覆盖（他们写了什么、排名了什么、流量从哪来）
  - **间接竞品**：侧重差异化参考（他们的定位和打法与你有什么不同）
- 所有竞品都会被分析，不跳过任何一个

**关键输出**：每个竞品的深度拆解文档（存放在 `1C-竞品深度拆解/` 目录）

> 1C 完成后进入 1D 网站结构分析、1E 内容策略分析。

#### 1D 网站结构分析（知道网站怎么建）

> **定位**：1D 是综合分析层，基于 1B 采集的 sitemap 数据 + 1C 深度拆解结论，进行跨竞品的结构对比分析。sitemap 数据采集已在 1B 完成，1D 不再包含数据采集任务。

**核心任务**：

1. **网站结构分析** — 导航结构、页面层级、布局优劣（综合 1B sitemap 数据 + 1C 深度拆解结论）
2. **产品分类模式识别** — 评估竞品分类方式优缺点

   | 竞品 | 分类方式 | 分类维度 | 优点 | 缺点 |
   |------|---------|---------|------|------|
   | 竞品X | [实际分类] | [按什么维度分] | [哪里好] | [哪里不足] |

3. **URL结构分析** — 命名规则、层级深度、关键词出现
4. **类目关键词确认** — 从竞品URL和类目名中提取核心词，确认搜索趋势（仅类目级别，不做完整关键词研究）

**关键输出**：竞品结构对比 + 类目关键词方向 + **架构层候选模仿点**（最终策略归入 1H，最终网站方案归入 02）

**工具链**：
- Skills: `frontend-design` — 分析竞品目录结构的 UX 优劣
- MCP: `webReader` — 按需复核竞品具体页面结构（轻量复核，不重新抓取全站 sitemap）

> 1D 完成后可直接进入步骤2网站规划。

#### 1E 内容策略分析

**核心任务**：
- 内容类型分析（博客/案例/教程/指南）
- 竞品内容优势与缺口识别
- 竞品共同弱点识别（找到 90%+ 竞品都没做的事情 = 最大差异化机会）

**关键输出**：内容策略分析（每个竞品的内容类型、优势、缺口、共同弱点）

> **边界约束**：1E 输出内容缺口和候选机会时使用"候选方向"语气（如"候选方向（→ 1H 策略清单决策）"），不使用"我们的策略""我们将"等决策语气。最终内容策略决策归入步骤3。

#### 1F 关键词研究

**核心任务**：
- 种子关键词提取 → SEMrush/Ahrefs 扩展 → 导出 CSV → AI 分析（意图分类、分组、优先级）
- 不需要对接付费关键词 API，SEMrush 手动导出已足够
- 详细流程参见 `01-关键词研究方法论.md`

**关键输出**：关键词库（Google Sheets，含分组、意图标注、优先级、Volume/KD/CPC）+ `关键词分组与优先级.md`

**工具链**：
- 数据获取: SEMrush/Ahrefs 导出 CSV → AI 分析
- MCP: `DataForSEO`（关键词搜索量/CPC）

> 1F 完成后可直接进入步骤3内容策略。

#### 1G 用户画像

**核心任务**（基于竞品数据+关键词数据推导，不凭空想象）：
- 目标用户画像
- 核心痛点
- 用户需求

**关键输出**：用户画像（每个画像可追溯到关键词意图、竞品页面、FAQ 或评论）

#### 1H 策略清单

**核心任务** — 必须独立成章：
- **模仿清单**：格式 "模仿X: [竞品]的[具体做法]"
- **超越清单**：格式 "超越X: [竞品不足的具体描述]"
- **差异化清单**：格式 "差异化X: [竞品空白的具体机会]"

**关键输出**：策略清单 → 汇入步骤2（网站规划）和步骤3（内容策略）

> **边界约束**：1H 不重复搬运竞品原始数据（导航、截图、URL等），只吸收来自 1D/1E/1F/1G 的分析结论，进行去重、排序、映射。1H 是唯一完整的策略中枢。

**工具链**：
- Skills: `claude-seo`（E-E-A-T标准）/ `seo-geo`（GEO优化）
- MCP: `GSC MCP`（网站上线后实时数据）

> **三层文档边界规则**（适用于所有RLM项目）：
>
> | 层级 | 文档 | 职责 | 禁止 |
> |------|------|------|------|
> | **分析层** | 1D、1E | 事实、优缺点、缺口、候选机会（"候选方向"语气） | 做最终策略决策（"我们将""我们的策略"） |
> | **策略层** | 1H | 去重后的模仿/超越/差异化策略，唯一完整策略中枢 | 重复搬运竞品原始数据 |
> | **执行层** | 02、03 | 最终网站方案和内容方案 | 重复搬运竞品对比表，应引用而非复制 |
>
> 核心原则：每个文档各司其职，上游输出分析判断，1H 汇聚为策略，下游引用策略做执行决策。

### 步骤2：网站规划（把网站建起来）

**目标**：基于竞品结构分析和关键词方向，规划并搭建 WordPress 网站的核心页面

> **边界约束**：步骤2文档可引用 1D/1H 的结论（如"参考 Energy Muse 的 Shop by Intention 分类"），但不重复搬运竞品对比表、截图、结构数据。引用时标注来源文件即可。

步骤2内部分4个子阶段：

```
2A: 架构规划     → 5个规划文档 + 竞品页面频率矩阵（页面候选池）
2B: 页面决策     → 决策表：做/不做/批次 + 竞品参考来源索引
2C: 逐页生成     → 每个页面：页面内容指南 → 单页依据包（复用1B/1C/1D/1H + 轻量复核URL）→ 写内容 → 图片资产清单/prompt → 构建脚本
2D: 上传与验收   → 逐页：图片生成/上传/URL回写 → 脚本上传 → 前端验收
```

> **步骤2数据流与边界规则**：
>
> | 环节 | 数据来源 | 可以做什么 | 不可以做什么 |
> |------|---------|-----------|------------|
> | 2A | 引用 1D 的竞品结构分析 | 从 1D 提炼频率矩阵、标注来源索引 | 重新搬运 1D 的竞品原始表格、截图、导航结构 |
> | 2B | 2A 频率矩阵 + 1H 策略清单 | 做页面决策，决定做/不做/批次 | 引入 1D/1E 没有支撑的"凭感觉"页面 |
> | 2C | 2B 决策表批准的页面清单 + 1B/1C/1D/1E/1H 已沉淀结论 | 建立单页依据包 → 对核心参考URL做轻量复核 → 写内容 → 图片资产清单/prompt → 构建脚本 | 重新深度拆解竞品；处理 2B 未批准的页面；复用同一图片填充多个关键视觉位 |
> | 2D | 2C 生成的脚本和图片资产清单 | 生成/上传图片 → 回写URL → 上传页面 → 前端验收 → 标记完成 | 只看 API 成功或 draft 创建成功就标记完成 |
>
> **核心原则**：2B 是 gate。没有通过 2B 决策表的页面，不能进入 2C 生成。这避免了"一口气生成 N 个页面但来源和优先级说不清"的问题。

> **竞品复用原则**：1B 数据采集和 1C 深度拆解是一次性资产库，1D/1E/1H 是对 1B+1C 的结构化沉淀。2C 不重新深度拆解竞品，只读取这些上游结论，并对 2B 选定的核心参考 URL 做轻量复核，确认该页面的结构、文案、CTA、动态模块没有遗漏。

#### 2A 架构规划

**核心任务**：

1. **网站结构设计** — 参考(模仿)竞品好的结构，优化(超越)竞品不足
2. **URL结构与内部链接** — 基于类目关键词方向设计
3. **产品类目规划** — 基于关键词命名，比竞品更清晰合理
4. **核心页面内容规划** — 每个页面标注：参考哪家竞品、模仿什么、超越什么、差异化创新
5. **产品页面规划** — 产品标题、描述、属性
6. **竞品页面频率矩阵**（2B 的直接输入） — 从 1D 竞品结构分析中提炼页面候选池，格式如下：

   | 页面类型 | 出现竞品数/总竞品数 | 普及率 | 出现的竞品 | 不出现的竞品 | 1D来源索引 | 备注 |
   |---------|-------------------|-------|-----------|------------|----------|------|

   > **边界**：原始证据（竞品导航截图、完整URL列表、详细结构图）留在 1D。2A 的频率矩阵只保留决策所需的摘要和引用（如"详见 1D §2.3"），不复制 1D 原始数据。

**关键输出**：`02-网站规划/` 下5个规划文档 + **竞品页面频率矩阵**（作为 2B 页面决策的直接输入）

#### 2B 页面决策（gate）

**目标**：基于 2A 的竞品频率矩阵 + 1H 策略清单，决定实际要做哪些页面、什么顺序。

**2B 是步骤2的 gate。** 没有通过 2B 决策的页面不能进入 2C 生成。

**强制输出 — 页面决策表**：

| 页面名称 | 页面类型 | 竞品覆盖率 | 对应竞品页面URL | 用户意图 | 1H策略匹配 | SEO/商业价值 | 实现复杂度 | 决策状态 | 执行批次 | 候选批次 | 不做/延后原因 |
|---------|---------|-----------|---------------|---------|-----------|------------|-----------|---------|---------|---------|------------|

> **字段规则**：
> - `决策状态` 只允许使用：`✅ 批准执行`、`⏸ 候选延后`、`❌ 不做`。
> - 只有 `✅ 批准执行` 的页面才能填写 `执行批次` 并进入 2C。
> - `⏸ 候选延后` 页面只能填写 `候选批次`，不能进入 2C；后续必须经过 2B 复审才能激活。
> - `对应竞品页面URL` 可以在 2B 阶段使用 URL 模板（如 `/products/[product]`），但进入 2C 前必须替换为实际可抓取 URL。
> - `核心参考竞品` 由 2B 选定：业务必要/低复杂度页面至少 2 家，标准核心页面建议 3-5 家，高价值SEO模板或SERP竞争强页面可扩展到 5-10 家。这里的数量指“单页参考来源”，不是要求在 2C 重新深度拆解。
> - 每个核心参考竞品必须能追溯到 1C 深度拆解文档、1D/1E 汇总结论或 1H 策略条目；没有上游依据的临时参考不能直接支撑页面结构和文案。

**决策规则**：

| 条件 | 决策 | 说明 |
|------|------|------|
| 竞品覆盖率 75%+ | **默认必须做** | 大多数竞品都有，不做等于缺失基础；除非与项目定位明显冲突 |
| 竞品覆盖率 25-75% | **进入判断** | 看 1H 策略价值和资源成本，决定"应该做 / 差异化做 / 延后做" |
| 竞品覆盖率 <25% | **默认不做/远期** | 除非它是 1H 明确要求的差异化页面（如 Crystal Quiz 只有 25% 竞品有，但 1H 明确要求做） |
| About / Contact / FAQ / Shipping / Returns | **业务必要页** | 不完全受频率阈值支配，属于网站信任基础，即使部分竞品没有也必须做 |

**执行批次排序原则**：

- B1a：上线骨架页（首页、About、FAQ、Contact、产品详情、核心分类入口等）— 网站存在与信任基础
- B1b：核心 SEO 模板（百科单品、按意图分类等可模板化矩阵）— 流量基础，但不阻塞 B1a 上线
- B2：上线后 1 个月内扩展页（更多分类、博客、轻量互动或转化页）— 扩展转化和内容覆盖
- B3：3 个月内差异化页（工具、资源、周期性内容）— 竞争优势验证
- B4：远期或模板工厂页（大规模矩阵、供应链依赖、蓝海验证不足页面）— 需要复审后再执行
- 每批内的页面按重要性、依赖关系和实现复杂度排序

**关键输出**：`02-网站规划/页面决策表.md` — 确定批准执行的页面清单 + 候选延后/不做页面及原因 + 执行批次 + 候选批次

#### 2C 逐页生成

**目标**：按 2B 确定的页面清单和执行批次，逐个页面完成：调用页面内容指南 → 建立单页依据包 → 写内容 → 图片资产清单/prompt → 构建脚本。

**每个页面的完整闭环（单页依据包）**：

1. **调用页面内容指南** — 先读取 `02-自动化工具库/03-WordPress建站/页面内容指南/` 中对应页面类型的指南，明确该页面类型的常见内容框架、标准 Section、数据来源规则、Elementor 生成规范和验收要点。页面内容指南只提供防错骨架，不替代竞品分析或 1H 策略。
2. **建立单页依据包** — 从 2B 决策表读取该页面的核心参考竞品和对应 URL，优先复用 1C 深度拆解、1D 网站结构分析、1E 内容策略分析、1H 策略清单中的既有结论；只对具体 URL 做轻量复核，不重新深度拆解竞品。参考数量规则：
   - 业务必要/低复杂度页面：至少 2 家核心参考
   - 标准核心页面：建议 3-5 家核心参考
   - 高价值SEO模板、SERP竞争强或页面形态复杂：可扩展到 5-10 家核心参考

   单页依据包需要提炼：
   - 页面内容指南（采用哪个指南、哪些标准 Section 保留、哪些可选模块采用或删除）
   - 模块组成（这个页面由哪些 Section 构成）
   - 标题句式（H1/H2/H3 的用词风格）
   - 正文风格（正式/亲切/权威/叙事）
   - CTA 策略（引导用户去哪、用什么文案）
   - 动态内容区规则（产品、博客、评论、相关文章等应调用真实组件/query，还是保留人工插入位）
3. **内容撰写** — 基于页面内容指南 + 单页依据包 + 1H 策略，写出该页面的全部文案
4. **图片资产清单** — 列出页面所有图片位置，标注用途、尺寸/比例、是否关键视觉位、资产键、复用理由
5. **图片prompt生成** — 基于页面内容，为每个图片位置创建独立prompt；关键视觉位必须独立，不复用、不共享
6. **构建脚本** — 生成 Elementor `.js` 脚本（如果使用 WordPress + Elementor），图片引用统一走资产键或页面配置，不在脚本中散落临时图片 URL

> **2C 开工验收**：页面进入 2C 前，必须确认其 `决策状态=✅ 批准执行`，且 `对应竞品页面URL` 已替换为实际可访问页面。仍包含 `[product]`、`[category]`、`[crystal]` 等模板占位符的页面，不得进入轻量复核和内容生成环节。单页依据包必须写清楚：页面内容指南、核心参考竞品、参考URL、上游来源索引（1B/1C/1D/1E/1H）、本页采用/规避/差异化的具体点。

> **同一批次内的并发**：允许同批次页面并行做单页依据包和轻量URL复核，但每个页面必须先完成"单页依据包"（竞品参考摘要 + 内容 + 图片prompt）再进入构建，不能跳过依据包直接写内容。

**内容与图片硬性要求**：
- **禁止**使用 Lorem ipsum、placeholder、"Sample text"、"Coming Soon"等占位文本
- 所有文案必须基于单页依据包 + 1H策略 + 业务事实，不能凭通用知识编造
- 页面结构不能只凭感觉生成；必须说明采用了哪个页面内容指南，以及哪些模块因竞品依据或业务事实被保留、删除或调整
- 每个图片位置必须有独立 prompt 或明确复用理由；Hero、CTA、产品/服务展示、信任背书等关键视觉位不得复用同一张图
- 图片资产必须能追踪状态：`needs_generation` → `generated` → `uploaded` → `url_written`

**关键输出**：
- 每个页面的单页依据包（页面内容指南、核心参考竞品、参考URL、上游来源索引、采用/规避/差异化点；内嵌在页面脚本注释或独立文档中）
- 每个页面的图片位置清单和资产键映射
- `02-网站规划/pages/*.js`（或对应的内容文件）
- `02-网站规划/assets/images/[page]-prompts.md`（图片prompt清单）
- `02-网站规划/assets/site-images.js`（或同等资产注册表，记录图片状态、media_id、URL、使用位置）

**参考文档**（执行 2C 时按需读取）：
- `02-自动化工具库/03-WordPress建站/页面内容指南/` — 页面类型的内容框架、标准 Section、数据来源规则和 Elementor 生成规范
- `02-自动化工具库/03-WordPress建站/Elementor REST API 操作手册.md` — 布局规则、Widget 选择、踩坑清单
- `02-自动化工具库/03-WordPress建站/elementor-upload.js` — 代码模式参考

#### 2D 上传与前端验收

**目标**：按 2B 的执行批次，逐个页面上传到 WordPress 并验收。

**每个页面的完整闭环**（不做完当前页面，不开始下一个）：
1. **图片生成** — 使用 Codex 内置生图或图片生成工具，按 2C 的 prompt 逐张生成
2. **图片质量验收** — 检查是否符合页面用途、品牌风格、主体清晰度、比例要求；不合格先重生，不上传
3. **图片上传** — 上传到 WordPress 媒体库，记录 media_id 和 URL
4. **URL 回写** — 将 WordPress 图片 URL 回写到 `site-images.js`、页面配置或对应资产注册表，避免页面继续引用本地路径、占位图或临时 URL
5. **页面脚本上传** — 运行 `node pages/xxx.js` 创建 draft 页面
6. **前端验收** — 打开 preview URL，检查桌面/移动端首屏、图片是否重复、内容是否像模板、布局是否错位、CTA 是否可见；不能只看 API 返回 200 或 draft 创建成功
7. **Elementor 同步检查** — 如页面使用 Elementor，确认 Elementor 编辑器可打开；必要时打开编辑器触发 CSS/数据同步后再复查预览
8. **标记完成** — 该页面通过前端验收后，在页面决策表中标记 2D 完成

**关键输出**：核心页面已上线（draft 或 published 状态），**每个页面的内容质量、图片独立性、前端预览效果已通过验收**

**里程碑**：核心页面已上传到 WordPress → 网站具备基本框架 → 可进入步骤3内容策略或同步启动

> 2C/2D 的详细操作流程（布局规则、Widget 选择策略、踩坑清单等）见 `02-自动化工具库/03-WordPress建站/` 目录下的文档。

### 步骤3：内容策略（一次性生成完整内容体系）

**目标**：基于关键词库和竞品策略，一次性生成覆盖所有搜索意图的完整内容清单

**前置输入**：来自步骤1的关键词库 + 用户画像 + 策略清单 + 网站结构规划

> **边界约束**：步骤3文档可引用 1E/1H 的结论（如"借鉴 Moon Omens 的免费PDF诱饵策略"），但不重复搬运竞品内容对比表、优势/缺口数据。引用时标注来源文件即可。

**AI时代的核心原则：一次性做完整，不分优先级逐步规划**

传统做法是人一天写1篇，所以必须分P0/P1/P2逐步规划。AI时代的关键认知转变：
- **Brief生成没有瓶颈** — AI可以一次性为几百甚至上千篇内容生成Brief
- **数据已经齐全** — 1F的关键词库已经包含了所有可覆盖的搜索意图
- **"优先级"改为"发布节奏"** — 不再在规划阶段筛选，而是给每篇内容标注"建议发布顺序"
- **关键词库可能不完整** — 如果内容策略覆盖了多品类（如水晶+星座+塔罗），每个品类需要单独的关键词研究

**核心任务**：

1. **内容类型规划** — 根据竞品实际内容类型 + 产品特性来规划，不预设固定类型

   *常见阶段*：
   - 认知阶段：教育类、原理解释类
   - 考虑阶段：产品评测、A vs B对比、使用技巧
   - 决策阶段：选购指南、合规类

2. **内容选题提取（一次性完成）** — 直接从关键词库提取所有可做内容的选题：
   - 从关键词库中按搜索意图分类（信息型/对比型/交易型）
   - 按关键词模式批量生成选题（如 `xxx meaning` → 水晶百科系列，`crystals for xxx` → 场景应用系列）
   - 每条选题必须绑定关键词库中的具体关键词（Volume/KD/CPC）
   - 补充竞品内容缺口和差异化选题

3. **完整内容清单** — 不分P0/P1/P2，直接生成全部选题清单，字段包括：
   - 目标关键词、搜索量、KD、CPC、意图
   - 内容类型、对应页面/URL
   - 建议发布节奏（首批/月度/季度） ← 替代原来的"优先级"
   - Brief状态

4. **检查关键词库是否覆盖所有内容品类** — 如果内容策略包含多个品类（如水晶+星座+塔罗+MBTI），每个品类需要确认1F中是否有对应关键词数据，缺失的需要补充关键词研究

**关键输出**：内容类型规划 + 选题方向 + 完整内容清单（Google Sheets，可能几百到上千条）+ 内容Brief

**工具链**：
```bash
# 批量分析竞品博客内容模式
python 02-自动化工具库/05-竞品内容分析工具/content_analyzer.py --input 03-sitemap_results/blog_urls.csv
```
- Skills: `claude-blog`（博客创作）/ `typefully`（社媒分发）/ `article-to-x`（内容转推文）
- MCP: `GSC MCP` — Content Decay 内容保鲜 / Content Recommendations 优先级建议 / Submit URL 加速收录

### 步骤4：社媒内容分发（可选 — 一次创作，多平台分发）

**目标**：将步骤3创作的博客文章再加工为多平台社媒内容，最大化内容ROI。

**前置输入**：步骤3的内容清单 + 已发布的博客文章 + 关键词库 + 用户画像

**核心思路**：
- **图文线**（Twitter / LinkedIn / Reddit / Instagram / Pinterest）：提炼核心论点，改写为各平台原生格式
- **音视频线**（YouTube / TikTok / Podcast）：博客→TTS口语化脚本→音频→配合画面/字幕合成视频

步骤4内部分4个子阶段：

```
4A: 分发策略规划     → 确定每篇文章适配哪些平台、什么格式
4B: 图文内容再加工   → 博客→推文线程/LinkedIn文章/Reddit帖子/IG轮播/Pinterest信息图
4C: 音视频内容再加工 → 博客→TTS脚本→YouTube视频/TikTok短视频/播客音频
4D: 排期与发布       → 批量排期、跨平台同步、效果追踪
```

**详细流程见**：`03-社媒内容分发工作流.md`（平台选择、再加工原则、TTS脚本改写、排期建议等完整SOP）

> 步骤4是可选步骤。内容量大的项目建议每10篇博客做一轮批量分发。B2B外贸站可优先LinkedIn + Twitter，B2C可优先Instagram + Pinterest。

> **项目简报规范**：`00-项目简报.md` 是项目唯一入口，必须包含以下结构：
> 1. **当前控制台** — 当前主线、并行准备、最大风险、下一次推荐动作（每次接手必看）
> 2. **执行状态（子步骤进度仪表板）** — 按 RLM 子步骤粒度追踪进度（如 `1B 数据采集: ✅ Sitemap 7/14成功`），使用 ✅完成 / 🔄进行中 / ⏳待开始 / ⏸暂停 图例。不写冗长描述，只突出进度和阻塞点。
> 3. **文档职责地图** — 每个核心文件的职责和读取时机
> 4. **工作清单（参考区）** — 历史执行记录和详细清单，放在仪表板下方，不作为入口
>
> 项目简报的"执行状态"是进度仪表板，不是分析报告。每次完成/开始一个子步骤时，应更新对应行的状态。

### 步骤1：竞品分析

## 5. 数据管理策略

RLM 项目中有两类数据，分别用不同载体管理：

| 载体 | 适用数据 | 原因 |
|------|---------|------|
| **Google Sheets** | 竞品清单、关键词库、内容清单等需要**频繁筛选/排序/更新**的结构化数据 | 在线协作、实时筛选、随时更新，不会产生本地快照过时的问题 |
| **本地 .md 文件** | 策略分析、竞品拆解、内容 Brief 等**以文字为主**的分析文档 | 方便 Git 版本管理、AI 按需读取 |

**规则**：
- 凡是会被当作"表格"反复查阅和修改的数据，优先放 Google Sheets
- 项目简报 `00-项目简报.md` 中必须记录所有 Google Sheets 链接（见"Google Sheets 数据源"章节）
- AI 执行时通过 Google Workspace MCP 或 curl + proxy 读取在线数据，不下载到本地
- 需要新增表格时，在项目简报中补上链接即可

## 10. 默认执行要求

AI每次输出必须满足：

- **有结构**：输出组织清晰，不散漫
- **有依据**：每个判断能追溯到竞品、关键词或业务事实
- **有动作**：策略具体到可执行的动作，不空泛
- **有文件名**：明确输出到哪个文件
- **能进入下一步**：当前输出是下一步的输入

不合格输出：
- 没有数据来源的判断
- 只给结论不给依据
- 只有标题没有关键词
- 只有策略没有动作
- 只说"优化用户体验"但不说明怎么优化

## 11. 每步验收标准

| 阶段 | 完成标准 |
|------|---------|
| 竞品发现（1A） | 至少 40 家竞品入池；每个竞品有人工标注的类型标签；大平台已过滤 |
| 数据采集（1B） | 竞品清单数据字段完整（AS、流量、截图、渠道）+ Sitemap 解析完成（至少覆盖直接竞品）+ 三级 fallback 方案已尝试 |
| 竞品深度拆解（1C） | 每个竞品有深度拆解文档（按类型侧重）；使用了1B全部数据（SEMrush + traffic.cv + Sitemap） |
| 网站结构分析（1D） | 能提炼竞品导航、URL、类目、页面层级；有结构图；架构模仿点使用"候选"语气，不含最终网站方案 |
| 内容策略分析（1E） | 每个竞品的内容类型、优势、缺口已记录；候选方向使用"候选"语气，不含"我们的策略"决策表述 |
| 关键词研究（1F） | 关键词库（Google Sheets）有分组、有意图标注、有优先级、有 Volume/KD/CPC、有推荐页面类型 |
| 用户画像（1G） | 每个画像可追溯到关键词意图、竞品页面、FAQ 或评论 |
| 策略清单（1H） | 每条策略有动作、有依据、有对应页面或内容；不重复搬运竞品原始数据；1D/1E 的候选模仿点已全部汇入并去重 |
| 架构规划（2A） | 导航结构确定、URL规则确定、核心页面有目标关键词和参考竞品；**竞品页面频率矩阵已完成**；不包含竞品对比表，竞品数据通过引用标注来源 |
| 页面决策（2B） | **每个页面有明确决策状态：批准执行/候选延后/不做**；做的原因追溯到竞品频率或1H策略；候选延后和不做的原因明确；执行批次与候选批次分离；没有未经批准的页面进入2C |
| 逐页生成（2C） | 每个页面有单页依据包：已读取并记录对应页面内容指南；实际可访问的核心参考URL、上游来源索引（1B/1C/1D/1E/1H）、提炼结论、采用/规避/差异化点；模板URL已替换；不重复深度拆解竞品；遵循 Flexbox 布局规则；可复用区块已提取到 `templates/`；文案基于页面内容指南 + 单页依据包写作，无 Lorem ipsum 或占位文本；每个页面有图片位置清单，关键视觉位有独立 prompt 和资产键，复用图片必须有明确理由 |
| 上传WordPress（2D） | 核心页面已创建为 draft；图片已生成、验收、上传并回写 URL；预览 URL 可访问；桌面/移动端前端预览已检查；Elementor 编辑器可正常打开编辑；**每个页面内容质量、图片独立性和前端效果已验收**（无占位文本、无图片跨关键页面复用、无明显模板感） |
| 内容策略（步骤3） | 内容清单一次性生成，覆盖关键词库中所有可做内容的选题；每篇绑关键词、意图、类型、建议发布节奏；高搜索量内容有Brief；多品类项目需确认关键词库覆盖完整；不包含竞品内容对比表 |
| 分发策略（4A） | 已确定主攻平台（2-4个）；每篇文章有平台-格式适配方案 |
| 图文再加工（4B） | 每篇博客至少适配2个图文平台；内容为再加工而非直接搬运 |
| 音视频再加工（4C） | TTS脚本已口语化改写；音频/视频已生成并可播放 |
| 排期与发布（4D） | 发布排期已制定；错峰发布；有引流回网站的追踪机制 |

## 12. RLM核心原则

1. **复杂任务分而治之**：不要让AI一次性完成模糊的大任务，拆成明确步骤。
2. **数据采集交给工具**：批量抓取、Sitemap解析、关键词导出等重复任务优先交给Python脚本。
3. **策略判断基于证据**：AI可以归纳和生成策略，但不能虚构数据。
4. **每一步输出都是下一步输入**：竞品清单→拆解→报告，关键词库→网站规划→内容策略，策略清单→页面规划→内容Brief。
