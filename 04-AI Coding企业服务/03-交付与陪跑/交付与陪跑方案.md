# 交付与陪跑方案

> 最后更新：2026-06-03

## 一、为什么"培训不够，需要陪跑"

来自那篇公众号文章的核心洞察：

> Claude Code 工具层面的培训，其实没啥门槛，离企业真正关心的**老旧项目接手、代码安全、测试闭环、团队规范、研发效率评估**还有差距。

**培训的局限**：
- 学员现场会了，回去不用
- 通用案例，和企业真实项目脱节
- 没有持续的反馈和迭代机制

**陪跑的价值**：
- 在真实项目中验证方法论
- 逐步建立企业内部的模板和规范
- 有问题随时解决，避免"回到老路"

## 四、陪跑节奏

### 月度陪跑节奏

```
第1周：诊断
├── 团队访谈（每人30分钟）
├── 代码库分析
├── 工具使用现状摸底
└── 输出：诊断报告

第2周：培训 + 配置
├── 1天现场培训
├── CLAUDE.md 模板定制
├── MCP/Hooks 配置
└── 输出：培训课件 + 配置模板

第3周：实操 + 答疑
├── 学员在真实项目中实践
├── 线上答疑（2次，每次1小时）
├── 收集问题和反馈
└── 输出：问题清单 + 优化建议

第4周：复盘 + 报告
├── 效果数据收集
├── 团队复盘会（线上1小时）
├── 优化配置和流程
└── 输出：效果评估报告
```

### 关键里程碑

| 里程碑 | 时间 | 标准 |
|--------|------|------|
| 全员环境就绪 | 第1周 | 每人可正常使用AI Coding工具 |
| 首个功能交付 | 第2周 | 用AI Coding完成1个功能开发 |
| 团队规范定稿 | 第3周 | CLAUDE.md团队版通过评审 |
| 效果可量化 | 第4周 | 有明确的效率提升数据 |

## 七、内部培训师培养

### 培养目标
让企业内部有人能持续推动AI Coding实践，减少对外部顾问的依赖。

### 培养路径

```
第1阶段：优秀学员识别（培训中观察）
    ↓
第2阶段：深度辅导（陪跑期间每周1对1）
    ↓
第3阶段：助教实践（下次内训时担任助教）
    ↓
第4阶段：独立授课（在内部做小范围培训）
    ↓
第5阶段：认证（通过考核，成为内部AI Coding教练）
```

### 内部培训师要求
- 熟练使用至少1个AI Coding工具
- 能编写和优化CLAUDE.md
- 理解Vibe Coding方法论
- 能独立解答团队常见问题
- 有意愿推动团队AI转型

## 八、AI 原生工作法工具箱（陪跑过程中逐步引入）

> 来源：Claude Code 工程总监 Fiona Fung 演讲「5 条工作原则」（2025）
>
> 以下方法论不是一次性灌输，而是在陪跑的不同阶段逐步引入

### 工具 1：JIT 规划 — 原型先行，砍掉仪式

**适用阶段**：第 2 周（实操阶段）

**核心逻辑**：在 AI 时代，写代码成本极低，写代码前花大量时间做设计文档是浪费。

**引入方式**：
1. 选一个真实需求，让团队用传统方式写设计文档（对照组）
2. 同时用 AI 直接出原型（实验组）
3. 对比时间成本和产出质量，让团队自己感受差异

**落地方案**：
```
旧流程：需求 → 设计文档（2天） → 评审（0.5天） → 开发（5天） → 测试
新流程：需求 → 原型（半天） → 内部试用 → 迭代 → 补文档（如果需要）
```

**注意**：不是所有场景都适用。安全敏感、合规要求高的项目仍需前期规划。JIT 规划适用于大多数业务功能开发。

### 工具 2：自动化肌肉记忆 — 重复 3 次就自动化

**适用阶段**：第 1 周（诊断阶段启动）

**核心逻辑**：以前自动化成本高，只有高频高价值的事才值得。现在自动化成本趋近于零，几乎所有重复 3 次以上的事都应该自动化。

**引入方式**：
1. 诊断周让每个成员列出"每周重复做 3 次以上的事"
2. 选出最有共鸣的 3-5 项
3. 现场用 AI 在 10 分钟内自动化其中 1 项，建立信心
4. 后续每周复盘时追踪"本周自动化了几件事"

**示例（直接可用）**：
| 重复工作 | 自动化方案 | 预估时间 |
|---------|-----------|---------|
| 每天手动拉取最新代码 | Git hook：打开项目自动 pull | 3 分钟配置 |
| 手动写周报/日报 | AI 读取 commit log 自动生成 | 5 分钟配置 |
| 手动做代码风格检查 | CI 集成 AI Code Review | 15 分钟配置 |
| 手动整理客户反馈 | AI 自动汇总各渠道反馈 | 30 分钟配置 |

### 工具 3：Trust but Verify — 分层评审

**适用阶段**：第 3 周（实操深化阶段）

**核心逻辑**：AI 处理 60-70% 的评审工作（风格/lint/基础 bug），人只做真正需要专业判断的部分。

**引入方式**：
```python
# 分层评审框架
AI 负责：                              人类负责：
  - 代码风格一致性                        - 法律合规风险评估
  - Linting 和静态检查                    - 安全敏感代码
  - 基础 bug 捕捉和修复                   - 产品方向和用户体验判断
  - 测试覆盖率检查                        - 架构级决策
  - PR 摘要生成                          - 性能关键路径

关键原则：这条线是动态的。
今天的"人做"事项，下一个模型版本可能就能做了。
每个季度重新评估一次分层。
```

### 工具 4：流程杀戮 — 主动删除无用流程

**适用阶段**：第 4 周（复盘阶段）

**核心逻辑**：人不会主动删除流程，只会在旧流程上叠新流程。必须有人主动站出来，点名道姓地说出哪些流程可以砍。

**引入方式**：
1. 复盘会上做一次"流程审计"
2. 列出团队所有例会、审批、报告、文档要求
3. 逐一问：这个流程解决了什么问题？AI 时代还需要吗？
4. 砍掉至少 1 个没有实际价值的流程

**判断标准**：
- 如果所有人都在开会时看电脑，这个会就不需要开
- 如果一个文档没人读但每个人都写，这个文档就不需要
- 如果一个审批平均通过时间 < 1 分钟且从未拒绝过，这个审批就不需要
