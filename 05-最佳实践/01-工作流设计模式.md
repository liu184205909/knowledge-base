# 工作流设计模式

> **RLM工作流 + AI开发方式的设计模式总结** | 跨领域通用 | 2026-01-06

---

## 🎯 什么是工作流设计模式？

**工作流设计模式** = 从成功案例中提炼的、可复用的设计方案

**为什么需要设计模式？**
- ❌ 每次从零开始设计工作流 (效率低)
- ✅ 使用验证过的模式 (效率高)
- ❌ 重复踩同样的坑 (风险高)
- ✅ 避免已知问题 (风险低)

---

## 📋 核心设计原则

### 原则1: 单一职责

**定义**: 每个Skill只负责一件事

**示例**:
- ❌ 一个Skill既做选品又做竞品分析
- ✅ 选品Skill + 竞品分析Skill

**好处**:
- 易于维护
- 易于复用
- 易于测试

---

### 原则2: 递归调用

**定义**: 主控Skill递归调用专业Skills

**模式**:
```
主控Skill
  ├─ 调用 Skill A
  ├─ 调用 Skill B
  ├─ 调用 Skill C
  └─ 汇总结果
```

**示例**: 产品分析全流程
```
产品分析主控Skill
  ├─ 调用 选品评估Skill → 评分92分
  ├─ 调用 竞品分析Skill → 10个竞品
  ├─ 调用 用户洞察Skill → 3个痛点
  └─ 汇总 → 完整报告
```

---

### 原则3: 结果汇总

**定义**: 最后必须汇总所有子任务结果

**模式**:
```
1. 分解任务
2. 递归调用
3. 收集结果
4. 汇总生成
```

**示例**: 内容生产流水线
```
选题 → 大纲 → 撰写 → SEO → 质检
        ↓
    汇总: 最终文章
```

## 🔧 8大设计模式

### 模式1: 串行流水线

**适用**: 任务有明确的先后顺序

**示例**: 内容生产流水线
```
选题 → 大纲 → 撰写 → SEO → 质检 → 发布
```

**实现**:
```markdown
## 工作流程

### Step 1: 选题研究
调用Skill: 选题研究Skill

### Step 2: 大纲生成
基于Step 1结果，调用Skill: 大纲生成Skill

### Step 3: 内容撰写
基于Step 2结果，调用Skill: 内容撰写Skill

### ...
```

**优点**:
- 流程清晰
- 易于理解
- 易于调试

**缺点**:
- 总时间 = 各步骤时间之和
- 无法并行优化

---

### 模式2: 并行分发

**适用**: 多个独立任务同时执行

**示例**: 批量选品分析
```
同时分析:
  ├─ 产品A
  ├─ 产品B
  └─ 产品C
```

**实现**:
```markdown
## 工作流程

### 并发分析多个产品
对每个产品执行:
  调用Skill: 选品评估Skill

### 汇总对比
对比所有产品结果
```

**优点**:
- 效率高
- 适合批量任务

**缺点**:
- 需要汇总逻辑
- 结果可能不一致

---

### 模式3: 迭代优化

**适用**: 需要多次迭代才能完成

**示例**: feature-dev流程
```
代码 → 测试 → 失败 → 修改 → 测试 → 成功
```

**实现**:
```markdown
## 工作流程

### 循环直到成功
1. 尝试实现
2. 测试验证
3. 如果失败，分析原因，修改代码
4. 重复1-3，直到成功
```

**优点**:
- 自动修复错误
- 直到成功为止

**缺点**:
- 时间不可控
- 可能陷入循环

---

### 模式4: 专家协作

**适用**: 需要多个专业AI协作

**示例**: Schaltwerk多AI协同
```
项目经理AI
  ├─ 架构师AI
  ├─ 开发者AI
  └─ 测试AI
```

**实现**:
```markdown
## 工作流程

### 项目经理AI协调
1. 分解任务
2. 分配给专业AI
3. 协调进度
4. 汇总结果
```

**优点**:
- 各司其职
- 适合大型项目

**缺点**:
- 复杂度高
- 协调成本大

---

### 模式5: 决策树

**适用**: 根据条件选择不同路径

**示例**: AI开发方式选择
```
复杂吗？
  ├─ 是 → 大型项目？
  │        ├─ 是 → Schaltwerk多AI协同
  │        └─ 否 → feature-dev流程
  └─ 否 → 简单任务？
           ├─ 是 → CSV+MCP自动化
           └─ 否 → 单AI开发
```

**实现**:
```markdown
## 决策流程

### 判断任务特征
IF 复杂且大型:
  → Schaltwerk多AI协同
ELSE IF 复杂但小型:
  → feature-dev流程
ELSE IF 简单批量:
  → CSV+MCP自动化
ELSE:
  → 单AI开发
```

**优点**:
- 智能选择
- 灵活适应

**缺点**:
- 决策逻辑复杂
- 需要明确判断条件

---

### 模式6: 模板化

**适用**: 相同结构，不同内容

**示例**: 批量生成产品页
```
模板 + 产品数据 → 产品页
```

**实现**:
```markdown
## 工作流程

### 使用模板
1. 加载模板
2. 填充数据
3. 生成结果
```

**优点**:
- 效率高
- 结构统一

**缺点**:
- 灵活性低
- 依赖模板质量

---

### 模式7: 错误恢复

**适用**: 容错要求高的场景

**示例**: 关键业务流程
```
执行 → 失败 → 重试 → 降级方案
```

**实现**:
```markdown
## 工作流程

### 带重试的执行
1. 尝试执行
2. IF 失败:
     - 重试3次
     - 如果仍失败，使用降级方案
```

**优点**:
- 可靠性高
- 容错能力强

**缺点**:
- 复杂度增加
- 时间可能延长

---

### 模式8: 混合模式

**适用**: 复杂场景，组合多种模式

**示例**: 社媒营销全流程
```
串行流水线 (创作→发布→分析)
  + 并行分发 (多平台适配)
  + 迭代优化 (基于数据优化)
```

**优点**:
- 适应复杂场景
- 灵活组合

**缺点**:
- 复杂度最高
- 难以维护

## 💡 设计技巧

### 技巧1: 明确输入输出

**每个Skill必须明确**:
- 输入: 需要什么信息？
- 输出: 产生什么结果？

**示例**:
```markdown
## 选品评估Skill

**输入**:
- 产品名称
- 目标市场
- 预算范围

**输出**:
- 100分制评分
- 优劣势分析
- 推荐建议
```

---

### 技巧2: 错误处理

**必须考虑错误情况**:
- Skill调用失败怎么办？
- 结果为空怎么办？
- 结果不符合预期怎么办？

**示例**:
```markdown
## 错误处理

### Skill调用失败
- 重试3次
- 记录错误日志
- 返回默认值或跳过

### 结果为空
- 使用默认值
- 提示用户补充信息
```

---

### 技巧3: 进度反馈

**长时间任务必须提供进度**:

**示例**:
```markdown
## 执行进度

✅ Step 1: 选品评估 (92分) - 完成
✅ Step 2: 竞品分析 (10个竞品) - 完成
✅ Step 3: 用户洞察 (3个痛点) - 完成
🔄 Step 4: 内容生产 - 进行中...
⏳ Step 5: 质量审核 - 待开始
```

## 🚀 如何设计自己的工作流？

### Step 1: 明确目标

**问题**: 你想解决什么问题？

**示例**:
- 我想批量生产SEO文章
- 我想分析多个产品
- 我想自动发布到社媒

---

### Step 2: 分解任务

**方法**: 将复杂任务分解成小任务

**示例**:
```
批量生产SEO文章
  ├─ 选题研究
  ├─ 大纲生成
  ├─ 内容撰写
  ├─ SEO优化
  └─ 质量审核
```

---

### Step 3: 选择模式

**根据任务特点选择设计模式**:

| 任务特点 | 推荐模式 |
|---------|---------|
| 有明确顺序 | 串行流水线 |
| 独立任务 | 并行分发 |
| 需要迭代 | 迭代优化 |
| 大型项目 | 专家协作 |
| 条件分支 | 决策树 |
| 相同结构 | 模板化 |
| 高可靠性 | 错误恢复 |
| 复杂场景 | 混合模式 |

---

### Step 4: 设计Skill

**为每个子任务设计专业Skill**:

**示例**:
```
选题研究Skill
  ├─ 输入: 产品/主题
  ├─ 处理: 关键词研究
  └─ 输出: 10个选题

大纲生成Skill
  ├─ 输入: 选题
  ├─ 处理: 生成大纲
  └─ 输出: 文章大纲
```

---

### Step 5: 实现主控Skill

**设计主控Skill协调所有子任务**:

**示例**:
```markdown
# 批量内容生产主控Skill

## 工作流程

### Step 1: 选题研究
调用Skill: 选题研究Skill
输入: [产品/主题]
输出: [10个选题]

### Step 2: 并行生成大纲
对每个选题，调用Skill: 大纲生成Skill

### Step 3: 并行撰写内容
对每个大纲，调用Skill: 内容撰写Skill

### Step 4: SEO优化
对所有文章，调用Skill: SEO优化Skill

### Step 5: 质量审核
对所有文章，调用Skill: 质量审核Skill

### 汇总
返回: 10篇SEO优化文章
```

---

### Step 6: 测试优化

**测试**:
- 用简单案例测试
- 检查每个步骤输出
- 验证最终结果

**优化**:
- 记录性能瓶颈
- 优化慢的步骤
- 调整参数

## 📖 案例研究

### 案例1: 产品分析全流程

**模式**: 串行流水线

**流程**:
```
选品 → 竞品 → 用户 → 内容
```

**为什么成功?**
- ✅ 单一职责: 每个Skill专注一件事
- ✅ 递归调用: 主控协调4个专业Skills
- ✅ 结果汇总: 最后汇总成完整报告

---

### 案例2: 批量选品分析

**模式**: 并行分发 + 结果汇总

**流程**:
```
同时分析多个产品 → 汇总对比
```

**为什么成功?**
- ✅ 并行效率: 同时分析，节省时间
- ✅ 统一标准: 所有产品用同一评估体系
- ✅ 横向对比: 容易对比多个产品

---

### 案例3: 内容生产流水线

**模式**: 串行流水线 + 模板化

**流程**:
```
选题 → 大纲 → 撰写 → SEO → 质检
```

**为什么成功?**
- ✅ 标准化: 每篇文章质量一致
- ✅ 高效: 流水线作业，批量生产
- ✅ 质控: 每个环节都有质量检查
